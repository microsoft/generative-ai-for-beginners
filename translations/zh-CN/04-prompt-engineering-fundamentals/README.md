# Prompt 工程基础

[![Prompt 工程基础](../../../translated_images/zh-CN/04-lesson-banner.a2c90deba7fedacd.webp)](https://youtu.be/GElCu2kUlRs?si=qrXsBvXnCW12epb8)

## 介绍
本模块涵盖了在生成式 AI 模型中创建有效提示的基本概念和技术。你向大型语言模型（LLM）编写提示的方式同样重要。精心设计的提示可以获得更高质量的响应。但诸如 _prompt_ 和 _prompt engineering_ 这样的术语到底是什么意思？我如何改进发送给 LLM 的提示 _输入_？这些是我们将在本章及下一章尝试回答的问题。

_生成式 AI_ 能够根据用户请求创建新内容（例如文本、图像、音频、代码等）。它使用像 OpenAI 的 GPT（“预训练生成式转换器”）系列这样的大型语言模型，这些模型经过自然语言和代码的训练。

用户现在可以使用熟悉的范式（如聊天）与这些模型交互，无需任何技术专长或培训。模型是基于 _提示_ 的——用户发送文本输入（提示），并获得 AI 响应（完成）。然后他们可以“与 AI 聊天”，进行多轮对话，不断优化其提示，直到响应符合预期。

“提示”如今成为生成式 AI 应用的主要 _编程接口_，告诉模型该做什么并影响返回响应的质量。“提示工程”是一个快速发展的研究领域，专注于提示的 _设计与优化_，以实现一致且高质量的响应输出。

## 学习目标

在本课中，我们将了解什么是提示工程，为什么它重要，以及如何为特定模型和应用目标设计更有效的提示。我们将理解提示工程的核心概念和最佳实践——并了解一个交互式 Jupyter 笔记本“沙盒”环境，在那里可以看到这些概念如何应用于实际案例。

到本课结束时，我们将能够：

1. 解释什么是提示工程以及为什么它重要。
2. 描述提示的组成部分及其使用方式。
3. 学习提示工程的最佳实践和技巧。
4. 使用 OpenAI 端点，将学到的技巧应用于实际案例。

## 关键词

提示工程：设计和优化输入以引导 AI 模型产生期望输出的实践。  
分词（Tokenization）：将文本转换为更小单元（称为标记）的过程，模型可以理解和处理这些标记。  
指令调优大型语言模型（Instruction-Tuned LLMs）：通过特定指令进行微调的大型语言模型，以提高响应的准确性和相关性。

## 什么是提示工程？

我们从定义 <strong>提示工程</strong> 开始，指的是为特定应用目标和模型 _设计与优化_ 文本输入（提示），以提供一致且高质量的响应（完成）的过程。我们可以把它看作一个两步过程：

- 为特定模型和目标 _设计_ 初始提示  
- 迭代 _优化_ 提示以提升响应质量

这必然是一个需要用户直觉和努力的试错过程，以获得最佳结果。那么，为什么它重要？为回答这个问题，我们先了解三个概念：

- _分词_ = 模型如何“看待”提示  
- _基础大型语言模型_ = 基础模型如何“处理”提示  
- _指令调优大型语言模型_ = 模型如何“理解任务”

### 分词

LLM 将提示视为一个 _标记序列_，不同模型（或模型的不同版本）可以用不同方式分词同一个提示。因为 LLM 是基于标记（而非原始文本）训练的，提示的分词方式直接影响生成响应的质量。

想直观感受分词如何工作？试试下面的 [OpenAI 分词工具](https://platform.openai.com/tokenizer?WT.mc_id=academic-105485-koreyst)。复制你的提示文本，看看它是如何转换成标记的，注意空白字符和标点符号的处理。注意此示例显示的是旧版 LLM（GPT-3），用新版模型试可能会有不同结果。

![分词](../../../translated_images/zh-CN/04-tokenizer-example.e71f0a0f70356c5c.webp)

### 概念：基础模型

提示分词后，["基础 LLM"](https://blog.gopenai.com/an-introduction-to-base-and-instruction-tuned-large-language-models-8de102c785a6?WT.mc_id=academic-105485-koreyst)（或基础模型）的主要功能是预测该序列中的下一个标记。由于 LLM 在庞大的文本数据集上训练，它们对标记之间的统计关系有较好把握，能有一定信心地做出预测。请注意，它们并不理解提示或标记中单词的 _含义_；它们只是看到可“完成”的模式。它们可以连续预测序列，直到用户干预或达到预设条件终止。

想看看基于提示的完成是如何工作的吗？将上述提示输入 Azure OpenAI Studio [_Chat Playground_](https://oai.azure.com/playground?WT.mc_id=academic-105485-koreyst)，使用默认设置。系统设为将提示视为信息请求——你应该看到符合该上下文的完成结果。

但如果用户想看到符合特定条件或任务目标的内容呢？这时就用到 _指令调优_ LLM 了。

![基础 LLM 聊天完成](../../../translated_images/zh-CN/04-playground-chat-base.65b76fcfde0caa67.webp)

### 概念：指令调优大型语言模型

[指令调优 LLM](https://blog.gopenai.com/an-introduction-to-base-and-instruction-tuned-large-language-models-8de102c785a6?WT.mc_id=academic-105485-koreyst) 以基础模型为起点，通过示例或输入/输出对（比如多轮“消息”）进行微调，这些对中包含清晰指令，AI 对响应尝试遵循该指令。

这使用了如人类反馈强化学习（RLHF）等技术，训练模型 _遵循指令_ 并 _从反馈中学习_，使响应更贴合实际应用，更符合用户目标。

## 为什么我们需要提示工程？

既然了解了提示如何被 LLM 处理，我们来谈谈 _为什么_ 需要提示工程。原因在于当前 LLM 存在一些挑战，使得在不投入提示构建和优化努力的情况下，难以实现 _可靠且一致的完成_。例如：

1. **模型响应具有随机性。** _同一个提示_ 可能在不同模型或版本下产生不同响应，甚至同一模型在不同时间的响应也不同。_提示工程技巧可以帮助我们通过设置更好的护栏减少这种波动_。

1. **模型可能编造响应。** 模型预训练时使用的是 _庞大但有限_ 的数据集，因此对训练之外的概念缺乏了解。结果它们可能生成不准确、虚构或与已知事实相矛盾的完成。_提示工程技巧帮助用户识别和减少此类编造，例如要求 AI 提供引证或推理_。

1. **模型能力千差万别。** 新一代模型功能更丰富，但也带来独特的怪癖，以及成本和复杂性的权衡。_提示工程可以帮助我们开发最佳实践和工作流，抽象差异并适应模型特定需求，实现可扩展、无缝的使用_。

让我们在 OpenAI 或 Azure OpenAI Playground 亲自体验：

- 用不同的 LLM 部署（如 OpenAI、Azure OpenAI、Hugging Face）测试同一提示——你看到的变化了吗？  
- 用同一 LLM 部署（如 Azure OpenAI Playground）多次测试同一提示——这些变化有何不同？

### 编造示例

本课程中，我们用 **“编造”** 一词指代 LLM 有时因训练限制或其他约束，生成事实错误信息的现象。你可能也见过它被称为 _“幻觉”_，但我们强烈建议使用 _“编造”_，避免将机器行为拟人化，赋予其人类特质。这也从术语层面支持了[负责任 AI 指南](https://www.microsoft.com/ai/responsible-ai?WT.mc_id=academic-105485-koreyst)，剔除可能被认为冒犯性或不包容的用词。

## 提示构造

我们已经了解了为何提示工程如此重要——现在让我们理解提示是如何_构造_的，从而评估多种技术以实现更有效的提示设计。

### 基本提示

先从基本提示开始：将文本输入发送给模型，不带任何其他上下文。举个例子——当我们将美国国歌的前几个词发送给 OpenAI 的[Completion API](https://platform.openai.com/docs/api-reference/completions?WT.mc_id=academic-105485-koreyst)时，它会立刻_补全_接下来的几行，展示基本预测行为。

| 提示（输入）        | 补全（输出）                                                                                                                               |
| :----------------- | :----------------------------------------------------------------------------------------------------------------------------------------- |
| Oh say can you see | 听起来你正在开始唱《星条旗永不落》的歌词，这是美国的国歌。完整歌词是...                                                                     |

### 复杂提示

现在让我们为基本提示添加上下文和指令。[Chat Completion API](https://learn.microsoft.com/azure/ai-services/openai/how-to/chatgpt?WT.mc_id=academic-105485-koreyst)允许我们将复杂提示构造成一组_消息_，包括：

- 反映_用户_输入与_助手_回应的输入/输出对。
- 设置助手行为或个性的系统消息。

请求现在采用如下形式，其中_分词_有效捕捉了上下文和对话中相关信息。改变系统上下文对补全质量的影响，可以与用户提供的输入同样重要。

```python
response = openai.chat.completions.create(
    model="gpt-3.5-turbo",
    messages=[
        {"role": "system", "content": "You are a helpful assistant."},
        {"role": "user", "content": "Who won the world series in 2020?"},
        {"role": "assistant", "content": "The Los Angeles Dodgers won the World Series in 2020."},
        {"role": "user", "content": "Where was it played?"}
    ]
)
```

### 说明型提示

在上述示例中，用户提示是可解释为信息请求的简单文本查询。使用_说明型_提示时，我们可以用文本更详细地指定任务，为 AI 提供更好的指导。举个例子：

| 提示（输入）                                                                                                                                                                                                         | 补全（输出）                                                                                                                | 说明类型          |
| :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :--------------------------------------------------------------------------------------------------------------------------- | :---------------- |
| 写一段关于内战的描述                                                                                                                                                                                                   | _返回了一段简单的段落_                                                                                                      | 简单              |
| 写一段关于内战的描述。提供关键日期和事件并描述其意义                                                                                                                                                                       | _返回了一段段落，随后是带说明的关键事件日期列表_                                                                             | 复杂              |
| 用一段话描述内战。提供3个要点，列出关键日期及其意义。再提供3个要点，介绍重要历史人物及其贡献。将输出以 JSON 文件形式返回                                                                                           | _返回更详尽内容，格式为 JSON 代码块，你可以复制粘贴到文件中并根据需要验证_                                                      | 复杂，格式化版    |

## 主要内容

在以上示例中，提示仍然比较开放，让大型语言模型自行决定其预训练数据集中哪些部分相关。使用_主要内容_设计模式时，输入文本分为两部分：

- 一个指令（动作）
- 相关内容（影响动作）

例如指令是“用两句话总结”，示例如下：

| 提示（输入）                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    | 补全（输出）                                                                                                                                                                                                                                                                |
| :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 木星是太阳系第五颗行星，也是体积最大的行星。它是一个气态巨行星，质量是太阳的千分之一，但却是太阳系其他所有行星质量的两倍半。木星是夜空中裸眼可见的最亮天体之一，自有文字记载前的古代文明时代便被人类认识。它以罗马神朱庇特命名。[19]从地球看来，木星的反射光线有时能投射可见阴影，[20]平均而言，它是月亮和金星之后夜空中第三亮的天然天体。<br/><strong>用两句话简要总结此文</strong> | 木星是太阳系中第五颗行星，也是体积最大的行星，以罗马神朱庇特命名，是夜空中最亮的天体之一。它是一个气态巨行星，质量是其他所有行星质量的两倍半。                                                                                                                                    |

主要内容段可以用多种方式来驱动更有效的指令：

- <strong>示例</strong> — 不直接告诉模型做什么，而是给模型一些示例，启发它推断所需输出的模式。
- <strong>提示</strong> — 在指令后跟一个“提示”，为生成补全做铺垫，引导模型产出更相关的回答。
- <strong>模板</strong> — 这些是带占位符（变量）的可重复使用“配方”提示，可以用特定数据定制以满足特定场景。

接下来让我们实际演示。

### 使用示例

这种方法用主要内容“喂给模型”一些所需输出的示例，让模型推断出期望输出的模式。根据示例数量的不同，可分为零-shot提示、单-shot提示、多-shot提示等。

提示现在由三部分组成：

- 任务描述
- 若干期望输出示例
- 新示例开头（作为隐含任务描述）

| :-------- | :----------------------------------------------------------------------------------------------------------------------------------------- | :----------------------- |
| 零-shot   | “The Sun is Shining” 翻译成西班牙语                                                                                                       | “El Sol está brillando”. |

| 多-shot   | 球员围绕垒跑 => 棒球 <br/> 球员发出“ACE”球 => 网球 <br/> 球员击出“六分球” => 板球 <br/> 球员完成灌篮 => 篮球                                          | 篮球                      |

注意零-shot提示中需要明确指令（“翻译成西班牙语”），但在单-shot提示示例中，模型能自行推断。在多-shot中，加入更多示例帮助模型在没有明确说明的情况下准确推断。

### 提示模板

提示模板是一个 _预定义的提示配方_，可以存储并按需重用，以推动更一致的用户体验。在最简单的形式下，它只是一个包含提示示例的集合，比如[OpenAI的这个示例](https://cookbook.openai.com/examples/gpt4-1_prompting_guide?WT.mc_id=academic-105485-koreyst)，提供了交互式提示组件（用户和系统消息）及基于API的请求格式，支持复用。

在更复杂的形式中，比如[LangChain的这个示例](https://python.langchain.com/docs/concepts/prompt_templates/?WT.mc_id=academic-105485-koreyst)，它包含可用多种数据源（用户输入、系统上下文、外部数据源等）替换的 _占位符_，以动态生成提示。这使我们能够创建可编程的可复用提示库，用于驱动一致的用户体验。

最终，模板的真正价值在于其支持创建和发布针对垂直应用领域的 _提示库_ —— 提示模板经过_优化_，反映应用特定上下文或示例，使响应对目标用户更相关、更准确。[Prompts For Edu](https://github.com/microsoft/prompts-for-edu?WT.mc_id=academic-105485-koreyst)仓库就是这种方法的优秀范例，为教育领域策划了一个提示库，重点涵盖课程设计、课程规划、学生辅导等关键目标。

## 支持内容

如果我们将提示构建看作包含指令（任务）和目标（主要内容），那么 _次要内容_ 就像是我们提供的额外上下文，用以 <strong>某种方式影响输出</strong>。它可以是调整参数、格式说明、主题分类法等，帮助模型 _调整_ 回答以符合预期的用户目标或期望。

举例：有一份课程目录，包含大量元数据（名称、描述、级别、标签、讲师等）供整套课程：

- 我们可以定义指令“总结2023年秋季课程目录”
- 使用主要内容提供几个示例以呈现期望输出
- 利用次要内容注明最关注的前五个“标签”

这样，模型可以按示例中的格式输出摘要——如果某个结果含多个标签，可优先考虑次要内容中的五个标签。

## 提示最佳实践

既然我们知道如何 _构建_ 提示，就可以开始考虑如何 _设计_ 它们以体现最佳实践。可以分两部分来看——具备正确的 _心态_ 和应用正确的 _技术_。

### 提示工程心态

提示工程是一个反复试验的过程，请牢记以下三条指导原则：

1. **领域理解很重要。** 响应的准确性和相关性取决于应用或用户所在的 _领域_。应用你的直觉和领域专长来 <strong>定制技术</strong>。例如，在系统提示中定义 _领域特定角色_，或在用户提示中使用 _领域特定模板_。提供反映领域上下文的次要内容，或使用 _领域特定线索和示例_ 引导模型采用熟悉的用法模式。

2. **了解模型很重要。** 模型本质上是随机的，但具体实现可能因训练数据集（预训练知识）、提供的能力（如API或SDK）及优化内容类型（代码、图像或文本）不同而异。了解你所用模型的优势和局限，利用这些知识 _优先处理任务_ 或构建 _定制模板_，以发挥模型最佳能力。

3. **迭代与验证很重要。** 模型和提示工程技术发展极快。作为领域专家，你可能有自己特定应用的额外上下文或标准，这些可能不适用于更广泛的社区。利用提示工程工具与技术“快速启动”提示构建，然后用你的直觉与领域专长迭代验证结果。记录你的见解，创建 <strong>知识库</strong>（如提示库），供他人日后以此为新基线快速迭代。
