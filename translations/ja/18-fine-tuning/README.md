# LLMのファインチューニング

大規模言語モデルを利用して生成AIアプリケーションを構築するには、新たな課題があります。重要な問題は、ユーザーのリクエストに対してモデルが生成するコンテンツの応答品質（正確さと関連性）を確保することです。前のレッスンでは、既存のモデルに対して_プロンプト入力を修正する_ことで問題を解決しようとするプロンプトエンジニアリングやリトリーバル拡張生成の技術について説明しました。

本日のレッスンでは、3つ目の技法である <strong>ファインチューニング</strong> について議論します。これは追加のデータで_モデル自体を再訓練する_ことで課題に対処しようとするものです。詳細を見ていきましょう。

## 言語モデルのファインチューニングとは？

定義によれば、大規模言語モデルはインターネットなど多様なソースから取得した大量のテキストで_事前訓練_されています。前のレッスンで学んだように、ユーザーの質問（「プロンプト」）に対する応答の品質向上のためには_プロンプトエンジニアリング_や_リトリーバル拡張生成_の技術が必要です。

人気のあるプロンプトエンジニアリング技術には、_指示_（明示的なガイダンス）を与えたり、_いくつかの例を示す_（暗黙的なガイダンス）ことでモデルに期待される応答をより具体的に案内する方法があります。これは_少数ショット学習_と呼ばれますが、2つの制約があります：

- モデルのトークン制限により提供できる例の数が制限され、その効果を下げることがある。
- 例を毎回のプロンプトに追加するとトークンコストが高くなり、柔軟性を制限する可能性がある。

ファインチューニングは、機械学習システムでよく使われる手法で、事前訓練済みモデルを利用して、新しいデータで再訓練し、特定のタスクの性能を向上させるものです。言語モデルの文脈では、与えられたタスクや応用領域に適した例を精選したデータセットで事前訓練済みモデルをファインチューニングし、その特定のタスクや領域に対してより正確かつ関連性が高い<strong>カスタムモデル</strong>を作成できます。ファインチューニングの副次的な利点として、少数ショット学習に必要な例の数を減らせるため、トークン使用量および関連コストの削減が期待できます。

## いつ、なぜモデルをファインチューニングすべきか？

ここで言うファインチューニングとは、<strong>監督付きファインチューニング</strong>のことであり、元の訓練データセットには含まれていなかった新しいデータを<strong>追加して再訓練</strong>することを指します。これは、元のデータで異なるハイパーパラメータを使って再訓練する非監督型のファインチューニングとは異なります。

ファインチューニングは高度な技術であり、目標とする成果を得るには一定の専門知識が必要であることを覚えておいてください。誤った方法で行うと期待した改善が得られず、特定のドメインにおけるモデル性能が低下することさえあります。

したがって、言語モデルのファインチューニング方法を学ぶ前に、「なぜ」この手法を選ぶべきか、「いつ」ファインチューニングのプロセスを開始すべきかを理解することが必要です。以下の質問から始めましょう：

- <strong>ユースケース</strong>：ファインチューニングの_ユースケース_は何か？現在の事前訓練済みモデルのどの側面を改善したいか？
- <strong>代替案</strong>：望む結果を得るために_他の技術_を試したか？比較のための基準を作る。
  - プロンプトエンジニアリング：少数ショットのプロンプト例を用いた手法を試し、応答の品質を評価する。
  - リトリーバル拡張生成：データ検索結果をプロンプトに補強して応答の品質を評価。
- <strong>コスト</strong>：ファインチューニングのコストを把握しているか？
  - チューナビリティ：事前訓練済みモデルがファインチューニング可能か。
  - 労力：学習データの準備、モデル評価・改良に必要な作業量。
  - 計算資源：ファインチューニングジョブの実行及びモデル展開のコスト。
  - データ：ファインチューニングに影響を与える十分な質の例へのアクセス。
- <strong>メリット</strong>：ファインチューニングの利点を確認しているか？
  - 品質：ファインチューニング済みモデルは基準モデルより優れているか？
  - コスト：プロンプトを簡素化してトークン使用量を削減できるか？
  - 拡張性：基盤モデルを新たなドメインに再利用できるか？

これらの質問に答えることで、ファインチューニングがあなたのユースケースに適しているかどうか判断できます。理想的には、利点がコストを上回る場合のみこのアプローチが妥当です。実行を決めたら、次に事前訓練済みモデルを_どのように_ファインチューニングするかを考えましょう。

意思決定プロセスの洞察を深めたい方は、[To fine-tune or not to fine-tune](https://www.youtube.com/watch?v=0Jo-z-MFxJs)をご覧ください。

## 事前訓練済みモデルをどうやってファインチューニングする？

事前訓練済みモデルをファインチューニングするには、以下が必要です：

- ファインチューニング対象の事前訓練済みモデル
- ファインチューニングに使うデータセット
- ファインチューニングジョブを実行する訓練環境
- ファインチューニング済みモデルをデプロイするホスティング環境

## Microsoft Foundryでのファインチューニング

[Microsoft Foundry](https://ai.azure.com?WT.mc_id=academic-105485-koreyst) は、Azure上でカスタムモデルのファインチューニング、デプロイ、管理を行うプラットフォームです（以前のAzure OpenAI StudioとAzure AI Studioの機能を統合）。ジョブ開始前にFoundryが提供する選択肢やプラットフォームの推奨ベストプラクティスを理解しておくと良いでしょう。Foundryでは効率的なファインチューニングを可能にする<strong>LoRA（低ランク適応）</strong>を採用しており、すべての重みを再訓練するよりも高速かつ低コストな訓練を実現しています。

### ステップ 1: 訓練手法を選択する

Foundryは3つのファインチューニング手法をサポートしています。まずは<strong>SFT</strong>から始めましょう。最も幅広いシナリオをカバーします。

| 手法 | 機能 | 使いどころ |
| --- | --- | --- |
| **監督付きファインチューニング (SFT)** | 入力/出力の例対で訓練し、望む応答を生成するようモデルを学習させます。 | ドメイン特化、タスク性能、文体・トーン調整、指示理解、言語適応など、ほとんどのタスクのデフォルト。 |
| **直接好み最適化 (DPO)** | _好ましい応答 vs. 好ましくない応答_のペアから学び、出力を人間の好みに合わせます。 | 比較フィードバックを持つ場合に応答品質、安全性、整合性の向上に。 |
| **強化ファインチューニング (RFT)** | _評価者_からの報酬信号を用いて強化学習で複雑な振る舞いを最適化。 | 明確な正解・誤答がある数学、化学、物理など推論重視分野に適し、より高度なML専門知識が必要。 |

### ステップ 2: 訓練レベルを選ぶ

Foundryでは訓練の実行場所と方法を選択できます：

- <strong>標準</strong> - リソースが属するリージョンで訓練し、データの所在を保証します。データが特定の地域に留まる必要がある場合に利用。
- <strong>グローバル</strong> - リージョン外の容量も使用し、安価でキュー入りが速い（データと重みは訓練地域に複製）。データ所在が重要でない場合の良い通常選択肢。
- <strong>開発者用</strong> - 最も低コストで、アイドル状態の容量を使用。レイテンシ・SLAなし、ジョブは中断・再開可能。実験に最適。

### ステップ 3: ベースモデルを選ぶ

ファインチューニング可能なモデルには、OpenAIの `gpt-4o-mini`、`gpt-4o`、`gpt-4.1`、`gpt-4.1-mini`、`gpt-4.1-nano`（SFT。4o/4.1 familyはDPOも対応）、推論モデルの `o4-mini` と `gpt-5`（RFT）、およびオープンソースの `Ministral-3B`、`Qwen-32B`、`Llama-3.3-70B-Instruct`、`gpt-oss-20b`（Foundryリソース上でのSFT）があります。利用可能な手法、リージョン、利用可否は常に最新の[ファインチューニングモデル一覧](https://learn.microsoft.com/azure/ai-foundry/foundry-models/concepts/models-sold-directly-by-azure?WT.mc_id=academic-105485-koreyst#fine-tuning-models)をご確認ください。

> Foundryには2種類のモードがあります：<strong>サーバーレス</strong>（消費課金制、GPUクォータ不要、OpenAIと選択モデル対応）と、<strong>マネージドコンピュート</strong>（Azure Machine Learning経由で自前VMを使い最も幅広いモデルに対応）。多くの方はまずサーバーレスから始めることをおすすめします。

### Foundryのベストプラクティス

- **まずベースラインを確立。** ファインチューニング前にプロンプトエンジニアリングやRAGでベースモデルの性能を測定し、効果を証明しましょう。
- **小規模から開始し、スケールアップ。** 最初は50-100件の高品質な例から始め検証し、生産環境には500件以上へ増やす。質は量に勝るため低品質例は除去します。
- **データ形式を正しく。** 訓練と検証ファイルはJSONL、UTF-8の<strong>BOM付き</strong>、512MB未満で、チャット補完メッセージ形式を使用。過学習防止のため検証ファイルは必ず含めましょう。
- **推論時のシステムプロンプトは訓練時と同じに。** モデル呼び出しの際、訓練時に使ったシステムメッセージを利用します。
- **チェックポイントの評価 - 最後のエポックを盲目的に使わない。** Foundryは最後の3エポックを展開可能なチェックポイントとして保持します。`train_loss` / `valid_loss`やトークン精度を観察し最適なものを選びましょう。
- **品質と共にトークンコストも評価。** ファインチューニングモデルとベースライン比較ではトークン使用量も考慮します。
- **継続的ファインチューニングで反復。** 既にファインチューニング済みモデルをさらに新データで訓練可能（OpenAIモデル対応）。
- **ホスティングコストに注意。** 配備されたカスタムモデルは時間単位で課金され、非アクティブ状態のデプロイは15日後に削除されるため、不要なものは整理してください。

[Customize a model with fine-tuning](https://learn.microsoft.com/azure/ai-foundry/openai/how-to/fine-tuning?WT.mc_id=academic-105485-koreyst)のエンドツーエンドウォークスルーを進め、準備ができたら[DPO](https://learn.microsoft.com/azure/ai-foundry/openai/how-to/fine-tuning-direct-preference-optimization?WT.mc_id=academic-105485-koreyst)や[RFT](https://learn.microsoft.com/azure/ai-foundry/openai/how-to/reinforcement-fine-tuning?WT.mc_id=academic-105485-koreyst)のガイドも参照してください。
