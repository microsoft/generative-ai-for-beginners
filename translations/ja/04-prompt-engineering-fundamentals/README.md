# プロンプトエンジニアリング基礎

[![Prompt Engineering Fundamentals](../../../translated_images/ja/04-lesson-banner.a2c90deba7fedacd.webp)](https://youtu.be/GElCu2kUlRs?si=qrXsBvXnCW12epb8)

## はじめに
本モジュールでは、生成AIモデルで効果的なプロンプトを作成するための基本概念と手法を扱います。LLMにプロンプトを書く方法も重要です。注意深く作られたプロンプトはより質の高い応答を得ることができます。しかし、_プロンプト_や_プロンプトエンジニアリング_という用語は具体的に何を意味するのでしょうか？また、LLMに送るプロンプトの_input_をどのように改善すればよいのでしょうか？これらの質問にこの章と次章で答えようとします。

_生成AI_は、ユーザーのリクエストに応じて新しいコンテンツ（例：テキスト、画像、音声、コードなど）を作成できます。これは、自然言語とコードの使用を学習したOpenAIのGPT（「Generative Pre-trained Transformer」）シリーズのような_Large Language Models_を用いて達成されます。

ユーザーは今や、チャットのような馴染みのある形式でこれらのモデルと対話でき、専門的な技術や訓練は不要です。モデルは_プロンプトベース_で、ユーザーはテキスト入力（プロンプト）を送り、AIの応答（補完）を受け取ります。その後、期待する応答になるまでプロンプトを修正しながら、多回の会話でAIと「チャット」できます。

「プロンプト」は、生成AIアプリケーションの主要な_プログラミングインターフェース_となり、モデルに何をすべきか指示し、返される応答の質に影響を与えます。「プロンプトエンジニアリング」は、スケールで一貫性のある質の高い応答を提供するために、プロンプトの_設計と最適化_に焦点を当てた急成長中の研究分野です。

## 学習目標

本レッスンでは、プロンプトエンジニアリングとは何か、なぜ重要かを学び、特定のモデルおよびアプリケーションの目的に合った効果的なプロンプトの作り方を理解します。コア概念とベストプラクティスを理解し、実際の例に適用できる対話型Jupyterノートブックの「サンドボックス」環境も紹介します。

レッスン終了時には、以下が可能になります：

1. プロンプトエンジニアリングとは何か、なぜ重要なのか説明できる。
2. プロンプトの構成要素とその使われ方を説明できる。
3. プロンプトエンジニアリングのベストプラクティスと手法を学べる。
4. OpenAIエンドポイントを使い、学んだ手法を実際の例に適用できる。

## 重要用語

プロンプトエンジニアリング: AIモデルが望ましい出力を生成するように導く入力を設計・洗練する実践。
トークナイゼーション: モデルが理解・処理可能な小さな単位（トークン）にテキストを変換する過程。
指示調整済みLLM: 特定の指示で微調整され、応答の正確性と関連性が向上した大規模言語モデル（LLM）。

## プロンプトエンジニアリングとは何か？

本レッスンは、<strong>プロンプトエンジニアリング</strong>を、特定のアプリケーション目的とモデルに対して一貫した質の高い応答（補完）を提供するため、テキスト入力（プロンプト）を_設計し最適化する_プロセスとして定義しました。これを2段階のプロセスと考えられます：

- 特定のモデルと目的に対して初期プロンプトを_設計_する
- 応答の質を改善するためにプロンプトを反復的に_洗練_する

これは最適な結果を得るためにユーザーの直感と努力が必要な試行錯誤のプロセスです。では、なぜ重要なのでしょうか？その答えにはまず、3つの概念を理解する必要があります：

- _トークナイゼーション_ = モデルがプロンプトを「どのように見るか」
- _基礎LLM_ = 基礎モデルがプロンプトを「どのように処理するか」
- _指示調整済みLLM_ = モデルが「タスク」として認識できる仕組み

### トークナイゼーション

LLMはプロンプトを_トークンの並び_として見ますが、モデルやモデルのバージョンによって同じプロンプトが異なる方法でトークン化されます。LLMはトークン単位（生テキストではなく）で学習しているため、プロンプトのトークン化方法は生成される応答の質に直接影響します。

トークナイゼーションの感覚を掴むために、下記のような[OpenAI Tokenizer](https://platform.openai.com/tokenizer?WT.mc_id=academic-105485-koreyst)のツールを使ってみてください。プロンプトをコピーし、どのようにトークンに変換されるのか、空白や句読点の扱いに注目してください。この例は古いLLM（GPT-3）を示しているので、新しいモデルで試すと異なる結果が出るかもしれません。

![Tokenization](../../../translated_images/ja/04-tokenizer-example.e71f0a0f70356c5c.webp)

### 概念：基礎モデル

プロンプトがトークン化された後、["基礎LLM"](https://blog.gopenai.com/an-introduction-to-base-and-instruction-tuned-large-language-models-8de102c785a6?WT.mc_id=academic-105485-koreyst)（または基礎モデル）の主な機能は、その列のトークンを予測することです。LLMは巨大なテキストデータセットで訓練されているため、トークン間の統計的関係を把握しており、その予測をある程度の確信を持って行えます。彼らはプロンプトやトークンの_意味_を理解しているわけではなく、次に来るトークンを「補完」するパターンを見ているだけです。その予測をユーザーの操作や事前の条件で終了するまで続けます。

プロンプトベースの補完動作を見たいですか？上記のプロンプトを[Microsoft Foundry playground](https://ai.azure.com?WT.mc_id=academic-105485-koreyst)のデフォルト設定で入力してみてください。システムはプロンプトを情報要求として扱う設定なので、この文脈が満たされた補完が得られるでしょう。

しかし、もしユーザーが特定の基準やタスク目的を満たす結果を見たい場合はどうなるでしょうか？ここで_指示調整済み_LLMが登場します。

![Base LLM Chat Completion](../../../translated_images/ja/04-playground-chat-base.65b76fcfde0caa67.webp)

### 概念：指示調整済みLLM

[指示調整済みLLM](https://blog.gopenai.com/an-introduction-to-base-and-instruction-tuned-large-language-models-8de102c785a6?WT.mc_id=academic-105485-koreyst)は、基礎モデルをスタート地点として、明確な指示を含む例や入出力ペア（例：多段階「メッセージ」）で微調整されます。AIの応答はその指示に従おうとします。

これは、人間のフィードバックを用いた強化学習（RLHF）などの技術を用い、モデルが_指示に従い_、_フィードバックから学習_するように訓練することで、実用的なアプリケーションに適した、ユーザーの目的により関連した応答を生成するためのものです。

試してみましょう。上記のプロンプトに戻り、_システムメッセージ_を以下の指示に変更してください：

> _提供された内容を小学校2年生向けに要約してください。結果は1段落で、3～5の箇条書きにまとめてください。_

結果が望ましい目的とフォーマットを反映して調整されているのがわかりますか？教育者はこの応答をそのままクラスのスライドで使うことができます。

![Instruction Tuned LLM Chat Completion](../../../translated_images/ja/04-playground-chat-instructions.b30bbfbdf92f2d05.webp)

## なぜプロンプトエンジニアリングが必要か？

プロンプトがLLMでどのように処理されるかを知ったところで、なぜプロンプトエンジニアリングが必要なのか説明しましょう。その答えは、現行のLLMが抱える様々な課題にあります。これらの課題により、プロンプトの構成と最適化に努力を注がなければ、_信頼でき一貫した補完_を得ることが難しくなっています。例を挙げると：

1. **モデルの応答は確率的。** 同じプロンプトでも異なるモデルやバージョンで異なる応答を生成する可能性があります。また、_同じモデル_でも異なる時に異なる結果を出すことがあります。_プロンプトエンジニアリング手法はこれらの変動を最小化するためのより良いガードレールを提供します_。

1. **モデルは誤情報を作り出すことがある。** モデルは_大規模だが有限な_データセットで事前訓練されているため、その訓練範囲外の概念を知らないことがあります。その結果、不正確だったり、架空だったり、既知の事実と矛盾する補完を生成することがあります。_プロンプトエンジニアリングの手法は、AIに出典や推論を求めるなどの方法で、こうした誤情報の特定と軽減を促します_。

1. **モデルの能力は異なる。** 新しいモデルや世代はより多彩な能力を持ちますが、コストや複雑さの面で独自の特性やトレードオフも伴います。_プロンプトエンジニアリングは、差異を抽象化し、モデル特有の要件に適応するベストプラクティスとワークフローの構築を助けます_。

OpenAIまたはAzure OpenAI Playgroundでこれを試してみましょう：

- 同じプロンプトを異なるLLMデプロイメント（例：OpenAI、Azure OpenAI、Hugging Face）で使い比べると変動は見られましたか？
- 同じプロンプトを_同じ_LLMデプロイメント（例：Azure OpenAI playground）で繰り返し使った場合の変動はどう違いましたか？

### 誤情報の例

本コースでは、LLMが訓練やその他の制約によって時折事実と異なる情報を生成する現象を<strong>「誤情報」</strong>と呼んでいます。一般的な記事や研究論文ではこれが_「幻覚」_と呼ばれることもありますが、人間の性質を機械の結果に誤って付与しないために、_「誤情報」_という用語の使用を強く推奨します。これは[Responsible AIガイドライン](https://www.microsoft.com/ai/responsible-ai?WT.mc_id=academic-105485-koreyst)の用語面での遵守にもなり、文脈によっては攻撃的または非包括的に見なされる語の排除に寄与します。

誤情報の働きを理解したいですか？存在しないトピックのコンテンツ生成をAIに指示するプロンプトを考えてみてください（訓練データセットに存在しないことを保証するため）。例えば、私は次のプロンプトを試しました：

> **プロンプト:** 2076年の火星戦争に関するレッスンプランを作成してください。

ウェブ検索では火星戦争を題材にしたフィクション（テレビシリーズや書籍など）はありましたが、2076年に関するものはありませんでした。また、常識的に2076年は_未来_であるため、実際の出来事とは結びつきません。


では、このプロンプトを異なるLLMプロバイダーで実行するとどうなるでしょうか？

> **回答 1**: OpenAI Playground (GPT-35)

![Response 1](../../../translated_images/ja/04-fabrication-oai.5818c4e0b2a2678c.webp)

> **回答 2**: Azure OpenAI Playground (GPT-35)

![Response 2](../../../translated_images/ja/04-fabrication-aoai.b14268e9ecf25caf.webp)

> **回答 3**: : Hugging Face Chat Playground (LLama-2)

![Response 3](../../../translated_images/ja/04-fabrication-huggingchat.faf82a0a51278956.webp)

予想通り、各モデル（またはモデルバージョン）は確率的な挙動やモデルの能力差により、わずかに異なる回答を生成します。例えば、あるモデルは中学2年生を対象とし、もう一方は高校生を想定しています。しかし、3つのモデルすべてが、情報を持たないユーザーにその出来事が本当だと納得させる回答を生成しました。

_メタプロンプティング_や_温度設定_などのプロンプトエンジニアリング技術により、モデルの虚偽生成はある程度抑えられる可能性があります。新しいプロンプトエンジニアリングの_アーキテクチャ_は、新しいツールや技術をプロンプトフローにシームレスに組み込み、これらの影響を緩和または軽減します。

## プロンプト構成

プロンプトエンジニアリングが重要な理由は見てきました。次に、プロンプトがどのように_構成_されるかを理解し、効果的なプロンプト設計のための異なる技術を評価しましょう。

### 基本的なプロンプト

まず基本的なプロンプトから始めます。これは他のコンテキストなしにモデルに送るテキスト入力です。例として、米国国歌の最初の数語をOpenAIの[Completion API](https://platform.openai.com/docs/api-reference/completions?WT.mc_id=academic-105485-koreyst)に送ると、次の数行を即座に_補完_して返します。これは基本的な予測動作を示しています。

| プロンプト（入力） | 補完（出力）                                                                                                                        |
| :----------------- | :----------------------------------------------------------------------------------------------------------------------------------------- |
| Oh say can you see | これは「星条旗」の歌詞の始まりのようです。米国の国歌です。全文は...                                                                 |

### 複雑なプロンプト

次に、基本プロンプトにコンテキストと指示を追加します。[Chat Completion API](https://learn.microsoft.com/azure/ai-foundry/openai/how-to/chatgpt?WT.mc_id=academic-105485-koreyst)では、複雑なプロンプトを_メッセージ_の集合として構成できます：

- ユーザー入力とアシスタント応答を反映した入出力ペア。
- アシスタントの振る舞いまたは人格を設定するシステムメッセージ。

リクエストは以下の形式となり、_トークン化_はコンテキストや会話の関連情報を効果的に捉えています。このようにシステムコンテキストの変更は、提供されるユーザー入力と同様に補完の質に大きく影響します。

```python
response = client.responses.create(
    model="gpt-5-mini",
    input=[
        {"role": "system", "content": "You are a helpful assistant."},
        {"role": "user", "content": "Who won the world series in 2020?"},
```

プロンプトは以下の3つの要素で構成されます：

- タスクの説明
- 望ましい出力のいくつかの例
- 新しい例の開始（暗黙のタスク説明となる）

### プロンプトテンプレート

プロンプトテンプレートは、必要に応じて保存し再利用できる_事前定義されたプロンプトのレシピ_であり、大規模に一貫したユーザー体験を促進します。最も単純な形では、[OpenAIのこちらの例](https://cookbook.openai.com/examples/gpt4-1_prompting_guide?WT.mc_id=academic-105485-koreyst)のように、対話型のプロンプト構成（ユーザーとシステムメッセージ）とAPI駆動のリクエストフォーマットの両方を提供する一連のプロンプト例のコレクションです。

より複雑な形態としては、[LangChainのこの例](https://python.langchain.com/docs/concepts/prompt_templates/?WT.mc_id=academic-105485-koreyst)のように、_プレースホルダー_を含み、これをユーザー入力、システムコンテキスト、外部データソースなど、さまざまなソースのデータで置き換え、動的にプロンプトを生成します。これにより、プログラム的に一貫したユーザー体験を大規模に促進できる再利用可能なプロンプトのライブラリーを作成できます。

最後に、テンプレートの本当の価値は、縦断的なアプリケーションドメイン向けに_プロンプトライブラリー_を作成・公開できる点にあります。ここでプロンプトテンプレートは、アプリケーション固有のコンテキストや例を反映するよう_最適化_され、対象ユーザー向けにより関連性が高く正確な応答を生成します。[Prompts For Edu](https://github.com/microsoft/prompts-for-edu?WT.mc_id=academic-105485-koreyst)リポジトリは、このアプローチの優れた例であり、レッスンプラン作成、カリキュラム設計、学生指導などの主要目的に焦点を当てた教育領域向けのプロンプトのライブラリをキュレーションしています。

## 補助コンテンツ

プロンプト構築を指示（タスク）と対象（主要コンテンツ）を持つと考えると、_副次的コンテンツ_は<strong>出力を何らかの方法で影響させるための</strong>追加コンテキストのようなものです。チューニングパラメータ、フォーマット指示、トピック分類などがこれに該当し、モデルが応答を望ましいユーザー目標や期待に合わせて_調整_するのに役立ちます。

例えば：カリキュラム内の利用可能なすべてのコースについて、名前、説明、レベル、メタデータタグ、講師などの詳細なメタデータを含むコースカタログがあるとします。

- 「2023年秋学期のコースカタログを要約せよ」という指示を定義できます。
- 主要コンテンツを使い、望ましい出力の例をいくつか提示できます。
- 副次的コンテンツを使い、関心のある上位5つの「タグ」を特定できます。

これでモデルは、いくつかの例で示した形式で要約を提供できますが、結果に複数のタグがある場合、副次的コンテンツで特定された5つのタグを優先できます。

## プロンプティングのベストプラクティス

プロンプトの_構築_方法が分かったので、今度はベストプラクティスを反映するように_設計_することを考えましょう。これには、大きく分けて適切な_マインドセット_を持つことと、適切な_技術_を適用することの2つがあります。

### プロンプトエンジニアリングのマインドセット

プロンプトエンジニアリングは試行錯誤のプロセスなので、以下の3つの広範な指針を心に留めてください：

1. **ドメイン理解が重要です。** 応答の正確性と関連性は、そのアプリケーションやユーザーが活動する_ドメイン_の影響を受けます。直感やドメイン専門知識を活用して<strong>技術をさらにカスタマイズ</strong>してください。例えば、システムプロンプトに_ドメイン固有のパーソナリティ_を定義したり、ユーザープロンプトに_ドメイン固有のテンプレート_を使ったりします。副次的コンテンツにドメイン固有のコンテキストを反映させたり、_ドメイン固有の合図や例_を使ってモデルを馴染みのある利用パターンへ導きます。

2. **モデル理解が重要です。** モデルは本質的に確率的であることは知られていますが、トレーニングデータセット（事前学習知識）、提供される機能（APIやSDK経由など）、最適化されているコンテンツタイプ（コード、画像、テキストなど）は実装により異なります。使用しているモデルの強みや制限を理解し、その知識を活用して_タスクの優先順位付け_やモデルの能力に最適化された_カスタマイズテンプレート_を作成してください。

3. **反復と検証が重要です。** モデルもプロンプトエンジニアリング技術も急速に進化しています。ドメイン専門家として、あなたにはコミュニティ全体には当てはまらない独自の文脈や基準があるかもしれません。プロンプトエンジニアリングのツールや技術を使って「素早く」プロンプト構築を始め、その後、自身の直感やドメイン知識で結果を反復・検証してください。洞察を記録し、他の人が将来的により速く反復できるように<strong>知識ベース</strong>（例：プロンプトライブラリ）を作成しましょう。
