## Вступ

AI-агенти представляють захопливий розвиток у галузі Генеративного ШІ, дозволяючи Великі Мовні Моделі (LLM) еволюціонувати з помічників у агентів, здатних виконувати дії. Фреймворки AI-агентів дають розробникам можливість створювати застосунки, які надають LLM доступ до інструментів і керування станом. Ці фреймворки також підвищують прозорість, дозволяючи користувачам і розробникам контролювати дії, які планують LLM, покращуючи управління досвідом.

- Розуміти, як функціонують AI-агенти, щоб будувати застосунки з їх використанням.

## Що таке AI-агенти?

AI-агенти — дуже захоплива сфера у світі Генеративного ШІ. З цим захопленням іноді виникає плутанина в термінах і їхньому застосуванні. Щоб усе було просто та включало більшість інструментів, які називаються AI-агентами, ми використаємо таке визначення:

AI-агенти дозволяють Великим Мовним Моделям (LLM) виконувати завдання, надаючи їм доступ до **стану** та **інструментів**.

![Agent Model](../../../translated_images/uk/what-agent.21f2893bdfd01e6a.webp)

Давайте визначимо ці терміни:

**Великі Мовні Моделі** — це моделі, про які йдеться у цьому курсі, такі як GPT-3.5, GPT-4, Llama-2 тощо.

**Стан** — це контекст, у якому працює LLM. Модель використовує контекст своїх минулих дій і поточний контекст, що допомагає їй приймати рішення для подальших дій. Фреймворки AI-агентів полегшують розробникам підтримувати цей контекст.

**Інструменти** — для виконання завдання, яке користувач запросив і яке планує LLM, модель потребує доступ до інструментів. Прикладами інструментів можуть бути база даних, API, зовнішній застосунок або навіть інша LLM!

Сподіваємось, ці визначення дадуть вам хорошу основу, коли ми далі розглядатимемо їх реалізацію. Давайте ознайомимося з кількома різними фреймворками AI-агентів:

## LangChain Agents

Для керування **станом** він використовує вбудовану функцію `AgentExecutor`. Вона приймає визначений `agent` та доступні для нього `tools`.

`Agent Executor` також зберігає історію чату для підтримки контексту бесіди.

![Langchain Agents](../../../translated_images/uk/langchain-agents.edcc55b5d5c43716.webp)

LangChain пропонує [каталог інструментів](https://integrations.langchain.com/tools?WT.mc_id=academic-105485-koreyst), які можна імпортувати у ваш застосунок і до яких LLM може отримати доступ. Ці інструменти створені спільнотою та командою LangChain.

Ви можете визначити ці інструменти й передати їх `Agent Executor`.

## AutoGen

Наступний фреймворк AI-агентів, який ми розглянемо, — [AutoGen](https://microsoft.github.io/autogen/?WT.mc_id=academic-105485-koreyst). Головна увага AutoGen — розмови. Агенти є як **спілкуваними**, так і **налаштовуваними**.

**Спілкувані —** LLM можуть починати та продовжувати розмову з іншою LLM, щоб виконати завдання. Це робиться шляхом створення `AssistantAgents` і надання їм конкретного системного повідомлення.

```python

autogen.AssistantAgent( name="Coder", llm_config=llm_config, ) pm = autogen.AssistantAgent( name="Product_manager", system_message="Creative in software product ideas.", llm_config=llm_config, )

```

**Налаштовувані** — агенти можуть бути визначені не тільки як LLM, але й як користувач або інструмент. Як розробник, ви можете визначити `UserProxyAgent`, який відповідає за взаємодію з користувачем для зворотного зв’язку у виконанні завдання. Цей зворотний зв’язок може продовжувати виконання завдання або зупинити його.

```python
user_proxy = UserProxyAgent(name="user_proxy")
```

![AutoGen](../../../translated_images/uk/autogen.dee9a25a45fde584.webp)

#### LLM визначена системним повідомленням

Це системне повідомлення направляє цю конкретну LLM на те, які функції релевантні для її завдання. Пам’ятайте, що з AutoGen ви можете мати кілька визначених AssistantAgents з різними системними повідомленнями.

#### Чат ініційований користувачем

```python
user_proxy.initiate_chat( chatbot, message="I am planning a trip to NYC next week, can you help me pick out what to wear? ", )
```

## Microsoft Agent Framework

[Microsoft Agent Framework](https://learn.microsoft.com/agent-framework/?WT.mc_id=academic-105485-koreyst) — це опенсорс SDK від Microsoft для побудови AI-агентів та систем з кількома агентами на **Python** і **.NET**. Він об’єднує сильні сторони двох попередніх проєктів Microsoft — корпоративні функції **Semantic Kernel** та мультіагентну оркестрацію **AutoGen** — в один підтримуваний фреймворк. Якщо ви починаєте новий проєкт агента сьогодні, це рекомендований наступник AutoGen.

Фреймворк масштабований від одного **чат-агента** до складних **багатоагентських робочих процесів**, і він інтегрується безпосередньо з Microsoft Foundry, Azure OpenAI та OpenAI. Він також забезпечує вбудовану спостережуваність через OpenTelemetry, щоб ви могли точно відслідкувати, що роблять ваші агенти.

### Стан і інструменти

**Стан** — фреймворк керує контекстом розмови через **потоки (threads)**. Агент відстежує історію повідомлень (запити користувача, виклики інструментів і їх результати), тож кожен крок будується на попередніх. Потоки можна зберігати, що дозволяє призупиняти і відновлювати розмову.

**Інструменти** — ви даєте агенту інструменти, передаючи прості Python-функції. Параметри з анотаціями типів автоматично перетворюються в схему, тож модель знає, як і коли їх викликати (function calling). Фреймворк також підтримує сервери Model Context Protocol (MCP) і розміщені інструменти, наприклад, інтерпретатор коду.

Ось приклад одного агента з кастомним інструментом:

```python
import asyncio
from typing import Annotated

from pydantic import Field
from agent_framework import Agent
from agent_framework.openai import OpenAIChatClient


def get_weather(
    location: Annotated[str, Field(description="The location to get the weather for.")],
) -> str:
    """Get the weather for a given location."""
    return f"The weather in {location} is sunny with a high of 22°C."


async def main():
    agent = Agent(
        client=OpenAIChatClient(),
        instructions="You are a helpful assistant that can answer weather questions.",
        tools=[get_weather],
    )

    response = await agent.run("What's the weather in Amsterdam?")
    print(response)


asyncio.run(main())
```

Щоб підключитися до Azure OpenAI у Microsoft Foundry, передайте свій endpoint і облікові дані клієнту:

```python
from azure.identity.aio import AzureCliCredential
from agent_framework.openai import OpenAIChatClient

client = OpenAIChatClient(
    model="my-gpt-4o-deployment",
    azure_endpoint="https://my-resource.openai.azure.com",
    credential=AzureCliCredential(),
)
```

### Робочі процеси з кількома агентами

Найкраще фреймворк проявляє себе в оркестрації кількох агентів разом. Наприклад, можна запускати агентів один за одним (передаючи контекст наступному), або запускати кілька агентів паралельно і збирати їхні результати:

```python
from agent_framework.orchestrations import SequentialBuilder, ConcurrentBuilder

# Запустити агентів послідовно, передаючи контекст розмови уздовж ланцюга
sequential = SequentialBuilder(participants=[researcher, writer, editor]).build()

# Запустити агентів паралельно, а потім агрегувати їхні відповіді
concurrent = ConcurrentBuilder(participants=[analyst_a, analyst_b, analyst_c]).build()
```

## Taskweaver

Наступний фреймворк агентів, який ми розглянемо, — [Taskweaver](https://microsoft.github.io/TaskWeaver/?WT.mc_id=academic-105485-koreyst). Його називають «code-first» агентом, бо замість роботи виключно зі `strings` може працювати з DataFrames у Python. Це дуже корисно для завдань аналізу та генерації даних. Наприклад, створення графіків і діаграм або генерація випадкових чисел.

### Стан і інструменти

Для керування станом розмови TaskWeaver використовує концепцію `Planner`. `Planner` — це LLM, яка приймає запит користувача і розбиває його на завдання, які потрібно виконати для його задоволення.

Для виконання завдань `Planner` має доступ до колекції інструментів під назвою `Plugins`. Це можуть бути Python-класи або загальний інтерпретатор коду. Ці плагіни зберігаються у вигляді ембеддінгів, щоб LLM краще знаходила потрібний плагін.

![Taskweaver](../../../translated_images/uk/taskweaver.da8559999267715a.webp)

Ось приклад плагіна для виявлення аномалій:

```python
class AnomalyDetectionPlugin(Plugin): def __call__(self, df: pd.DataFrame, time_col_name: str, value_col_name: str):
```

Код перевіряється перед виконанням. Ще одна функція для керування контекстом у Taskweaver — це `experience`. Експерієнс дозволяє зберігати контекст розмови на тривалий термін у YAML файлі. Це можна налаштувати так, щоб LLM з часом покращувалася у певних завданнях, оскільки вона має доступ до попередніх розмов.

## JARVIS

Останній фреймворк агентів, який ми розглянемо, — [JARVIS](https://github.com/microsoft/JARVIS?tab=readme-ov-file&WT.mc_id=academic-105485-koreyst). Унікальність JARVIS полягає в тому, що він використовує LLM для керування `станом` розмови, а `інструментами` є інші AI-моделі. Кожна з AI-моделей спеціалізована на певних завданнях, таких як виявлення об’єктів, транскрипція або написання підписів до зображень.

![JARVIS](../../../translated_images/uk/jarvis.762ddbadbd1a3a33.webp)

LLM, будучи моделлю загального призначення, отримує запит від користувача і визначає конкретне завдання та будь-які аргументи/дані, необхідні для його виконання.

```python
[{"task": "object-detection", "id": 0, "dep": [-1], "args": {"image": "e1.jpg" }}]
```

Потім LLM форматує запит у форматі, який спеціалізована AI-модель може інтерпретувати, наприклад JSON. Після того, як AI-модель повертає свій прогноз за завданням, LLM отримує відповідь.

Якщо для виконання завдання потрібні кілька моделей, LLM також інтерпретує їхні відповіді, перш ніж зібрати їх разом для формування відповіді користувачу.
