Чат-додатки стали невід'ємною частиною нашого повсякденного життя, пропонуючи більше, ніж просто засіб для звичайної розмови. Деякі питання, на які нам потрібно відповісти:

- **Моніторинг**. Після розгортання, як можна моніторити та забезпечити, що додатки працюють на найвищому рівні якості, як у функціональному плані, так і у дотриманні [шести принципів відповідального ШІ](https://www.microsoft.com/ai/responsible-ai?WT.mc_id=academic-105485-koreyst)?

Цей урок розгляне архітектурні аспекти, які підтримують ці складні системи, дослідить методи їх тонкого налаштування для конкретних доменних задач та оцінить метрики і фактори, важливі для відповідального впровадження ШІ.

### Чатбот чи чат-додаток?

Перш ніж перейти до створення чат-додатків, давайте порівняємо «чатботи» та «чат-додатки на базі ШІ», які виконують різні ролі й функції. Основна мета чатбота — автоматизація певних розмовних завдань, наприклад, відповідь на часті питання або відстеження посилки. Зазвичай він керується логікою, заснованою на правилах, або складними алгоритмами ШІ. Натомість чат-додаток на базі ШІ — це значно ширше середовище, спроєктоване для забезпечення різних форм цифрової комунікації, таких як текстові, голосові та відеочати між користувачами. Його визначальною рисою є інтеграція генеративної моделі ШІ, яка імітує тонкі, подібні до людських, розмови, генеруючи відповіді на основі різного роду вхідних і контекстних сигналів. Чат-додаток із генеративним ШІ може брати участь у відкритих дискусіях, адаптуватися до змінюваних контекстів розмови та навіть створювати творчі або складні діалоги.

| Орієнтований на завдання і керується правилами | Контекстно-усвідомлений                    |

### Використання готових функцій із SDK та API

При створенні чат-додатка важливо спершу оцінити, що вже існує. Використання SDK і API для створення чат-додатків — це вигідна стратегія з багатьох причин. Інтегруючи добре документовані SDK та API, ви стратегічно позиціонуєте ваш додаток на довгостроковий успіх, враховуючи питання масштабованості та підтримки.

- **Прискорення розробки та зниження навантаження**: Використання готових функцій замість дорогого процесу їх створення дозволяє вам зосередитися на інших аспектах додатку, які можуть бути важливішими, наприклад на бізнес-логіці.
- **Краща продуктивність**: При створенні функціоналу з нуля з часом виникає питання: «Як це масштабується? Чи може додаток впоратися з раптовим збільшенням користувачів?» Добре підтримувані SDK і API часто мають вбудовані рішення для подібних питань.
- **Легший супровід**: Оновлення та вдосконалення керувати простіше, оскільки більшість API та SDK вимагають лише оновлення бібліотеки при виході нової версії.
- **Доступ до новітніх технологій**: Використання моделей, які були тонко налаштовані та навчені на великих масивах даних, забезпечує ваш додаток можливостями обробки природної мови.

Доступ до функціоналу SDK або API зазвичай вимагає отримання дозволу на використання послуг, що часто реалізується через унікальний ключ або токен автентифікації. Ми використаємо бібліотеку OpenAI для Python, щоб показати, як це виглядає. Ви також можете спробувати самостійно у наведених [зошитах для OpenAI](./python/oai-assignment.ipynb?WT.mc_id=academic-105485-koreyst) або [для Azure OpenAI Services](./python/aoai-assignment.ipynb?WT.mc_id=academic-105485-koreys) до цього уроку.

## Користувацький досвід (UX)

Загальні принципи UX застосовуються до чат-додатків, але є додаткові фактори, що набувають особливої важливості через компоненти машинного навчання.

- **Механізм вирішення неоднозначностей**: Моделі генеративного ШІ іноді створюють неоднозначні відповіді. Функція, що дозволяє користувачам просити уточнень, може бути корисною у таких випадках.
- **Збереження контексту**: Розвинені генеративні моделі ШІ можуть зберігати контекст розмови, що є необхідним для користувацького досвіду. Надання користувачам можливості контролювати та керувати контекстом покращує UX, але водночас несе ризик збереження конфіденційної інформації. Фактори, як-от тривалість зберігання (політика зберігання), можуть збалансувати потребу в контексті та конфіденційність.
- **Персоналізація**: Здатність моделей навчатися і адаптуватися пропонує індивідуалізований досвід користувача. Налаштування UX за допомогою таких функцій, як профілі користувачів, не лише дає почуття розуміння, а й допомагає швидше знаходити конкретні відповіді, створюючи ефективну і приємну взаємодію.

## Налаштування та тонке налаштування для доменно-специфічних мовних моделей

Уявіть чат-додаток, який розуміє жаргон вашої компанії та передбачає конкретні запитання, які часто виникають у його користувачів. Існує кілька підходів, які варто згадати:

- **Використання моделей DSL**. DSL означає доменно-специфічна мова. Ви можете використовувати так звану модель DSL, навчену на конкретній доменній області, щоб розуміти її поняття і сценарії.
- **Застосування тонкого налаштування**. Тонке налаштування — це процес подальшого навчання вашої моделі з використанням спеціалізованих даних.

## Налаштування: Використання DSL

Використання моделей доменно-специфічної мови (DSL Models) може покращити взаємодію користувачів, забезпечуючи спеціалізовані, контекстно доречні взаємодії. Це модель, навчена або тонко налаштована так, щоб розуміти та генерувати текст, пов’язаний із конкретною галуззю, індустрією або темою. Варіанти використання моделі DSL можуть варіюватися від навчання з нуля до використання наявних через SDK і API. Інший варіант — тонке налаштування, що передбачає адаптацію існуючої попередньо навчених моделі до конкретного домену.

## Налаштування: Застосування тонкого налаштування

Тонке налаштування часто розглядається, коли попередньо навчена модель не справляється у спеціалізованій доменній області або для конкретного завдання.

Наприклад, медичні запити є складними та потребують великої кількості контексту. Коли медичний фахівець ставить діагноз, він базується на різних факторах, таких як спосіб життя або наявні захворювання, а також може звертатися до останніх медичних журналів для підтвердження діагнозу. У таких складних випадках загального чат-додатка на основі ШІ недостатньо.

### Сценарій: медичний додаток

Розглянемо чат-додаток, призначений для допомоги медичним працівникам, який надає швидкі посилання на керівництва з лікування, взаємодії ліків або останні наукові дослідження.

Загальна модель може бути достатньою для відповіді на базові медичні питання або надання загальних порад, але їй буде важко в таких випадках:

- **Особливо специфічні або складні випадки**. Наприклад, невропатолог може запитати додаток: «Які найкращі практики контролю резистентної до ліків епілепсії у педіатричних пацієнтів?»
- **Відсутність урахування останніх досягнень**. Загальна модель може не надати актуальної відповіді, враховуючи останні досягнення в галузі неврології та фармакології.

У таких випадках тонке налаштування моделі зі спеціалізованим медичним набором даних може значно покращити її здатність більш точно та надійно обробляти ці складні медичні запити. Це вимагає доступу до великого і релевантного набору даних, який відтворює доменно-специфічні виклики та питання, які потрібно розв’язати.

## Параметри для високоякісного досвіду чат-додатку на базі ШІ

### Ключові метрики

Для підтримання високої якості роботи додатку важливо відслідковувати ключові метрики та фактори. Ці вимірювання не тільки забезпечують функціональність додатку, а й оцінюють якість моделі ШІ та користувацький досвід. Нижче наведено список базових, ШІ- та UX-метрик для розгляду.

| Метрика                       | Визначення                                                                                                            | Роздуми для розробника чату                                            |
| ----------------------------- | --------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------- |
| **Час безвідмовної роботи**  | Вимірює час, протягом якого додаток працює і доступний для користувачів.                                               | Як ви будете мінімізувати час простою?                                |
| **Час відповіді**             | Час, потрібний додатку для відповіді на запит користувача.                                                            | Як оптимізувати обробку запиту для покращення часу відповіді?         |
| **Точність**                 | Відношення істинно позитивних прогнозів до загальної кількості позитивних прогнозів                                    | Як ви будете перевіряти точність вашої моделі?                        |
| **Повнота (чутливість)**      | Відношення істинно позитивних прогнозів до фактичної кількості позитивних випадків                                      | Як ви будете вимірювати та покращувати повноту?                       |
