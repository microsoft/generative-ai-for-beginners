## Підсилене Пошуком Генерування (RAG)

Чатбот на основі LLM обробляє запити користувачів для генерації відповідей. Він розрахований на інтерактивність і спілкування з користувачами з різних тем. Втім, його відповіді обмежені контекстом і базовими навчальними даними. Наприклад, датою останнього оновлення знань GPT-4 є вересень 2021 року, тож він не знає про події після цього періоду. Крім того, дані для навчання LLM не включають конфіденційну інформацію, таку як особисті нотатки або продуктову документацію компанії.

### Як працюють RAG (Підсилене Пошуком Генерування)

![діаграма, що показує, як працює RAG](../../../translated_images/uk/how-rag-works.f5d0ff63942bd3a6.webp)

Припустимо, ви хочете розгорнути чатбота, який створює тести з ваших нотаток — вам потрібне підключення до бази знань. Тут на допомогу приходить RAG. RAG працює так:

- **База знань:** Перед пошуком документи потрібно завантажити та опрацювати, зазвичай розбиваючи великі документи на менші частини, перетворюючи їх у текстові ембеддінги та зберігаючи у базі даних.

- **Запит користувача:** користувач ставить питання

- **Пошук:** коли користувач ставить запитання, модель ембеддінгу знаходить релевантну інформацію у базі знань для забезпечення контексту, який буде додано до запиту.

- **Підсилене Генерування:** LLM покращує свою відповідь, використовуючи отримані дані. Це дозволяє генерувати відповіді, базовані не лише на попередньо навчених даних, а й на релевантній інформації з доданого контексту. Отримані дані використовуються для підсилення відповідей LLM. Потім LLM видає відповідь на запит користувача.

![діаграма, що показує архітектуру RAG](../../../translated_images/uk/encoder-decode.f2658c25d0eadee2.webp)

Архітектура RAG реалізується з використанням трансформерів, що складаються з двох частин: енкодера та декодера. Наприклад, коли користувач ставить запит, вхідний текст «кодується» у вектори, що передають значення слів, після чого вектори «декодуються» у наш пошуковий індекс документів та генерують новий текст, заснований на запиті користувача. LLM використовує модель енкодер-декодер для генерації результату.

Два підходи до реалізації RAG згідно з запропонованою статтею: [Retrieval-Augmented Generation for Knowledge intensive NLP Tasks](https://arxiv.org/pdf/2005.11401.pdf?WT.mc_id=academic-105485-koreyst):

- **_RAG-Sequence_** - використання отриманих документів для передбачення найкращої відповіді на запит користувача

- **RAG-Token** - генерування наступного токена із документів, потім їх повторне отримання для відповіді на запит користувача

### Чому варто використовувати RAG? 

- **Насиченість інформацією:** забезпечує актуальні та сучасні текстові відповіді. Це покращує результати на завданнях у конкретних доменах завдяки доступу до внутрішньої бази знань.

- Зменшує вигадки, використовуючи **перевірені дані** з бази знань для створення контексту до запитів користувача.

- **Економічність** — це більш вигідний варіант порівняно з тонким налаштуванням LLM

### Векторні бази даних

Векторна база даних, на відміну від традиційних, є спеціалізованою базою, призначеною для зберігання, керування та пошуку ембеддінгів векторів. Вона зберігає числові представлення документів. Перетворення даних у числові ембеддінги полегшує нашій ШІ-системі розуміння і обробку цих даних.

Ми зберігаємо ембеддінги у векторних базах даних, оскільки LLM мають обмеження на кількість токенів, які вони приймають на вхід. Оскільки неможливо передати всю ембеддінг до LLM, ми розбиваємо їх на частини, і коли користувач ставить питання, повертаються ембеддінги, найбільш подібні до питання, разом із запитом. Також такий поділ знижує витрати на кількість токенів, які проходять через LLM.

Деякі популярні векторні бази даних — Azure Cosmos DB, Clarifyai, Pinecone, Chromadb, ScaNN, Qdrant і DeepLake. Ви можете створити модель Azure Cosmos DB, використовуючи Azure CLI, з такою командою:

### Від тексту до ембеддінгів

Перед збереженням даних їх потрібно конвертувати в векторні ембеддінги. Якщо ви працюєте з великими документами або довгими текстами, ви можете розбити їх на частини відповідно до очікуваних запитів. Розбиття можна робити на рівні речень або параграфів. Оскільки сенс частини залежить від оточуючих слів, можна додати до частини контекст, наприклад, назву документа або вставити текст до чи після частини. Ви можете розбивати дані так:

```python
def split_text(text, max_length, min_length):
    words = text.split()
    chunks = []
    current_chunk = []

    for word in words:
        current_chunk.append(word)
        if len(' '.join(current_chunk)) < max_length and len(' '.join(current_chunk)) > min_length:
            chunks.append(' '.join(current_chunk))
            current_chunk = []

    # Якщо останній шматок не досяг мінімальної довжини, додайте його у будь-якому випадку
    if current_chunk:
        chunks.append(' '.join(current_chunk))

    return chunks
```

Після розбиття можна виконати ембеддінг тексту за допомогою різних моделей ембеддінгів. Деякі з них: word2vec, ada-002 від OpenAI, Azure Computer Vision і багато інших. Вибір моделі залежить від мов, якими ви працюєте, типу кодуємого контенту (текст/зображення/аудіо), розміру вхідних даних і довжини ембеддінгу на виході.

## Пошук і Векторний Пошук

Коли користувач ставить запит, ретрівер перетворює його у вектор за допомогою кодувальника запитів, потім шукає у нашому індексі документів релевантні вектори у документах, пов’язані з вхідним запитом. Після цього і вхідний вектор, і вектори документів конвертуються в текст і передаються у LLM.

### Пошук

Пошук відбувається, коли система швидко знаходить документи у індексі, які відповідають критеріям пошуку. Мета ретрівера — отримати документи, які будуть використані для контексту і підкріплення LLM вашими даними.

Існує кілька способів пошуку у базі даних, таких як:

- **Пошук ключовими словами** — використовується для текстового пошуку

- **Векторний пошук** — перетворює документи з тексту у векторні представлення за допомогою моделей ембеддінгу, що дозволяє виконувати **семантичний пошук** за значенням слів. Пошук здійснюється за документами, векторні представлення яких найближчі до запиту користувача.

- **Гібридний** — поєднання пошуку за ключовими словами і векторного

Проблема з пошуком виникає, коли у базі немає схожої відповіді на запит, тоді система повертає найкращу можливу інформацію. Ви можете застосувати стратегії, такі як встановлення максимальної відстані релевантності або гібридний пошук, що сполучає ключові слова та векторний пошук. У цьому уроці ми використовуватимемо гібридний пошук — поєднання обох типів. Дані ми збережемо у dataframe з колонками, що містять частини тексту і ембеддінги.

### Векторна подібність

Ретрівер шукатиме у базі знань ембеддінги, які розташовані поруч, найближчого сусіда, адже це тексти, схожі один на одного. У сценарії, коли користувач ставить запит — спочатку він ембеддиться, а потім шукається найбільш схожий ембеддинг. Найпоширеніша міра для визначення схожості векторів — косинусна подібність, заснована на куті між двома векторами.

Ми можемо використовувати інші альтернативи для вимірювання подібності: євклідова відстань (пряма лінія між кінцями векторів) та скалярний добуток (сума добутків відповідних елементів двох векторів).

### Пошуковий індекс

Для пошуку потрібно спершу створити пошуковий індекс для нашої бази знань. Індекс зберігатиме ембеддінги і швидко знайде найбільш схожі частини навіть у великій базі. Ми можемо створити індекс локально за допомогою:

```python
from sklearn.neighbors import NearestNeighbors

embeddings = flattened_df['embeddings'].to_list()

# Створити індекс пошуку
nbrs = NearestNeighbors(n_neighbors=5, algorithm='ball_tree').fit(embeddings)

# Для запиту індексу ви можете використовувати метод kneighbors
distances, indices = nbrs.kneighbors(embeddings)
```

### Заново Ранжування (Re-ranking)

Після запиту у базу даних вам може знадобитись відсортувати результати за релевантністю. LLM для переранжування використовує машинне навчання для покращення релевантності результатів, упорядковуючи їх від найбільш релевантних. За допомогою Azure AI Search переранжування відбувається автоматично за допомогою семантичного ранжувальника. Приклад того, як працює переранжування за допомогою найближчих сусідів:

```python
# Знайдіть найбільш схожі документи
distances, indices = nbrs.kneighbors([query_vector])

index = []
# Виведіть найбільш схожі документи
for i in range(3):
    index = indices[0][i]
    for index in indices[0]:
        print(flattened_df['chunks'].iloc[index])
        print(flattened_df['path'].iloc[index])
        print(flattened_df['distances'].iloc[index])
    else:
        print(f"Index {index} not found in DataFrame")
```

## Об’єднання всього разом

Останній крок — додати LLM, щоб отримувати відповіді, підкріплені нашими даними. Це можна реалізувати так:

```python
user_input = "what is a perceptron?"

def chatbot(user_input):
    # Перетворити питання у вектор запиту
    query_vector = create_embeddings(user_input)

    # Знайти найбільш схожі документи
    distances, indices = nbrs.kneighbors([query_vector])

    # додати документи до запиту для забезпечення контексту
    history = []
    for index in indices[0]:
        history.append(flattened_df['chunks'].iloc[index])

    # поєднати історію та введення користувача
    history.append(user_input)

    # створити об'єкт повідомлення
    messages=[
        {"role": "system", "content": "You are an AI assistant that helps with AI questions."},
        {"role": "user", "content": "\n\n".join(history) }
    ]

    # використовувати API відповідей для генерації відповіді
    response = client.responses.create(
        model="gpt-4o-mini",
        temperature=0.7,
        max_output_tokens=800,
        input=messages,
        store=False,
    )

    return response.output_text

chatbot(user_input)
```

## Варіанти використання RAG (Підсилене Пошуком Генерування) та векторних баз даних

Існує багато різних варіантів, де виклики функцій можуть покращити ваш застосунок, такі як:

- Відповіді на запитання: підкріплення даних компанії у чат, який можуть використовувати співробітники для питань.

- Системи рекомендацій: створення системи, що підбирає найбільш схожі варіанти, наприклад, фільми, ресторани тощо.

- Чатбот-сервіси: можна зберігати історію чату і персоналізувати розмову на основі даних користувача.

- Пошук зображень на основі векторних ембеддінгів, корисний під час розпізнавання зображень і виявлення аномалій.
