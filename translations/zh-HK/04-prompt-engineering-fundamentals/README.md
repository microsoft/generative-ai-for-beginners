# 提示工程基礎

[![提示工程基礎](../../../translated_images/zh-HK/04-lesson-banner.a2c90deba7fedacd.webp)](https://youtu.be/GElCu2kUlRs?si=qrXsBvXnCW12epb8)

## 介紹
本單元涵蓋在生成式 AI 模型中創建有效提示的基本概念和技術。如何撰寫提示給大型語言模型（LLM）也很重要。精心設計的提示可達到更高質量的回應。但到底像 _prompt_ 和 _prompt engineering_ 這些術語是什麼意思？我又如何改進傳送給 LLM 的提示 _輸入_ 呢？這些是本章與下一章將嘗試回答的問題。

_生成式 AI_ 能夠根據用戶請求創建新內容（例如文字、影像、聲音、程式碼等）。它利用像 OpenAI 的 GPT（「生成式預訓練轉換器」）系列這類大型語言模型來實現，這些模型經過自然語言及程式碼的訓練。

使用者現在可以使用熟悉的對話形式與這些模型互動，無需任何技術專長或訓練。這些模型基於 _prompt_ ——用戶發送文字輸入（提示），模型回傳 AI 回應（完成）。用戶可以透過多輪不斷迭代「與 AI 對話」，反覆調整提示直到回應符合預期。

「提示」現已成為生成式 AI 應用的主要 _程式介面_，用以告訴模型該執行什麼，並影響返回回應的質量。「提示工程」是一個快速成長的領域，專注於 _設計與優化_ 提示，以規模化地達成一致且高質量的回應。

## 什麼是提示工程？

我們從定義<strong>提示工程</strong>開始，意指 _設計與優化_ 文字輸入（提示），以針對特定應用目標和模型，產出一致且高品質的回應（完成）。可視為一個兩步驟的過程：

- 為特定模型和目標 _設計_ 初始提示
- 反覆 _細化_ 提示以提升回應品質

這是需要使用者直覺與努力透過試錯來達成最佳結果的過程。那為何重要？先讓我們了解三個概念：

- _斷詞_ = 模型如何「看見」提示
- _基礎大型語言模型_ = 基礎模型如何「處理」提示
- _指令調校型大型語言模型_ = 模型如何「理解任務」

### 斷詞（Tokenization）

LLM 是將提示視為 _token_ 序列，不同模型（或同模型不同版本）對相同提示的斷詞方式可能不同。由於 LLM 是針對 token（非原始文字）訓練，因此提示的斷詞方式會直接影響生成回應的質量。

想了解斷詞原理，可嘗試下面的 [OpenAI Tokenizer](https://platform.openai.com/tokenizer?WT.mc_id=academic-105485-koreyst) 工具。複製提示文字，看如何被轉換為 token，注意空白字元及標點的處理方式。此範例顯示較舊的 LLM（GPT-3），以較新模型測試可能會有不同結果。

![斷詞示例](../../../translated_images/zh-HK/04-tokenizer-example.e71f0a0f70356c5c.webp)

### 概念：基礎模型（Foundation Models）

提示斷詞後，["基礎大型語言模型"](https://blog.gopenai.com/an-introduction-to-base-and-instruction-tuned-large-language-models-8de102c785a6?WT.mc_id=academic-105485-koreyst)（或稱基礎模型）的主要功能是預測序列中的下一個 token。由於 LLM 以龐大文本資料訓練，能理解 token 間的統計關係，並有信心做出預測。注意，它們不理解提示或 token 的 _意義_，僅見到可「完成」的模式。預測會持續直到用戶中止或符合預設條件。

想看看基於提示的完成如何運作？將上述提示貼入 [Microsoft Foundry playground](https://ai.azure.com?WT.mc_id=academic-105485-koreyst)（預設設定）。系統將提示視為資訊請求，應會產生符合該情境的回應。

那如果用戶想看到符合某條件或任務目標的特定資訊呢？這時就派上 _指令調校型_ LLM 的用場了。

![基礎 LLM 聊天完成示例](../../../translated_images/zh-HK/04-playground-chat-base.65b76fcfde0caa67.webp)

### 概念：指令調校型大型語言模型

[指令調校型 LLM](https://blog.gopenai.com/an-introduction-to-base-and-instruction-tuned-large-language-models-8de102c785a6?WT.mc_id=academic-105485-koreyst)以基礎模型為起點，透過含清楚指令的範例或輸入/輸出對（例如多輪「訊息」）微調，使 AI 回應嘗試遵循該指令。

利用如人類回饋的強化學習（RLHF）技術，訓練模型 _遵循指令_ 及 _從回饋學習_，產出更適合實際應用且更符合使用者目標的回答。

來試試看——回到前述提示，將 _系統訊息_ 改成提供以下指令作為上下文：

> _為二年級學生總結你收到的內容。結果限定一段文字，包含 3-5 個重點條列。_

看說明回應已調整來反映目標與格式，教育工作者可直接用於課堂簡報。

![指令調校型 LLM 聊天完成示例](../../../translated_images/zh-HK/04-playground-chat-instructions.b30bbfbdf92f2d05.webp)

## 為什麼需要提示工程？

了解 LLM 如何處理提示後，我們來討論 _為何_ 需要提示工程。原因在於目前的 LLM 面臨多項挑戰，若不花力氣建構和優化提示，很難達到 _可靠且一致的完成結果_。例如：

1. **模型回應具有隨機性。** _相同提示_ 在不同模型或版本可能產生不同回應，同一模型的不同時間也可能不同。_提示工程技巧可幫助最小化這些變異，提供更好的護欄_。

1. **模型可能偽造回應。** 模型經訓練於 _龐大但有限_ 的資料集，對訓練範圍外的概念缺乏知識，導致生成不正確、虛構或與已知事實相矛盾的內容。_提示工程技巧協助用戶識別及減輕此類偽造，舉例包含請 AI 提供引用或推理_。

1. **模型能力會有差異。** 新一代模型具豐富能力，但也帶來成本與複雜度上的權衡。_提示工程有助於建立最佳實務與作業流程，抽象差異並能以可擴展、無縫方式適應不同模型需求_。

讓我們在 OpenAI 或 Azure OpenAI Playground 實際看看：

- 對不同 LLM 部署（例如 OpenAI、Azure OpenAI、Hugging Face），使用相同提示 — 你看到差異了嗎？
- 對 _相同_ LLM 部署多次使用同一提示（例如 Azure OpenAI Playground） — 變異如何不同？

### 偽造示例

在本課程中，我們用 **「偽造」** 指 LLM 由於訓練或其他限制，有時會生成事實錯誤的資訊。這在流行文章或研究論文中被稱為 _「幻覺」_。然而，我們強烈建議使用 _「偽造」_ 一詞，避免誤將此行為擬人化，賦予機器類似人類的特質。這也符合 [負責任 AI 指南](https://www.microsoft.com/ai/responsible-ai?WT.mc_id=academic-105485-koreyst) 的術語使用原則，避免使用在某些情境中可能被視為冒犯或不包容的詞彙。

想了解偽造如何發生？試想一個指示 AI 產生不存在主題內容的提示（確保該主題不在訓練資料中）。舉例來說—我嘗試了此提示：

> **提示：** 產生關於 2076 年火星戰爭的課程計劃。

網路搜索顯示有關火星戰爭的虛構敘述（例如電視劇或書籍），但沒有提到 2076 年。常識告訴我們 2076 年 _還在未來_，不可能是真實事件。

如預期，每個模型（或模型版本）因為隨機性行為和模型能力差異，產生稍有不同的回應。例如，有一個模型的目標讀者是八年級學生，而另一個則假設是高中生。但三個模型都產生了能夠說服不知情用戶該事件是真的的回應。

提示工程技巧如 _元提示設計_ 和 _溫度設定_ 可能在某種程度上減少模型的捏造。新的提示工程 _架構_ 也能將新工具和技術無縫融入提示流程，以緩解或減少這些效應。

## 提示構造

我們已了解提示工程的重要性——現在讓我們明白提示是如何 _構造_ 的，這樣才能評估不同技巧以設計出更有效的提示。

### 基本提示

先從基本提示開始：向模型傳送無其他上下文的文字輸入。舉例來說，當我們將美國國歌開頭幾個字傳送至 OpenAI [Completion API](https://platform.openai.com/docs/api-reference/completions?WT.mc_id=academic-105485-koreyst)，它會立即 _補全_ 後續歌詞，說明了基本的預測行為。

| 提示（輸入）         | 補全（輸出）                                                                                                                           |
| :----------------- | :------------------------------------------------------------------------------------------------------------------------------------ |
| Oh say can you see | 聽起來你開始唱《星條旗永不落》，美國國歌。完整歌詞是...                                                                                 |

### 複雜提示

現在我們來給基本提示增加上下文和指令。[Chat Completion API](https://learn.microsoft.com/azure/ai-services/openai/how-to/chatgpt?WT.mc_id=academic-105485-koreyst) 讓我們將複雜提示構造成由多條 _訊息_ 組成的集合，這些訊息包含：

- 反映 _用戶_ 輸入和 _助手_ 回應的輸入/輸出配對。
- 系統訊息，設定助手的行為或個性上下文。

現在請求呈現如下形式，其中 _斷詞(tokenization)_ 有效擷取對話和上下文中的相關資訊。更改系統上下文對補全品質的影響可與提供的用戶輸入一樣重要。

```python
response = client.responses.create(
    model="gpt-4o-mini",
    input=[
        {"role": "system", "content": "You are a helpful assistant."},
        {"role": "user", "content": "Who won the world series in 2020?"},
        {"role": "assistant", "content": "The Los Angeles Dodgers won the World Series in 2020."},
        {"role": "user", "content": "Where was it played?"}
    ]
)
```

### 指令提示

在上述例子中，用戶提示是可以被當作訊息請求的簡單文字查詢。使用 _指令_ 提示，我們可用文字更詳細指定任務，給 AI 更佳指引。例如：

| 提示（輸入）                                                                                                                                                                                                                             | 補全（輸出）                                                                                 | 指令類型        |
| :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :------------------------------------------------------------------------------------------ | :-------------- |
| Write a description of the Civil War                                                                                                                                                                                                 | _返回簡單段落_                                                                              | 簡單            |
| Write a description of the Civil War. Provide key dates and events and describe their significance                                                                                                                                   | _返回段落並附上重要事件日期列表與描述_                                                     | 複雜            |
| Write a description of the Civil War in 1 paragraph. Provide 3 bullet points with key dates and their significance. Provide 3 more bullet points with key historical figures and their contributions. Return the output as a JSON file | _以 JSON 格式於文字框中呈現更詳細內容，可複製貼上成檔案並驗證_                            | 複雜且格式化    |

## 主體內容

在上述例子中，提示仍相當開放式，讓大型語言模型自行決定其預訓練資料中哪些部分是相關的。使用 _主體內容_ 設計模式時，輸入文字被分成兩部分：

- 指令（動作）
- 相關內容（影響動作的資訊）

以下是一個範例：指令是「用兩句話總結這段」。

### 提示範本

提示範本是 _預先定義的提示配方_，可以儲存並在需要時重複使用，以規模化推動更一致的使用者體驗。最簡單的形式，就是像[這個來自OpenAI的例子](https://cookbook.openai.com/examples/gpt4-1_prompting_guide?WT.mc_id=academic-105485-koreyst)那樣的提示範例集合，提供互動提示元件（使用者與系統訊息）和透過API驅動的請求格式——以支援重用。

在更複雜的形式中，如[這個LangChain的例子](https://python.langchain.com/docs/concepts/prompt_templates/?WT.mc_id=academic-105485-koreyst)包含 _佔位符_，可用來替換來自各種來源的資料（使用者輸入、系統上下文、外部資料來源等）以動態生成提示。這讓我們能建立可程式化規模使用的一系列可重用提示庫，推動一致的使用者體驗。

最後，模板的真正價值在於能為垂直應用領域創建並發布 _提示庫_—此類提示範本會針對應用特定上下文或範例進行 _優化_，使回應更切合且準確地迎合目標使用者。如[Prompts For Edu](https://github.com/microsoft/prompts-for-edu?WT.mc_id=academic-105485-koreyst)專案，就是這種方法的絕佳範例，策劃了一套教育領域的提示庫，著重於課程規劃、課程設計、學生輔導等關鍵目標。

## 支援內容

如果我們將提示結構看作有指令（任務）和目標（主要內容），那麼 _次要內容_ 就像是我們提供的額外上下文，用以<strong>以某種方式影響輸出</strong>。它可能是調校參數、格式指示、主題分類法等，能幫助模型 _調整_ 回應，使其符合預期的使用者目標或期待。

例如：給定一份課程目錄，含有豐富的元資料（名稱、描述、等級、元資料標籤、授課教師等）涵蓋所有課程內容：

- 我們可以定義一個指令為「總結2023年秋季的課程目錄」
- 我們可以使用主要內容提供幾個預期輸出範例
- 我們可以用次要內容標示前五個「標籤」重點。

現在，模型可以用示例顯示的格式提供總結——但若結果包含多個標籤，則會優先考慮次要內容中標定的五個標籤。

## 提示最佳實務

既然我們知道如何 _建構_ 提示，現在可以開始思考如何 _設計_ 它們以符合最佳實務。我們可以從兩大層面來思考——擁有正確的 _心態_ 和應用正確的 _技巧_。

### 提示工程心態

提示工程是個試錯過程，請記住三個廣泛的指導要點：

1. **領域理解很重要。** 回應的準確性與相關性取決於應用或使用者所在的 _領域_。運用你的直覺和領域專長來<strong>進一步定制技巧</strong>。例如，在系統提示中定義 _領域特定的人格_，或在使用者的提示中使用 _領域特定的範本_。提供反映領域特定上下文的次要內容，或使用 _領域特定的提示與範例_，引導模型朝熟悉的使用模式。

2. **理解模型很重要。** 我們知道模型本質是隨機性的。但模型實作也會因訓練資料集（預訓練知識）、提供的功能（例如透過API或SDK）、及其優化的內容類型（例如代碼、圖片或文字）而異。了解你使用的模型的優缺點，並利用該知識 _優先處理任務_ 或建立 _依模型能力優化的定制模板_。

3. **迭代與驗證很重要。** 模型和提示工程技巧正快速演進。作為領域專家，你可能會有其他背景或標準，適用於 _你_ 的特定應用，而非廣泛社群。利用提示工程工具與技巧「快速啟動」提示建構，然後用你的直覺與領域專長反覆迭代與驗證結果。記錄你的見解並建立一個 <strong>知識庫</strong>（例如提示庫），可作為他人未來快速迭代的新基線。
