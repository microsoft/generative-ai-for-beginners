# 提示工程基礎

[![提示工程基礎](../../../translated_images/zh-HK/04-lesson-banner.a2c90deba7fedacd.webp)](https://youtu.be/GElCu2kUlRs?si=qrXsBvXnCW12epb8)

## 介紹
本單元涵蓋生成式 AI 模型中創建有效提示的基本概念和技術。向大型語言模型 (LLM) 編寫提示的方式同樣重要。精心設計的提示能夠獲得更高質量的回應。但「提示」和「提示工程」這些術語究竟是什麼意思？我如何改進發送給 LLM 的提示輸入？這些都是我們在本章及下一章嘗試解答的問題。

_生成式 AI_ 能夠根據用戶請求創造新的內容（例如，文字、圖像、音頻、程式碼等）。它是利用像 OpenAI 的 GPT（「生成式預訓練變換器」）系列這樣的 _大型語言模型_ 來達成的，這些模型經過自然語言和程式碼的訓練。

用戶現在可以通過類似聊天的熟悉範式與這些模型互動，無需任何技術專長或訓練。這些模型是 _基於提示的_ ——用戶傳送文本輸入（提示），並獲得 AI 回應（完成）。用戶可以迭代地「與 AI 聊天」，進行多輪對話，調整提示直到回應符合期望。

「提示」現已成為生成式 AI 應用的主要 _程式設計介面_，指示模型該做什麼並影響回應品質。「提示工程」是一個快速成長的研究領域，專注於 _設計和優化_ 提示，以大規模 deliver 一致且優質的回應。

## 什麼是提示工程？

我們從定義 <strong>提示工程</strong> 開始，指為了給定應用目標和模型，_設計與優化_ 文本輸入（提示）以交付一致且優質回應（完成）的過程。我們可以將此視為一個兩步驟流程：

- 為特定模型和目標 _設計_ 初步提示
- 反覆 _調整_ 提示以提升回應質量

這必須透過反覆試錯過程進行，需用戶直覺與努力才能達到最佳效果。那為什麼重要？首先我們要了解三個概念：

- _分詞_ = 模型「看見」提示的方式
- _基礎 LLM_ = 基礎模型「處理」提示的方式
- _指令微調 LLM_ = 模型現在如何「看見任務」

### 分詞

大型語言模型將提示視為 _詞元序列_，不同模型（或同一模型的不同版本）可對同一提示執行不同方式的分詞。由於 LLM 是基於詞元訓練的（而非原始文字），提示如何被分詞會直接影響生成回應的質量。

要理解分詞原理，可以試用下方的 [OpenAI 分詞器](https://platform.openai.com/tokenizer?WT.mc_id=academic-105485-koreyst) 工具。將你的提示貼入，觀察其如何轉換成詞元，特別留意空白字符和標點符號的處理方式。請注意此範例使用的是較舊的 LLM（GPT-3），以較新模型試驗可能有不同結果。

![分詞示例](../../../translated_images/zh-HK/04-tokenizer-example.e71f0a0f70356c5c.webp)

### 概念：基礎模型

提示一旦被分詞，["基礎 LLM"](https://blog.gopenai.com/an-introduction-to-base-and-instruction-tuned-large-language-models-8de102c785a6?WT.mc_id=academic-105485-koreyst)（或稱基礎模型）的主要功能是預測該序列中的下一個詞元。由於 LLM 受大量文本資料訓練，對詞元間的統計關係有良好理解，可以較有信心地做出預測。請注意，它們不理解提示或詞元中的 _意義_，只是看到可以透過下一個預測來「完成」的模式。直到用戶干預或達到預先設定條件，模型會持續預測序列。

想看看基於提示的完成是如何運作的嗎？將上述提示輸入 [Microsoft Foundry playground](https://ai.azure.com?WT.mc_id=academic-105485-koreyst)，使用預設設定。系統會將提示視為資訊請求，因此你應該會看到符合該上下文的完成内容。

但如果用戶想要看到符合某些條件或任務目標的特定內容呢？這時就有 _指令微調_ 的大型語言模型登場了。

![基礎 LLM 聊天完成示例](../../../translated_images/zh-HK/04-playground-chat-base.65b76fcfde0caa67.webp)

### 概念：指令微調 LLM

[指令微調 LLM](https://blog.gopenai.com/an-introduction-to-base-and-instruction-tuned-large-language-models-8de102c785a6?WT.mc_id=academic-105485-koreyst)由基礎模型開始，透過範例或輸入/輸出對（例如多輪「訊息」），帶有明確指令，進行微調，AI 回應會嘗試遵從該指令。

這利用了例如帶有人類回饋的強化學習（RLHF）等技術，訓練模型 _遵循指令_ 並 _從回饋中學習_，使其產生更適合實際應用且更貼合用戶目標的回應。

讓我們試一試——回到上述提示，但這次將 _系統訊息_ 更改為下述指令做為上下文：

> _為二年級學生總結你所提供的內容。保持結果為一段文字，包含 3-5 點重點條列。_

看看結果如何現在調整為反映所需目標和格式？教育工作者現在可以直接在該堂課的投影片中使用這回應。

![指令微調 LLM 聊天完成示例](../../../translated_images/zh-HK/04-playground-chat-instructions.b30bbfbdf92f2d05.webp)

## 為什麼需要提示工程？

既然我們知道提示如何被 LLM 處理，讓我們談談 _為什麼_ 需要提示工程。原因在於當前 LLM 存在多項挑戰，使 _可靠且一致的完成_ 難以達成，除非在提示構造和優化上投入努力。例如：

1. **模型回應具隨機性。** _相同提示_ 在不同模型或模型版本間可能產生不同回應。即使是同一模型，也可能在不同時間出現不同結果。_提示工程技巧可以幫助我們透過更好的護欄來減少這些變異_。

1. **模型可能會捏造回應。** 模型經過 _龐大但有限_ 的數據集預訓練，缺乏訓練範圍外概念的知識。所以它們可能產生不準確、虛構或與已知事實直接矛盾的完成。_提示工程技巧能幫助用戶識別並減輕此類捏造，例如透過要求 AI 提供引用或推理_。

1. **模型能力有變異。** 較新模型或新一代模型功能更豐富，但也帶來獨特的特性與成本、複雜度上的權衡。_提示工程可以幫助我們發展最佳實踐和工作流程，抽象化差異並以可擴展、無縫方式適應模型特定需求_。

讓我們在 OpenAI 或 Azure OpenAI Playground 中實際操作看看：

- 使用相同提示搭配不同 LLM 部署（例如 OpenAI、Azure OpenAI、Hugging Face）——你觀察到變異了嗎？
- 使用相同提示重複搭配同一 LLM 部署（例如 Azure OpenAI playground）——這些變異有哪些差異？

### 捏造範例

在本課程中，我們用 **「捏造」** 一詞指代 LLM 由於訓練或其他限制，有時會產生事實錯誤資訊的現象。你或許也在流行文章或研究論文中聽過它被稱為 _「幻覺」_。但我們強烈建議使用 _「捏造」_ 這個術語，以免將機器產生的結果擬人化，賦予機器人類特質。這也從術語角度強化了[負責任 AI 指導原則](https://www.microsoft.com/ai/responsible-ai?WT.mc_id=academic-105485-koreyst)，剔除了可能在某些場合被認為具冒犯性或不包容的詞彙。

想了解捏造是如何發生的？想像一個提示指示 AI 生成不存在主題的內容（以確保該主題不在訓練數據中）。例如，我試過這個提示：

> **提示：** 生成一份有關 2076 年火星戰爭的課程計劃。

網絡搜尋顯示火星戰爭有虛構敘述（例如電視劇或書籍），但不是在 2076 年。常識告訴我們 2076 年是在未來，因此不可能與真實事件相關。


那麼當我們用不同的 LLM 供應商執行這個提示時會發生什麼事？

> **回應 2**：Azure OpenAI Playground（GPT-35）

正如預期的，每個模型（或模型版本）因為隨機行為和模型能力的差異，而產生略有不同的回應。例如，一個模型針對的是八年級學生，而另一個則假設是高中生。但這三個模型都生成了足以使未受過相關資訊的用戶相信事件是真實的回答。

提示工程技術如 _元提示_ 和 _溫度設定_ 可能在某種程度上減少模型的捏造。新的提示工程 _架構_ 也將新工具和技術無縫整合到提示流程中，以緩解或減少這些影響。

## 提示組成

我們已經知道為何提示工程很重要—現在讓我們了解提示是如何被_組成_，以便能評估不同技術，設計更有效的提示。

### 基本提示

讓我們從基本提示開始：沒有其他背景上下文，單純送給模型的文字輸入。以下是例子——當我們將美國國歌的前幾個字送入 OpenAI 的[ Completion API ](https://platform.openai.com/docs/api-reference/completions?WT.mc_id=academic-105485-koreyst) 時，它會立刻_補全文字_為接下來的幾行，展示基本的預測行為。

| 提示（輸入）        | 補全（輸出）                                                                                                                             |
| :----------------- | :---------------------------------------------------------------------------------------------------------------------------------------- |
| Oh say can you see | 聽起來你正在開始唱美國國歌「星條旗永不落」（The Star-Spangled Banner）的歌詞，完整歌詞是 ...                                           |

### 複雜提示

現在來給這個基本提示加入上下文和指示。[Chat Completion API](https://learn.microsoft.com/azure/ai-foundry/openai/how-to/chatgpt?WT.mc_id=academic-105485-koreyst)允許我們組成一個複雜提示，作為多個 _訊息_ 的集合：

- 輸入/輸出對，反映 _使用者_ 輸入和 _助手_ 回應。
- 系統訊息設定助手的行為或人格上下文。

請求現依呈現為以下形式，其中 _分詞_ 有效捕捉來自上下文與對話的相關資訊。改變系統上下文對完成品質的影響，和使用者提供的輸入一樣重大。

```python
response = client.responses.create(
    model="gpt-5-mini",
    input=[
        {"role": "system", "content": "You are a helpful assistant."},
        {"role": "user", "content": "Who won the world series in 2020?"},
        {"role": "assistant", "content": "The Los Angeles Dodgers won the World Series in 2020."},
        {"role": "user", "content": "Where was it played?"}
    ]
)
```

### 指令提示

在上述例子中，使用者提示是一個簡單的文字查詢，可視為請求資訊。使用_指令_ 提示，我們可以使用該文字來更細緻地指定任務，提供更好的 AI 指導。舉例：

| 提示（輸入）                                                                                                                                                                                                                         | 補全（輸出）                                                                                                        | 指令類型            |
| :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :-----------------------------------------------------------------------------------------------------------------| :------------------ |
| 撰寫一段美國內戰的描述                                                                                                                                                                                                             | _返回一段簡單的文字_                                                                                              | 簡單                |
| 撰寫一段美國內戰的描述。提供重要日期和事件並說明其意義                                                                                                                                                                          | _返回一段文字，接著列出關鍵事件日期與描述_                                                                       | 複雜                |

### 提示範本

提示範本是 _預先定義的提示配方_，可以儲存並按需重用，以推動更一致且大規模的用戶體驗。它最簡單的形式是像[這個來自 OpenAI 的範例](https://cookbook.openai.com/examples/gpt4-1_prompting_guide?WT.mc_id=academic-105485-koreyst)那樣，提供交互式的提示元件（用戶和系統訊息）以及 API 驅動的請求格式——以支持重用。

在更複雜的形式中，如[這個來自 LangChain 的範例](https://python.langchain.com/docs/concepts/prompt_templates/?WT.mc_id=academic-105485-koreyst)，它包含可以用來替換來自各種來源資料（用戶輸入、系統上下文、外部數據源等）的 _佔位符_，用以動態生成提示。這讓我們能建立一個可重用的提示庫，能 <strong>以程式化方式</strong> 大規模地推動一致的用戶體驗。

最後，模板的真正價值在於能創建並發布垂直應用領域的 _提示庫_——此時提示模板經過 _優化_，反映特定應用的上下文或範例，使回應對目標用戶更具相關性和準確度。[Prompts For Edu](https://github.com/microsoft/prompts-for-edu?WT.mc_id=academic-105485-koreyst) 倉庫就是這方面的優秀例子，策劃了一套教育領域相關的提示庫，重點放在教學計劃、課程設計、學生輔導等關鍵目標。

## 支援內容

如果我們將提示建構視為有一個指令（任務）和一個目標（主要內容），那麼 _次要內容_ 就像是我們提供的額外上下文，用以 <strong>某種方式影響輸出</strong>。它可以是調整參數、格式指示、主題分類等，有助於模型 _針對_ 期望的用戶目標或期待調整回應。

例如：給定一個含有詳細元資料（名稱、描述、級別、元標籤、講師等）的課程目錄，涵蓋所有課程：

- 我們可以定義一個指令為「總結 2023 秋季學期課程目錄」
- 我們可以用主要內容給出幾個想要的輸出範例
- 我們可以用次要內容標示感興趣的前 5 個「標籤」

現在，模型可以以範例中展示的格式提供摘要——但若一條結果有多個標籤，它能優先考慮次要內容中標示出的 5 個標籤。

## 提示最佳實務

現在我們知道如何 _構建_ 提示，可以開始思考如何 _設計_ 它們以反映最佳實務。我們可以從兩方面來考慮——擁有正確的 _心態_ 和運用正確的 _技術_。

### 提示工程心態

提示工程是個試錯過程，請牢記三大指導因素：

1. **領域理解很重要。** 回應的準確與相關性取決於該應用或用戶操作的 _領域_。發揮你的直覺和領域專長來<strong>進一步定制技術</strong>。例如，在系統提示中定義 _領域專屬個性_，或在用戶提示中使用 _領域專屬模板_。提供反映領域上下文的次要內容，或用 _領域專屬線索和示例_ 引導模型習慣用法。

2. **模型了解很重要。** 我們知道模型本質上是隨機的。但模型實作也會因所用訓練數據集（預訓練知識）、支持的能力（例如透過 API 或 SDK）及其優化的內容類型（如程式碼、圖像、文字）而異。了解你使用的模型的優缺點，並用此知識 _優先任務_ 或建立 _自定義模板_，優化以契合該模型的能耐。

3. **反覆與驗證很重要。** 模型及提示工程技術都在快速演進。作為領域專家，你可能有針對 _特定應用_ 的其他背景或標準，未必適用於更廣的社群。使用提示工程工具與技術來「快速啟動」提示構建，然後用直覺和領域專長反覆試驗驗證結果。紀錄你的心得，建立一個 <strong>知識庫</strong> （如提示庫），作為日後他人更快迭代的新基準。
