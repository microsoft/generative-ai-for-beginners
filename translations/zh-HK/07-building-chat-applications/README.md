### 聊天機器人還是聊天應用？

在深入建置聊天應用程式之前，我們先比較「聊天機器人」和「AI 驅動的聊天應用程式」，它們扮演著不同的角色與功能。聊天機器人的主要功能是自動化特定對話任務，例如回答常見問題或追蹤包裹。它通常由規則邏輯或複雜的 AI 算法控制。相比之下，AI 驅動的聊天應用是一個更廣闊的數位通訊環境，能促進如文字、語音及視頻等多種人類用戶間的交流。其核心特色是整合生成式 AI 模型，模擬細緻如人類般的對話，根據多樣輸入及上下文線索產生回應。生成式 AI 驅動的聊天應用能從事開放領域的對話，適應變化的對話情境，甚至產生創意或複雜的交流。

下表概述主要差異與相似點，幫助我們理解它們在數位通訊中的獨特角色。

| 聊天機器人                               | 生成式 AI 驅動的聊天應用程式         |
| ------------------------------------- | -------------------------------------- |
| 聚焦任務且基於規則                     | 瞭解上下文                            |
| 常整合於較大系統中                     | 可能承載一個或多個聊天機器人          |
| 限於已編程功能                         | 包含生成式 AI 模型                    |
| 專門且結構化的互動                     | 可進行開放領域對話                    |

### 利用 SDK 與 API 使用預建功能

建立聊天應用程式時，一個很好的起點是評估現有資源。使用 SDK 和 API 建構聊天應用是有利的策略，原因多樣。透過整合有良好文件的 SDK 和 API，你讓應用程式在長期成功上佔據有利位置，解決擴展性與維護問題。

- <strong>加速開發並減少負擔</strong>：依賴預建功能而非自行巨資開發，可讓你專注於應用程式更重要的部分，例如商業邏輯。
- <strong>較佳表現</strong>：自行構建功能時，最終會問自己「它如何擴展？應用是否能應付突增的使用者？」妥善維護的 SDK 和 API 通常內建解決方案來應對這些問題。
- <strong>較易維護</strong>：更新與改進管理較容易，大部分 API 及 SDK 僅需在新版本釋出時更新函式庫。
- <strong>取得尖端技術</strong>：利用在大量資料上微調與訓練的模型，賦予應用程式自然語言能力。

使用 SDK 或 API 功能通常需取得授權，大多以獨特鑰匙或驗證 token 方式授權。我們將用 OpenAI Python 函式庫示範相關作法。你也可以嘗試本課程的以下[OpenAI 筆記本](./python/oai-assignment.ipynb?WT.mc_id=academic-105485-koreyst)或[Azure OpenAI Services 筆記本](./python/aoai-assignment.ipynb?WT.mc_id=academic-105485-koreys)。

```python
import os
from openai import OpenAI

API_KEY = os.getenv("OPENAI_API_KEY","")

client = OpenAI(
    api_key=API_KEY
    )

response = client.responses.create(model="gpt-4o-mini", input="Suggest two titles for an instructional lesson on chat applications for generative AI.", store=False)
print(response.output_text)
```

上例使用 GPT-4o mini 模型的 Responses API 來完成提示，但請注意先設定好 API 金鑰。若未設定金鑰，會收到錯誤。

## 使用者體驗 (UX)

一般 UX 原則適用於聊天應用程式，但由於機器學習元件的加入，下列額外考量變得特別重要。

- <strong>解決模糊性的機制</strong>：生成式 AI 模型偶爾會產生模稜兩可的回答。若用戶遇此問題，允許他們請求澄清的功能非常有用。
- <strong>上下文保留</strong>：先進生成式 AI 模型能記憶對話上下文，這可大幅提升用戶體驗。允許用戶控制和管理上下文提升體驗，但也可能留存敏感資訊。考慮資訊存放時間，如引入保留政策，能在上下文需求與隱私間取得平衡。
- <strong>個人化</strong>：AI 模型能學習與適應，為用戶提供個別化體驗。透過用戶檔案等功能調整體驗，不僅讓用戶感到被理解，還助於快速找到特定答案，創造更有效率且令人滿意的互動。

### 無障礙

不論用戶有視覺、聽覺、動作或認知障礙，設計良好的聊天應用程式都應便於所有人使用。以下列表說明針對不同障礙用戶增強無障礙特性的功能。

- <strong>視覺障礙功能</strong>：高對比主題和可調整文字大小、支援螢幕閱讀器。
- <strong>聽覺障礙功能</strong>：文字轉語音與語音轉文字功能、音訊通知的視覺提示。
- <strong>動作障礙功能</strong>：鍵盤導航支援、語音指令。
- <strong>認知障礙功能</strong>：簡化語言選項。

## 針對特定領域語言模型的客製化與微調

想像一個聊天應用程式，能理解你公司的術語並預測用戶常見特定詢問。有幾種方法值得提及：

- **利用 DSL 模型**。DSL 指的是特定領域語言模型。你可以利用訓練於特定領域的 DSL 模型，理解其概念與情境。
- <strong>應用微調</strong>。微調是進一步以特定資料訓練你的模型的過程。

## 客製化：使用 DSL

利用特定領域語言模型（DSL 模型）能增進用戶參與度，提供專業且上下文相關的互動。此類模型針對特定領域、行業或主題進行訓練或微調。使用 DSL 模型的選項包括從零開始訓練，或透過 SDK 和 API 使用現有模型，另一選擇是對既有預訓練模型進行微調以適應特定領域。

## 客製化：應用微調

當預訓練模型在專業領域或特定任務中表現不足時，常考慮微調。

例如醫療詢問十分複雜，需要大量上下文。醫生為病人診斷基於生活習慣、既往病史，甚至需要參考最新醫學期刊。在這種細緻情境下，一般用途的 AI 聊天應用或許無法成為可靠來源。

### 情境示例：醫療應用程式

想像一款協助醫療人員，能快速查詢治療指南、藥物交互作用或最新研究結果的聊天應用程式。

一般模型可能足以解答基本醫療問題或提供一般建議，但可能無法妥善處理：

- <strong>高度專業或複雜案例</strong>。例如神經科醫師可能詢問：「兒童耐藥性癲癇目前的最佳管理方案是什麼？」
- <strong>缺乏最新進展</strong>。一般模型可能無法提供包含神經學與藥理學最新進展的答案。

在此類情況下，針對專門醫療資料集進行微調，可顯著提升模型處理細緻醫療詢問的準確性與可靠性。這需要取得大型且相關的資料集，以代表該領域的挑戰與問題。

## 高品質 AI 驅動聊天體驗的考量

本節說明「高品質」聊天應用程式的標準，包括捕捉具體可行的指標及遵循負責任運用 AI 技術的框架。

### 關鍵指標

為維持應用程式的高品質表現，必須持續追蹤關鍵指標和考量因素。這些衡量不僅確保應用功能性，也評估 AI 模型及用戶體驗的品質。以下列表涵蓋基本指標、AI 指標與用戶體驗指標。

| 指標                         | 定義                                                                                                                 | 聊天開發者需注意事項                                                  |
| ----------------------------- | ---------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------- |
| **正常運作時間 (Uptime)**    | 測量應用程式可運作並供用戶存取的時間比例。                                                                           | 如何將停機時間降至最低？                                                |
| <strong>回應時間</strong>                 | 應用程式回覆使用者查詢所需時間。                                                                                     | 如何優化查詢處理以提升回應速度？                                        |
| **精確度 (Precision)**       | 真陽性預測佔所有陽性預測的比例。                                                                                     | 如何驗證模型的精確度？                                                  |
| **召回率 (Recall / 敏感度)** | 真陽性預測佔實際陽性的比例。                                                                                         | 如何衡量及改善召回率？                                                  |
| **F1 分數**                  | 精確度與召回率的調和平均數，平衡兩者的取捨。                                                                          | 你的目標 F1 分數為何？如何在精確度與召回率間取得平衡？                  |
| **困惑度 (Perplexity)**      | 測量模型預測的機率分布與數據實際分布的吻合程度。                                                                      | 如何降低困惑度？                                                        |
| <strong>用戶滿意度指標</strong>           | 衡量用戶對應用程式的感知，通常透過問卷調查收集。                                                                       | 你會多頻繁收集用戶反饋？如何根據反饋做出調整？                          |
| <strong>錯誤率</strong>                   | 模型在理解或輸出上的錯誤發生頻率。                                                                                   | 你有哪些策略來降低錯誤率？                                             |
| <strong>再訓練週期</strong>               | 模型更新頻率，用以納入新數據與洞察。                                                                                 | 你會多久再訓練一次模型？什麼情況會觸發再訓練？                        |

| <strong>異常檢測</strong>         | 用於識別不符合預期行為的異常模式的工具和技術。                        | 你會如何回應異常？                                        |

### 在聊天應用中實施負責任的 AI 實踐

微軟負責任 AI 的方法確定了六項原則，應指導 AI 的開發和使用。以下是這些原則、它們的定義，以及聊天開發者應該考慮的事項及其重要原因。

| 原則             | 微軟的定義                                | 聊天開發者的考慮                                      | 重要原因                                                                     |
| ---------------------- | ----------------------------------------------------- | ---------------------------------------------------------------------- | -------------------------------------------------------------------------------------- |
| 公平性               | AI 系統應公平對待所有人。            | 確保聊天應用不會基於用戶資料有所歧視。  | 建立用戶之間的信任和包容性；避免法律問題。                |

| 包容性          | AI 系統應賦能每個人並讓人們參與。 | 設計適合多元用戶的易用界面(UI/UX)。 | 確保更廣泛的人士能有效使用應用程式。                   |
