## Введение

AI-агенты представляют собой захватывающее развитие в области генеративного ИИ, позволяя крупным языковым моделям (LLM) эволюционировать от помощников к агентам, способным выполнять действия. Фреймворки AI-агентов позволяют разработчикам создавать приложения, которые предоставляют LLM доступ к инструментам и управлению состоянием. Эти фреймворки также улучшают видимость, позволяя пользователям и разработчикам отслеживать действия, запланированные LLM, тем самым улучшая управление опытом.

## Что такое AI-агенты?

AI-агенты — это очень захватывающая область в мире генеративного ИИ. Вместе с этим волнением иногда возникает путаница в терминах и их применении. Чтобы упростить понимание и охватить большинство инструментов, которые называют AI-агентами, мы будем использовать следующее определение:

AI-агенты позволяют крупным языковым моделям (LLM) выполнять задачи, предоставляя им доступ к **состоянию** и **инструментам**.

![Agent Model](../../../translated_images/ru/what-agent.21f2893bdfd01e6a.webp)

Давайте определим эти термины:

**Крупные языковые модели** — Это модели, упоминаемые в этом курсе, такие как GPT-5, GPT-4o, Llama 3.3 и другие.

**Состояние** — Это контекст, в котором работает LLM. LLM использует контекст своих прошлых действий и текущий контекст, чтобы направлять принятие решений для последующих действий. Фреймворки AI-агентов облегчают разработчикам поддержание этого контекста.

**Инструменты** — Чтобы выполнить задачу, запрошенную пользователем и запланированную LLM, LLM нужен доступ к инструментам. Примерами таких инструментов могут быть база данных, API, внешнее приложение или даже другая LLM!

Эти определения, надеемся, дадут вам хорошую базу для дальнейшего изучения того, как они реализованы. Давайте рассмотрим несколько различных фреймворков AI-агентов:

## LangChain Agents

[LangChain Agents](https://python.langchain.com/docs/how_to/#agents?WT.mc_id=academic-105485-koreyst) — это реализация упомянутых выше определений.

Для управления **состоянием** используется встроенная функция под названием `AgentExecutor`. Она принимает определённого `агента` и `инструменты`, доступные этому агенту.

`Agent Executor` также сохраняет историю чата, чтобы предоставить контекст разговора.

![Langchain Agents](../../../translated_images/ru/langchain-agents.edcc55b5d5c43716.webp)

LangChain предлагает [каталог инструментов](https://integrations.langchain.com/tools?WT.mc_id=academic-105485-koreyst), которые можно импортировать в ваше приложение для доступа LLM к ним. Эти инструменты созданы сообществом и командой LangChain.

Вы можете определить эти инструменты и передать их `Agent Executor`.

Видимость — ещё один важный аспект при обсуждении AI-агентов. Разработчикам приложений важно понимать, каким инструментом пользуется LLM и зачем. Для этого команда LangChain разработала LangSmith.

## AutoGen

Следующий фреймворк AI-агентов, который мы рассмотрим — [AutoGen](https://microsoft.github.io/autogen/?WT.mc_id=academic-105485-koreyst). Основное внимание в AutoGen уделяется разговорам. Агенты являются как **разговорчивыми**, так и **настраиваемыми**.

**Разговорчивые** — LLM могут начинать и продолжать разговор с другим LLM для выполнения задачи. Это делается путём создания `AssistantAgents` с определённым системным сообщением.

```python

autogen.AssistantAgent( name="Coder", llm_config=llm_config, ) pm = autogen.AssistantAgent( name="Product_manager", system_message="Creative in software product ideas.", llm_config=llm_config, )

```

**Настраиваемые** — Агенты могут быть определены не только как LLM, но и как пользователь или инструмент. Как разработчик, вы можете определить `UserProxyAgent`, который отвечает за взаимодействие с пользователем для получения обратной связи при выполнении задачи. Эта обратная связь может либо продолжить выполнение задачи, либо остановить его.

```python
user_proxy = UserProxyAgent(name="user_proxy")
```

### Состояние и инструменты

Для изменения и управления состоянием помощник-агент генерирует Python-код для выполнения задачи.

Вот пример процесса:

![AutoGen](../../../translated_images/ru/autogen.dee9a25a45fde584.webp)

#### LLM определён с системным сообщением

```python
system_message="For weather related tasks, only use the functions you have been provided with. Reply TERMINATE when the task is done."
```

Это системное сообщение направляет конкретный LLM, какие функции релевантны для его задачи. Помните, в AutoGen можно иметь несколько AssistantAgents с разными системными сообщениями.

#### Разговор инициирован пользователем

```python
user_proxy.initiate_chat( chatbot, message="I am planning a trip to NYC next week, can you help me pick out what to wear? ", )

```

Это сообщение от user_proxy (человека) запускает процесс, в ходе которого агент исследует возможные функции для выполнения.

#### Выполнение функции

```bash
chatbot (to user_proxy):

***** Suggested tool Call: get_weather ***** Arguments: {"location":"New York City, NY","time_periond:"7","temperature_unit":"Celsius"} ******************************************************** --------------------------------------------------------------------------------

>>>>>>>> EXECUTING FUNCTION get_weather... user_proxy (to chatbot): ***** Response from calling function "get_weather" ***** 112.22727272727272 EUR ****************************************************************

```

После обработки начального сообщения агент отправит предложение вызвать инструмент. В данном случае — функцию под названием `get_weather`. В зависимости от вашей конфигурации, эта функция может быть автоматически выполнена и прочитана агентом или выполнена на основе ввода пользователя.

Вы можете найти список [примеров кода AutoGen](https://microsoft.github.io/autogen/docs/Examples/?WT.mc_id=academic-105485-koreyst), чтобы подробнее изучить, как начать разработку.

## Microsoft Agent Framework

[Microsoft Agent Framework](https://learn.microsoft.com/agent-framework/?WT.mc_id=academic-105485-koreyst) — это открытый SDK Microsoft для создания AI-агентов и систем с несколькими агентами на **Python** и **.NET**. Он объединяет сильные стороны двух предыдущих проектов Microsoft — корпоративные функции **Semantic Kernel** и оркестрацию с несколькими агентами из **AutoGen** — в единую поддерживаемую платформу. Если вы начинаете новый проект агента сегодня, это рекомендованный преемник AutoGen.

Фреймворк масштабирующийся от одного **чат-агента** до сложных **многоагентных рабочих процессов**, и интегрируется напрямую с Microsoft Foundry, Azure OpenAI и OpenAI. Он также обеспечивает встроенную наблюдаемость через OpenTelemetry, позволяющую точно отслеживать действия ваших агентов.

### Состояние и инструменты

**Состояние** — фреймворк управляет контекстом разговора за вас через **потоки** (threads). Агент следит за историей сообщений (запросы пользователя, вызовы инструментов и результаты), так что каждый ход строится на предыдущих. Потоки можно сохранять, позволяя приостанавливать и возобновлять разговор.

**Инструменты** — вы даёте агенту инструменты, передавая обычные функции Python. Параметры с аннотациями типов автоматически превращаются в схему, так что модель знает, как и когда их вызывать (вызов функций). Фреймворк также поддерживает серверы Model Context Protocol (MCP) и размещённые инструменты, такие как интерпретатор кода.

Вот пример одного агента с пользовательским инструментом:

```python
import asyncio
from typing import Annotated

from pydantic import Field
from agent_framework import Agent
from agent_framework.openai import OpenAIChatClient


def get_weather(
    location: Annotated[str, Field(description="The location to get the weather for.")],
) -> str:
    """Get the weather for a given location."""
    return f"The weather in {location} is sunny with a high of 22°C."


async def main():
    agent = Agent(
        client=OpenAIChatClient(),
        instructions="You are a helpful assistant that can answer weather questions.",
        tools=[get_weather],
    )

    response = await agent.run("What's the weather in Amsterdam?")
    print(response)


asyncio.run(main())
```

Для подключения к Azure OpenAI в Microsoft Foundry передайте конечную точку и учётные данные клиенту:

```python
from azure.identity.aio import AzureCliCredential
from agent_framework.openai import OpenAIChatClient

client = OpenAIChatClient(
    model="my-gpt-5-mini-deployment",
    azure_endpoint="https://my-resource.openai.azure.com",
    credential=AzureCliCredential(),
)
```

### Многоагентные рабочие процессы

Фреймворк особенно выделяется возможностями оркестровки нескольких агентов. Например, вы можете запускать агентов последовательно (каждый передаёт контекст следующему) или параллельно распределять на несколько агентов и агрегировать их результаты:

```python
from agent_framework.orchestrations import SequentialBuilder, ConcurrentBuilder

# Запускайте агентов последовательно, передавая контекст разговора по цепочке
sequential = SequentialBuilder(participants=[researcher, writer, editor]).build()

# Разветвляйте на агентов параллельно, затем объединяйте их ответы
concurrent = ConcurrentBuilder(participants=[analyst_a, analyst_b, analyst_c]).build()
```

## Taskweaver

Следующий фреймворк, который мы рассмотрим — [Taskweaver](https://microsoft.github.io/TaskWeaver/?WT.mc_id=academic-105485-koreyst). Он известен как "код-ориентированный" агент, потому что вместо работы исключительно со `строками` может работать с DataFrame в Python. Это становится чрезвычайно полезным для задач анализа и генерации данных. Это могут быть такие задачи, как создание графиков и диаграмм или генерация случайных чисел.

### Состояние и инструменты

Для управления состоянием разговора TaskWeaver использует концепцию `Planner` (планировщика). `Planner` — это LLM, который принимает запросы пользователей и выстраивает задачи, необходимые для выполнения этого запроса.

Для выполнения задач `Planner` получает доступ к набору инструментов, называемому `Plugins`. Это могут быть классы Python или общий интерпретатор кода. Эти плагины хранятся в виде эмбеддингов, чтобы LLM лучше находил нужный плагин.

![Taskweaver](../../../translated_images/ru/taskweaver.da8559999267715a.webp)

Вот пример плагина для обнаружения аномалий:

```python
class AnomalyDetectionPlugin(Plugin): def __call__(self, df: pd.DataFrame, time_col_name: str, value_col_name: str):
```

Код проверяется перед выполнением. Ещё одна особенность для управления контекстом в Taskweaver — это `experience` (опыт). Опыт позволяет сохранять контекст разговора в долгосрочной перспективе в YAML-файле. Его можно настроить так, чтобы LLM со временем улучшался в определённых задачах, учитывая предыдущие разговоры.

## JARVIS

Последний фреймворк агентов, который мы рассмотрим, — [JARVIS](https://github.com/microsoft/JARVIS?tab=readme-ov-file&WT.mc_id=academic-105485-koreyst). Особенность JARVIS в том, что он использует LLM для управления `состоянием` разговора, а `инструментами` являются другие AI-модели. Каждая из AI-моделей специализирована и выполняет определённые задачи, такие как обнаружение объектов, транскрипция или создание подписей к изображениям.

![JARVIS](../../../translated_images/ru/jarvis.762ddbadbd1a3a33.webp)

LLM, являясь универсальной моделью, получает запрос от пользователя и определяет конкретную задачу и любые аргументы/данные, необходимые для её выполнения.

```python
[{"task": "object-detection", "id": 0, "dep": [-1], "args": {"image": "e1.jpg" }}]
```

Затем LLM форматирует запрос так, чтобы специализированная AI-модель могла его интерпретировать, например, в формате JSON. Как только AI-модель возвращает прогноз по задаче, LLM получает ответ.

Если для выполнения задачи требуется несколько моделей, LLM также интерпретирует ответы этих моделей, прежде чем объединить их для формирования ответа пользователю.
