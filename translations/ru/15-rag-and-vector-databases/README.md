## Генерация с поддержкой поиска (RAG)

Чат-бот с поддержкой LLM обрабатывает пользовательские запросы для генерации ответов. Он разработан для интерактивного взаимодействия и общения с пользователями по широкому кругу тем. Однако его ответы ограничены предоставленным контекстом и базовыми тренировочными данными. Например, знание GPT-4 ограничено сентябрем 2021 года, то есть он не знает событий после этого периода. Кроме того, данные для тренировки LLM исключают конфиденциальную информацию, такую как личные заметки или руководство по продукту компании.

### Как работает RAG (генерация с поддержкой поиска)

![рисунок, показывающий, как работают RAG](../../../translated_images/ru/how-rag-works.f5d0ff63942bd3a6.webp)

Предположим, вы хотите развернуть чат-бота, который создает тесты по вашим заметкам, для этого потребуется подключение к базе знаний. Здесь на помощь приходит RAG. RAG работает следующим образом:

- **База знаний:** Перед поиском документы необходимо загрузить и предварительно обработать, обычно разбивая большие документы на небольшие фрагменты, преобразовывая их в текстовые эмбеддинги и сохраняя в базе данных.

- **Пользовательский запрос:** пользователь задает вопрос

- **Поиск:** Когда пользователь задает вопрос, модель для создания эмбеддингов извлекает релевантную информацию из нашей базы знаний, чтобы добавить контекст в запрос.

- **Улучшенная генерация:** LLM улучшает свой ответ на основе полученных данных. Это позволяет сгенерированному ответу основываться не только на предварительно обученных данных, но и на релевантной информации из добавленного контекста. Извлеченные данные используются для улучшения ответов LLM. Затем LLM возвращает ответ на вопрос пользователя.

![рисунок, показывающий архитектуру RAG](../../../translated_images/ru/encoder-decode.f2658c25d0eadee2.webp)

Архитектура RAG реализована с помощью трансформеров, состоящих из двух частей: энкодера и декодера. Например, когда пользователь задает вопрос, входной текст 'кодируется' в векторы, улавливающие значение слов, потом эти векторы 'декодируются' в наш индекс документов и генерируют новый текст на основе запроса пользователя. LLM использует модель энкодер-декодер для генерации ответа.

Существует два подхода к реализации RAG согласно предложенной статье: [Retrieval-Augmented Generation for Knowledge intensive NLP (natural language processing software) Tasks](https://arxiv.org/pdf/2005.11401.pdf?WT.mc_id=academic-105485-koreyst):

- **_RAG-Sequence_** — использование извлеченных документов для предсказания наилучшего ответа на запрос пользователя

- **RAG-Token** — использование документов для генерации следующего токена, затем извлечение для ответа на запрос пользователя

### Зачем использовать RAG?

- **Богатство информации:** обеспечивает актуальность и современность текстовых ответов. Это повышает производительность в задачах, специфичных для домена, за счет доступа к внутренней базе знаний.

- Снижает искажения, используя **проверяемые данные** из базы знаний для предоставления контекста пользовательским запросам.

- Это **экономично**, поскольку требует меньших затрат по сравнению с дообучением LLM.

### Векторные базы данных

Векторная база данных, в отличие от традиционных баз, это специализированная база для хранения, управления и поиска встроенных векторов. Она хранит числовые представления документов. Преобразование данных в числовые эмбеддинги облегчает понимание и обработку информации нашей AI-системой.

Мы храним наши эмбеддинги во векторных базах данных, поскольку LLM ограничены по количеству токенов, которые могут принимать на вход. Поскольку передать весь эмбеддинг целиком нельзя, нам нужно разбивать его на части, и при запросе пользователя будут возвращаться наиболее похожие эмбеддинги вместе с запросом. Разбиение также снижает затраты на количество токенов, передаваемых через LLM.

Некоторые популярные векторные базы данных включают Azure Cosmos DB, Clarifyai, Pinecone, Chromadb, ScaNN, Qdrant и DeepLake. Вы можете создать модель Azure Cosmos DB с помощью Azure CLI с помощью следующей команды:

```bash

az cosmosdb create -n <cosmos-db-name> -r <resource-group-name>
az cosmosdb list-keys -n <cosmos-db-name> -g <resource-group-name>
```

### От текста к эмбеддингам

Перед тем как хранить данные, их необходимо преобразовать в векторные эмбеддинги. Если вы работаете с большими документами или длинными текстами, их можно разбивать на части в зависимости от ожидаемых запросов. Разбиение может происходить на уровне предложений или абзацев. Поскольку смысл определяется словами вокруг, к части можно добавить некоторый контекст, например, название документа или текст перед или после части. Разбиение данных можно выполнить следующим образом:

```python
def split_text(text, max_length, min_length):
    words = text.split()
    chunks = []
    current_chunk = []

    for word in words:
        current_chunk.append(word)
        if len(' '.join(current_chunk)) < max_length and len(' '.join(current_chunk)) > min_length:
            chunks.append(' '.join(current_chunk))
            current_chunk = []

    # Если последний блок не достиг минимальной длины, добавьте его в любом случае
    if current_chunk:
        chunks.append(' '.join(current_chunk))

    return chunks
```

После разбиения мы можем создать эмбеддинги текста с помощью различных моделей эмбеддингов. Некоторые из них: word2vec, ada-002 от OpenAI, Azure Computer Vision и многие другие. Выбор модели зависит от используемых языков, типа кодируемого контента (текст/изображения/аудио), размера входных данных и длины выходного эмбеддинга.

## Поиск и векторный поиск

Когда пользователь задает вопрос, ретривер преобразует его в вектор с помощью кодировщика запроса, а затем ищет в нашем поисковом индексе документы с релевантными векторами, связанными с входным запросом. После этого преобразует как входной вектор, так и векторы документов в текст и передает их LLM.

### Поиск

Поиск происходит, когда система пытается быстро найти документы из индекса, удовлетворяющие критериям поиска. Цель ретривера — получить документы, которые будут использоваться для предоставления контекста и закрепления LLM на ваших данных.

Существует несколько способов выполнения поиска в нашей базе данных, например:

- **Поиск по ключевым словам** — используется для текстовых поисков

- **Векторный поиск** — преобразует документы из текста в векторные представления с помощью моделей эмбеддинга, что позволяет выполнять **семантический поиск** по смыслу слов. Поиск осуществляется путем запроса документов с векторами, наиболее близкими к вопросу пользователя.

- **Гибридный** — сочетание поиска по ключевым словам и векторного поиска.

Проблема поиска возникает, когда в базе нет похожего ответа на запрос, тогда система возвращает максимально релевантную информацию. Вы можете настроить максимальное расстояние для релевантности или использовать гибридный поиск, который комбинирует ключевые слова и векторный поиск. В этом уроке мы используем гибридный поиск и будем хранить данные в dataframe с колонками для частей текста и эмбеддингов.

### Векторное сходство

Ретривер ищет в базе знаний эмбеддинги, которые находятся близко друг к другу, как ближайшие соседи, поскольку они соответствуют похожим текстам. В нашем сценарии пользователь задает запрос, он сначала преобразуется в эмбеддинг, который затем сопоставляется с похожими эмбеддингами. Обычно для измерения схожести используется косинусное сходство, основанное на угле между двумя векторами.

Также можно использовать другие способы измерения схожести: эвклидово расстояние — это прямая между концами векторов, и скалярное произведение — сумма произведений соответствующих элементов двух векторов.

### Поисковый индекс

При выполнении поиска необходимо построить поисковый индекс для базы знаний перед поиском. Индекс хранит наши эмбеддинги и быстро находит наиболее похожие части даже в большой базе данных. Индекс можно создать локально с помощью:

```python
from sklearn.neighbors import NearestNeighbors

embeddings = flattened_df['embeddings'].to_list()

# Создать поисковый индекс
nbrs = NearestNeighbors(n_neighbors=5, algorithm='ball_tree').fit(embeddings)

# Для запроса к индексу можно использовать метод kneighbors
distances, indices = nbrs.kneighbors(embeddings)
```

### Переоценка (Re-ranking)

После запроса к базе, возможно, потребуется отсортировать результаты по убыванию релевантности. Модель переоценки LLM использует машинное обучение для повышения релевантности результатов поиска, упорядочивая их от самых релевантных. В Azure AI Search переоценка выполняется автоматически с помощью семантического переоценщика. Пример работы переоценки с использованием ближайших соседей:

```python
# Найдите наиболее похожие документы
distances, indices = nbrs.kneighbors([query_vector])

index = []
# Выведите наиболее похожие документы
for i in range(3):
    index = indices[0][i]
    for index in indices[0]:
        print(flattened_df['chunks'].iloc[index])
        print(flattened_df['path'].iloc[index])
        print(flattened_df['distances'].iloc[index])
    else:
        print(f"Index {index} not found in DataFrame")
```

## Объединяем всё вместе

```python
user_input = "what is a perceptron?"

def chatbot(user_input):
    # Преобразовать вопрос в вектор запроса
    query_vector = create_embeddings(user_input)

    # Найти наиболее похожие документы

    # добавить документы к запросу для предоставления контекста

    # создать объект сообщения

        {"role": "system", "content": "You are an AI assistant that helps with AI questions."},
        {"role": "user", "content": "\n\n".join(history) }
    ]
```
