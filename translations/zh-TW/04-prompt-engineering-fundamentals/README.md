# 提示工程基礎

[![提示工程基礎](../../../translated_images/zh-TW/04-lesson-banner.a2c90deba7fedacd.webp)](https://youtu.be/GElCu2kUlRs?si=qrXsBvXnCW12epb8)

## 介紹
本模組涵蓋在生成式 AI 模型中創建有效提示的基本概念和技術。你給大型語言模型 (LLM) 的提示寫法也很重要。經過精心設計的提示可以產生更高質量的回應。但像是「提示 (prompt)」和「提示工程 (prompt engineering)」這些詞究竟是什麼意思？我該如何改善送到 LLM 的提示輸入呢？這些是我們在本章和下一章會嘗試解答的問題。

_生成式 AI_ 能夠根據使用者請求創造新內容（例如文字、圖片、音訊、程式碼等）。它透過像是 OpenAI 的 GPT（「生成式預訓練轉換器」）系列大型語言模型，這些模型是經由自然語言和程式碼的訓練而成。

使用者現在可以用熟悉的對話方式與這些模型互動，無需任何技術專長或訓練。這些模型是基於「提示」的，使用者傳送文字輸入（提示），模型就會回應 AI 產生的結果（補全）。使用者能透過多回合對話「與 AI 聊天」，不斷微調其提示直到回應符合期望。

「提示」現在成為生成式 AI 應用的主要 _程式介面_，告訴模型要做什麼，並影響回應品質。「提示工程」是一個快速成長的研究領域，專注於設計和優化提示，旨在大規模地提供穩定且高品質的回應。

## 什麼是提示工程？

我們從定義<strong>提示工程</strong>開始，指的是為給定的應用目標和模型_設計與優化_文字輸入（提示），以產出一致且高品質的回應（補全）的過程。我們可以將其視為兩步驟：

- 為特定模型與目標_設計_初始提示
- 迭代_微調_提示以提升回應品質

這必然是一個試錯過程，需靠使用者直覺和努力以取得最佳效果。那為什麼它重要呢？為了解答，我們先來理解三個概念：

- _分詞_ = 模型如何「看到」提示
- _基礎 LLM_ = 基礎模型如何「處理」提示
- _指令微調 LLM_ = 模型如何「理解任務」

### 分詞

大型語言模型將提示視為_一連串的 token_，不同模型（或模型版本）對同一提示的分詞方式可能不同。由於 LLM 是以 token（而非原始文字）進行訓練，提示的分詞方式會直接影響生成回應的品質。

想要了解分詞的運作直覺嗎？試試下面示例的 [OpenAI Tokenizer](https://platform.openai.com/tokenizer?WT.mc_id=academic-105485-koreyst) 工具。貼上你的提示－看看它如何被轉換為 token，特別留意空白字元與標點符號的處理。此示例用的是舊版 LLM（GPT-3），用新版模型可能會得出不同結果。

![分詞示例](../../../translated_images/zh-TW/04-tokenizer-example.e71f0a0f70356c5c.webp)

### 概念：基礎模型

一旦提示被分詞，["基礎 LLM"](https://blog.gopenai.com/an-introduction-to-base-and-instruction-tuned-large-language-models-8de102c785a6?WT.mc_id=academic-105485-koreyst)（或稱基礎模型）的核心功能是預測序列中下一個 token。由於 LLM 是基於龐大文字資料集訓練，能很好地掌握 token 間統計關係，並以一定信心完成預測。請注意，模型並不「理解」提示或 token 的_意義_，它只是看到可「補完」的模式，並不斷預測序列直到使用者終止或達既定條件。

想看看基於提示的補全如何運作？將上面提示輸入 [Microsoft Foundry playground](https://ai.azure.com?WT.mc_id=academic-105485-koreyst) 的預設設定。系統把提示視為資訊請求，你應該會看到符合上下文的補全結果。

但如果使用者想要特定符合某些條件或任務目標的回應呢？這時候就需要用到_指令微調_的 LLM。

![基礎 LLM 聊天補全](../../../translated_images/zh-TW/04-playground-chat-base.65b76fcfde0caa67.webp)

### 概念：指令微調 LLM

[指令微調 LLM](https://blog.gopenai.com/an-introduction-to-base-and-instruction-tuned-large-language-models-8de102c785a6?WT.mc_id=academic-105485-koreyst) 基於基礎模型，使用帶有明確指令的範例或輸入/輸出對（例如多回合「訊息」）進行微調，AI 的回應會嘗試遵循該指令。

它運用像是人類回饋強化學習 (RLHF) 的技術，可教模型_遵循指令_、_從回饋學習_，進而產生更切合實際應用、更符合使用者目標的回應。

我們試試看－回到上面提示，將 _系統訊息_ 改成以下指令作為上下文：

> _請將所提供內容為二年級學生做摘要。結果限定在一段文字，並以 3-5 個重點列出。_

看看結果如何調整為符合指定目標和格式？教育工作者現在可以直接使用這段回應於該堂課的簡報。

![指令微調 LLM 聊天補全](../../../translated_images/zh-TW/04-playground-chat-instructions.b30bbfbdf92f2d05.webp)

## 為什麼我們需要提示工程？

瞭解提示如何被 LLM 處理後，我們來談談_為何_需要提示工程。理由在於現有 LLM 存在一些挑戰，使得不花心力進行提示建構和優化，很難達成_可靠且一致的補全_。例如：

1. **模型回應具隨機性。**_相同的提示_在不同模型或版本中，可能產生不同回應。同一模型在不同時間也會有不同結果。_提示工程技巧有助於降低這些變異，提供更好的護欄_。

1. <strong>模型可能編造回應。</strong>模型是基於_龐大但有限_的資料集訓練，因此對超出訓練範圍的概念不甚了解。結果可能生成錯誤、虛構或與既知事實相反的補全。_提示工程可幫助使用者識別並減輕這類編造，例如向 AI 詢問引用來源或推理_。

1. <strong>模型能力各異。</strong>新版或新一代模型功能更豐富，但同時帶來獨特的特性及成本與複雜度的權衡。_提示工程能助我們發展最佳實務與工作流程，抽象化差異並適應特定模型需求，以規模化和無縫方式調整_。

讓我們在 OpenAI 或 Azure OpenAI Playground 中看看這些現象：

- 用相同提示嘗試不同 LLM 部署（如 OpenAI、Azure OpenAI、Hugging Face）－你看到變化了嗎？
- 用相同提示重複嘗試同一 LLM 部署（例如 Azure OpenAI playground）－這些變化有何不同？

### 編造範例

在本課程中，我們用<strong>「編造」</strong>一詞指代 LLM 有時因訓練限制或其他制約，產生事實不正確資訊的現象。你可能在流行文章或研究論文中聽過此現象被稱為_"幻覺"（hallucination）_。但我們強烈建議使用「編造」一詞，以避免不小心將此行為人格化，誤將機器生成結果視為人類特質。這同時強化了從用語角度出發的[負責任 AI 指南](https://www.microsoft.com/ai/responsible-ai?WT.mc_id=academic-105485-koreyst)，剔除可能被視為冒犯或不包容的詞彙。

想了解編造是如何運作嗎？想像一個要求 AI 產生不存在主題內容的提示（確保該主題未在訓練資料中）。例如－我試了這條提示：

網路搜尋顯示，有關火星戰爭的虛構作品（例如電視劇或書籍），但沒有發生在 2076 年的。常識也告訴我們，2076 年_屬於未來_，因此不可能是真實事件。

提示工程技術如 _元提示（metaprompting）_ 和 _溫度設定（temperature configuration）_ 可能在一定程度上減少模型捏造。新的提示工程 _架構_ 也無縫整合新工具和技術進入提示流程，以緩解或減少部分這些效應。

## 提示構造

我們已了解為什麼提示工程重要—現在讓我們理解提示如何 _構造_，以便評估不同技術來設計更有效的提示。

### 基本提示

先從基本提示開始：一個沒有其他上下文的文字輸入送給模型。舉例來說，當我們將美國國歌開頭幾句送到 OpenAI [Completion API](https://platform.openai.com/docs/api-reference/completions?WT.mc_id=academic-105485-koreyst)，它會瞬間 _補完_ 接下來的幾句，展示基本的預測行為。

| 提示（輸入）       | 補完（輸出）                                                                                                                           |
| :----------------- | :------------------------------------------------------------------------------------------------------------------------------------- |
| Oh say can you see | 聽起來你是在開始唱《星條旗永不落》（The Star-Spangled Banner），美國的國歌。完整歌詞是 ...                                       |

### 複雜提示

現在讓我們在基本提示上加入上下文和指示。[Chat Completion API](https://learn.microsoft.com/azure/ai-services/openai/how-to/chatgpt?WT.mc_id=academic-105485-koreyst) 讓我們能建構為一組 _訊息_ 的複雜提示，包含：

- 反映 _使用者_ 輸入和 _助手_ 回應的輸入/輸出對。
- 設定助手行為或個性的系統訊息。

請求現在以下列形式呈現，其中 _分詞（tokenization）_ 有效擷取上下文和對話中的相關資訊。改變系統上下文對補完品質的影響，與提供的使用者輸入同等重要。

```python
response = client.responses.create(
    model="gpt-4o-mini",
    input=[
        {"role": "system", "content": "You are a helpful assistant."},
        {"role": "user", "content": "Who won the world series in 2020?"},
        {"role": "assistant", "content": "The Los Angeles Dodgers won the World Series in 2020."},
        {"role": "user", "content": "Where was it played?"}
    ]
)
```

### 指令提示

上述範例中，使用者提示是簡單的文字查詢，可詮釋為資訊請求。藉由 _指令_ 提示，我們能用文字更詳細規定任務，提供更佳指引給 AI。以下是範例：

| 提示（輸入）                                                                                                                                                                                                                         | 補完（輸出）                                                                                                        | 指令類型            |
| :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :------------------------------------------------------------------------------------------------------------------------- | :------------------ |
| Write a description of the Civil War                                                                                                                                                                                                   | _回傳簡單段落_                                                                                                   | 簡單                |
| Write a description of the Civil War. Provide key dates and events and describe their significance                                                                                                                                     | _回傳段落後附重要事件日期與描述的列表_                                                                           | 複雜                |
| Write a description of the Civil War in 1 paragraph. Provide 3 bullet points with key dates and their significance. Provide 3 more bullet points with key historical figures and their contributions. Return the output as a JSON file | _回傳更詳盡的文字框內容，格式為 JSON，可複製貼上到檔案並視需要驗證_                                       | 複雜且格式化        |

## 主要內容

在上述範例中，提示仍相當開放，讓大型語言模型決定其預訓練資料中哪些部分是相關的。透過 _主要內容_ 設計模式，輸入文字分為兩部分：

- 指令（動作）
- 相關內容（影響動作的部分）

以下範例中，指令是「用兩句話摘要此內容」。

- 任務描述

### 提示範本

提示範本是 _預先定義好的提示_recipe_，可儲存並按需重複使用，以推動更具一致性的用戶體驗。在最簡單的形式中，它只是像[這個來自 OpenAI 的範例](https://cookbook.openai.com/examples/gpt4-1_prompting_guide?WT.mc_id=academic-105485-koreyst)一樣，提供互動式提示元件（用戶和系統訊息）和 API 請求格式－以支持重用。

在更複雜的形式中，如[這個來自 LangChain 的範例](https://python.langchain.com/docs/concepts/prompt_templates/?WT.mc_id=academic-105485-koreyst)，它包含 _可替換的佔位符_，可用來替換來自各種來源的資料（用戶輸入、系統上下文、外部資料來源等），從而動態生成提示。這讓我們能創建一個可重複使用的提示庫，<strong>以程式化方式</strong>規模化推動一致的用戶體驗。

最後，範本的最大價值在於能創建和發布 _垂直應用領域的提示庫_－提示範本被 _優化_ 以反映特定應用場景或範例，讓回應對特定用戶群更相關且準確。[Prompts For Edu](https://github.com/microsoft/prompts-for-edu?WT.mc_id=academic-105485-koreyst) 專案就是很好的示例，策畫了一個專注於教育領域提示的庫，強調課程規劃、課程設計、學生輔導等關鍵目標。

## 支援內容

如果我們把提示結構視為由指令（任務）和目標（主要內容）組成，那麼 _次要內容_ 就像是額外提供的上下文，能夠<strong>以某種方式影響輸出</strong>。它可能包括調整參數、格式指示、主題分類等，幫助模型 _調整_ 回應，使其更符合期望或用戶目標。

例如：給定一個課程目錄，內含豐富的元資料（名稱、描述、等級、元資料標籤、講師等）涵蓋整個課程：

- 我們可以定義一個指令叫「總結 2023 年秋季課程目錄」
- 我們可以用主要內容給出一些期望輸出的範例
- 我們可以用次要內容來辨識最關注的五個「標籤」

現在模型可以依據該範例格式去總結，但當結果包含多個標籤時，會優先考慮次要內容中識別出的五個標籤。

## 提示工程最佳實務

現在我們知道了提示如何被 _建構_，可以開始思考如何 _設計_ 它們以反映最佳實務。我們可以從兩個面向思考－具備正確的 _思維模式_ 和應用正確的 _技術_。

### 提示工程思維模式

提示工程是個反覆試錯的過程，請記住三大指導要點：

1. **領域理解很重要。** 回應的準確性和相關性取決於該應用或用戶運作的 _領域_。運用你的直覺和領域專業技術來<strong>針對技巧做客製化</strong>。例如，在系統提示中定義 _領域專屬人格_，或在用戶提示中使用 _領域專屬範本_。提供反映領域特定背景的次要內容，或用 _領域提示和範例_ 導引模型朝熟悉用法前進。

2. **模型理解很重要。** 我們知道模型本質上是隨機性的，但不同模型根據其訓練資料集（預訓練知識）、所提供的功能（例如透過 API 或 SDK）和優化內容類型（如程式碼、影像、文本）存在差異。了解你使用的模型優缺點，並利用這些認知去 _優先排序任務_ 或構建 _客製化範本_ 以發揮模型能力。

3. **迭代與驗證很重要。** 模型和提示工程技巧都在快速進步。作為領域專家，你可能擁有其他特定應用的背景或判斷標準，這些未必適用於更廣泛的社群。使用提示工程工具及方法來快速「起步」提示建構，然後用你的直覺和領域專業不斷迭代與驗證結果。記錄你的見解並建立 <strong>知識庫</strong>（如提示庫），供他人未來當作新基準使用，以加速迭代。

## 最佳實務

現在來看看 [OpenAI](https://help.openai.com/en/articles/6654000-best-practices-for-prompt-engineering-with-openai-api?WT.mc_id=academic-105485-koreyst) 和 [Azure OpenAI](https://learn.microsoft.com/azure/ai-services/openai/concepts/prompt-engineering#best-practices?WT.mc_id=academic-105485-koreyst) 實務者推薦的常見最佳做法。

| 做什麼                          | 為什麼                                                                                                                                                                                                                                               |
| :-------------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 評估最新模型                    | 新一代模型可能有更好的功能與品質，但可能伴隨較高成本。評估其影響後再作遷移決策。                                                                                                              |
| 分離指令與上下文              | 檢查你的模型或服務提供者是否定義了 _分隔符_，用以更清楚區分指令、主要與次要內容。這能幫助模型更精準分配權重給詞元。                                                                           |
| 具體且清楚                    | 提供更多想要的上下文、結果、長度、格式、風格等細節。這會改善回應品質與一致性。將流程建成可重用範本。                                                                                         |
| 描述性強，使用範例            | 模型對「示範與說明」法回應可能更佳。先用 `零次學習`（給指令但不示範），再用 `少次學習` 精煉，提供數個期望範例。可用類比說明。                                                                |
| 使用提示詞引導完成            | 給模型一些引導詞或短語，作為回應的起點，推動其朝特定結果前進。                                                                                                                                |
| 重複強調                      | 有時需重複指令給模型。可在主要內容前後都放指令，或用指令加提示詞。透過迭代與驗證找出奏效做法。                                                                                              |
| 呈現順序很重要                | 你呈現給模型資訊的順序會影響輸出，包含學習範例的順序，這是因為近期偏向效應。嘗試不同選項找出最適合的。                                                                                          |
| 給模型「退路」                | 給模型一個 _備援_ 回應，以便無法完成任務時使用。這能減少模型生成錯誤或杜撰回應的機會。                                                                                                      |
|                              |                                                                                                                                                                                                                                                   |

和任何最佳實務一樣，請記住基於模型、任務和領域的不同，_你的情況可能會不同_。將這些作為起點，並不斷迭代以找出最適合你的做法。隨著新模型和工具的推出，要持續重新評估你的提示工程流程，聚焦流程擴展性和回應品質。

## 知識檢核

以下哪一個是遵循合理最佳實務的好提示？

1. 給我一張紅色汽車的圖片
2. 給我一張在懸崖旁夕陽下停著的 Volvo XC90 型號紅色汽車圖片
3. 給我一張 Volvo XC90 型號紅色汽車的圖片

答案：2，是最佳提示，因為它說明「什麼樣的車」且細節具體（不是任何車，而是特定廠牌與型號），還描述了整體場景。3 次之，也具描述性。
