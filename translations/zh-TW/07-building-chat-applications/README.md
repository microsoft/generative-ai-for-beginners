### 聊天機器人還是聊天應用？

在深入建構聊天應用之前，讓我們比較「聊天機器人」和「AI 驅動聊天應用」，它們在角色和功能上有所不同。聊天機器人的主要目的是自動化特定的對話任務，例如回覆常見問題或包裹追蹤。它通常由規則邏輯或複雜的 AI 演算法控制。相較之下，AI 驅動的聊天應用是一個更廣泛的環境，旨在促進各種形式的數位溝通，如人際間的文字、語音及視訊聊天。其核心特點是整合生成式 AI 模型，能模擬細膩且類似人類的對話，根據多樣輸入和情境提示產生回應。生成式 AI 驅動的聊天應用可以進行開放領域討論，適應不斷演變的對話內容，甚至產生創意或複雜的對話。

下表概述了兩者的主要差異與共通點，幫助我們了解它們在數位交流中的獨特角色。

| 聊天機器人                         | 生成式 AI 驅動的聊天應用                     |
| ----------------------------------- | ------------------------------------------- |
| 專注任務且基於規則                 | 具情境感知能力                             |
| 常整合於較大型系統                 | 可能同時承載一個或多個聊天機器人           |
| 功能受限於程式設定                 | 採用生成式 AI 模型                         |
| 專業與結構化互動                   | 能進行開放領域討論                         |

### 利用 SDK 與 API 內建功能

建立聊天應用時，評估現有資源是個良好的起點。利用 SDK 和 API 建構聊天應用具多種優勢。整合文件齊全的 SDK 與 API，有助於你的應用長期成功，並解決擴展與維護問題。

- <strong>加速開發流程並降低負擔</strong>：依靠內建功能取代自行昂貴打造，使你能專注於應用中更重要的其他部分，如商業邏輯。
- <strong>效能更佳</strong>：自行打造功能時，終將會思考「它如何擴展？這個應用能應付用戶突然大量湧入嗎？」良好維護的 SDK 和 API 通常內建這些疑慮的解決方案。
- <strong>維護更容易</strong>：更新與改進容易管理，大多數 API 和 SDK 僅需在新版本釋出時更新函式庫。
- <strong>使用尖端技術</strong>：利用已經微調並以龐大資料集訓練的模型，賦予應用自然語言能力。

訪問 SDK 或 API 功能通常涉及取得使用授權，常透過獨特密鑰或認證令牌進行。我們將使用 OpenAI Python 函式庫來示範其運作。你也可以在本課程的 [OpenAI 筆記本](./python/oai-assignment.ipynb?WT.mc_id=academic-105485-koreyst) 或 [Azure OpenAI 服務筆記本](./python/aoai-assignment.ipynb?WT.mc_id=academic-105485-koreys) 中自行嘗試。

```python
import os
from openai import OpenAI

API_KEY = os.getenv("OPENAI_API_KEY","")

client = OpenAI(
    api_key=API_KEY
    )

response = client.responses.create(model="gpt-4o-mini", input="Suggest two titles for an instructional lesson on chat applications for generative AI.", store=False)
print(response.output_text)
```

上例使用 GPT-4o mini 模型與 Responses API 完成提示，但請注意 API 金鑰必須預先設定，否則會出錯。

## 使用者體驗 (UX)

一般 UX 原則適用於聊天應用，但因涉及機器學習元件，還有以下額外重要考量。

- <strong>處理歧義的機制</strong>：生成式 AI 模型偶爾會產生模糊答案。加入讓用戶可要求澄清的功能，能在遇到此情況時提供幫助。
- <strong>情境保留</strong>：先進生成式 AI 模型能記憶對話中的上下文，對使用者體驗具重要助益。讓使用者能控制與管理情境提升體驗，但也帶來保留敏感資訊的風險。考慮資料保存時間，例如引入保存政策，可以平衡情境需求與隱私。
- <strong>個人化</strong>：AI 模型具學習與適應能力，提供個別化體驗。透過使用者設定等功能量身打造，不僅讓用戶感到被理解，還幫助他們更快找到答案，創造更有效且滿意的互動。

個人化的一個例子是 OpenAI ChatGPT 中的「自訂指令」功能。它允許你提供關於自己的資訊，成為輸入提示的重要上下文。以下是自訂指令的範例。

### 無障礙

不論使用者有視覺、聽覺、動作或認知障礙，設計良好的聊天應用都應對所有人可用。以下列表細分了增進各種使用者障礙無障礙性的特定功能。

- <strong>視覺障礙功能</strong>：高對比主題與可調整文字大小，支援螢幕朗讀器。
- <strong>聽覺障礙功能</strong>：文字轉語音與語音轉文字功能，音訊通知的視覺提示。
- <strong>動作障礙功能</strong>：鍵盤導航支持，語音指令。
- <strong>認知障礙功能</strong>：簡化語言選項。

## 針對特定領域語言模型的客製化與微調

想像一個聊天應用理解你公司的行話，且能預測使用者常見問題。這裡有兩種值得一提的方法：

- **利用 DSL 模型**。DSL 是特定領域語言（domain specific language）的簡稱。你可以利用所謂用特定領域資料訓練的 DSL 模型理解其概念與場景。
- <strong>套用微調</strong>。微調是使用特定資料進一步訓練你的模型的過程。

## 客製化：使用 DSL

利用特定領域語言模型 (DSL 模型) 可以透過提供專門且具上下文相關的互動，提升用戶參與度。該模型訓練或微調於特定領域、產業或主題，能理解並生成相關文本。使用 DSL 模型有多種選擇，從自行從零開始訓練，到透過 SDK 和 API 使用預訓練模型。另一方案是微調，將既有預訓練模型調整至特定領域。

## 客製化：進行微調

當預訓練模型在特定領域或任務上表現不足時，通常會考慮微調。

例如，醫療查詢複雜且需大量上下文。醫生診斷患者時會考量多種因素，如生活習慣或既有病史，甚至依賴最新醫學期刊驗證診斷。在此類細微情境下，一般用途的 AI 聊天應用無法成為可靠資訊來源。

### 情境：醫療應用

考慮一個設計來協助醫療人員快速查詢治療指南、藥物交互作用或最新研究成果的聊天應用。

一般模型或許足以回答基本醫療問題或提供一般建議，但可能在以下情況力不從心：

- <strong>高度專業或複雜案例</strong>。例如，神經學家可能會問應用：「目前兒科藥物抗性癲癇管理的最佳實務是什麼？」
- <strong>缺乏最新進展</strong>。一般模型難以提供涵蓋神經學和藥理學最新發展的答案。

在這些情況下，使用專門醫療資料微調模型，可以顯著提升其處理複雜醫療詢問的精確度與可靠性。此類微調需依賴龐大且具相關性的資料集，反映領域特定的挑戰與問題。

## 高品質 AI 驅動聊天體驗考量

本節列出「高品質」聊天應用的標準，包括可採集的行為指標和謹慎使用 AI 技術的框架。

### 關鍵指標

為保持應用的高效能，必須追蹤關鍵指標與相關考量。這些量度不僅確保應用功能正常，也評估 AI 模型與使用者體驗的品質。以下列表涵蓋基本、AI 及使用者體驗的指標。

| 指標                         | 定義                                                                                                                | 聊天開發者的考量                                                       |
| ---------------------------- | ------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------- |
| **運作時間 (Uptime)**        | 測量應用可用且運作的時間。                                                                                          | 如何將停機時間降至最低？                                               |
| <strong>回應時間</strong>                 | 應用回覆用戶查詢所需的時間。                                                                                        | 如何優化查詢處理以縮短回應時間？                                      |
| **精確率 (Precision)**       | 真陽性預測占所有陽性預測的比例。                                                                                    | 如何驗證你的模型精確率？                                               |
| **召回率 (Recall/靈敏度)**  | 真陽性預測占實際陽性數量的比例。                                                                                    | 如何測量並提升召回率？                                                 |
| **F1 分數**                 | 精確率與召回率的調和平均數，平衡兩者取捨。                                                                          | 你的目標 F1 分數為何？如何平衡精確率與召回率？                        |
| **困惑度 (Perplexity)**      | 衡量模型預測機率分布與資料實際分布的一致性。                                                                        | 如何將困惑度降至最低？                                                 |
| <strong>用戶滿意度指標</strong>           | 測量用戶對應用的感知，通常通過調查收集。                                                                            | 多常收集用戶反饋？如何根據反饋調整？                                  |
| <strong>錯誤率</strong>                   | 模型在理解或輸出上出錯的頻率。                                                                                        | 你有哪些策略來降低錯誤率？                                             |
| <strong>重訓週期</strong>                 | 模型更新以納入新資料與洞見的頻率。                                                                                    | 多常重訓模型？觸發重訓的條件是什麼？                                  |

| <strong>異常檢測</strong>               | 用於識別不符合預期行為的異常模式的工具和技術。                                       | 你將如何應對異常？                                                         |
