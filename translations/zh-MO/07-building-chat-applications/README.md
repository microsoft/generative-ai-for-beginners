### 聊天機器人還是聊天應用？

在深入建立聊天應用前，讓我們對比「聊天機器人」與「AI 驅動聊天應用」，它們扮演不同角色與功能。聊天機器人的主要目標是自動化特定對話任務，如回答常見問題或追蹤包裹，通常依規則邏輯或複雜 AI 演算法控制。相對地，AI 驅動的聊天應用是一個更廣泛的環境，設計用於促進多種數位溝通形式，如人與人之間的文字、語音及視訊聊天。其特徵在於整合生成式 AI 模型，能模擬細膩且類似人類的對話，根據多樣輸入及上下文線索生成回應。生成式 AI 驅動的聊天應用可展開開放領域討論、適應不斷演變的對話語境，甚至產生創意或複雜對話。

下表列出關鍵差異與相似點，幫助理解它們在數位溝通中的獨特角色。

| 聊天機器人                             | 生成式 AI 驅動聊天應用                          |
| ------------------------------------- | ---------------------------------------------- |
| 任務導向且基於規則                     | 上下文感知                                      |
| 常整合於較大系統                       | 可承載一個或多個聊天機器人                      |
| 限於程式化功能                         | 包含生成式 AI 模型                              |
| 專門且結構化的互動                     | 能進行開放領域討論                              |

### 利用 SDK 與 API 的預建功能

建立聊天應用的好第一步是評估市場已有資源。透過整合文件完善的 SDK 與 API，策略性地為應用建立良好基礎，涵蓋擴展性與維護性問題，這是有效策略。

- <strong>加速開發流程並降低負擔</strong>：依賴預建功能省去自己建立昂貴過程，讓你能專注於應用程式中你認為更重要的業務邏輯等其他部分。
- <strong>性能更佳</strong>：從零開始構建功能時，難免會問自己「如何擴展？此應用能應付使用者激增嗎？」良好維護的 SDK 和 API 通常已內建解決方案。
- <strong>維護更輕鬆</strong>：更新和改進更容易管理，大部分 API 和 SDK 只需更新函式庫至新版。
- <strong>取得尖端技術</strong>：利用經過細緻調教並基於龐大數據訓練的模型，賦予應用自然語言能力。

存取 SDK 或 API 功能通常需取得使用許可，通常透過唯一金鑰或驗證令牌。我們將使用 OpenAI Python 库展示這是如何操作。你也可以在本課程的[OpenAI 筆記本](./python/oai-assignment.ipynb?WT.mc_id=academic-105485-koreyst)或[Azure OpenAI 服務筆記本](./python/aoai-assignment.ipynb?WT.mc_id=academic-105485-koreys)自行嘗試。

```python
import os
from openai import OpenAI

API_KEY = os.getenv("OPENAI_API_KEY","")

client = OpenAI(
    api_key=API_KEY
    )

response = client.responses.create(model="gpt-5-mini", input="Suggest two titles for an instructional lesson on chat applications for generative AI.", store=False)
print(response.output_text)
```

上例使用 GPT-5 mini 模型搭配 Responses API 完成提示，但請注意，必須先設定 API 金鑰，否則會出錯。

## 使用者體驗 (UX)

一般用户体验原则适用于聊天应用程序，但因涉及機器學習元件，以下是一些特別重要的注意事項。

- <strong>解決歧義的機制</strong>：生成式 AI 模型偶爾會產生含糊回答，若有讓用戶可要求澄清的功能，會非常有幫助。
- <strong>上下文保留</strong>：先進生成式 AI 模型能記憶對話上下文，對使用者體驗非常重要。讓用戶管理上下文，雖提升體驗，但也有保留敏感資訊風險。可考慮設置資料保留政策，以求在上下文需求與隱私間取得平衡。
- <strong>個人化</strong>：AI 模型具學習與適應能力，為用戶提供個別化體驗。透過用戶檔案等功能客製化體驗，不僅讓用戶感覺被理解，也提升尋找特定答案的效率與滿意度。

### 無障礙設計

無論用戶有視覺、聽覺、運動或認知障礙，設計良好的聊天應用應易於所有用戶使用。下列列表針對不同障礙提供提升無障礙性的具體功能。

- <strong>視覺障礙功能</strong>：高對比主題與可調大小文字，屏幕閱讀器相容性。
- <strong>聽覺障礙功能</strong>：文字轉語音與語音轉文字功能，音訊通知的視覺提示。
- <strong>運動障礙功能</strong>：鍵盤導航支援，語音指令。
- <strong>認知障礙功能</strong>：簡化語言選項。

## 領域專用語言模型的客製化與微調

想像一個能理解你公司行話並預測用戶常見查詢的聊天應用。有幾種方法值得一提：

- **利用 DSL 模型**。DSL 指領域專用語言。你可以利用訓練於特定領域的 DSL 模型來理解其概念與情境。
- <strong>應用微調</strong>。微調是透過特定資料繼續訓練模型的過程。

## 客製化：使用 DSL

利用領域專用語言模型（DSL 模型）能提升用戶黏著度，提供特殊且語境相關的互動。這種模型針對特定領域、產業或主題進行訓練或微調。使用 DSL 模型的選項從自行從頭訓練，到透過 SDK 和 API 使用現成的模型皆可。另一種選擇是微調，也就是拿現有的預訓練模型，調整用於特定領域。

## 客製化：應用微調

當預訓練模型在特定領域或任務上不理想時，常考慮進行微調。

例如，醫療問診複雜且需大量上下文。醫療專業人員診斷患者時會考慮生活習慣、既有病況，甚至需參考最新醫學期刊來確認診斷。在這些細緻情境中，通用 AI 聊天應用無法成為可靠來源。

### 情境：醫療應用

想像一個協助醫療人員快速查詢治療指南、藥物交互作用或最新研究的聊天應用。

通用模型或許足以回答基本醫學問題或給出一般建議，但會遇到以下挑戰：

- <strong>高度專門或複雜案例</strong>。譬如一位神經科醫師問應用程式：「目前兒童藥物難治性癲癇的最佳管理實務是什麼？」
- <strong>缺乏最新進展資訊</strong>。通用模型可能難以納入神經科學與藥理學最新進展來提供及時解答。

針對這些情況，以專門醫療資料集進行微調能大幅提升模型應對複雜醫療問題的精準與可靠性，但這需要取得龐大且相關的數據集，代表需解決的特定領域挑戰與問題。

## 高品質 AI 聊天體驗的考量

本節概述「高品質」聊天應用的標準，包含可捕捉可操作指標及遵守負責任運用 AI 技術的架構。

### 主要指標

為維持應用高品質表現，必須追蹤關鍵指標及考量。這些衡量不僅確保應用功能正常，也評估 AI 模型與使用者體驗品質。以下列出基礎指標、AI 和用戶體驗度量供參考。

| 指標                         | 定義                                                                                                         | 聊天開發者考量                                                              |
| ---------------------------- | ------------------------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------- |
| **正常運行時間 (Uptime)**    | 衡量應用可操作及用戶可接取的時間長度。                                                                     | 如何將停機時間降至最低？                                                     |
| **回應時間 (Response Time)** | 應用回覆用戶查詢所需時間。                                                                                   | 如何優化查詢處理以提升回應速度？                                           |
| **精準度 (Precision)**       | 真陽性預測與所有陽性預測的比例。                                                                             | 如何驗證模型精準度？                                                        |
| **召回率 (Recall)**          | 真陽性預測與所有實際陽性的比例。                                                                             | 如何衡量及提升召回率？                                                      |
| **F1 分數**                  | 精準度與召回率的調和平均數，平衡兩者間的取捨。                                                               | 目標 F1 分數是多少？如何在精準度和召回率間找到平衡？                      |
| **困惑度 (Perplexity)**     | 衡量模型預測的機率分布與資料的實際分布吻合程度。                                                             | 如何降低困惑度？                                                            |
| <strong>用戶滿意度指標</strong>           | 衡量用戶對應用的感受，經常透過調查捕捉。                                                                     | 多常收集用戶回饋？如何根據回饋調整？                                       |
| **錯誤率 (Error Rate)**      | 模型理解或輸出錯誤的比率。                                                                                     | 有何策略降低錯誤率？                                                        |
| **重訓周期 (Retraining Cycles)** | 模型更新以納入新資料與見解的頻率。                                                                             | 多常重訓模型？有哪些觸發重訓的因素？                                       |

| <strong>異常檢測</strong>               | 用於識別不符合預期行為的異常模式的工具和技術。                                                    | 你將如何應對異常？                                                       |
