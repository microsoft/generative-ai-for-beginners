# 提示工程基礎

[![提示工程基礎](../../../translated_images/zh-MO/04-lesson-banner.a2c90deba7fedacd.webp)](https://youtu.be/GElCu2kUlRs?si=qrXsBvXnCW12epb8)

## 介紹
本模組涵蓋在生成式 AI 模型中創建有效提示的基本概念和技術。您向大型語言模型（LLM）撰寫提示的方式同樣重要。精心設計的提示能獲得更高質量的回應。但到底什麼是 _提示_ 和 _提示工程_？我如何改進傳送給 LLM 的提示 _輸入_？這些是我們將在本章及下一章試圖回答的問題。

_生成式 AI_ 能根據用戶請求創造新內容（例如，文字、圖像、音訊、代碼等）。它透過使用像 OpenAI 的 GPT（「生成式預訓練轉換器」）系列這類大型語言模型，利用自然語言和代碼進行訓練來實現。

現在用戶可以透過熟悉的聊天模式與這些模型互動，無需技術專業知識或培訓。這些模型是 _基於提示_ 的——用戶發送文本輸入（提示），並獲得 AI 回應（完成）。用戶可以透過多輪對話反覆「與 AI 聊天」，不斷調整提示，直到回應符合預期。

「提示」現已成為生成式 AI 應用的主要 _程式設計介面_，告訴模型該做什麼並影響回應的品質。「提示工程」是一個快速發展的研究領域，專注於 _設計和優化_ 提示，以在大規模下提供一致且高品質的回應。

## 關鍵術語

## 什麼是提示工程？

本課從定義 <strong>提示工程</strong> 開始，將其視為一個過程：_設計和優化_ 文字輸入（提示），以為特定應用目標和模型產生一致且高品質的回應（完成）。我們可將其視為兩個步驟：

- 為特定模型和目標 _設計_ 初始提示
- 反覆_精煉_提示以提升回應品質

這必然是一個反覆試錯的過程，需要用戶有直覺和努力以達最佳結果。那它為何重要呢？為了解答，首先需理解三個概念：

- _分詞_＝模型如何「看見」提示
- _基礎 LLM_＝基礎模型如何「處理」提示
- _指令微調 LLM_＝模型如何「看見任務」

### 分詞

LLM 將提示視為一串 _tokens_，不同模型（或同模型的不同版本）會以不同方式分詞同一提示。因 LLM 是以 tokens 為訓練單位（非原始文本），提示分詞方式直接影響生成回應的品質。

想了解分詞如何運作，試試像下面展示的 [OpenAI Tokenizer](https://platform.openai.com/tokenizer?WT.mc_id=academic-105485-koreyst) 工具。將提示貼上，看看它如何被轉成 tokens，注意空白字元和標點符號的處理方式。此例使用舊版 LLM（GPT-3）示範，用新模型試結果可能不同。

![分詞示例](../../../translated_images/zh-MO/04-tokenizer-example.e71f0a0f70356c5c.webp)

### 概念：基礎模型

提示被分詞後，["基礎 LLM"](https://blog.gopenai.com/an-introduction-to-base-and-instruction-tuned-large-language-models-8de102c785a6?WT.mc_id=academic-105485-koreyst)（或基礎模型）的主要功能是預測序列中的下一個 token。由於 LLM 經大量文字資料訓練，對 tokens 之間的統計關係有良好認識，可有相當信心地完成預測。它們並不真正理解提示或 token 的 _意義_，僅看到可用下一步予測「補全」的模式。它會持續作出預測直到用戶終止或符合預設條件。

想看看基於提示的補全怎麼運作？將上述提示貼到 [Microsoft Foundry playground](https://ai.azure.com?WT.mc_id=academic-105485-koreyst) 並使用預設設定。系統會將提示視為詢問資訊的請求——因此您應會看到滿足該上下文的補全結果。

但若用戶想看到符合特定條件或任務目標的回應呢？這就是 _指令微調_ LLM 發揮作用的地方。

![基礎 LLM 對話補全](../../../translated_images/zh-MO/04-playground-chat-base.65b76fcfde0caa67.webp)

### 概念：指令微調 LLM

[指令微調 LLM](https://blog.gopenai.com/an-introduction-to-base-and-instruction-tuned-large-language-models-8de102c785a6?WT.mc_id=academic-105485-koreyst) 是在基礎模型之上，利用包含明確指令的輸入/輸出範例（如多輪「訊息」）進行微調，AI 回應會嘗試遵循該指令。

這使用像是人類回饋強化學習（RLHF）等技術，訓練模型遵循指令並從回饋中學習，產生更適合實際應用且與用戶目標更相關的回應。

讓我們試試——再次輸入上述提示，但現在將 _系統訊息_ 改成以下指令作為上下文：

> _將您提供的內容總結給二年級學生。結果保持一段文字，附帶3-5個要點。_

看結果如何被調整以符合期望目標和格式？教育工作者現在能直接將此回應用於課堂簡報中。

![指令微調 LLM 對話補全](../../../translated_images/zh-MO/04-playground-chat-instructions.b30bbfbdf92f2d05.webp)

## 為什麼需要提示工程？

現在我們知道提示如何被 LLM 處理，讓我們講講 _為何_ 需要提示工程。原因在於目前 LLM 存在多種挑戰，使得 _可靠且一致的補全_ 在未經過提示建構與優化的努力下較難達成。例如：

1. **模型回應具隨機性。**_同一提示_在不同模型或版本可能產生不同回應，甚至同一模型不同時間亦然。_提示工程技術可幫助降低這些差異，提供更好保護_。

1. <strong>模型可能造假回應。</strong>模型透過 _龐大但有限_ 的資料訓練，缺乏訓練範圍外概念知識。因此，回應可能不準確、虛構甚至與已知事實矛盾。_提示工程可幫助用戶識別並減輕造假，例如向 AI 要求引用或推理_。

1. <strong>模型能力不一。</strong>更新模型或新世代模型能力更強，但也帶來成本、複雜度及特殊特質的權衡。_提示工程可助我們建立最佳實踐與工作流程，抽象差異並無縫適應模型需求_。

讓我們在 OpenAI 或 Azure OpenAI Playground 實際看看運作：

- 用同一提示搭配不同 LLM 部署（例如 OpenAI、Azure OpenAI、Hugging Face）——有看到差異嗎？
- 重複用同一提示搭配 _同一_ LLM 部署（例如 Azure OpenAI playground）——這些差異如何變化？

### 造假示例

本課程中，我們用 **「造假」** 來描述 LLM 有時因訓練限制或其他制約，產生事實錯誤資訊的現象。您或許曾在文章或研究中聽聞稱為 _「幻覺」_。然而，我們強烈建議使用 _「造假」_ 一詞，以免不慎將此行為擬人化，給予機器人類特質。此詞也符合 [負責任 AI 指導原則](https://www.microsoft.com/ai/responsible-ai?WT.mc_id=academic-105485-koreyst)的用詞規範，避免部分語境下可能具冒犯性或不包容的詞彙。

想了解造假現象如何發生？想像一個提示指示 AI 產生不存在主題的內容（確保不在訓練資料內）。例如，我嘗試以下提示：

> **提示：** 產生 2076 火星戰爭的教案。

網路搜尋顯示有關火星戰爭的虛構作品（如電視劇或書籍）——但並無 2076 年的相關描述。常識亦提醒我們，2076 年尚未來到，因此無法對應真實事件。

提示工程技術如 _元提示_ 和 _溫度配置_ 可能在某種程度上減少模型虛構。新的提示工程 _架構_ 也將新的工具和技術毫無縫隙地融入提示流程，以緩解或減少其中一些影響。

## 提示構造

我們已了解提示工程的重要性— 現在讓我們了解提示是如何 _構造_ 的，好讓我們評估不同技術以設計更有效的提示。

### 基本提示

先從基本提示開始：一個純文字輸入送到模型，沒有其他上下文。這裡有一個例子—當我們將美國國歌的前幾個字送到 OpenAI [Completion API](https://platform.openai.com/docs/api-reference/completions?WT.mc_id=academic-105485-koreyst)，它會立刻 _補全_ 接下來的幾行，展示基本的預測行為。

| 提示（輸入）          | 補全（輸出）                                                                                                                          |
| :----------------- | :----------------------------------------------------------------------------------------------------------------------------------- |
| Oh say can you see | 聽起來你正在開始念「星條旗旗歌」的歌詞，這是美國的國歌。完整歌詞是...                                                        |

### 複雜提示

現在讓我們為基本提示加入上下文和指示。[Chat Completion API](https://learn.microsoft.com/azure/ai-services/openai/how-to/chatgpt?WT.mc_id=academic-105485-koreyst) 允許我們將複雜提示構成為多條 _訊息_，包括：

- 反映 _用戶_ 輸入和 _助理_ 回應的輸入/輸出對。
- 設定助理行為或人格的系統訊息。

請求的格式如下，_標記化_ 有效捕捉上下文與對話中的相關資訊。改變系統上下文對補全質量的影響和提供的用戶輸入一樣重要。

```python
response = client.responses.create(
    model="gpt-4o-mini",
    input=[
        {"role": "system", "content": "You are a helpful assistant."},
        {"role": "user", "content": "Who won the world series in 2020?"},
        {"role": "assistant", "content": "The Los Angeles Dodgers won the World Series in 2020."},
        {"role": "user", "content": "Where was it played?"}
    ]
)
```

### 指示提示

在上述例子中，用戶提示是簡單的文字查詢，可解釋為資訊請求。使用 _指示_ 提示，我們可以用該文字更詳細地指定任務，為 AI 提供更好指導。舉例：

| 提示（輸入）                                                                                                                                                                                                                         | 補全（輸出）                                                                                                        | 指示類型            |
| :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :------------------------------------------------------------------------------------------------------------------------- | :------------------ |
| Write a description of the Civil War                                                                                                                                                                                                   | _回傳簡單段落_                                                                                                      | 簡單               |
| Write a description of the Civil War. Provide key dates and events and describe their significance                                                                                                                                     | _回傳段落加上關鍵事件日期與描述清單_                                                                             | 複雜               |
| Write a description of the Civil War in 1 paragraph. Provide 3 bullet points with key dates and their significance. Provide 3 more bullet points with key historical figures and their contributions. Return the output as a JSON file | _回傳更多詳盡內容於文字框，以 JSON 格式，可複製貼上存為檔案並按需驗證_                                        | 複雜。格式化。    |

## 主要內容

在上述例子中，提示仍相當開放，讓 LLM 決定其預訓練資料中哪些部份相關。使用 _主要內容_ 設計模式，輸入文字分為兩部分：

- 一個指示（動作）
- 相關內容（影響動作）

這是個例子，其中指示是「用兩句話總結這個」。

| 提示（輸入）                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      | 補全（輸出）                                                                                                                                                                                                                                                                               |
| :-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Jupiter is the fifth planet from the Sun and the largest in the Solar System. It is a gas giant with a mass one-thousandth that of the Sun, but two-and-a-half times that of all the other planets in the Solar System combined. ### 提示範本

提示範本是為了推動更具一致性的用戶體驗而預先定義的提示「配方」，可以存儲並按需重用。最基本的形式，它只是一組提示範例，就像[OpenAI這個範例](https://cookbook.openai.com/examples/gpt4-1_prompting_guide?WT.mc_id=academic-105485-koreyst)，提供互動式提示組件（用戶和系統消息）和基於 API 的請求格式——以支持重用。

複雜一些的形式，例如[LangChain這個範例](https://python.langchain.com/docs/concepts/prompt_templates/?WT.mc_id=academic-105485-koreyst)，包含可由各種來源（用戶輸入、系統上下文、外部數據源等）替換的_佔位符_，以動態生成提示。這使我們能夠建立一個可程式化地推動一致用戶體驗的可重用提示庫。

範本的真正價值在於能夠為特定垂直應用領域創建並發布_提示庫_——這些提示範本現已_優化_以反映特定應用上下文或示例，使回應更貼近目標用戶群的需求和準確性。[Prompts For Edu](https://github.com/microsoft/prompts-for-edu?WT.mc_id=academic-105485-koreyst) 倉庫就是這種方法的絕佳範例，策劃出一套專注於教育領域的重要目標（例如課程規劃、課綱設計、學生輔導等）的提示庫。

## 支持內容

如果我們將提示建構視為有一個指令（任務）和一個目標（主要內容），那麼_次要內容_就像是我們提供的額外上下文，用以<strong>在某種程度上影響輸出</strong>。它可能包括調節參數、格式說明、主題分類等，有助模型將回答_量身定制_成符合期望的用戶目標或期望。

例如：給定一份包含豐富元數據（名稱、描述、等級、元數據標籤、講師等）的課程目錄：

- 我們可以定義一個指令為「彙整2023年秋季課程目錄」
- 我們可以利用主要內容提供一些預期輸出的示例
- 我們可以使用次要內容來識別最重要的5個「標籤」

現在，模型可以以幾個範例示範的格式提供摘要——但當一條結果包含多重標籤時，優先顯示次要內容指定的5個標籤。

## 提示最佳實踐

現在我們了解提示可以如何_構建_，可以開始思考如何_設計_提示以體現最佳實踐。我們可以從兩個角度思考——擁有正確的_心態_和應用合適的_技術_。

### 提示工程心態

提示工程是一個反覆試驗的過程，要注意以下三個大方向：

1. **領域理解很重要。** 回應的準確度和相關性反映應用或用戶所處的_領域_。運用直覺與領域專業知識以<strong>進一步自訂技巧</strong>。例如，在系統提示中定義_領域特定的人格_，或在用戶提示中使用_領域特定範本_。提供反映特定領域上下文的次要內容，或運用_領域特定的提示與示例_，引導模型朝熟悉的使用模式。

2. **模型理解很重要。** 我們知道模型本質上是隨機的。但模型實現也可能因其使用的訓練數據集（預訓練知識）、提供的功能（如透過 API 或 SDK）及其優化的內容類型（如程式碼、圖像或文本）而異。了解你所使用模型的優勢與限制，並據此_優先排序任務_或建構_適合該模型的自訂範本_。

3. **迭代與驗證很重要。** 模型和提示工程技術都在迅速演進。身為領域專家，你可能擁有其他特定於_你的_應用的上下文或標準，這些未必適用於更廣泛的社群。使用提示工程工具和技術「快速啟動」提示建構，然後以自己的直覺和領域專業反覆驗證結果。記錄你的見解並建立<strong>知識庫</strong>（例如提示庫），供他人作為新基準以加快未來迭代。
