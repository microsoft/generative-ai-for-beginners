# ការសម្រួលលម្អិតលើ LLM របស់អ្នក

ការប្រើប្រាស់ម៉ូឌែលភាសាធំព្រមទាំងសាងសង់កម្មវិធី AI បង្កើតថ្មីមានជាមួយនឹងបញ្ហាថ្មីៗ។ បញ្ហាសំខាន់ៗគឺការធានាគុណភាពនៃចម្លើយ (ភាពត្រឹមត្រូវ និងពាក់ព័ន្ធ) ក្នុងអំពីខ្លឹមសារដែលម៉ូឌែលបង្កើតឡើងសម្រាប់សំណើរបស់អ្នកប្រើ។ ក្នុងមេរៀនមុនៗ យើងបានពិភាក្សាអំពីបច្ចេកទេសដូចជា វិជ្ជាជីវៈប្រើប្រាស់ស្នើសុំ (prompt engineering) និង ការបង្កើតបន្ថែមដោយយកទិន្នន័យមកបន្ថែ (retrieval-augmented generation) ដែលព្យាយាមដោះស្រាយបញ្ហានេះដោយ _កែប្រែបញ្ចូលស្នើសុំ_ ទៅម៉ូឌែលដែលមានរួច។

ក្នុងមេរៀនថ្ងៃនេះ យើងនឹងពិភាក្សាអំពីបច្ចេកទេសទីបី ការសម្រួលលម្អិត (**fine-tuning**), ដែលព្យាយាមដោះស្រាយបញ្ហានេះដោយ _បណ្តុះបណ្តាលម៉ូឌែលឡើងវិញ_ ជាមួយទិន្នន័យបន្ថែម។ តោះចូលទៅកាន់ព័ត៌មានលម្អិត។

## គោលបំណងរៀន

មេរៀននេះណែនាំអំពីគំនិតនៃការសម្រួលលម្អិតសម្រាប់ម៉ូឌែលភាសាមុនបានបណ្តុះបណ្តាលរួច, ស្វែងយល់អំពីអត្ថប្រយោជន៍ និងបញ្ហារបស់វិធីនេះ ហើយផ្ដល់ការណែនាំពីពេលវេលា និងវិធីដែលត្រូវប្រើការសម្រួលលម្អិតដើម្បីធ្វើឱ្យម៉ូឌែល AI បង្កើតរបស់អ្នកមានប្រសិទ្ធភាពកាន់តែប្រសើរ។

នៅចុងមេរៀននេះ អ្នកគួរតែអាចឆ្លើយបានចំលើយសំណួរខាងក្រោម៖

- ការសម្រួលលម្អិតសម្រាប់ម៉ូឌែលភាសាជាអ្វី?
- ពេលណា និងហេតុអ្វីបានជា ការសម្រួលលម្អិតមានប្រយោជន៍?
- តើខ្ញុំអាចសម្រួលលម្អិតម៉ូឌែលដែលបានបណ្តុះបណ្តាលរួចបានយ៉ាងដូចម្តេច?
- កំណត់ខ្ពស់នៃការសម្រួលលម្អិតមានអ្វីខ្លះ?

## មគ្គុទេសក៍ជារូបភាព

ចង់មើលទេសភាពជាធំមុនពេលចូលលម្អិតទេ? សូមពិនិត្យមើលមគ្គុទេសក៍ជារូបភាពដែលបង្ហាញពីដំណើរការរៀនសម្រាប់មេរៀននេះ - ចាប់ពីការស្វែងយល់គំនិតមូលដ្ឋាន និងមូលហេតុចម្បងនៃការសម្រួលលម្អិត ដល់ការយល់ដឹងអំពីដំណើរការ និងការអនុវត្តិល្អបំផុតសម្រាប់ការបញ្ចុះបញ្ចូលការសម្រួលលម្អិត។ នេះជាប្រធានបទគួរឱ្យចាប់អារម្មណ៍សម្រាប់ការស្រាវជ្រាវ ហើយកុំភ្លេចស្ដាប់ពីទំព័រ [ធនធាន](./RESOURCES.md?WT.mc_id=academic-105485-koreyst) សម្រាប់តំណខ្សែបន្ថែមដើម្បីគាំទ្រដល់ដំណើរការរៀនដោយខ្លួនឯងរបស់អ្នក!

## ការសម្រួលលម្អិតសម្រាប់ម៉ូឌែលភាសាជាអ្វី?

តាមនិយមន័យ ម៉ូឌែលភាសាធំព្រមគឺ _បានបណ្តុះបណ្តាលរួច_ លើបរិមាណអក្សរធំៗ ដែលប្រភពមកពីប្រភពផ្សេងៗជាច្រើន រួមទាំងអ៊ីនធឺណែត។ ដូចដែលយើងបានរៀននៅមេរៀនមុនៗ គួរត្រូវប្រើបច្ចេកទេសដូចជា _វិជ្ជាជីវៈប្រើប្រាស់ស្នើសុំ_ និង _ការបង្កើតបន្ថែមដោយយកទិន្នន័យមកបន្ថែ_ ដើម្បីធ្វើអោយគុណភាពនៃចម្លើយម៉ូឌែលទៅសំណួររបស់អ្នកប្រើ ("សំនួរ") ប្រសើរឡើង។

បច្ចេកទេស prompt-engineering ពេញនិយមមួយគឺការផ្ដល់ការណែនាំលំអិតទៅម៉ូឌែលអំពីអ្វីដែលគួរតែជាចម្លើយ ប្រាកដតាម _ការណែនាំ_ (គន្លងបញ្ជាក់ច្បាស់) ឬ _ផ្ដល់ឧទាហរណ៍ពីច្រើន_ (គន្លងបញ្ជាក់រលូន)។ នេះហៅថា _ការសិក្សាជាមួយគំរូយ៉ាងតិច_ ប៉ុន្តែវាមានកំណត់ពីរគឺ៖

- កំណត់របស់ token ម៉ូឌែលអាចដាក់កំណត់ចំនួនឧទាហរណ៍ដែលអាចផ្ដល់ និងមានប្រសិទ្ធភាពកម្រិតកំណត់។
- តម្លៃ token ម៉ូឌែលអាចធ្វើឱ្យការបន្ថែមឧទាហរណ៍លើសំនួរនីមួយៗមានថ្លៃថ្នូរ ហើយកំណត់បំណុលភាព។

ការសម្រួលលម្អិតគឺជាបញ្ហាធម្មតានៅក្នុងប្រព័ន្ធរៀនម៉ាស៊ីន ដែលយើងយកម៉ូឌែលបានបណ្តុះបណ្តាលរួច ហើយបណ្តុះបណ្តាលឡើងវិញជាមួយទិន្នន័យថ្មី ដើម្បីធ្វើឱ្យមានប្រសិទ្ធភាពកាន់តែល្អសម្រាប់ភារកិច្ចដែលបានកំណត់។ ក្នុងបរិបទរបស់ម៉ូឌែលភាសា អាចសម្រួលលម្អិតម៉ូឌែលដែលបានបណ្តុះបណ្តាលរួចដោយ _ជ្រើសយកឧទាហរណ៍ដែលបានជ្រើសរើសយ៉ាងល្អសម្រាប់ភារកិច្ច ឬដែនកម្មវិធីបញ្ជាក់មួយ_ ដើម្បីបង្កើតម៉ូឌែលផ្ទាល់ខ្លួនដែលអាចមានភាពត្រឹមត្រូវ និងពាក់ព័ន្ធច្រើនសម្រាប់ភារកិច្ចឬដែននោះ។ អត្ថប្រយោជន៍ពីលើនៃការសម្រួលលម្អិតគឺវាក៏អាចធ្វើយកចំនួនឧទាហរណ៍ដែលត្រូវការសម្រាប់ការសិក្សាជាមួយគំរូយ៉ាងតិចបានបន្ថយ—បន្ថយការប្រើ token និងចំណាយដែលទាក់ទង។

## ពេលណា និងហេតុអ្វីបានជា យើងគួរតែសម្រួលលម្អិតម៉ូឌែល?

ក្នុង _បរិបទនេះ_, ពេលដែលយើងនិយាយអំពីការសម្រួលលម្អិត យើងកំពុងនិយាយពី **fine-tuning** ដែលគ្រប់គ្រងដោយ **ម៉ាស៊ីនពិនិត្យ** ដែលការបណ្តុះឡើងវិញត្រូវធ្វើដោយ **បន្ថែមទិន្នន័យថ្មី** ដែលមិនមែនជាផ្នែកមួយនៃទិន្នន័យបណ្តុះបណ្តាលដើមទេ។ វាត្រូវខុសពីវិធីសាស្រ្ត fine-tuning លើបរិបទគ្មានការត្រួតពិនិត្យ ដែលម៉ូឌែលត្រូវបានបណ្តុះឡើងវិញលើទិន្នន័យដើម ប៉ុន្តែជាមួយប៉ារ៉ាម៉ែត្រ hyperparameters ផ្សេង។

អ្វីដែលគួរចាប់អារម្មណ៍គឺថា fine-tuning គឺជាបច្ចេកទេសខ្ពស់មួយដែលតម្រូវឲ្យមានជំនាញខ្ពស់ខ្លះដើម្បីទទួលបានលទ្ធផលដែលចង់បាន។ ប្រសិនបើធ្វើខុស វាអាចមិនផ្តល់នូវការកែលម្អដែលរំពឹងទុកទេ និងអាចបណ្តាលឲ្យការសម្រួលម៉ូឌែលខូចខាតសម្រាប់ដែនដែលអ្នកមានបំណងគោលដៅ។

ដូច្នេះ មុននឹងរៀន "វិធីសាស្រ្ត" សម្រាប់សម្រួលលម្អិតម៉ូឌែលភាសា អ្នកត្រូវតែកំណត់ "ហេតុអ្វី" ត្រូវបើកផ្លូវនេះ និង "ពេលណា" ដែលត្រូវចាប់ផ្ដើមដំណើរការសម្រួលលម្អិត។ ចាប់ផ្ដើមដោយសួរខ្លួនឯងសំណួរទាំងនេះ៖

- **ករណីប្រើប្រាស់**៖ តើអ្នកមាន _ករណីប្រើប្រាស់_ សម្រាប់ការសម្រួលលម្អិតអ្វី? តើអ្នកចង់កែលម្អមុខងារម៉ូឌែលដែលបានបណ្តុះរួចនៅផ្នែកណា?
- **ជម្រើសផ្សេងទៀត**៖ តើអ្នកមានការសាកល្បង _បច្ចេកទេសផ្សេងទៀត_ ដើម្បីទទួលបានលទ្ធផលដូចចង់បានទេ? ប្រើវាដើម្បីបង្កើតមូលដ្ឋានសម្រាប់ការប្រៀបធៀប។
  - វិជ្ជាជីវៈប្រើប្រាស់ស្នើសុំ: សាកល្បងបច្ចេកទេសដូចជា few-shot prompting ជាមួយឧទាហរណ៌នៃចម្លើយស្នើសុំដែលពាក់ព័ន្ធ។ វាយតម្លៃគុណភាពចម្លើយ។
  - ការបង្កើតបន្ថែមដោយយកទិន្នន័យមកបន្ថែ: ព្យាយាមបន្ថែមសំនួរជាមួយលទ្ធផលស្វែងរកតាមបណ្ដាញទិន្នន័យរបស់អ្នក។ វាយតម្លៃគុណភាពចម្លើយ។
- **ថ្លៃចំណាយ**៖ តើអ្នកបានកំណត់ថ្លៃចំណាយសម្រាប់សម្រួលលម្អិតទេ?
  - ពិចារណាបានថា ម៉ូឌែលបានបណ្តុះរួចអាចសម្រួលលម្អិតបានឬទេ?
  - ផលិតភាព - សម្រាប់ការរៀបចំទិន្នន័យបណ្តុះបណ្តាល ការវាយតម្លៃ និងការកែលម្អម៉ូឌែល។
  - កំណត់កម្លាំងគណនា - សម្រាប់ដំណើរការងារសម្រួលលម្អិត និងដាក់បង្ហាញម៉ូឌែលដែលបានសម្រួលលម្អិត។
  - ទិន្នន័យ - ប្រកបដោយគុណភាពគ្រប់គ្រាន់សម្រាប់មានឥទ្ធិពលលើការសម្រួលលម្អិត។
- **អត្ថប្រយោជន៍**៖ តើអ្នកបានបញ្ជាក់អត្ថប្រយោជន៍នៃការសម្រួលលម្អិតទេ?
  - គុណភាព - តើម៉ូឌែលដែលបានសម្រួលលម្អិតមានលទ្ធផលល្អជាងមូលដ្ឋានទេ?
  - ថ្លៃចំណាយ - តើវាធ្វើអោយកាត់បន្ថយការប្រើ token ដោយធ្វើឲ្យសំណួរងាយស្រួលឡើងទេ?
  - ភាពអាចបន្ថែម - តើអ្នកអាចប្រើម៉ូឌែលមូលដ្ឋានសម្រាប់ដែនថ្មីៗបានទេ?

ដោយការឆ្លើយសំណួរទាំងនេះ អ្នកគួរអាចសម្រេចចិត្តបានថា ការសម្រួលលម្អិតគឺជាវិធីសាស្រ្តដែលត្រឹមត្រូវសម្រាប់ករណីប្រើប្រាស់របស់អ្នកឬទេ។ តាមអនុសាសន៍ វិធីសាស្រ្តនេះត្រឹមត្រូវបើអត្ថប្រយោជន៍លើសថ្លៃចំណាយ។ បន្ទាប់ពីអ្នកសម្រេចចិត្តបន្ត ការចាប់ផ្ដើមគិតពី _វិធីសាស្រ្ត_ ផងដែរ ក្នុងការសម្រួលលម្អិតម៉ូឌែលស្រាប់។

## តើយើងអាចសម្រួលលម្អិតម៉ូឌែលដែលបានបណ្តុះរួចបានយ៉ាងដូចម្តេច?

ដើម្បីសម្រួលលម្អិតម៉ូឌែលដែលបានបណ្តុះរួច អ្នកត្រូវមាន៖

- សំណុំទិន្នន័យសម្រាប់ប្រើសម្រួលលម្អិត
- បរិបទបណ្តុះបណ្តាលសម្រាប់ដំណើរការងារសម្រួលលម្អិត
