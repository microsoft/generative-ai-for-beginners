# គ្រឹះការណ៍រចនាសំណួរ

## ការណែនាំ
ម៉ូឌុលនេះគ្របដណ្តប់ពីយោគយល់ និងបច្ចេកទេសសំខាន់ៗសម្រាប់បង្កើតសំណួរដែលមានប្រសិទ្ធភាពនៅក្នុងម៉ូដែល AI សំរាប់បង្កើតមាតិកាថ្មី។ វិធីសាស្រ្តនៃការសរសេរសំណួរដល់ LLM ក៏មានអត្ថន័យដែរ។ សំណួរដែលបានបង្កើតយ៉ាងម៉ត់ចត់អាចទទួលបានចម្លើយដែលមានគុណភាពល្អប្រសើរ។ តែការពន្យល់ពាក្យដូចជា _prompt_ និង _prompt engineering_ មានន័យដូចម្តេច? ហើយតើខ្ញុំធ្វើដូចម្តេចដើម្បីធ្វើឱ្យការបញ្ចូលសំណួរ _input_ ដែលខ្ញុំផ្ញើទៅ LLM ល្អប្រសើរឡើង? នេះគឺជាសំណួរដែលយើងនឹងព្យាយាមឆ្លើយខណៈដែលពិនិត្យនៅក្នុងជំពូកនេះ និងជំពូកបន្ទាប់។

_Generative AI_ អាចបង្កើតមាតិកាថ្មី (ដូចជា អត្ថបទ រូបភាព សំឡេង កូដ លើកនេះ) ដោយឆ្លើយតបនឹងការស្នើសុំពីអ្នកប្រើប្រាស់។ វាធ្វើដំណើរការនេះដោយប្រើ _Large Language Models_ ដូចជា ស៊េរី GPT របស់ OpenAI ("Generative Pre-trained Transformer") ដែលត្រូវបានបណ្តុះបណ្តាលសម្រាប់ប្រើភាសា ឬកូដ។

ឥឡូវនេះអ្នកប្រើប្រាស់អាចធ្វើអន្តរកម្មជាមួយម៉ូដែលទាំងនេះដោយប្រើប្រព័ន្ធដែលស្គាល់ដូចជា ការសន្ទនា ដោយមិនចាំបាច់មានជំនាញបច្ចេកទេសឬការបណ្តុះបណ្តាលឡើយ។ ម៉ូដែលទាំងនេះគឺ _របៀបសំណួរ_ - អ្នកប្រើផ្ញើអត្ថបទ (prompt) ហើយទទួលបានចម្លើយ AI (completion) ថយវិញ។ បន្ទាប់មកពួកគេអាច "សន្ទនាជាមួយ AI" តាមរយៈការជជែកជាបន្តបន្ទាប់ ដើម្បីកែប្រែសំណួររហូតចម្លើយបានតាមចិត្តគួរឱ្យពេញចិត្ត។

"សំណួរ" ឥឡូវក្លាយជាចំណុចសំខាន់ក្នុង _ចំណុចប្រទាក់កម្មវិធី_ សម្រាប់កម្មវិធី generative AI ដែលប្រាប់ម៉ូដែលអ្វីដែលត្រូវធ្វើ ហើយមានឥទ្ធិពលលើគុណភាពនៃចម្លើយត្រលប់មកវិញ។ "Prompt Engineering" គឺជាវិស័យសិក្សាដែលកំពុងអភិវឌ្ឍយ៉ាងលឿន ដែលផ្តោតលើ _រចនា និងបង្កើនសមត្ថភាព_ នៃសំណួរដើម្បីផ្តល់ចម្លើយដែលមានគុណភាពនិងមានសំរួល។

## គោលបំណងរៀន

ក្នុងម៉េនុយនេះ យើងនឹងរៀនអំពីតើ Prompt Engineering ជាអ្វី ហើយហេតុអ្វីវាមានសារៈសំខាន់ និងរបៀបដែលយើងអាចបង្កើតសំណួរដែលមានប្រសិទ្ធភាពសម្រាប់ម៉ូដែល និងគោលបំណងកម្មវិធីណាមួយ។ យើងនឹងយល់រយៈដំណើរការចម្បង និងការអនុវត្តល្អបំផុតសម្រាប់បច្ចេកទេសរចនាសំណួរ - ហើយរៀនពីបរិដ្ឋានអន្តរកម្ម Jupyter Notebooks "sandbox" ដែលអាចមើលឃើញគំនិតទាំងនេះដែលបានអនុវត្តទៅលើគំរូពិត។

នៅចុងម៉េនុយនេះយើងនឹងអាចធ្វើបាន៖

1. ពន្យល់ថា prompt engineering ជាអ្វី ហើយហេតុអ្វីវាជារឿងសំខាន់។
2. ពណ៌នាប្រភេទមុខនៃសំណួរ និងរបៀបដែលវាត្រូវបានប្រើ។
3. រៀនពីអនុវត្តល្អបំផុត និងបច្ចេកទេសសម្រាប់ prompt engineering។
4. អនុវត្តបច្ចេកទេសដែលបានរៀនទៅឧទាហរណ៍ពិត ដោយប្រើ OpenAI endpoint។

## ពាក្យសំខាន់ៗ

Prompt Engineering: ជាអនុក្រឹតនៃការរចនា និងកែសម្រួលការបញ្ចូល (input) ដើម្បីណែនាំម៉ូដែល AI ផ្តល់ចេញលទ្ធផលដែលចង់បាន។
Tokenization: ដំណើរការបម្លែងអត្ថបទទៅជាឯកតាចំនួនតូចៗ ហៅថា token ដែលម៉ូដែលអាចយល់ និងកម្រងបាន។
Instruction-Tuned LLMs: ម៉ូដែលភាសាធំ (LLMs) ដែលត្រូវបានបង្រៀនបន្ថែមជាមួយសេចក្តីណែនាំពិសេស ដើម្បីបង្កើនភាពត្រឹមត្រូវ និងសក្ដានុពលក្នុងចម្លើយ។

## បរិដ្ឋានរៀន

Prompt engineering គឺជាការប្រណាំងសិល្បៈមួយជាងវិទ្យាសាស្រ្ត។ វិធីដ៏ល្អបំផុតក្នុងការកែល $\unicode{x178f}$ឹយសម្រួលនេះគឺ _ហាត់ប្រាណច្រើន_ និងអនុវត្តវិធីសាស្រ្តព្យាយាម និងខុសដែលលាយជាមួយជំនាញ និងបច្ចេកទេសដែលបានណែនាំ និងការបង្រៀនជាក់លាក់សម្រាប់ម៉ូដែល។

ទំព័រ Jupyter Notebook ដែលភ្ជាប់ទៅនឹងមេនុយនេះផ្តល់បរិដ្ឋាន _sandbox_ ដែលអ្នកអាចសាកល្បងអ្វីដែលបានរៀន - ទោះនៅពេលណាក៏ដោយ ឬជាផ្នែកនៃការប្រកួតកូដនៅចុងម៉េនុយ។ ដើម្បីអនុវត្ត អ្នកត្រូវការ៖

1. **កូនសោ API Azure OpenAI** - គឺជាចំណុចបញ្ចប់សេវាសម្រាប់ម៉ូដែល LLM ដែលបានបញ្ជូន។
2. **Python Runtime** - សម្រាប់បញ្ចូលចូលដំណើរការ Notebook។
3. **អថេរបរិយាកាសក្នុងមូលដ្ឋាន** - _បញ្ចប់ជំហាន [SETUP](./../00-course-setup/02-setup-local.md?WT.mc_id=academic-105485-koreyst) ឥឡូវនេះដើម្បីរៀបចំ_។

Notebook មកជាមួយលំហាត់ _ដំបូង_ - ប៉ុន្តែអ្នកត្រូវលើកទឹកចិត្តក្នុងការបន្ថែមអត្ថបទ _Markdown_ (ពិពណ៌នា) និង _Code_ (សំណើសំណួរ) ដើម្បីសាកល្បងឧទាហរណ៍ឬគំនិតបន្ថែម - ហើយកែលតម្រូវនូវការយល់ដឹងរបស់អ្នកក្នុងការរចនាសំណួរ។

## មេរៀនត្រូវមានរូបភាព

ចង់ទទួលបានទិដ្ឋភាពធំនៃអ្វីដែលមេរៀននេះគ្របដណ្តប់មុនពេលចូលកាន់មុខវិជ្ជា? សូមមើលមេរៀនដែលមានរូបភាព អ្នកនឹងមានគំនិតអំពីប្រធានបទសំខាន់ៗ និងចំណុចសង្ខេបសម្រាប់ការឆែកទំនោរនៅក្នុងមួយមុខវិជ្ជា។ ផែនទីមេរៀននាំអ្នកពីការយល់ដឹងគ្រឹះនៃយោគយល់ និងបញ្ហាទៅរកការដោះស្រាយពួកវាជាមួយបច្ចេកទេសនិងការអនុវត្តល្អបំផុត។ ចំណាំថា ផ្នែក "បច្ចេកទេសកម្រិតខ្ពស់" នៅក្នុងមេរៀននេះពាក់ព័ន្ធនឹងមាតិកាដែលបានគ្របដណ្តប់នៅជំពូកខាងក្រោយ។

## ស្តាតអាប់របស់យើង

ឥឡូវនេះ មកពិភាក្សាអំពីរបៀបដែល _ប្រធានបទនេះ_ មានទំនាក់ទំនងទៅនឹងបេសកកម្មស្តាតអាប់របស់យើងក្នុងការនាំយកបញ្ចូល AI ចូលក្នុង​វិស័យអប់រំ [បញ្ចូល AI ក្នុងអប់រំ](https://educationblog.microsoft.com/2023/06/collaborating-to-bring-ai-innovation-to-education?WT.mc_id=academic-105485-koreyst)។ យើងចង់បង្កើតកម្មវិធីដែលមាន AI គ្រប់គ្រង និងផ្តល់ការសិក្សាប្រកបដោយការប្ដូរតាមបុគ្គល - ដូច្នេះចាំបាច់ត្រូវគិតពីរបៀបដែលអ្នកប្រើប្រាស់ផ្សេងៗអាច "រចនា" សំណួរផ្ទាល់ខ្លួនរបស់ពួកគេ៖

- **អ្នកអភិបាល** អាចស្នើ AI ដើម្បី _វិភាគទិន្នន័យមេរៀនដើម្បីកំណត់ចន្លោះក្នុងការរួមបញ្ចូល_។ AI អាចសង្ខេបលទ្ធផលឬបង្ហាញជារូបភាពជាមួយកូដ។
- **គ្រូបង្រៀន** អាចស្នើ AI ដើម្បី _បង្កើតផែនការរៀនសម្រាប់គោលដៅនិងប្រធានបទមួយ_។ AI អាចបង្កើតផែនការផ្ទាល់ខ្លួនក្នុងទ្រង់ទ្រាយកំណត់។
- **សិស្ស** អាចស្នើ AI ដើម្បី _បង្រៀនពួកគេក្នុងមុខវិជ្ជាដ៏យ៉ាងពិបាក_។ AI អាចនាំផ្លូវសិស្សជាមួយមេរៀន ការណែនាំ និងឧទាហរណ៍ដែលសមស្របនឹងកម្រិតរបស់ពួកគេ។

នេះគ្រាន់តែជាផ្នែកតូចបំផុត។ សូមពិនិត្យមើល [Prompts For Education](https://github.com/microsoft/prompts-for-edu/tree/main?WT.mc_id=academic-105485-koreyst) - បណ្ណាល័យសំណួរបើកចំហរ ដែលបានរៀបចំដោយអ្នកជំនាញអប់រំ - ដើម្បីទទួលបានចំណេះដឹងទូលំទូលាយ​​​​​​​! _សូមសាកល្បងដំណើរការសំណួរនោះនៅក្នុង sandbox ឬប្រើ OpenAI Playground ដើម្បីមើលថាតើមានអ្វីកើតឡើង!_

## Prompt Engineering ជាអ្វី?

យើងបានចាប់ផ្តើមម៉េរៀននេះដោយកំណត់ **Prompt Engineering** ជាដំណើរការនៃ _រចនា និងបង្កើនសមត្ថភាព_ បញ្ចូលអត្ថបទ (សំណួរ) ដើម្បីផ្តល់ចម្លើយដែលមានលក្ខណៈថេរ និងគុណភាពល្អសម្រាប់គោលបំណងកម្មវិធី និងម៉ូដែលកំណត់។ អាចគិតថា វាជាដំណើរការពីរដំណាក់កាល៖

- _រចនា_ សំណួរដំបូងសម្រាប់ម៉ូដែល និងគោលបំណងដែលបានកំណត់
- _កែសម្រួល_ សំណួរ​ឡើងវិញ ដើម្បីធ្វើឱ្យគុណភាពចម្លើយប្រសើរឡើង

នេះជាដំណើរការព្យាយាម និងខុស ដែលទាមទារជំនាញ និងការខិតខំរបស់អ្នកប្រើសម្រាប់ទទួលលទ្ធផលល្អបំផុត។ ដូច្នេះហេតុអ្វីវាសំខាន់? ដើម្បីឆ្លើយសំណួរនេះ យើងត្រូវយល់ពីគំនិតបីទៀតជាមុន៖

- _Tokenization_ = របៀបដែលម៉ូដែល "ឃើញ" សំណួរ
- _Base LLMs_ = របៀបដែលម៉ូដែលមូលដ្ឋាន "ដំណើរការសំណួរ"
- _Instruction-Tuned LLMs_ = របៀបដែលម៉ូដែលឥឡូវនឹងឃើញ "ការងារ"

### Tokenization

LLM មើលសំណួរដូចជា _ជាបន្ទាត់នៃ token_ ដែលម៉ូដែលផ្សេងគ្នា (ឬជំនាន់ម៉ូដែល) អាច tokenize សំណួរដូចគ្នានៅលើវិធីផ្សេងៗគ្នា។ ព្រោះ LLM បានបណ្តុះលើ token (មិនមែនអត្ថបទដើម) វិធី tokenize នៃសំណួរនេះមានឥទ្ធិពលផ្ទាល់លើគុណភាពនៃចម្លើយបានបង្កើត។

ដើម្បីទទួលបានគំនិតថា tokenization ដំណើរការយ៉ាងដូចម្តេច សូមសាកល្បងឧបករណ៍ដូចជា [OpenAI Tokenizer](https://platform.openai.com/tokenizer?WT.mc_id=academic-105485-koreyst) ដែលបង្ហាញខាងក្រោម។ ចំណាំលង់សំណួររបស់អ្នក ហើយមើលពីរបៀបសំណួរបម្លែងទៅជាតួ token ដោយចាប់អារម្មណ៍ប្រចាំលើការគ្រប់គ្រងតួអក្សរពន្លឺ និងសញ្ញាវិរុទ្ធ។ ចំណាំថាជាឧទាហរណ៍នេះបង្ហាញ LLM ចាស់មួយ (GPT-3) ដូច្នេះសាកល្បងជាមួយម៉ូដែលថ្មីអាចបង្កើតលទ្ធផលខុស។

![Tokenization](../../../translated_images/km/04-tokenizer-example.e71f0a0f70356c5c.webp)

### គំនិត៖ ម៉ូដែលមូលដ្ឋាន

ពេលដែលសំណួរត្រូវបាន tokenize រួច មុខងារសំខាន់នៃ ["Base LLM"](https://blog.gopenai.com/an-introduction-to-base-and-instruction-tuned-large-language-models-8de102c785a6?WT.mc_id=academic-105485-koreyst) (ឬម៉ូដែលមូលដ្ឋាន) គឺទស្សនាតួ token នៅក្នុងខ្សែបន្ទាត់នោះ។ ព្រោះ LLM បានបណ្តុះលើសំណុំទិន្នន័យអត្ថបទធំៗ វាអាចទាក់ទងស្ថិតិរវាង token ហើយអាចទស្សនាទុកដោយមានភាពទំនុកចិត្តខ្លះៗ។ ចំណាំថាវាមិនបានយល់ពី _ន័យ_ នៃពាក្យនៅក្នុងសំណួរឬ token ទេ; វាស៊ីញាប៊ែមម្ដងទៀតលំនាំដែលវាអាច "បញ្ចប់" ជាមួយការទស្សនាថាតទៅ។ វាអាចបន្តទស្សនាផ្សេងទៀតរហូតដល់មានការចាប់ផ្តើមដោយអ្នកប្រើឬលក្ខណ:បានកំណត់កន្លងមក។

ចង់ឃើញថាចម្លើយមូលដ្ឋានរបៀបសំណួរធ្វើដូចម្តេច? ចូរបញ្ចូលសំណួរខាងលើទៅក្នុង [Microsoft Foundry playground](https://ai.azure.com?WT.mc_id=academic-105485-koreyst) ជាមួយការកំណត់លំនាំដើម។ ប្រព័ន្ធត្រូវបានកំណត់ឱ្យដាក់សំណួរជាការស្នើសុំព័ត៌មាន - ដូច្នេះអ្នកគួរតែឃើញចម្លើយដែលសម្របសម្រួលរួចជាមួយបរិបទនេះ។

តែបើអ្នកប្រើចង់ឃើញអ្វីមួយជាក់លាក់ដែលផ្គូរផ្គងក្រុមក្រសោបឬគោលបំណងអង្គភាព? នេះជាទីតាំងដែល _instruction-tuned_ LLMs ចូលចិត្ត។

### គំនិត៖ Instruction Tuned LLMs

[Instruction Tuned LLM](https://blog.gopenai.com/an-introduction-to-base-and-instruction-tuned-large-language-models-8de102c785a6?WT.mc_id=academic-105485-koreyst) សូម្បីតែចាប់ផ្តើមជាមួយម៉ូដែលមូលដ្ឋាន ហើយបរិច្ឆេទវាថយក្រោយជាមួយឧទាហរណ៍ឬគូ input/output (ដូចជា "សារ" បញ្ចូលច្រើនជុំ) ដែលអាចមានសេចក្តីណែនាំច្បាស់ - ហើយចម្លើយពី AI ព្យាយាមធ្វើតាមសេចក្តីណែនាំនោះ។

វាការប្រើបច្ចេកទេសដូចជា Reinforcement Learning with Human Feedback (RLHF) ដែលអាចបណ្តុះបណ្តាលម៉ូដែលឱ្យ _ធ្វើតាមសេចក្តីណែនាំ_ និង _រៀនពីមតិយោបល់_ ដើម្បីផ្តល់ចម្លើយដែលសមស្របជាងសម្រាប់កម្មវិធីជាក់លាក់ និងសក្ដានុពលកាន់តែខ្ពស់ជាមួយគោលបំណងអ្នកប្រើ។

សូមសាកល្បងម្ដងទៀត - ត្រឡប់ទៅសំណួរខាងលើ ប្តូរសារ _system message_ ដើម្បីផ្តល់នូវសេចក្តីណែនាំក្នុងបរិបទដូចខាងក្រោម៖

> _សង្ខេបមាតិកាដែលអ្នកបានផ្តល់ជូនសម្រាប់សិស្សថ្នាក់ទី ២។ ចាំបាច់អោយលទ្ធផលមានមួយវេទិកាមួយមានចំនុចបញ្ចាំងចំនួន ៣-៥។_

មើលបើការចម្លើយឥឡូវបានត្រូវបង្រៀបទៅតាមគោលបំណង និងទ្រង់ទ្រាយដែលចង់បាន? អ្នកបង្រៀនអាចនាំចម្លើយនេះទៅប្រើផ្ទាល់ក្នុងស្លាយសម្រាប់ថ្នាក់នោះ។

## ហេតុអ្វីយើងត្រូវការការរចនាសំណួរ?

ពេលនេះដែលយើងបានយល់ថាសំណួរត្រូវបានដំណើរការដោយ LLMs ដូចម្តេច សូមពិភាក្សាអំពី _ហេតុផល_ ដែលយើងត្រូវការការរចនាសំណួរ។ ចម្លើយស្ថិតក្នុងការពិតថា LLMs បច្ចុប្បន្នប្រឈមមុខនឹងបញ្ហាជាច្រើនដែលធ្វើឱ្យការទទួលបាន _ចម្លើយដែលអាចទុកចិត្ត និងមានភាពថេរ_ មានការលំបាក បើគ្មានកិច្ចខិតខំក្នុងការរចនានិងបង្កើនសមត្ថភាពសំណួរ។ ឧទាហរណ៍៖

1. **ចម្លើយម៉ូដែលមានភាពចៃដន្យ។** _សំណួរដូចគ្នា_ អាចធ្វើឱ្យមានចម្លើយខុសប្លែកគ្នាជាមួយម៉ូដែលផ្សេងៗ ឬជំនាន់ម៉ូដែលផ្សេងៗ។ ហើយវាអាចប្រសើរឡើងពេលប្រើ _ម៉ូដែលដូចគ្នា_ នៅពេលខុសៗគ្នា។ _បច្ចេកទេសរចនាសំណួរក្នុងការពារការប្រែប្រួលនេះអាចជួយកាត់បន្ថយភាពខុសគ្នានេះ_។

1. **ម៉ូដែលអាចបង្កើតចម្លើយមិនត្រូវបានពិត។** ម៉ូដែលត្រូវបានបណ្តុះលើសំណុំទិន្នន័យ _ធំ ប៉ុន្តែមិនពេញលេញ_ ដែលមានអត្ថន័យថាវាមិនមានចំណេះដឹងអំពីគំនិតនៅក្រៅអ្វីដែលបានបណ្តុះក្នុងការងារ។ ដូច្នេះ វាអាចផ្សងចាក់ចម្លើយមិនត្រឹមត្រូវ រាប់រោង ឬផ្ទុយពីការពិតដែលបានដឹង។ _បច្ចេកទេសរចនាសំណួរជួយអ្នកប្រើក្នុងការស្វែងរកនិងកាត់បន្ថយការបង្កើតចម្លើយមិនត្រឹមត្រូវ ដូចជាការស្នើរការដកសម្លឹងឬការតម្រូវយោបល់ពី AI_។

1. **សមត្ថភាពម៉ូដែលនឹងមិនស្ទាងគ្នាទាំងស្រុង។** ម៉ូដែលថ្មី ឬជំនាន់ម៉ូដែលនឹងមានសមត្ថភាពបន្ថែម ប៉ុន្តែត្រូវនាំមកនូវភាពខុសគ្នា និងការជួញដូរផលដូចជាគុណភាព និងភាពស្មុគស្មាញក្នុងការប្រើប្រាស់។ _បច្ចេកទេសរចនាសំណួរជួយពួកយើងបង្កើតនូវអនុវិធីល្អបំផុត និងលំនាំធ្វើការដែលអាចលាក់ល្នៅភាពខុសគ្នានិងអាចបំពេញតម្រូវការផ្តល់នូវលក្ខខណ្ឌវិនិយោគមូលដ្ឋានៗក្នុងវិធីមានប្រសិទ្ធភាព_។

យើងមកមើលវាជាការអនុវត្តនៅក្នុង OpenAI ឬ Azure OpenAI Playground៖

- ប្រើសំណួរដូចគ្នាជាមួយការដាក់បណ្ដោះអាសន្ន LLM ផ្សេងៗ (ដូចជា OpenAI, Azure OpenAI, Hugging Face) - តើអ្នកឃើញភាពខុសគ្នាទេ?
- ប្រើសំណួរដូចគ្នាជាវិញវិញជាមួយការដាក់បណ្ដោះអាសន្ន _ដូចគ្នា_ (ដូចជា Azure OpenAI playground) - តើភាពខុសគ្នាបានផ្សេងគ្នាយ៉ាងដូចម្តេច?

### ឧទាហរណ៍ចម្លើយបង្កើតមិនត្រឹមត្រូវ

នៅក្នុងវគ្គសិក្សានេះ យើងប្រើពាក្យ **"បង្កើតមិនត្រឹមត្រូវ"** ដើម្បីយោងដល់អាការៈដែល LLM មួយៗពេលខ្លះបង្កើតព័ត៌មានមិនត្រឹមត្រូវដោយសារការកំណត់ក្នុងការបណ្តុះបណ្តាល ឬកំណត់ផ្សេងៗ។ អ្នកអាចបានស្គាល់វាជា _"ភាពច្របូកច្របល់"_ ក្នុងអត្ថបទនិយម ឬឯកសារស្រាវជ្រាវ។ ប៉ុន្តែយើងណែនាំឱ្យប្រើពាក្យ _"បង្កើតមិនត្រឹមត្រូវ"_ ដើម្បីកុំឱ្យពួកយើងផ្ដល់ភាសារមានបែបសត្វចិត្ត ដោយផ្ដល់តួអង្គដូចមនុស្សទៅលើលទ្ធផលដែលគ្រប់គ្រងដោយម៉ាស៊ីន។ វាក៏ជាការបញ្ជាក់ផងដែរពី [មគ្គុទេសក៍ AI បន្ទុកចិត្ត](https://www.microsoft.com/ai/responsible-ai?WT.mc_id=academic-105485-koreyst) ចំពោះគំនិតពាក្យ ផ្ដាច់នោះ ព្រមទាំងយកចេញពាក្យដែលអាចមើលថាមិនសមរម្យ ឬមិនរួមបញ្ចូលនៅក្នុងបរិបទខ្លះៗ។

ចង់បានយល់ពីរបៀបដែលបង្កើតមិនត្រឹមត្រូវដំណើរការយ៉ាងដូចម្តេច? សូមគិតពីសំណួរមួយដែលណែនាំ AI ឲ្យបង្កើតមាតិកាសម្រាប់ប្រធានបទមិនមាននឹកស្មាន (ដើម្បីធានាមិនមានក្នុងសំណុំបណ្តុះបណ្តាល)។ ឧទាហរណ៍ ខ្ញុំបានសាកល្បងសំណួរនេះ៖

> **សំណួរ:** បង្កើតផែនការមេរៀនអំពីសង្រ្គាមមីតថាននៅឆ្នាំ 2076។

## ការសាងសង់សំណើ

យើងបានឃើញហេតុផលមួយដែលធ្វើអោយឧបករណ៍បង្ហាញមានសារៈសំខាន់ - ឥឡូវនេះចាំយល់ពីរបៀបដែលសំណើត្រូវ _សាងសង់_ ដើម្បីយើងអាចវាយតម្លៃបច្ចេកទេសផ្សេងៗសម្រាប់ការរចនាសំណើមានប្រសិទ្ធភាព។

### សំណើមូលដ្ឋាន

ចាប់ផ្តើមជាមួយសំណើមូលដ្ឋាន៖ អត្ថបទបញ្ចូលទៅម៉ូដែលដោយគ្មានបរិបទផ្សេងទៀត។ នេះជាឧទាហរណ៍មួយ - ពេលយើងផ្ញើពាក្យជាច្រើនដំបូងនៃអគ្គនាយក​សមរភូមិ​សហរដ្ឋ​អាមេរិកទៅកាន់ OpenAI [Completion API](https://platform.openai.com/docs/api-reference/completions?WT.mc_id=academic-105485-koreyst) វាបញ្ចប់ចម្លើយភ្លាមៗជាមួយបន្ទាត់បន្ទាប់ពីបង្ហាញអាកប្បកិរិយា​នៃ​ប្រតិបត្តិការព្យាករ។

| Oh say can you see | វាលឺដូចជាអ្នកកំពុងចាប់ផ្តើមចំរៀង "The Star-Spangled Banner," អគ្គនាយកសមរភូមិរបស់សហរដ្ឋអាមេរិក។ ចំរៀងពេញលេញគឺ ...                      |

### សំណើស្មុគស្មាញ

ឥឡូវនេះចូលបន្ថែមបរិបទនិងសេចក្ដីណែនាំទៅនឹងសំណើមូលដ្ឋាននោះ។ [Chat Completion API](https://learn.microsoft.com/azure/ai-foundry/openai/how-to/chatgpt?WT.mc_id=academic-105485-koreyst) អនុញ្ញាតឲ្យយើងសំរាប់សំណើស្មុគស្មាញជាសំណូល _សារ_ ជាច្រើនជាមួយ៖

- ឈុតបញ្ចូល/ចេញ​និយាយ​បង្ហាញ​ការ​បញ្ចូល​របស់ _អ្នក​ប្រើ_ និង​ផ្តល់​ចម្លើយ​របស់ _ជំនួយការ_។
- សារប្រព័ន្ធកំណត់បរិបទសម្រាប់អាកប្បកិរិយា ឬបុគ្គលិកលក្ខណៈរបស់ជំនួយការ។

សំណើឥឡូវនេះមានទំរង់ខាងក្រោម ដែល _tokenization_ ធ្វើការចាប់យកព័ត៍មាន​សំខាន់​ពី​បរិបទ​និង​សន្ទនាដោយមានប្រសិទ្ធភាព។ ឥឡូវនេះ ការផ្លាស់ប្តូបរិបទប្រព័ន្ធអាចមានឥទ្ធិពលដល់គុណភាពនៃការបញ្ចប់ ដូចជាការបញ្ចូលរបស់អ្នកប្រើ។

```python
response = client.responses.create(
    model="gpt-5-mini",
    input=[
        {"role": "system", "content": "You are a helpful assistant."},
        {"role": "user", "content": "Who won the world series in 2020?"},
        {"role": "assistant", "content": "The Los Angeles Dodgers won the World Series in 2020."},
        {"role": "user", "content": "Where was it played?"}
    ]
)
```

### គំរូបញ្ចូលសំណួរ

គំរូបញ្ចូលសំណួរ គឺជារ​ប្រមួនឯកសារត្រូវបានកំណត់ជាមុនសម្រាប់សំណួរ ដែលអាចផ្ទុកនិងប្រើឡើងវិញប្រកបដោយតម្រូវការ ដើម្បីបង្កើតបទពិសោធន៍អ្នកប្រើដែលមានសក្តានុពលជារឿយៗ។ នៅក្នុងទ្រង់ទ្រាយសាមញ្ញ វាជាការប្រមូលផ្តុំឧទាហរណ៍សំណួរដូចជា [ឧទាហរណ៍នេះពី OpenAI](https://cookbook.openai.com/examples/gpt4-1_prompting_guide?WT.mc_id=academic-105485-koreyst) ដែលផ្តល់នូវឧបករណ៍បញ្ចូលសំណួរប្រតិកម្ម (សារអ្នកប្រើនិងប្រព័ន្ធ) និងទ្រង់ទ្រាយសំណើ API ដើម្បីគាំទ្រការប្រើឡើងវិញ។

ក្នុងទ្រង់ទ្រាយស្មុគស្មាញបន្ថែមដូចជា [ឧទាហរណ៍នេះពី LangChain](https://python.langchain.com/docs/concepts/prompt_templates/?WT.mc_id=academic-105485-koreyst) វាមាន _ចំណុចទំនេរ_ ដែលអាចជំនួសដោយទិន្នន័យពីប្រភពផ្សេងៗ (ការបញ្ចូលអ្នកប្រើ សេចក្ដីផ្ដល់បរិបទប្រព័ន្ធ ប្រភពទិន្នន័យខាងក្រៅ ជាដើម) ដើម្បីបង្កើតសំណួរដោយអត្ថន័យ។ វាអនុញ្ញាតឱ្យយើងបង្កើតបណ្ណាល័យសំណួរដែលអាចប្រើឡើងវិញដើម្បីបំពេញបទពិសោធន៍អ្នកប្រើជាបន្តបន្ទាប់ **ដោយកម្មវិធី** នៅកម្រិតធំ។

នៅចុងបញ្ចប់ តម្លៃពិតនៃគំរូសម្រាប់សំណួរជាក់ស្តែងនៅក្នុងសមត្ថភាពបង្កើតនិងផ្សព្វផ្សាយ _បណ្ណាល័យសំណួរ_ សម្រាប់វិស័យ​កម្មវិធីជាក់លាក់ - ដែលគំរូសំណួរបាន _ធ្វើឲ្យប្រសើរ_ ដើម្បីបង្ហាញបរិបទឬឧទាហរណ៍ជាក់ស្តែងក្នុងកម្មវិធីដែលធ្វើឲ្យចម្លើយរីករាយ និងត្រឹមត្រូវសម្រាប់ក្រុមអ្នកប្រើដែលបានគោលដៅ។ ឃ្លាំង [Prompts For Edu](https://github.com/microsoft/prompts-for-edu?WT.mc_id=academic-105485-koreyst) ជាឧទាហរណ៍ល្អ ប្រមូលគំនិតបណ្ណាល័យសម្រាប់វិស័យអប់រំជាមួយការផ្ដោតលើគោលដៅសំខាន់ៗដូចជា ការរៀបចមគម្រោងមេរៀន ការរចនាម៉ោងសិក្សា ការបង្រៀនសិស្សជាដើម។

## ខ្លឹមសារគាំទ្រ

ប្រសិនបើយើងគិតពីការបង្កើតសំណួរជាអ្វីដែលមានណែនាំ (ភារកិច្ច) និងមាតិកា​គោលដៅ (មាតិកាសំខាន់) ក៏ប្រភេទ _មាតិកាជួយ_ គឺដូចជាបរិបទបន្ថែមដែលយើងផ្ដល់ដើម្បី **ឥទ្ធិពលលើចេញលទ្ធផលមួយវិធីណាមួយ**។ វាអាចជាប៉ារ៉ាម៉ែត្រការតំរូវការ ប្រតិបត្ដិការផ្នែកទ្រង់ទ្រាយ តារាងប្រធានបទ ជាដើម ដែលជួយម៉ូដែល _កែលំអ_ ចម្លើយឲ្យសមស្របនឹងគោលបំណង ឬការរំពឹងទុករបស់អ្នកប្រើ។

ឧទាហរណ៍៖ បើមានបញ្ជីមេរៀន មានមេតាឌាតាច្រើន (ឈ្មោះ ការពណ៌នា កម្រិត តម្លៃមេតាឌាតា អ្នកបង្រៀន ជាដើម) សម្រាប់មេរៀនទាំងអស់ក្នុងម៉ាស្សារៀន៖

- យើងអាចកំណត់ណែនាំមួយក្នុងការសំណូមពរ "សង្ខេបបញ្ជីមេរៀនរបស់រដូវស្លឹកឈើជ្រុះ ឆ្នាំ 2023"
- យើងអាចប្រើមាតិកាគោលដៅដើម្បីផ្ដល់ឧទាហរណ៍ខ្លះៗនៃលទ្ធផលដែលចង់បាន
- យើងអាចប្រើមាតិកាជួយដើម្បីកំណត់ ៥ "ស្លាក" ដែលមានចំណាប់អារម្មណ៍បំផុត។

ឥឡូវម៉ូដែលអាចផ្តល់សង្ខេបតាមទ្រង់ទ្រាយដែលបានបង្ហាញដោយឧទាហរណ៍ខ្លះៗ - ប៉ុន្តែប្រសិនបើលទ្ធផលមានស្លាកច្រើន អាចផ្តល់អាទិភាពដល់ ៥ ស្លាកដែលបានកំណត់ក្នុងមាតិកាជួយ។

## ច្បាប់ល្អសម្រាប់បញ្ចូលសំណួរ

ឥឡូវនេះ យើងបានស្គាល់របៀបដែលសំណួរអាច _ត្រូវបានបង្កើត_ យើងអាចចាប់ផ្តើមគិតពីរបៀបរៀបចំវាឲ្យមាន _ការរចនា_ ដែលបង្ហាញពីច្បាប់ល្អៗ។ យើងអាចគិតពីរបៀបនេះក្នុងពីរផ្នែក - មាន _ចិត្តវិធី_ ត្រឹមត្រូវ និង​​អនុវត្ត _បច្ចេកទេស_ ត្រឹមត្រូវ។

### ចិត្តវិធីបញ្ចូលសំណួរ

ការបញ្ចូលសំណួរជាប្រមូលនៃការបញ្ចេញកំហុសរហូត ដូច្នេះគួរតែកាន់ក្នុងចិត្តបីកត្តាធំៗ៖

1. **ការយល់ដឹងលើវិស័យសំខាន់។** ចម្លើយមានភាពត្រឹមត្រូវ និងពាក់ព័ន្ធ គឺបែងចែកដោយ _វិស័យ_ ដែលកម្មវិធី ឬអ្នកប្រើប្រាស់បំពាក់ទៅ។ ប្រើប្រាស់ឥទ្ធិពលនៃការយល់ដឹង និងជំនាញក្នុងវិស័យ ដើម្បី **ប្តូរបច្ចេកទេស** ឲ្យត្រូវគ្នាបន្ថែមទៀត។ ឧទាហរណ៍ កំណត់ _បុគ្គលិកលក្ខណៈសម្គាល់ក្នុវិស័យ_ នៅក្នុងសារប្រឹក្សាប្រព័ន្ធ រឺប្រើ _គំរូសម្រាប់វិស័យជាក់លាក់_ នៅក្នុងសំណួរអ្នកប្រើ។ ផ្តល់មាតិកាជួយដែលបង្ហាញបរិបទជាក់លាក់នៃវិស័យ ឬប្រើ _សញ្ញានិងឧទាហរណ៍វិស័យជាក់លាក់_ ដើម្បីណែនាំម៉ូដែលឲ្យស្គាល់នូវលំនាំប្រើប្រាស់។

2. **ការយល់ដឹងម៉ូដែលសំខាន់។** យើងដឹងថាម៉ូដែលមានលក្ខណៈស្ទុកស្ទង់ដោយធម្មជាតិ។ ប៉ុន្តែការ​អនុវត្តម៉ូដែលអាចបែបផ្សេងគ្នាសម្រាប់ទិន្នន័យបណ្តុះបណ្តាលដែលវាប្រើប្រាស់ (ចំណេះដឹងបានបណ្តុះបណ្តាលមុន) សមត្ថភាពដែលវាប្រើ (ឧទាហរណ៍ តាម API ឬ SDK) និងប្រភេទមាតិកាដែលវាបានធ្វើឲ្យប្រសើរ (ឧទាហរណ៍ កូដ បែបបទរូបភាព អត្ថបទ)។ យល់ដឹងពីកម្លាំង និងកំណត់ដែនលំបាករបស់ម៉ូដែលដែលអ្នកកំពុងប្រើ ហើយប្រើប្រាស់ចំណេះដឹងនោះដើម្បី _ផ្តល់អាទិភាពភារកិច្ច_ រឺបង្កើត _គំរូប្ដូរត្រៀមតាមបំណង_ ដែលបានធ្វើឲ្យសាកសមសម្រាប់សមត្ថភាពម៉ូដែល។

3. **ស្តាប់ថយវិល និង ផ្ទៀងផ្ទាត់សំខាន់។** ម៉ូដែលកំពុងអភិវឌ្ឍជាបន្ទាន់ ហើយបច្ចេកទេស​សម្រាប់បញ្ចូលសំណួរនឹងក៏ដូចគ្នា។ ជាជំនាញវិស័យ អ្នកអាចមានបរិបទឬវិធានការណ៍ផ្សេងទៀតសម្រាប់ _កម្មវិធីជាក់លាក់របស់អ្នក_ ដែលមិនត្រឹមត្រូវសម្រាប់សហគមន៍ទូលំទូលាយទេ។ ប្រើឧបករណ៍និងបច្ចេកទេសបញ្ចូលសំណួរដើម្បី "ចាប់ផ្តើម" ការបង្កើតសំណួរ បន្ទាប់មកស្តាប់ថយវិល និងផ្ទៀងផ្ទាត់លទ្ធផលដោយប្រើឥទ្ធិពលនិងជំនាញវិស័យរបស់អ្នក។ ចំណាំយកពីរបៀប និងបង្កើត **មូលដ្ឋានចំណេះដឹង** (ឧទាហរណ៍ បណ្ណាល័យសំណួរ) ដែលអាចប្រើជាមូលដ្ឋានថ្មីសម្រាប់អ្នកផ្សេងទៀត ដើម្បីកែលម្អលឿននៅថ្ងៃក្រោយ។

សម្រាប់ភារកិច្ចរបស់យើង យើងនឹងប្រើ Jupyter Notebook ដែលមានលំហាត់ដែលអ្នកអាចបញ្ចប់ដោយមានផ្លាស់ប្តូរក្នុងការប្រាស្រ័យទាក់ទង។ អ្នកក៏អាចបន្ថែម Markdown និងកូដផ្ទាល់ខ្លួនដើម្បីស្វែងយល់ពីគំនិតនិងបច្ចេកទេសបន្ថែម។

### ដើម្បីចាប់ផ្តើម សូម fork កម្មវិធី និងបន្ទាប់មក

- (ផ្តល់អនុសាសន៍) បើក GitHub Codespaces
- (ជាជម្រើសជំនួស) Clone កម្មវិធីទៅឧបករណ៍របស់អ្នក ហើយប្រើវាជាមួយ Docker Desktop
- (ជាជម្រើសជំនួស) បើក Notebook ជាមួយបរិយាកាសកំណត់សម្រាប់ Notebook ដែលអ្នកចូលចិត្ត។

### បន្ទាប់ ដំណើរការការកំណត់អថេរសាកល

- ចម្លងឯកសារ `.env.copy` ពី Root Folder ទៅ `.env` ហើយបញ្ចូលតម្លៃ `AZURE_OPENAI_API_KEY`, `AZURE_OPENAI_ENDPOINT` និង `AZURE_OPENAI_DEPLOYMENT` ។ ថយថយទៅកាន់ផ្នែក [Learning Sandbox](#បរិដ្ឋានរៀន) ដើម្បីរៀនពីរបៀប។

### បន្ទាប់ បើក Jupyter Notebook

- ជ្រើសរើស Kernel នៅពេលបើក។ ប្រសិនបើប្រើជម្រើសទី 1 ឬ 2 សូមជ្រើសរើស Kernel Python 3.10.x ដែលផ្តល់ដោយ Dev Container។

អ្នកស្រាប់តែរួចរាល់ដើម្បីធ្វើលំហាត់។ សម្គាល់ថា មិនមានចម្លើយត្រឹមត្រូវ និងមិនត្រឹមត្រូវនៅទីនេះទេ - គ្រាន់តែស្វែងរកជម្រើសតាមរបៀបសាកល្បង និងកសាងអារម្មណ៍ចំពោះអ្វីដែលដំណើរការសម្រាប់ម៉ូដែលនិងវិស័យកម្មវិធី។

_ដោយសារតែហេតុផលនេះ មេរៀននេះ មិនមានផ្នែកដំណោះស្រាយកូដទេ។ រឺប្រសិនបើមាន Notebook នឹងមានកូដ Markdown មានចំណងជើង "My Solution:" ដែលបង្ហាញឧទាហរណ៍ចម្លើយមួយសម្រាប់យោង។_

## ពិនិត្យសមត្ថភាព

តើខាងក្រោមណាមួយគឺជាសំណួរល្អបង្ហាញច្បាប់ល្អៗសមរម្យ?

1. បង្ហាញរូបភាពរថយន្តពណ៌ក្រហមមួយ
2. បង្ហាញរូបភាពរថយន្តពណ៌ក្រហមប្រភេទ Volvo ម៉ូដែល XC90 ដែលបានចតពីមូលជ្រេីងជាមួយព្រះអាទិត្យកំពុងលិច
3. បង្ហាញរូបភាពរថយន្តពណ៌ក្រហមប្រភេទ Volvo ម៉ូដែល XC90

ចម្លើយ៖ 2 ជាសំណួរល្អបំផុត ពីព្រោះវាបង្ហាញលម្អិតអំពី "អ្វី" ហើយបញ្ជាក់ជាក់លាក់ (មិនមែនរថយន្តណាមួយទេ ប៉ុន្តែជាប្រភេទនិងម៉ូដែលជាក់លាក់) ហើយវាក៏ពណ៌នាបរិបទទូទៅផងដែរ។ 3 ជាជម្រើសល្អបន្ទាប់ ព្រោះវាក៏មានការពិពណ៌នាច្រើនផងដែរ។
