# មូលដ្ឋានមេរៀនសម្រាប់បច្ចេកវិទ្យា Prompt Engineering

## ការណែនាំ
មូលដ្ឋានមេរៀននេះគ្របដណ្តប់ការស្រូបយល់ និងបច្ចេកទេសសំខាន់ៗ សម្រាប់បង្កើត prompt ដែលមានប្រសិទ្ធភាពក្នុងម៉ូដែល AI ជំនាញផលិត។ វិធីសាស្ដ្រដែលអ្នកសរសេរ prompt សម្រាប់ LLM ក៏មានសារៈសំខាន់ផងដែរ។ prompt ដែលត្រូវបានបង្កើតយ៉ាងប្រុងប្រយ័ត្នអាចឲ្យបានចម្លើយដែលមានគុណភាពល្អជាងមុន។ តើយ៉ាងដូចម្តេចដែលពាក្យបែបនេះ _prompt_ និង _prompt engineering_ មានន័យ? ហើយតើយើងអាចបង្កើត prompt ចូល (input) ដែលផ្ញើទៅ LLM ឲ្យប្រសើរឡើងដោយរបៀបណា? នេះជាសំណួរដែលយើងនឹងព្យាយាមឆ្លើយក្នុងជំពូកនេះ និងជំពូកបន្ទាប់។

_Generative AI_ អាចបង្កើតមាតិកាថ្មី (ដូចជា អត្ថបទ រូបភាព សំឡេង កូដ និងដូច្នេះ) ដើម្បីបំពេញតាមសំណើរបស់អ្នកប្រើ។ វាបានសម្រេចនេះដោយប្រើ _Large Language Models_ ដូចជាជួរប្រភេទ GPT របស់ OpenAI ("Generative Pre-trained Transformer") ដែលបានបណ្តុះបណ្តាលផ្អែកលើភាសាធម្មជាតិ និងកូដ។

ឥឡូវនេះ អ្នកប្រើអាចអន្តរកម្មជាមួយម៉ូដែលទាំងនេះតាមរយៈរបៀបជនបទដូចជា chat ដោយមិនចាំបាច់មានជំនាញបច្ចេកទេស ឬការបណ្តុះបណ្តាលណាមួយ។ ម៉ូដែលទាំងនេះគឺជា _prompt-based_ - អ្នកប្រើផ្ញើការបញ្ចូលអត្ថបទ (prompt) ហើយទទួលបានចម្លើយ AI (completion) តបតាម។ ពួកគេអាចបន្ត "chat ជាមួយ AI" ជាចំនួនជុំទូទៅ បន្ថែមតឹពេញលេញ prompt របស់ពួកគេសម្រាប់ធ្វើឲ្យចម្លើយសមស្របចំពោះការរំពឹងទុក។

"Prompt" បានក្លាយជាចំណុចប្រទាក់ _programming_ សំខាន់សម្រាប់កម្មវិធី generative AI ដើម្បីប្រាប់ម៉ូដែលថាអ្វីដែលត្រូវធ្វើ និងប៉ះពាល់ដល់គុណភាពនៃចម្លើយដែលត្រូវបានផ្តល់។ "Prompt Engineering" គឺជាវិស័យសិក្សាដែលកំពុងរីកចម្រើនយ៉ាងលឿនផ្ដោតលើ _ការរចនា និងធ្វើអោយមានប្រសិទ្ធិភាព_ នៃ prompt ដើម្បីផ្តល់ចម្លើយដែលមានគុណភាព និងឆាប់ចេញក្នុងកម្រិតធំ។

## គោលបំណងរៀន

ក្នុងមេរៀននេះ យើងនឹងរៀនអំពី Prompt Engineering គឺជាអ្វី ហើយហេតុអ្វីវាមានសារៈសំខាន់ និងវិធីសាស្ដ្ររចនា prompt ឲ្យមានប្រសិទ្ធភាពបំផុតសម្រាប់ម៉ូដែល និងគោលបំណងកម្មវិធីមួយ។ យើងនឹងយល់ពីគំនិតមូលដ្ឋាន និងអនុវត្តន៍ល្អៗសម្រាប់ prompt engineering - និងរៀនអំពីបរិស្ថានសម្រាប់សិក្សា Jupyter Notebooks "sandbox" ដែលអាចឲ្យយើងមើលឃើញការអនុវត្តន៍គំនិតទាំងនេះនៅលើឧទាហរណ៍ពិត។

នៅចុងបញ្ចប់មេរៀននេះ យើងនឹងអាចធ្វើបាន:

១. ពន្យល់អំពី prompt engineering និងហេតុអ្វីវាមានសារៈសំខាន់។
២. ពិពណ៌នាឧបករណ៍របស់ prompt និងរបៀបដែលវាត្រូវបានប្រើ។
៣. រៀនអនុវត្តខ្ពស់ និងបច្ចេកទេសសម្រាប់ prompt engineering។
៤. អនុវត្តបច្ចេកទេសដែលបានរៀនទៅឧទាហរណ៍ពិត ដោយប្រើ OpenAI endpoint។

## ពាក្យសំខាន់ៗ

Prompt Engineering: សកម្មភាពនៃការរចនា និងបន្តបន្ទាប់បញ្ចូលដើម្បីណែនាំម៉ូដែល AI ទៅរកការផលិតចម្លើយដែលបានចង់បាន។
Tokenization: ដំណើរការបំលែងអត្ថបទទៅជាឯកតាចម្រាស់តូចៗ ដែលហៅថា tokens ដើម្បីម៉ូដែលអាចយល់ និងពិនិត្យបាន។
Instruction-Tuned LLMs: ម៉ូដែលភាសាធំ (LLMs) ដែលត្រូវបានបណ្តុះបណ្តាលបន្ថែមជាមួយសេចក្តីណែនាំពិសេសដើម្បីបង្កើនភាពត្រឹមត្រូវ និងពាក់ព័ន្ធនៃចម្លើយ។

## សេវាសិក្សា Sandbox

Prompt engineering បច្ចុប្បន្នគឺជាអកុរសភាពធំជាងវិទ្យាសាស្ត្រ។ វិធីល្អបំផុតក្នុងការកែលម្អការយល់ដឹងនៃវា គឺត្រូវ _ហាត់ប្រាណ​តាម​អនុវត្ត​ន៍_ និងប្រើវិធីសាស្ដ្រជាប្រភេទសាកល្បងនិងកំហុសដែលរួមបញ្ចូលជាមួយជំនាញលើវិស័យការអនុវត្តន៍ និងបច្ចេកទេសផ្តល់អតិបរមា និងការផ្លាស់ប្តូរតាមម៉ូដែល។

សៀវភៅ Jupyter Notebook ដែលភ្ជាប់មកជាមួយមេរៀននេះផ្តល់បរិស្ថាន _sandbox_ ដែលអ្នកអាចសាកល្បងអ្វីដែលបានរៀនបាន - ប្រើលើកំណត់ផ្ទាល់ខ្លួន ឬជាផ្នែកនៃការប្រកួតកូដនៅចុងមេរៀន។ ដើម្បីអនុវត្ត វិធីសាស្ដ្រនេះ អ្នកត្រូវការ:

១. **កូនសោ Azure OpenAI API** - ចំណុចចេញសំរាប់ម៉ូដែល LLM ដែលបានប្រើប្រាស់។
២. **បរិយាកាស Python Runtime** - ដែលអាចដំណើរការ Notebook បាន។
៣. **អថេរបរិយាកាសក្នុងតំបន់បណ្ដាញ** - _បំពេញជំហាន [SETUP](./../00-course-setup/02-setup-local.md?WT.mc_id=academic-105485-koreyst) ពេលនេះដើម្បីត្រៀមខ្លួន_។

សៀវភៅ notebook ផ្តល់អនុស្សាវរីយ៍ _starter_ ប៉ុន្តែអ្នកត្រូវបានលើកទឹកចិត្ត ឲ្យបន្ថែមអត្ថបទ _Markdown_ (ពិពណ៌នា) និង _Code_ (សំណើ prompt) របស់អ្នក ដើម្បីសាកល្បងឧទាហរណ៍ និងគំនិតបន្ថែមជាច្រើន និងបង្កើតការយល់ដឹងល្អក្នុងការរចនា prompt។

## មេរៀនណែនាំជារូបភាព

ចង់ដឹងរូបមន្តធំៗនៃមេរៀននេះមុនចូលរួមទេ? សូមពិនិត្យមើលមេរៀនណែនាំជារូបភាពនេះ ដែលផ្តល់អារម្មណ៍ដល់អ្នកអំពីប្រធានបទសំខាន់ៗ និងចំណុចដែលត្រូវគិតក្នុងមួយចំណុច។ ផែនទីមេរៀននឹងដឹកនាំអ្នកពីការយល់ដឹង គំនិត និងបញ្ហាទៅរកការដោះស្រាយជាមួយបច្ចេកទេស prompt engineering និងអនុវត្តល្អៗ។ សូមចំណាំថា ផ្នែក "បច្ចេកទេសកម្រិតខ្ពស់" ក្នុងមេរៀននេះអشيرទៅដល់មាតិកាដែលគ្របដណ្តប់នៅជំពូកបន្ទាប់នៃកម្មវិធីសិក្សានេះ។

## កម្មវិធីស្តាតអាប់របស់យើង

ឥឡូវនេះ យើងនឹងនិយាយពី​ពីរបៀបដែល _ប្រធានបទនេះ_ មានទំនាក់ទំនងជាមួយបេសកកម្មអាជីវកម្មរបស់យើងក្នុងការនាំយកនូវ [ការបង្កើតថ្មី AI ទៅកាន់ការអប់រំ](https://educationblog.microsoft.com/2023/06/collaborating-to-bring-ai-innovation-to-education?WT.mc_id=academic-105485-koreyst)។ យើងចង់បង្កើតកម្មវិធីដែលបង្កើតដោយ AI សម្រាប់ _ការសិក្សាផ្ទាល់ខ្លួន_ - ដូច្នេះ យើងចង់គិតពីរបៀបអ្នកប្រើប្រាស់ផ្ទាល់ខ្លួននៃកម្មវិធីនឹង "រចនា" prompt ដូចខាងក្រោម៖

- **អ្នកគ្រប់គ្រង** អាចសួរបញ្ជាក់ AI ដើម្បី _វិភាគទិន្នន័យមាតិកាសិក្សាដើម្បីរកចន្លោះក្នុងការតំរង់_។ AI អាចសង្ខេបលទ្ធផល ឬបង្ហាញវាជាមួយកូដ។
- **គ្រូបង្រៀន** អាចសួរឲ្យ AI _បង្កើតផែនការសិក្សាដែលផ្តោតលើគោលដៅ និងប្រធានបទមួយ_។ AI អាចបង្កើតផែនការផ្ទាល់ខ្លួនក្នុងទ្រង់ទ្រាយកំណត់។
- **សិស្ស** អាចសួរឲ្យ AI _បង្រៀនមុខវិជ្ជាលំបាកមួយ_។ AI អាចណែនាំសិស្សជាមួយមេរៀន គន្លឹះ និងឧទាហរណ៍ដែលអាចសម្រួលតាមកម្រិតរបស់ពួកគេ។

នេះគឺគ្រាន់តែជា​ចំណុចខ្លីបង្ហាញពីអ្វីដែលអាចធ្វើទៅបាន។ សូមពិនិត្យមើល [Prompts For Education](https://github.com/microsoft/prompts-for-edu/tree/main?WT.mc_id=academic-105485-koreyst) - បណ្ណាល័យ prompt បើកចំហរដែលបានរៀបចំដោយអ្នកជំនាញក្នុងវិស័យអប់រំ - ដើម្បីបានមើលឃើញវិធីសាស្ត្រផ្សេងៗ! _សាកល្បងដំណើរការ prompt ទាំងនោះក្នុង sandbox ឬក្នុង OpenAI Playground ដើម្បីមើលអ្វីកើតមាន!_

## តើ Prompt Engineering មានអ្វីខ្លះ?

យើងបានចាប់ផ្តើមមេរៀននេះដោយកំណត់ពាក្យ **Prompt Engineering** គឺជាដំណើរការ _រចនា និងបន្ថែមប្រសិទ្ធភាព_ ការបញ្ចូលអត្ថបទ (prompt) ដើម្បីផ្តល់ចម្លើយដែលមានគុណភាព និងមានលំដាប់សម្រាប់គោលបំណងកម្មវិធី និងម៉ូដែលមួយ។ យើងអាចគិតមើលវាជាដំណាក់កាល ២ជំហាន៖

- _រចនា_ prompt ដើមសម្រាប់ម៉ូដែល និងគោលបំណងមួយ
- _បន្តបន្ទាប់_ prompt ដោយរំលែក ជាបណ្ដោះអាសន្នក្នុងការកែលម្អគុណភាពចម្លើយ

នេះគឺជាដំណើរការសាកល្បងនិងកំហុសដែលត្រូវការចំណេះដឹង និងកិច្ចខិតខំសំរាប់ទទួលលទ្ធផលល្អបំផុត។ ហេតុអ្វីវាមានសារៈសំខាន់? ដើម្បីឆ្លើយសំណួរនេះ អ្នកត្រូវយល់ពីគំនិតបីដូចខាងក្រោម៖

- _Tokenization_ = របៀបដែលម៉ូដែល "មើល" prompt
- _Base LLMs_ = របៀបដែលម៉ូដែលមូលដ្ឋាន "ដំណើរការ" prompt
- _Instruction-Tuned LLMs_ = របៀបដែលម៉ូដែលមើលឃើញ "ភារកិច្ច"

### Tokenization

ម៉ូដែល LLM មើលឃើញ prompt ជា _លំដាប់អក្សរ_ ដែលម៉ូដែលផ្សេងៗ (ឬកំណែផ្សេងៗនៃម៉ូដែល) អាច tokenization prompt ដូចគ្នាផ្សេងគ្នាទៅបាន។ ពីព្រោះ LLM ត្រូវបានបណ្តុះដោយ tokens (មិនមែនអត្ថបទដើម) របៀបនៃការបំបែក prompt ទៅជាឯកតា token មានផ្លែប៉ះយ៉ាងត្រង់ទៅលើគុណភាពនៃចម្លើយដែលផលិតបាន។

ដើម្បីបានការយល់ដឹងពីរបៀប tokenization ធ្វើការ សូមសាកល្បងឧបករណ៍ដូចជា [OpenAI Tokenizer](https://platform.openai.com/tokenizer?WT.mc_id=academic-105485-koreyst) ដែលមានរូបភាពខាងក្រោម។ ត្រូចចម្លើយរបស់អ្នក - ហើយមើលថាវាត្រូវបានបម្លែងទៅជាគ្រឿង token ដូចម្តេច ហើយសម្គាល់ពីរបៀបពណ៌នាតួរ និង យីហោតែងត្រូវបានដំណើរការ។ ចំណាំថាឧទាហរណ៍នេះបង្ហាញ LLM ដែលចាស់ (GPT-3) - ហេតុដូច្នេះការសាកល្បងជាមួយម៉ូដែលថ្មីអាចផ្តល់លទ្ធផលខុសគ្នា។

![Tokenization](../../../translated_images/km/04-tokenizer-example.e71f0a0f70356c5c.webp)

### គំនិត៖ ម៉ូដែលមូលដ្ឋាន

ពេលដែល prompt ត្រូវបាន tokenization រួច មុខងារ​សំខាន់របស់ ["Base LLM"](https://blog.gopenai.com/an-introduction-to-base-and-instruction-tuned-large-language-models-8de102c785a6?WT.mc_id=academic-105485-koreyst) (ឬម៉ូដែលមូលដ្ឋាន) គឺទាយ tokens នៅក្នុងលំដាប់នោះ។ ពីព្រោះ LLM ត្រូវបានបណ្តុះបណ្តាលលើដាតាបេសអត្ថបទធំៗ ជាមួយនឹងទំនាក់ទំនងស្ថិតិរវាង token ពួកគេចេះប៉ាន់ប្រមាណលទ្ធផលជាមួយភាពជឿជាក់មួយ។ ចំណាំថាពួកគេមិនយល់ពី _ន័យ_ នៃពាក្យក្នុង prompt ឬ token ទេ; ពួកគេចែកឃើញលំដាប់ដែលអាច "បញ្ចប់" ជាមួយការទាយបន្ទាប់របស់ពួកគេ។ ពួកគេចេះបន្តទាយលំដាប់រហូតដល់ការបញ្ឈប់ដោយការចូលមកចាប់បញ្ហារបស់អ្នកប្រើ ឬលក្ខខណ្ឌធម្មតា។

ចង់ឃើញរបៀប completion ដោយមូលដ្ឋាន prompt ត្រូវធ្វើដូចម្តេចមែនទេ? សូមបញ្ចូល prompt ខាងលើទៅក្នុង [Microsoft Foundry playground](https://ai.azure.com?WT.mc_id=academic-105485-koreyst) ជាមួយកំណត់តម្លៃលំនាំដើម។ ប្រព័ន្ធត្រូវបានកំណត់ឲ្យទទួល prompt ជាសំណើសម្រាប់ព័ត៌មាន - ដូច្នេះអ្នកគួរតែទទួលបានចម្លើយដែលបំពេញតាមបរិបទនេះ។

ប៉ុន្តែប្រសិនបើអ្នកប្រើចង់ឲ្យឃើញអ្វីមួយជាក់លាក់ដែលបំពេញតាមលក្ខខណ្ឌឬគោលបំណងភារកិច្ច? នេះជា​កន្លែងដែល _Instruction-tuned_ LLMs ចូលរួម។

### គំនិត៖ Instruction Tuned LLMs

[Instruction Tuned LLM](https://blog.gopenai.com/an-introduction-to-base-and-instruction-tuned-large-language-models-8de102c785a6?WT.mc_id=academic-105485-koreyst) ចាប់ផ្តើមពីម៉ូដែលមូលដ្ឋាន ហើយបង្ហាញវាជាមួយឧទាហរណ៍ឬគូថ្មី (input/output pairs) (ដូចជា "message" ច្រើនជុំ) ដែលមានសេចក្តីណែនាំច្បាស់ - ហើយចម្លើយពី AI ព្យាយាមអនុវត្តន៍សេចក្តីណែនាំនោះ។

វា​ប្រើ​បច្ចេកទេសដូចជា Reinforcement Learning with Human Feedback (RLHF) ដែលអាចបណ្តុះម៉ូដែលឲ្យ _អនុវត្តតាមសេចក្តីណែនាំ_ និង _រៀនពីមតិ​ឆ្លើយតប_ ដើម្បីផ PRODUCE responses ដែលសមរម្យនឹងការ អនុវត្តតាមការពិត និងពាក់ព័ន្ធនឹងគោលបំណងអ្នកប្រើ។

យើង​សាកល្បងវា - ត្រលប់ទៅ prompt ខាងលើ ប៉ុន្តែផ្លាស់ប្ដូរ _system message_ ដើម្បីផ្តល់សេចក្តីណែនាំដូចខាងក្រោមជាបរិបទ៖

> _សង្ខេបមាតិកាដែលអ្នកបានផ្តល់សម្រាប់សិស្សថ្នាក់ទី២។ រក្សាលទ្ធផលក្នុងមួយកថា មានចំណុចបញ្ចូល ៣-៥។_

មើលថាលទ្ធផលត្រូវបានកែប្រែ និងស្របតាមគោលបំណង និងទ្រង់ទ្រាយរបស់អ្នកដឹងទេ? គ្រូបង្រៀនអាចប្រើតែចម្លើយនេះដោយផ្ទាល់សម្រាប់ការបង្ហាញសម្រាប់ថ្នាក់នោះ។

## ហេតុអ្វីយើងត្រូវការបច្ចេកវិទ្យា Prompt Engineering?

ឥឡូវនេះដែលយើងបានយល់ពីរបៀបដែល prompt ត្រូវបំលែងដោយ LLM យើងនិយាយអំពី _ហេតុអ្វី_ ដែលយើងត្រូវការបច្ចេកវិទ្យា prompt engineering។ ចម្លើយគឺដោយសារតែ LLM សព្វថ្ងៃមានបញ្ហាច្រើនដែលធ្វើឲ្យ _ការបញ្ចប់ដែលទុកចិត្តបាន និងមានសភាពរឹងរូស_ មានភាពមានបញ្ហាប្រឈមសម្រាប់ទទួលបានលទ្ធផលល្អជាមុន។ ឧទាហរណ៍៖

១. **ចម្លើយម៉ូដែលមានលក្ខណៈ stochastic។** _prompt ដូចគ្នា_ អាចបង្កើតចម្លើយខុសគ្នាជាមួយម៉ូដែល ឬកំណែម៉ូដែលផ្សេងៗ។ ហើយវាអាចផ្តល់លទ្ធផលខុសគ្នា ជាមួយម៉ូដែល _ដូចគ្នា_ នៅពេលខុសៗគ្នា។ _បច្ចេកវិទ្យា prompt engineering អាចជួយយើងបន្ថយការប្រែប្រួលនេះដោយផ្តល់ណែនាំល្អជាង_។

១. **ម៉ូដែលអាចចេញចម្លើយបង្កើតស្រឡាយ។** ម៉ូដែលត្រូវបណ្តុះជាមួយ _ដាតាបេសធំ ប៉ុន្តែមិនអស់កំណត់_ ដែលមានបំណងហេតុខ្វះហេតុអ្វីខុសពីការបង្រៀន។ ដូច្នេះវាអាចបង្កើតចម្លើយដែលមិនត្រឹមត្រូវ មានការស្រឡាយ ឬផ្ទុយមុខខុសតម្លៃមួយចំនួននៃការពិត។ _បច្ចេកវិទ្យា prompt engineering ជួយអ្នកប្រើបញ្ជាក់ និងកាត់បន្ថយការស្រឡាយ ដូចជា ដាក់សំណួរឲ្យ AI អត្ថបទឯកសារយោង ឬហេតុផល_។

១. **សមត្ថភាពម៉ូដែលមានភាពខុសគ្នា។** ម៉ូដែលថ្មី ឬជំនាន់ម៉ូដែលថ្មីៗ មានសមត្ថភាពច្រើនជាង តែប្រៀបប្រដូចជាក៏នាំមកនូវលក្ខណៈពិសេស និងការជួបប្រទះព្រោះទៅលើការចំណាយ & ស្មុគស្មាញផងដែរ។ _prompt engineering ជួយយើងបង្កើតវិធីសាស្ដ្រ និងដំណើរការចំណាយពេលបំពេញឱ្យមានប្រសិទ្ធភាព និងអាចប្រើប្រាស់បានលើម៉ូដែលដែលមានតម្រូវការ វិធីសាស្ត្រជាក់លាក់ក្នុងវិស័យគ្រប់គ្រងបានយ៉ាងល្អ_។

យើងមកមើលវាទៅក្នុង OpenAI ឬ Azure OpenAI Playground៖

- ប្រើ prompt ដូចគ្នាជាមួយ DNS LLM ខុសៗគ្នា (ដូចជា OpenAI, Azure OpenAI, Hugging Face) - តើអ្នកបានឃើញការប្រែប្រួលទេ?
- ប្រើ prompt ដូចគ្នាដោយបន្តធ្វើជាមួយ LLM deployment _ដូចគ្នា_ (ឧទាហរណ៍ Azure OpenAI playground) - តើការផ្លាស់ប្តូរទាំងនេះខុសគ្នាយ៉ាងដូចម្តេច?

### ឧទាហរណ៍ Fabrications

ក្នុងវគ្គសិក្សានេះ យើងប្រើពាក្យ **"fabrication"** ដើម្បីសំដៅឲ្យដឹងពីស្ថានភាពដែល LLM ពេលខ្លះបង្កើតព័ត៌មានខុសដោយសារ​កំណត់ក្នុងការបង្រៀន ឬការកំណត់ផ្សេងៗ។ អ្នកក៏ប្រហែលអានសំដៅទៅលើវាជា _"hallucinations"_ ក្នុងអត្ថបទពេញនិយម ឬឯកសារស្រាវជ្រាវ។ ទោះជា។យើងខិតខំប្រើពាក្យ _"fabrication"_ ដើម្បីបញ្ជាក់ថាវាពុំបង្ហាញមនុស្សជាអ្នកបង្កើត ត្រូវការប្រើពាក្យនេះដើម្បីធ្វើឲ្យមានភាពទំនើបគ្នាពីទិដ្ឋភាពពាក្យពេញកំហុស ហើយមានតម្លាភាពត្រឹមត្រូវជាងការភ្ជាប់ជាមួយរឿងរ៉ាវមនុស្សផងដែរ។ នេះក៏ជាការពង្រឹង [មគ្គុទេសក៍ AI ទទួលខុសត្រូវ](https://www.microsoft.com/ai/responsible-ai?WT.mc_id=academic-105485-koreyst) ពីទស្សនៈពាក្យផ្សេងៗ ដោយលុបចោលពាក្យដែលអាចមានអត្ថន័យដូចជាគ្រោះថ្នាក់ ឬមិនរួមបញ្ចូលក្នុងបរិបទខ្លះ។

ចង់យល់ពីរបៀបដែលការសរសេរផលិតបានត្រូវកើតឡើង? សូមគិតពី prompt ដែលណែនាំ AI ដើម្បីបង្កើតមាតិកាក្នុងប្រធានបទមិនមានជាមួយ (ដំណើរការជៀសវាងពួកវាមិនមានក្នុង dataset បង្រៀន)។ ឧទាហរណ៍ - ខ្ញុំបានសាកល្បង prompt នេះ៖

## ការរៀបចំជម្រាប

យើងបានឃើញហេតុអ្វីបានជាការរចនាជម្រាបជារឿងសំខាន់ - ឥឡូវនេះយើងចង់យល់ពីរបៀបដែលជម្រាបត្រូវបាន _រៀបចំ_ ដូច្នេះយើងអាចបាយតម្លៃបច្ចេកទេសផ្សេងៗសម្រាប់ការរចនាជម្រាបដែលមានប្រសិទ្ធភាព។

### ជម្រាបមូលដ្ឋាន

ចាប់ផ្តើមជម្រាបមូលដ្ឋាន៖ អត្ថបទសូមបញ្ចូលទៅម៉ូដែលដោយគ្មានបរិបទផ្សេងទៀត។ នេះជាឧទាហរណ៍ - នៅពេលយើងផ្ញើពាក្យចាប់ផ្តើមរបស់អំនួតជាតិសហរដ្ឋអាមេរិកទៅកាន់ OpenAI [Completion API](https://platform.openai.com/docs/api-reference/completions?WT.mc_id=academic-105485-koreyst) វាធ្វើការ _បញ្ចប់_ ការឆ្លើយតបដោយខ្លួនឯងជាមួយបន្ទាត់បន្ទាប់បន្តិច ប្រើសំនួរជា click-to-continue ដើម្បីខណៈពេលវាបង្ហាញសមត្ថភាពច្រៀងវាគ្មិនមូលដ្ឋាន។

### ជម្រាបស្មុគស្មាញ

ឥឡូវនេះយើងបន្ថែមបរិបទនិងសេចក្តីណែនាំទៅក្នុងជម្រាបមូលដ្ឋាននោះ។ [Chat Completion API](https://learn.microsoft.com/azure/ai-services/openai/how-to/chatgpt?WT.mc_id=academic-105485-koreyst) អនុញ្ញាតឲ្យយើងរៀបចំជម្រាបស្មុគស្មាញជា ការប្រមូលផ្តុំ _សារ_ ជាមួយ៖

- គូអត្ថបទ/ចេញបង្ហាញបញ្ចេញសំរាប់ការបញ្ចូលអ្នកប្រើ និងការឆ្លើយតបរបស់ជំនួយការ។
- សេចក្តីសាររបស់ប្រព័ន្ធកំណត់បរិបទសម្រាប់ភាពប្រតិបត្តិរបស់ជំនួយការ ឬ បុគ្គលិកលក្ខណៈ។

សំណើឥឡូវនេះមានទំាងនេះ ដែល _tokenization_ ចាប់យកព័ត៌មានពាក់ព័ន្ធពីបរិបទ និងការសន្ទនា។ ឥឡូវនេះ ការប្រែប្រាស់បរិបទប្រព័ន្ធអាចមានឥទ្ធិពលដូចជាគុណភាពនៃការបញ្ចប់ ពីបានបញ្ចូលរបស់អ្នកប្រើ។

```python
response = client.responses.create(
    model="gpt-4o-mini",
    input=[
        {"role": "system", "content": "You are a helpful assistant."},
        {"role": "user", "content": "Who won the world series in 2020?"},
        {"role": "assistant", "content": "The Los Angeles Dodgers won the World Series in 2020."},
        {"role": "user", "content": "Where was it played?"}
    ]
)
```

- **ឧទាហរណ៍** - មិនបានប្រាប់ម៉ូដែលថាត្រូវធ្វើអ្វីជាមួយសេចក្តីណែនាំប៉ុណ្ណោះទេ ថែមទាំងផ្តល់ឧទាហរណ៍អំពីអ្វីដែលត្រូវធ្វើ ហើយអោយវាគណនាលំនាំ។
- **សន្លឹក** - ដើមសេចក្តីណែនាំជាមួយ "សន្លឹក" មួយដែលជារូបមន្តសម្រាប់ការបញ្ចប់ ដើម្បីណែនាំម៉ូដែលឲ្យឆ្លើយតបដូចគ្នា។
- **គំរូ** - នេះជារូបមន្តដែលអាចធ្វើម្ដងទៀតបានសម្រាប់ជម្រាបដែលមានអ្នកបញ្ចូល (អថេរ) ដែលអាចខ្លួនឯងបានយកព័ត៌មានសម្រាប់ករណីប្រើប្រាស់ជាក់លាក់។

### ការប្រើប្រាស់ឧទាហរណ៍

នេះជាវិធីដែលអ្នកប្រើមាតិកាគោល ដើម្បី "ផ្តល់ឧទាហរណ៍ដល់ម៉ូដែល" សម្រាប់លទ្ធផលដែលចង់បានសម្រាប់ការណែនាំជាក់លាក់ ហើយអោយវាកំណត់លំនាំសម្រាប់លទ្ធផលដែលចង់បាន។ ជាមូលដ្ឋានលើចំនួនឧទាហរណ៍ដែលបានផ្តល់ យើងអាចមានការជម្រាប zero-shot, one-shot, few-shot ល្បីៗ។

### គំរូសំណួរ

គំរូសំណួរគឺជាកម្មវិធីមួយដែលបានកំណត់ជាមុនសម្រាប់សំណួរ ដែលអាចរក្សាទុកនិងប្រើឡើងវិញតាមតម្រូវការដើម្បីជំរុញបទពិសោធន៍អ្នកប្រើប្រាស់ដែលមានភាពសមរម្យនៅលើវិសាលភាព។ ក្នុងរូបមន្តសាមញ្ញបំផុត វាជាការប្រមូលផ្តុំយ៉ាងសាមញ្ញនៃគំរូសំណួរដូចជា [នេះពី OpenAI](https://cookbook.openai.com/examples/gpt4-1_prompting_guide?WT.mc_id=academic-105485-koreyst) ដែលផ្តល់ទាំងធាតុលេងចំណុចសំណួរជាអន្តរកម្ម (សារអ្នកប្រើ និងប្រព័ន្ធ) និងទ្រង់ទ្រាយសំណើ API ដើម្បីគាំទ្រការប្រើឡើងវិញ។

នៅក្នុងរូបមន្តស្មុគស្មាញជាងនេះដូចជា [ឧទាហរណ៍ពី LangChain](https://python.langchain.com/docs/concepts/prompt_templates/?WT.mc_id=academic-105485-koreyst) វាបង្ហាប់ _កន្លែងដាក់_ ដែលអាចប្ដូរជាមួយទិន្នន័យពីប្រភពផ្សេងៗ (អ្នកប្រើ, បរិបទប្រព័ន្ធ, ប្រភពទិន្នន័យខាងក្រៅ) ដើម្បីបង្កើតសំណួរជាចលនា។ នេះអាចអនុញ្ញាតឱ្យយើងបង្កើតបណ្ណាល័យសំណួរដែលអាចប្រើឡើងវិញ ដើម្បីជំរុញបទពិសោធន៍អ្នកប្រើប្រាស់ដែលមានភាពសមរម្យ **បានដោយកម្មវិធី** នៅលើវិសាលភាព។

ចុងក្រោយ មានតម្លៃពិតប្រាកដនៃគំរូនៅក្នុងសមត្ថភាពបង្កើត និងបោះពុម្ព _បណ្ណាល័យសំណួរ_ សម្រាប់ដែនដឹកនាំកម្មវិធីថ្នាក់ជ្រុង - ដែលគំរូសំណួរបច្ចុប្បន្នត្រូវបាន _បង្រួមបញ្ចូល_ ដើម្បីបញ្ចូលបរិបទពាក់ព័ន្ធជាកម្មវិធីឬឧទាហរណ៍ដែលធ្វើឱ្យចម្លើយមានភាពសមរម្យ និងត្រឹមត្រូវសម្រាប់សាធារណជនអ្នកប្រើប្រាស់គោលដៅ។ បណ្ណាល័យ [Prompts For Edu](https://github.com/microsoft/prompts-for-edu?WT.mc_id=academic-105485-koreyst) ជាឧទាហរណ៍ល្អនៃវិធីនេះ ដែលជ្រើសរើសបណ្ណាល័យសំណួរសម្រាប់ដែនអប់រំ ដោយផ្ដោតលើគោលបំណងសំខាន់ៗដូចជា ការធ្វើផែនការថ្នាក់, ការរចនាសមាសបក្ត្រ, ការបង្រៀនសិស្ស។

## អត្ថបទគាំទ្រ

ប្រសិនបើយើងគិតពីការបង្កើតសំណួរជាការមានការណែនាំ (ភារៈកិច្ច) និងគោលដៅ (មាតិកាចម្បង) បន្ទាប់មក _មាតិកាមួយជាន់​ទីពីរ_ គឺដូចជាបរិបទបន្ថែមដែលយើងផ្តល់ដើម្បី **ប៉ះពាល់ចេញ​ផល**។ វាអាចជាផលប៉ះពាល់សម្រួលនានា, សេចក្តីណែនាំទ្រង់ទ្រាយ, បញ្ជីប្រធានបទជាដើម ដែលអាចជួយម៉ូឌែល _ប្តូរទម្រង់ចម្លើយ_ ឱ្យសមរម្យនឹងគោលដៅ ឬចំណង់ចំណូលចិត្តរបស់អ្នកប្រើ។

ឧទាហរណ៍៖ ភាគយោងពីកាតាឡុកខួបសិក្សាមួយដែលមានទិន្នន័យពហុមាតិកា (ឈ្មោះ, សេចក្ដីពិពណ៌នា, កម្រិត, ស្លាកពហុមាតិច, គ្រូបង្រៀនជាដើម) លើវគ្គសិក្សាទាំងអស់នៅក្នុងកម្មវិធីសិក្សា៖

- យើងអាចកំណត់ការណែនាំថា "សង្ខេបកាតាឡុកវគ្គសិក្សាសម្រាប់រដូវរងា ឆ្នាំ ២០២៣"
- យើងអាចប្រើមាតិកាចម្បងដើម្បីផ្តល់ឧទាហរណ៍បញ្ចេញចេញបង្រៀនចុងក្រោយ
- យើងអាចប្រើមាតិកាមួយជាន់​ទីពីរដើម្បីកំណត់ស្លាក ៥ នៃចំណាប់អារម្មណ៍

ឥឡូវម៉ូឌែលអាចផ្តល់សង្ខេបនៅក្នុងទ្រង់ទ្រាយដែលបានបង្ហាញដោយឧទាហរណ៍, ប៉ុន្តែនៅពេលលទ្ធផលមានស្លាកជាច្រើន វាអាចអាទិភាពទៅលើស្លាក ៥ ដែលបានកំណត់ក្នុងមាតិកាមួយជាន់​ទីពីរ។

## វិធីសាស្រ្តល្អបំព្រងសំណួរ

ឥឡូវដែលយើងដឹងថារូបមន្តសំណួរអាច _បង្កើត_, យើងអាចចាប់ផ្តើមគិតពីរបៀប _រចនា_ វាឱ្យបានត្រឹមត្រូវបំផុតនៃការអនុវត្តល្អ។ យើងអាចគិតពីវា ជាពីរផ្នែក - មាន _ចិត្តដូចម្តេច_ និងអនុវត្ត _បច្ចេកទេស_ ត្រឹមត្រូវ។

### ចិត្តដូចម្តេចនៃបច្ចេកទេសសំណួរ

បច្ចេកទេសសំណួរជាជំហានសាកល្បង-ខុស, ដូច្នេះត្រូវរក្សាកត្តាសំខាន់បីនេះក្នុងចិត្ត៖

1. **ការយល់ដឹងពីដែនសកម្មភាពសំខាន់។** ត្រឹមត្រូវនៃចម្លើយ និងភាពសមរម្យកើតមានដោយផ្អែកលើ _ដែនសកម្មភាព_ ដែលកម្មវិធី ឬអ្នកប្រើកំពុងដំណើរការ។ អនុវត្តការយុទ្ធសាស្ត្ររឹតបន្តឹងបន្ថែម ជាមួយការយល់ដឹង និងជំនាញក្នុងជំនាញនោះ។ ឧទាហរណ៍ កំណត់ _បុគ្គលិកលក្ខណៈដែនជាក់លាក់_ ក្នុងសំណួររបស់ប្រព័ន្ធ ឬប្រើ _រូបមន្តដែនជាក់លាក់_ ក្នុងសំណួរអ្នកប្រើ។ ផ្តល់មាតិកាជាន់​ទីពីរដែលបញ្ចេញបរិបទដែនជាក់លាក់ ឬប្រើ _សញ្ញានិងឧទាហរណ៍ដែនជាក់លាក់_ ដើម្បីណែនាំម៉ូឌែលទៅកាន់គំរូប្រើប្រាស់ដែលស្គាល់។

2. **ការយល់ដឹងម៉ូឌែលសំខាន់។** យើងដឹងថាម៉ូឌែលគឺមានលក្ខណៈតេស្តស្ថិតិ។ ប៉ុន្តែការអនុវត្តម៉ូឌែលអាចខុសគ្នានៅក្នុងទិន្នន័យហ្វឹកហាត់ (ចំណេះដឹងពីមុន), សមត្ថភាពដែលផ្តល់ (ឧទាហរណ៍, តាម API ឬ SDK), និងប្រភេទមាតិកាដែលម៉ូឌែលបង្កើតឡើងឲ្យមានប្រសិទ្ធភាព (ឧទាហរណ៍៖ កូដ, រូបភាព, ឬអត្ថបទ)។ យល់ដឹងពីកម្លាំងនិងកំណត់ខ្ទង់កំណត់របស់ម៉ូឌែលដែលអ្នកកំពុងប្រើ ហើយប្រើចំណេះដឹងនោះដើម្បី _អាទិភាពការងារ_ ឬបង្កើត _រូបមន្តតាមបំណង_ ដែលអាចបង្កើតបាននៅសមត្ថភាពម៉ូឌែល។

3. **ការចាប់ផ្តើមសាកល្បង និងការពិនិត្យសម្រួលសំខាន់។** ម៉ូឌែលកំពុងធ្វើអោយមានការកែលម្អយ៉ាងរហ័ស ហើយបច្ចេកទេសសំណួរជំនួយក៏ដូចគ្នា។ ជាអ្នកជំនាញដែនសកម្មភាព អ្នកអាចមានបរិបទផ្សេង ឬក្របខ័ណ្ឌច្បាស់លាស់សម្រាប់កម្មវិធីរបស់អ្នក ដែលគ្មាននៅក្នុងសហគមន៍ទាំងមូល។ ប្រើឧបករណ៍ និងបច្ចេកទេសសំណួរជំនួយដើម្បីចាប់ផ្តើមបង្កើតសំណួរ បន្ទាប់មកសាកល្បង និងបញ្ចាក់លទ្ធផលដោយការចេះដឹង និងជំនាញក្នុងដែនសះរបស់អ្នក។ កត់ត្រាចំណេះដឹងរបស់អ្នក និងបង្កើត **មូលដ្ឋានចំណេះដឹង** (ឧទាហរណ៍៖ បណ្ណាល័យសំណួរ) ដែលអាចប្រើជាស្ដង់ដាថ្មីសម្រាប់អ្នកផ្សេងទៀត ដើម្បីសំចាត់ដំណើរការ និងជំរុញប្រសិទ្ធភាពលឿនក្នុងអនាគត។

## ភារកិច្ច

សូមអបអរសាទរ! អ្នកបានឈានដល់ចុងបញ្ចប់មេរៀនហើយ! ពេលនេះគឺម៉ោងដើម្បីដាក់បញ្ហារបស់គំនិត និងបច្ចេកទេសទាំងនោះជាការប្រឡងជាក់ស្តែង!

សម្រាប់ភារកិច្ចរបស់យើង យើងនឹងប្រើ Jupyter Notebook ដែលមានការហាត់ប្រាណដែលអ្នកអាចបំពេញបានដោយផ្ទាល់។ អ្នកក៏អាចបន្តបន្ថែមសាច់រឿង Markdown និងកូដក្នុង Notebook ដើម្បីស្វែងយល់ពីគំនិត និងបច្ចេកទេសដោយខ្លួនឯង។

### ដើម្បីចាប់ផ្តើម សូម fork គម្រោង បន្ទាប់មក

- (ណែនាំ) បើក GitHub Codespaces
- (ជំនួស) តំឡើងគម្រោងទៅឧបករណ៍របស់អ្នក និងប្រើវាជាមួយ Docker Desktop
- (ជំនួស) បើក Notebook ជាមួយបរិស្ថានដំណើរការ Notebook ផ្ទាល់ខ្លួន។

### បន្ទាប់មក កំណត់អថេរបរិយាកាសរបស់អ្នក

- ចម្លងឯកសារ `.env.copy` នៅក្នុងដើមគម្រោងទៅ `.env` ហើយបំពេញតម្លៃ `AZURE_OPENAI_API_KEY`, `AZURE_OPENAI_ENDPOINT` និង `AZURE_OPENAI_DEPLOYMENT`។ ត្រឡប់ទៅផ្នែក [Learning Sandbox](#សេវាសិក្សា-sandbox) ដើម្បីរៀនពីរបៀបធ្វើ។

### បន្ទាប់មក បើក Jupyter Notebook

- ជ្រើសរើសដំណើរការល្បឿន kernel។ ប្រសិនបើប្រើជម្រើសលេខ 1 ឬ 2 គ្រាន់តែជ្រើសរើស Python 3.10.x kernel ដាច់ដោយឡែកដែលផ្តល់ដោយ dev container។

អ្នកបានរួចរាល់ក្នុងការរត់ការហាត់ប្រាណ។ សូមចំណាំថាគ្មានចម្លើយ _ត្រឹមត្រូវ ឬមិនត្រឹមត្រូវ_ នៅទីនេះទេ - គ្រាន់តែ ស្វែងយល់ជំរុញដោយការសាកល្បង-ខុស និងកសាងចំណេះដឹងដែលមានប្រសិទ្ធភាពសម្រាប់ម៉ូឌែល និងដែនកម្មវិធីណាមួយ។

## ការត្រួតពិនិត្យចំណេះដឹង

តើតើយ៉ាងណាជាសំណួរល្អដែលអនុវត្តវិធីសាស្រ្តល្អពាក់ព័ន្ធ?

1. បង្ហាញក្រុមរូបភាពរថយន្តក្រហមមួយ
2. បង្ហាញក្រុមរូបភាពរថយន្តក្រហមមានម៉ាក Volvo និងម៉ូដែល XC90 ចតនៅជាប់ដងអណ្តោយជាមួយព្រិលព្រះអាទិត្យលិច
3. បង្ហាញក្រុមរូបភាពរថយន្តក្រហមមានម៉ាក Volvo និងម៉ូដែល XC90

ការឆ្លើយ A៖ 2 គឺជាសំណួរល្អបំផុត ព្រោះវាបញ្ជាក់លម្អិតពី "អ្វី" និងលម្អិតជាក់លាក់ (មិនមែនការរថយន្តទូទៅនៅទីណាមួយទេ ប៉ុន្តែមកពីម៉ាក និងម៉ូដែលជាក់លាក់) និងវាក៏ពណ៌នាលម្អិតទាំងមូលទៀតផង។ 3 ជាជម្រើសល្អបន្ទាប់ ព្រោះវាក៏មានការពិពណ៌នាច្រើនដែរ។
