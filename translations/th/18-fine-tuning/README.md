# การปรับแต่ง LLM ของคุณ

การใช้โมเดลภาษาใหญ่เพื่อสร้างแอปพลิเคชัน AI เชิงสร้างสรรค์มาพร้อมกับความท้าทายใหม่ๆ ปัญหาหลักคือการรับประกันคุณภาพของการตอบกลับ (ความถูกต้องและความเกี่ยวข้อง) ในเนื้อหาที่โมเดลสร้างขึ้นตามคำขอของผู้ใช้ ในบทเรียนก่อนหน้านี้ เราได้พูดถึงเทคนิคต่างๆ เช่น การออกแบบพรอมต์และการสร้างผลลัพธ์แบบเสริมการดึงข้อมูล ที่พยายามแก้ปัญหาโดยการ _แก้ไขอินพุตพรอมต์_ ที่ส่งเข้าโมเดลเดิม

ในบทเรียนวันนี้ เราจะพูดถึงเทคนิคที่สาม คือ **การปรับแต่ง (fine-tuning)** ซึ่งพยายามแก้ไขความท้าทายนี้โดยการ _ฝึกโมเดลใหม่_ ด้วยข้อมูลเพิ่มเติม มาดูรายละเอียดกัน

## วัตถุประสงค์การเรียนรู้

บทเรียนนี้นำเสนอแนวคิดของการปรับแต่งสำหรับโมเดลภาษาที่ผ่านการฝึกมาแล้ว พร้อมทั้งสำรวจประโยชน์และความท้าทายของวิธีนี้ และให้คำแนะนำว่าเมื่อใดและอย่างไรควรใช้การปรับแต่งเพื่อปรับปรุงประสิทธิภาพของโมเดล AI เชิงสร้างสรรค์ของคุณ

เมื่อจบบทเรียนนี้ คุณควรสามารถตอบคำถามต่อไปนี้ได้:

- การปรับแต่งสำหรับโมเดลภาษา คืออะไร?
- เมื่อใด และทำไม การปรับแต่งจึงมีประโยชน์?
- ฉันจะปรับแต่งโมเดลที่ผ่านการฝึกมาแล้วได้อย่างไร?
- ข้อจำกัดของการปรับแต่งมีอะไรบ้าง?

พร้อมไหม? มาเริ่มกันเลย

## ไกด์ภาพประกอบ

## การปรับแต่งสำหรับโมเดลภาษา คืออะไร?

ตามคำจำกัดความ โมเดลภาษาใหญ่จะถูก _ฝึกล่วงหน้า_ ด้วยข้อความจำนวนมากจากแหล่งที่หลากหลายรวมถึงอินเทอร์เน็ต ตามที่เราได้เรียนรู้ในบทก่อนหน้า เราต้องใช้เทคนิคอย่าง _การออกแบบพรอมต์_ และ _การสร้างผลลัพธ์เสริมการดึงข้อมูล_ เพื่อปรับปรุงคุณภาพการตอบกลับของโมเดลต่อคำถามของผู้ใช้ ("พรอมต์")

เทคนิคการออกแบบพรอมต์ยอดนิยมคือการให้คำแนะนำกับโมเดลมากขึ้นในสิ่งที่คาดหวังในคำตอบ โดยการให้ _คำแนะนำ_ (คำแนะนำอย่างชัดเจน) หรือ _การให้ตัวอย่างบางส่วน_ (คำแนะนำแฝง) เทคนิคนี้เรียกว่า _few-shot learning_ แต่มีข้อจำกัดสองประการ:

- ขีดจำกัดของโทเค็นของโมเดลอาจจำกัดจำนวนตัวอย่างที่คุณสามารถให้ได้ และลดประสิทธิภาพของวิธีนี้
- ค่าใช้จ่ายของโทเค็นโมเดลอาจทำให้เพิ่มตัวอย่างในแต่ละพรอมต์มีค่าใช้จ่ายสูงและจำกัดความยืดหยุ่น

การปรับแต่งเป็นแนวทางปฏิบัติทั่วไปในระบบแมชชีนเลิร์นนิงที่เรานำโมเดลที่ผ่านการฝึกล่วงหน้ามาฝึกซ้ำด้วยข้อมูลใหม่เพื่อปรับปรุงประสิทธิภาพในงานเฉพาะในบริบทของโมเดลภาษา เราสามารถปรับแต่งโมเดลที่ผ่านการฝึกล่วงหน้าด้วย _ชุดตัวอย่างที่คัดสรรสำหรับงานหรือโดเมนเฉพาะ_ เพื่อสร้าง **โมเดลเฉพาะทาง** ที่อาจแม่นยำและเกี่ยวข้องมากขึ้นสำหรับงานหรือสาขานั้นๆ ผลพลอยได้จากการปรับแต่งคือช่วยลดจำนวนตัวอย่างที่ต้องใช้ในการเรียนรู้แบบ few-shot ลดการใช้โทเค็นและค่าใช้จ่ายที่เกี่ยวข้อง

## เมื่อไรและทำไมเราควรปรับแต่งโมเดล?

ใน _กรณีนี้_ เมื่อเราพูดถึงการปรับแต่ง เราหมายถึงการปรับแต่งแบบ **มีผู้สอน** ที่การฝึกซ้ำทำโดยการ **เพิ่มข้อมูลใหม่** ที่ไม่ได้อยู่ในชุดข้อมูลฝึกต้นฉบับ ซึ่งต่างจากการปรับแต่งแบบไม่มีผู้สอนที่โมเดลถูกฝึกใหม่จากข้อมูลเดิมแต่มีค่าพารามิเตอร์อื่น

สิ่งสำคัญที่ต้องจำคือ การปรับแต่งเป็นเทคนิคขั้นสูงที่ต้องการความเชี่ยวชาญระดับหนึ่งเพื่อให้ได้ผลลัพธ์ตามที่ต้องการ ถ้าทำผิดวิธีอาจไม่เกิดการปรับปรุงตามคาด และอาจทำให้ประสิทธิภาพโมเดลในโดเมนเป้าหมายลดลงด้วย

ดังนั้น ก่อนที่คุณจะเรียนรู้ "วิธี" ปรับแต่งโมเดลภาษา คุณควรรู้ "ทำไม" คุณควรเลือกเส้นทางนี้ และ "เมื่อใด" ที่เหมาะสมสำหรับการเริ่มปรับแต่ง เริ่มต้นด้วยการถามตัวเองดังนี้:

- **กรณีใช้งาน**: กรณีการใช้งานสำหรับการปรับแต่งคืออะไร? คุณต้องการปรับปรุงด้านใดของโมเดลที่ผ่านการฝึกในปัจจุบัน?
- **ทางเลือก**: คุณเคยลอง _เทคนิคอื่นๆ_ เพื่อบรรลุผลลัพธ์ที่ต้องการหรือไม่? ใช้มันเพื่อสร้างฐานเปรียบเทียบ
  - การออกแบบพรอมต์: ลองเทคนิคเช่น few-shot prompting กับตัวอย่างการตอบที่เกี่ยวข้อง ประเมินคุณภาพการตอบกลับ
  - การสร้างผลลัพธ์เสริมการดึงข้อมูล: ลองเพิ่มพรอมต์ด้วยผลลัพธ์ที่ค้นพบโดยการค้นหาข้อมูลของคุณ ประเมินคุณภาพการตอบกลับ
- **ค่าใช้จ่าย**: คุณระบุค่าใช้จ่ายสำหรับการปรับแต่งหรือยัง?
  - ความสามารถในการปรับแต่ง - โมเดลที่ฝึกล่วงหน้าพร้อมสำหรับการปรับแต่งหรือไม่?
  - ความพยายาม - สำหรับการเตรียมข้อมูลฝึก, การประเมินและปรับปรุงโมเดล
  - การคำนวณ - สำหรับการทำงานปรับแต่งและการปรับใช้โมเดลที่ปรับแต่งแล้ว
  - ข้อมูล - การเข้าถึงตัวอย่างคุณภาพเพียงพอสำหรับผลกระทบของการปรับแต่ง
- **ประโยชน์**: คุณยืนยันประโยชน์ของการปรับแต่งหรือยัง?
  - คุณภาพ - โมเดลที่ปรับแต่งดีกว่าฐานข้อมูลหรือไม่?
  - ต้นทุน - ลดการใช้โทเค็นโดยการทำพรอมต์ให้ง่ายขึ้นไหม?
  - ความสามารถในการขยาย - คุณสามารถนำโมเดลพื้นฐานไปใช้กับโดเมนใหม่ได้หรือไม่?

โดยการตอบคำถามเหล่านี้ คุณควรจะสามารถตัดสินใจได้ว่าการปรับแต่งเป็นวิธีที่เหมาะสมกับกรณีใช้งานของคุณหรือไม่ โดยทั่วไป วิธีนี้จะเหมาะสมก็ต่อเมื่อประโยชน์มีมากกว่าต้นทุน เมื่อตัดสินใจเดินหน้าต่อ ก็ถึงเวลาคิดว่า _อย่างไร_ คุณจะปรับแต่งโมเดลที่ผ่านการฝึกล่วงหน้า

ต้องการข้อมูลเชิงลึกเพิ่มเติมในกระบวนการตัดสินใจหรือไม่? ดู [To fine-tune or not to fine-tune](https://www.youtube.com/watch?v=0Jo-z-MFxJs)

## เราจะปรับแต่งโมเดลที่ผ่านการฝึกมาแล้วได้อย่างไร?

ในการปรับแต่งโมเดลที่ผ่านการฝึกมาแล้ว คุณจำเป็นต้องมี:

- โมเดลที่ผ่านการฝึกแล้วสำหรับการปรับแต่ง
- ชุดข้อมูลที่ใช้สำหรับการปรับแต่ง
- สภาพแวดล้อมสำหรับฝึกซ้ำเพื่อรันงานปรับแต่ง
- สภาพแวดล้อมสำหรับการโฮสต์เพื่อนำโมเดลที่ปรับแต่งแล้วไปใช้

## การปรับแต่งบน Microsoft Foundry

[Microsoft Foundry](https://ai.azure.com?WT.mc_id=academic-105485-koreyst) คือสถานที่ที่คุณปรับแต่ง เปิดใช้งาน และจัดการโมเดลเฉพาะบน Azure ในปัจจุบัน (รวมสิ่งที่เคยเป็น Azure OpenAI Studio และ Azure AI Studio ไว้ด้วยกัน) ก่อนเริ่มงาน ควรทำความเข้าใจตัวเลือกที่ Foundry ให้มาและแนวทางปฏิบัติที่ดีที่สุดที่แพลตฟอร์มแนะนำ Foundry ใช้ **LoRA (low-rank adaptation)** เพื่อการปรับแต่งโมเดลที่มีประสิทธิภาพ ซึ่งช่วยให้การฝึกซ้ำเร็วขึ้นและประหยัดกว่าการฝึกซ้ำทุกพารามิเตอร์

### ขั้นตอนที่ 1: เลือกเทคนิคการฝึก

Foundry รองรับสามเทคนิคการปรับแต่ง **เริ่มต้นที่ SFT** - เหมาะกับสถานการณ์ที่หลากหลายที่สุด

| เทคนิค | มันทำอะไร | ใช้เมื่อไร |
| --- | --- | --- |
| **การปรับแต่งแบบมีผู้สอน (SFT)** | ฝึกด้วยตัวอย่างคู่ขาเข้า/ขาออกเพื่อให้โมเดลเรียนรู้ผลิตคำตอบที่คุณต้องการ | เริ่มต้นสำหรับงานส่วนใหญ่: การเชี่ยวชาญโดเมน, ประสิทธิภาพงาน, สไตล์และโทนเสียง, การปฏิบัติตามคำสั่ง, และการปรับภาษา |
| **การปรับแต่งโดยตรงด้วยการเพิ่มพฤติกรรมที่ชอบ (DPO)** | เรียนรู้จากคู่คำตอบ _ที่ชอบกับไม่ชอบ_ เพื่อตรงกับความชอบของมนุษย์ | ปรับปรุงคุณภาพคำตอบ, ความปลอดภัย และการจัดแนวเมื่อมีข้อมูลเปรียบเทียบ |
| **การปรับแต่งด้วยการเสริมแรง (RFT)** | ใช้สัญญาณรางวัลจาก _ผู้ตรวจสอบ_ เพื่อเพิ่มประสิทธิภาพพฤติกรรมซับซ้อนโดยใช้การเรียนรู้เสริมแรง | โดเมนที่ต้องใช้เหตุผลเชิงวัตถุประสงค์และเข้มข้น (คณิตศาสตร์, เคมี, ฟิสิกส์) ที่มีคำตอบชัดเจน ต้องการความเชี่ยวชาญ ML สูงกว่า |

### ขั้นตอนที่ 2: เลือกระดับการฝึก

Foundry ให้คุณเลือกวิธีและสถานที่รันการฝึกได้:

- **มาตรฐาน** - ฝึกภายในภูมิภาคของทรัพยากรของคุณและรับรองให้อยู่ในภูมิภาคเดียวกัน ใช้เมื่อข้อมูลต้องอยู่ในภูมิภาคที่ระบุเท่านั้น
- **ระดับโลก** - ราคาถูกกว่าและรอคิวน้อยกว่าโดยใช้กำลังการผลิตนอกภูมิภาคของคุณ (ข้อมูลและพารามิเตอร์ถูกคัดลอกไปยังภูมิภาคการฝึก) เป็นค่าเริ่มต้นที่ดีเมื่อไม่ต้องการจำกัดการเก็บข้อมูลในภูมิภาค
- **สำหรับนักพัฒนา** - ราคาต่ำที่สุด ใช้กำลังการผลิตที่ว่างไม่มีการรับประกันความหน่วงเวลา/ SLA (งานอาจถูกแทรกแซงและทำต่อได้) เหมาะสำหรับการทดลอง

### ขั้นตอนที่ 3: เลือกโมเดลฐาน

โมเดลที่ปรับแต่งได้รวมถึง OpenAI `gpt-4o-mini`, `gpt-4o`, `gpt-4.1`, `gpt-4.1-mini`, และ `gpt-4.1-nano` (SFT; ตระกูล 4o/4.1 ยังรองรับ DPO), โมเดลเหตุผล `o4-mini` และ `gpt-5` (RFT), พร้อมทั้งโมเดลโอเพ่นซอร์ส เช่น `Ministral-3B`, `Qwen-32B`, `Llama-3.3-70B-Instruct`, และ `gpt-oss-20b` (SFT บน Foundry) ตรวจสอบรายการ [โมเดลปรับแต่ง](https://learn.microsoft.com/azure/ai-foundry/foundry-models/concepts/models-sold-directly-by-azure?WT.mc_id=academic-105485-koreyst#fine-tuning-models) ล่าสุดเพื่อดูวิธีรองรับ, ภูมิภาคและความพร้อมใช้งานเสมอ

> Foundry มีสองโหมด: **serverless** (ราคาตามการใช้งาน, ไม่มีการจัดการโควต้าการ์ดจอ, OpenAI และโมเดลที่เลือก) และ **managed compute** (ใช้ VM ของตัวเองผ่าน Azure Machine Learning เพื่อรองรับโมเดลหลากหลายที่สุด) ส่วนใหญ่ควรเริ่มด้วย serverless

### แนวทางปฏิบัติที่ดีที่สุดของ Foundry

- **เริ่มจากฐาน** วัดโมเดลฐานด้วยการออกแบบพรอมต์และ RAG _ก่อน_ ปรับแต่งเพื่อพิสูจน์ประโยชน์
- **เริ่มน้อย แล้วขยาย** เริ่มจากตัวอย่างคุณภาพสูง 50-100 ตัวอย่างเพื่อยืนยันวิธี จากนั้นเพิ่มเป็น 500+ สำหรับใช้งานจริง เน้นคุณภาพมากกว่าปริมาณ - ตัดตัวอย่างคุณภาพต่ำออก
- **จัดรูปแบบข้อมูลให้ถูกต้อง** ไฟล์ฝึกและไฟล์ยืนยันต้องเป็น JSONL, UTF-8 **พร้อม BOM**, ขนาดไม่เกิน 512 MB, ใช้รูปแบบข้อความแชทเสร็จสมบูรณ์ และควรมีไฟล์ยืนยันเพื่อติดตามการ overfitting
- **เก็บพรอมต์ระบบต้นทางไว้ที่การอนุมาน** ใช้ข้อความระบบเดียวกันเมื่อเรียกโมเดลเหมือนกับในการฝึก
- **ประเมินจุดเช็คพอยต์ - อย่าเพิ่งปรับใช้โดยไม่ดู** Foundry เก็บจุดเช็คพอยต์ 3 รอบสุดท้ายไว้สำหรับปรับใช้ เลือกอันที่ทำนองทั่วไปดีที่สุดโดยดู `train_loss` / `valid_loss` และความถูกต้องของโทเค็น
- **วัดค่าโทเค็นควบคู่กับคุณภาพ** เมื่อต้องเปรียบเทียบโมเดลที่ปรับแต่งกับฐาน
- **ปรับแต่งต่อเนื่องได้** คุณสามารถปรับแต่งโมเดลที่ปรับแต่งแล้วซ้ำอีกด้วยข้อมูลใหม่ (รองรับสำหรับโมเดล OpenAI)
- **ระวังค่าใช้จ่ายโฮสต์** โมเดลเฉพาะที่ปรับแต่งแล้วคิดค่าบริการเป็นรายชั่วโมง และการถอดถอนโมเดลที่ไม่ใช้งานจะเกิดขึ้นหลัง 15 วัน - ทำความสะอาดสิ่งที่ไม่ต้องการ

ทำงานตามขั้นตอนเต็มใน [Customize a model with fine-tuning](https://learn.microsoft.com/azure/ai-foundry/openai/how-to/fine-tuning?WT.mc_id=academic-105485-koreyst) และดูไกด์สำหรับ [DPO](https://learn.microsoft.com/azure/ai-foundry/openai/how-to/fine-tuning-direct-preference-optimization?WT.mc_id=academic-105485-koreyst) และ [RFT](https://learn.microsoft.com/azure/ai-foundry/openai/how-to/reinforcement-fine-tuning?WT.mc_id=academic-105485-koreyst) เมื่อพร้อมสำหรับเทคนิคอื่นๆ
