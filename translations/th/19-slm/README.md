# บทนำสู่โมเดลภาษาเล็กสำหรับ AI เชิงสร้างสรรค์สำหรับผู้เริ่มต้น
AI เชิงสร้างสรรค์เป็นสาขาที่น่าทึ่งของปัญญาประดิษฐ์ที่มุ่งเน้นการสร้างระบบที่สามารถสร้างเนื้อหาใหม่ได้ เนื้อหานี้อาจครอบคลุมตั้งแต่ข้อความและภาพไปจนถึงเพลงและแม้กระทั่งสภาพแวดล้อมเสมือนจริงทั้งมวล หนึ่งในแอปพลิเคชันที่น่าตื่นเต้นที่สุดของ AI เชิงสร้างสรรค์อยู่ในขอบเขตของโมเดลภาษา

## โมเดลภาษาเล็กคืออะไร?

โมเดลภาษาเล็ก (SLM) หมายถึงรุ่นย่อขนาดของโมเดลภาษาขนาดใหญ่ (LLM) ที่นำหลักการและเทคนิคหลายอย่างของ LLM มาใช้ในขณะที่มีรอยเท้าการคำนวณที่ลดลงอย่างมาก

SLM คือชุดย่อยของโมเดลภาษาที่ออกแบบมาเพื่อสร้างข้อความที่เหมือนมนุษย์ แตกต่างจากคู่ขนาดใหญ่ เช่น GPT-4 SLM มีขนาดกะทัดรัดและมีประสิทธิภาพมากกว่า ทำให้เหมาะสำหรับแอปพลิเคชันที่มีทรัพยากรคำนวณจำกัด แม้จะมีขนาดเล็กกว่า พวกมันก็ยังสามารถทำหน้าที่หลากหลายได้ โดยปกติแล้ว SLM ถูกสร้างขึ้นโดยการบีบอัดหรือกลั่นกรอง LLM โดยมีเป้าหมายเพื่อรักษาฟังก์ชันและความสามารถทางภาษาเดิมไว้เป็นส่วนใหญ่ การลดขนาดโมเดลนี้ช่วยลดความซับซ้อนโดยรวม ทำให้ SLM มีประสิทธิภาพทั้งในด้านการใช้หน่วยความจำและความต้องการทางคำนวณ แม้จะมีการปรับให้เหมาะสมเหล่านี้ SLM ก็ยังสามารถทำงานด้านการประมวลผลภาษาธรรมชาติ (NLP) ได้หลากหลายประเภท:

- การสร้างข้อความ: การสร้างประโยคหรือย่อหน้าที่สอดคล้องและเหมาะสมกับบริบท
- การเติมข้อความ: การทำนายและเติมประโยคตามข้อความต้นแบบที่กำหนด
- การแปลภาษา: การแปลงข้อความจากภาษาหนึ่งไปยังอีกภาษา
- การสรุปข้อความ: การย่อข้อความยาว ๆ ให้เป็นสรุปที่สั้นลงและเข้าใจง่ายขึ้น

แม้จะมีการแลกเปลี่ยนด้านประสิทธิภาพหรือความลึกในการเข้าใจเมื่อเทียบกับโมเดลขนาดใหญ่กว่า

## โมเดลภาษาเล็กทำงานอย่างไร?
SLM ถูกฝึกด้วยข้อมูลข้อความจำนวนมาก ในระหว่างการฝึก พวกมันเรียนรู้รูปแบบและโครงสร้างของภาษา ทำให้สามารถสร้างข้อความที่ถูกต้องตามไวยากรณ์และเหมาะสมกับบริบทได้ กระบวนการฝึกประกอบด้วย:

- การเก็บข้อมูล: รวบรวมชุดข้อมูลขนาดใหญ่ของข้อความจากแหล่งต่าง ๆ
- การเตรียมข้อมูล: ทำความสะอาดและจัดระเบียบข้อมูลให้เหมาะสมสำหรับการฝึก
- การฝึก: ใช้อัลกอริธึมการเรียนรู้ของเครื่องเพื่อสอนโมเดลให้เข้าใจและสร้างข้อความ
- การปรับแต่งเพิ่มเติม: ปรับแต่งโมเดลเพื่อเพิ่มประสิทธิภาพในการทำงานเฉพาะงาน

การพัฒนา SLM สอดคล้องกับความต้องการที่เพิ่มขึ้นสำหรับโมเดลที่สามารถนำไปใช้ในสภาพแวดล้อมที่มีทรัพยากรจำกัด เช่น อุปกรณ์มือถือหรือแพลตฟอร์ม edge computing ที่โมเดล LLM ขนาดเต็มอาจไม่เหมาะสมเนื่องจากการใช้ทรัพยากรหนัก ด้วยการมุ่งเน้นที่ประสิทธิภาพ SLM จึงสร้างสมดุลระหว่างประสิทธิภาพและการเข้าถึง ทำให้สามารถใช้งานได้อย่างกว้างขวางในหลายโดเมน

## ความแตกต่างระหว่าง Large Language Models (LLMs) และ Small Language Models (SLMs)

ทั้ง LLM และ SLM สร้างขึ้นบนหลักการพื้นฐานของการเรียนรู้เครื่องแบบมีความน่าจะเป็น โดยมีแนวทางที่เหมือนกันในการออกแบบสถาปัตยกรรม วิธีการฝึก กระบวนการสร้างข้อมูล และเทคนิคการประเมินโมเดล อย่างไรก็ตาม มีปัจจัยสำคัญหลายประการที่ทำให้โมเดลสองประเภทนี้แตกต่างกัน

## การใช้งานของโมเดลภาษาเล็ก

SLM มีการใช้งานหลากหลาย เช่น:

- แชทบอท: ให้การสนับสนุนลูกค้าและโต้ตอบกับผู้ใช้ในลักษณะสนทนา
- การสร้างเนื้อหา: ช่วยนักเขียนสร้างไอเดียหรือร่างบทความทั้งหมด
- การศึกษา: ช่วยนักเรียนในการเขียนงานหรือเรียนรู้ภาษาใหม่
- การเข้าถึง: สร้างเครื่องมือสำหรับบุคคลที่มีความบกพร่อง เช่น ระบบแปลงข้อความเป็นเสียง

**ขนาด**
  
ความแตกต่างหลักระหว่าง LLM และ SLM อยู่ที่ขนาดของโมเดล LLM เช่น ChatGPT (GPT-4) มีประมาณ 1.76 ล้านล้านพารามิเตอร์ ในขณะที่ SLM แบบโอเพนซอร์ส เช่น Mistral 7B ออกแบบให้มีพารามิเตอร์น้อยกว่ามาก ประมาณ 7 พันล้าน ความแตกต่างนี้เกิดจากสถาปัตยกรรมและกระบวนการฝึกต่างกัน ตัวอย่างเช่น ChatGPT ใช้กลไก self-attention ภายในกรอบ encoder-decoder ในขณะที่ Mistral 7B ใช้ attention แบบหน้าต่างเลื่อนซึ่งช่วยให้ฝึกได้มีประสิทธิภาพมากขึ้นภายในโมเดลแบบ decoder-only ความแตกต่างในสถาปัตยกรรมนี้ส่งผลลึกซึ้งต่อความซับซ้อนและประสิทธิภาพของโมเดลเหล่านี้

**ความเข้าใจ**

SLM มักถูกปรับให้เหมาะสมสำหรับประสิทธิภาพในโดเมนเฉพาะ ทำให้มีความเชี่ยวชาญสูงแต่จำกัดในความสามารถในการให้ความเข้าใจบริบทกว้าง ๆ ในหลากหลายสาขาความรู้ ในทางกลับกัน LLM มุ่งหวังเลียนแบบความฉลาดเหมือนมนุษย์ในระดับที่ครอบคลุมมากขึ้น โดยได้รับการฝึกกับชุดข้อมูลที่ใหญ่และหลากหลาย LLM ถูกออกแบบมาให้ทำงานได้ดีในโดเมนต่าง ๆ จึงมีความยืดหยุ่นและปรับตัวได้มากกว่า ดังนั้น LLM จึงเหมาะกับงานหลากหลายมากขึ้น เช่น การประมวลผลภาษาธรรมชาติ และการเขียนโปรแกรม

**การคำนวณ**

การฝึกและใช้งาน LLM เป็นกระบวนการที่ต้องใช้ทรัพยากรจำนวนมาก มักต้องการโครงสร้างพื้นฐานด้านคำนวณสูง รวมถึงคลัสเตอร์ GPU ขนาดใหญ่ เช่น การฝึกโมเดลอย่าง ChatGPT ตั้งแต่ต้นอาจต้องใช้ GPU หลายพันตัวเป็นเวลานาน ในทางกลับกัน SLM ด้วยจำนวนพารามิเตอร์ที่น้อยกว่าสามารถเข้าถึงได้ง่ายกว่าในแง่ของทรัพยากรคำนวณ โมเดลอย่าง Mistral 7B สามารถฝึกและใช้งานบนเครื่องท้องถิ่นที่มี GPU ระดับปานกลางได้ แม้ว่าการฝึกยังต้องใช้เวลาหลายชั่วโมงบนหลาย GPU

**อคติ**

อคติเป็นปัญหาที่รู้จักกันใน LLM ส่วนใหญ่เนื่องจากลักษณะของข้อมูลฝึก โมเดลเหล่านี้มักพึ่งพาข้อมูลดิบที่เปิดเผยบนอินเทอร์เน็ต ซึ่งอาจขาดการแทนกลุ่มคนบางกลุ่ม หรือนำเสนอข้อมูลผิดพลาด หรือสะท้อนอคติทางภาษา เช่น ผลกระทบจากภาษาถิ่น ภูมิศาสตร์ และกฎเกรามาติก นอกจากนี้ ความซับซ้อนของสถาปัตยกรรม LLM อาจทำให้อคติรุนแรงขึ้นโดยที่ไม่ถูกสังเกตหากไม่มีการปรับแต่งอย่างระมัดระวัง ในทางตรงกันข้าม SLM ซึ่งถูกฝึกด้วยชุดข้อมูลที่จำกัดเฉพาะโดเมน มักจะได้รับผลกระทบน้อยกว่าจากอคติเหล่านี้ แม้ว่าจะไม่ใช่ว่าจะปลอดภัยทั้งหมด

**การอนุมาน**

ขนาดที่เล็กลงของ SLM ทำให้มีข้อได้เปรียบสำคัญด้านความเร็วในการอนุมาน ช่วยให้สามารถสร้างผลลัพธ์ได้อย่างมีประสิทธิภาพบนฮาร์ดแวร์ท้องถิ่นโดยไม่ต้องใช้การประมวลผลแบบขนานอย่างกว้างขวาง ในทางตรงกันข้าม LLM ด้วยขนาดและความซับซ้อนมักต้องการทรัพยากรการประมวลผลแบบขนานมากเพื่อให้ได้เวลาตอบสนองที่ยอมรับได้ การมีผู้ใช้หลายคนพร้อมกันก็ยิ่งทำให้เวลาตอบสนองของ LLM ช้าลงโดยเฉพาะเมื่อนำไปใช้ในขนาดใหญ่

กล่าวโดยสรุป แม้ LLM และ SLM จะใช้ฐานหลักการเรียนรู้เครื่องเหมือนกัน แต่มีความแตกต่างสำคัญในขนาดโมเดล ความต้องการทรัพยากร ความเข้าใจบริบท ความเปราะบางต่ออคติ และความเร็วในการอนุมาน ความแตกต่างเหล่านี้สะท้อนความเหมาะสมสำหรับการใช้งานที่แตกต่างกัน โดย LLM มีความหลากหลายแต่ใช้ทรัพยากรมาก ขณะที่ SLM มีประสิทธิภาพในโดเมนเฉพาะพร้อมความต้องการการคำนวณที่น้อยลง

***หมายเหตุ: ในบทเรียนนี้ เราจะนำเสนอ SLM โดยใช้ Microsoft Phi-3 / 3.5 เป็นตัวอย่าง***

## แนะนำตระกูล Phi-3 / Phi-3.5

ตระกูล Phi-3 / 3.5 มุ่งเน้นไปที่สถานการณ์ใช้งานในด้านข้อความ รูปภาพ และ Agent (MoE)

### Phi-3 / 3.5 Instruct

ส่วนใหญ่สำหรับการสร้างข้อความ การเติมบทสนทนา และการสกัดข้อมูลเนื้อหา ฯลฯ

**Phi-3-mini**

โมเดลภาษาขนาด 3.8B พร้อมใช้งานบน Microsoft Foundry, Hugging Face และ Ollama โมเดล Phi-3 มีประสิทธิภาพดีกว่าโมเดลภาษาขนาดเท่ากันและใหญ่กว่าในเกณฑ์มาตรฐานที่สำคัญ (ดูตัวเลขเกณฑ์ด้านล่าง ยิ่งตัวเลขสูงยิ่งดี) Phi-3-mini ดีกว่าโมเดลที่มีขนาดสองเท่า ขณะที่ Phi-3-small และ Phi-3-medium ดีกว่าโมเดลที่ใหญ่กว่า รวมถึง GPT-3.5

**Phi-3-small & medium**

ด้วยพารามิเตอร์เพียง 7B Phi-3-small สามารถเอาชนะ GPT-3.5T ในเกณฑ์มาตรฐานด้านภาษา การให้เหตุผล การเขียนโค้ด และคณิตศาสตร์ต่าง ๆ

Phi-3-medium ซึ่งมีพารามิเตอร์ 14B ยังคงแนวโน้มนี้และมีประสิทธิภาพดีกว่า Gemini 1.0 Pro

**Phi-3.5-mini**

เราสามารถคิดว่าเป็นการอัปเกรดจาก Phi-3-mini แม้ว่าพารามิเตอร์จะไม่เปลี่ยนแปลง แต่ปรับปรุงความสามารถในการรองรับหลายภาษา (รองรับมากกว่า 20 ภาษา: อาหรับ จีน เช็ก เดนมาร์ก ดัตช์ อังกฤษ ฟินแลนด์ ฝรั่งเศส เยอรมัน ฮิบรู ฮังการี อิตาเลียน ญี่ปุ่น เกาหลี นอร์เวย์ โปแลนด์ โปรตุเกส รัสเซีย สเปน สวีเดน ไทย ตุรกี ยูเครน) และเพิ่มการรองรับบริบทที่ยาวขึ้น

Phi-3.5-mini ที่มีพารามิเตอร์ 3.8B มีประสิทธิภาพเหนือกว่าโมเดลภาษาขนาดเดียวกันและเทียบเท่ากับโมเดลที่มีขนาดสองเท่า

### Phi-3 / 3.5 Vision

เราสามารถคิดว่าโมเดล Instruct ของ Phi-3/3.5 เป็นความสามารถในการเข้าใจของ Phi ส่วน Vision คือสิ่งที่ให้ Phi มีตาเพื่อเข้าใจโลก


**Phi-3-Vision**

Phi-3-vision ด้วยพารามิเตอร์เพียง 4.2B ยังคงแนวโน้มนี้และมีประสิทธิภาพเหนือกว่าโมเดลที่ใหญ่กว่า เช่น Claude-3 Haiku และ Gemini 1.0 Pro V ในงานการให้เหตุผลภาพทั่วไป OCR และการเข้าใจตารางและแผนผัง


**Phi-3.5-Vision**

Phi-3.5-Vision ยังเป็นการอัปเกรดจาก Phi-3-Vision เพิ่มการรองรับภาพหลายภาพ คุณสามารถคิดว่าเป็นการปรับปรุงวิสัยทัศน์ ไม่เพียงแต่ดูภาพนิ่งแต่ยังดูวิดีโอได้ด้วย

Phi-3.5-vision มีประสิทธิภาพเหนือกว่าโมเดลที่ใหญ่กว่า เช่น Claude-3.5 Sonnet และ Gemini 1.5 Flash ในงาน OCR ตารางและการเข้าใจแผนภูมิ และเทียบเท่าในงานให้เหตุผลความรู้ภาพทั่วไป รองรับการป้อนข้อมูลหลายเฟรม หมายถึงการให้เหตุผลจากภาพหลายภาพพร้อมกัน


### Phi-3.5-MoE

***Mixture of Experts(MoE)*** ทำให้โมเดลสามารถฝึกล่วงหน้าด้วยการใช้ทรัพยากรน้อยลงมาก ซึ่งหมายความว่าคุณสามารถขยายขนาดโมเดลหรือชุดข้อมูลได้อย่างมากด้วยงบประมาณคำนวณเท่ากันกับโมเดลหนาแน่น โดยเฉพาะ โมเดล MoE ควรบรรลุคุณภาพเทียบเท่าโมเดลหนาแน่นในเวลาที่เร็วกว่าอย่างมากในระหว่างการฝึกล่วงหน้า

Phi-3.5-MoE ประกอบด้วยโมดูลผู้เชี่ยวชาญ 16x3.8B Phi-3.5-MoE ที่มีพารามิเตอร์ที่ใช้งานอยู่เพียงแค่ 6.6B สามารถบรรลุระดับการให้เหตุผล ความเข้าใจภาษา และคณิตศาสตร์ที่เทียบเท่าโมเดลที่มีขนาดใหญ่กว่าอย่างมาก

เราสามารถใช้โมเดลตระกูล Phi-3/3.5 โดยอิงตามสถานการณ์ต่าง ๆ แตกต่างจาก LLM ที่คุณสามารถปรับใช้ Phi-3/3.5-mini หรือ Phi-3/3.5-Vision บนอุปกรณ์ edge ได้

- **ความสามารถในการขยาย:** NIM รองรับการปรับขนาดอัตโนมัติบน Kubernetes ช่วยให้จัดการกับภาระงานที่หลากหลายได้อย่างมีประสิทธิภาพ
- **ความปลอดภัยและการควบคุม:** องค์กรสามารถรักษาการควบคุมข้อมูลและแอปพลิเคชันของตนเองโดยการโฮสต์ไมโครเซอร์วิส NIM บนโครงสร้างพื้นฐานที่จัดการเองได้

**Hugging Face Transformer**
Hugging Face Transformers เป็นไลบรารีที่ทรงพลังออกแบบมาสำหรับการประมวลผลภาษาธรรมชาติ (NLP) และงานแมชชีนเลิร์นนิงอื่นๆ นี่คือประเด็นสำคัญบางประการเกี่ยวกับมัน:

1. **โมเดลที่ผ่านการฝึกมาแล้ว**: มีโมเดลผ่านการฝึกนับพันที่สามารถใช้สำหรับงานต่างๆ เช่น การจำแนกข้อความ การระบุชื่อเอนทิตี การตอบคำถาม การสรุป การแปล และการสร้างข้อความ

2. **ความเข้ากันได้หลากหลายเฟรมเวิร์ก:** ไลบรารีรองรับหลายเฟรมเวิร์กสำหรับการเรียนรู้เชิงลึก เช่น PyTorch, TensorFlow, และ JAX ทำให้สามารถฝึกโมเดลในเฟรมเวิร์กหนึ่งและใช้งานในอีกเฟรมเวิร์กหนึ่งได้

3. **ความสามารถมัลติโมดอล:** นอกจาก NLP แล้ว Hugging Face Transformers ยังรองรับงานคอมพิวเตอร์วิทัศน์ (เช่น การจำแนกรูปภาพ การตรวจจับวัตถุ) และการประมวลผลเสียง (เช่น การรู้จำเสียง การจำแนกเสียง)

4. **ใช้งานง่าย:** ไลบรารีมี API และเครื่องมือสำหรับการดาวน์โหลดและปรับแต่งโมเดลอย่างง่ายดาย เหมาะสำหรับผู้เริ่มต้นและผู้เชี่ยวชาญ

5. **ชุมชนและแหล่งข้อมูล:** Hugging Face มีชุมชนที่แข็งแกร่งและเอกสาร คู่มือ และบทเรียนมากมายเพื่อช่วยให้ผู้ใช้เริ่มต้นและใช้งานไลบรารีได้อย่างเต็มที่
[เอกสารอย่างเป็นทางการ](https://huggingface.co/docs/transformers/index?WT.mc_id=academic-105485-koreyst) หรือที่ [GitHub repository](https://github.com/huggingface/transformers?WT.mc_id=academic-105485-koreyst)

นี่เป็นวิธีที่ใช้กันทั่วไปที่สุด แต่ต้องใช้การเร่งความเร็ว GPU ด้วย เพราะสถานการณ์เช่น Vision และ MoE ต้องการการคำนวณจำนวนมาก ซึ่งจะช้ามากบน CPU ถ้าไม่ได้ประมวลผลด้วยการลดความแม่นยำ (quantized)

**ONNX Runtime for GenAI**

[ONNX Runtime](https://github.com/microsoft/onnxruntime-genai?WT.mc_id=academic-105485-koreyst) เป็นโปรแกรมเร่งความเร็วสำหรับการสรุปผลและการฝึกโมเดลแมชชีนเลิร์นนิงข้ามแพลตฟอร์ม ONNX Runtime for Generative AI (GENAI) เป็นเครื่องมือที่ทรงพลังที่ช่วยให้คุณรันโมเดล AI สร้างสรรค์ได้อย่างมีประสิทธิภาพบนแพลตฟอร์มต่างๆ  

## ONNX Runtime คืออะไร?
ONNX Runtime เป็นโครงการโอเพนซอร์สที่ช่วยให้การสรุปผลของโมเดลแมชชีนเลิร์นนิงมีประสิทธิภาพสูง รองรับโมเดลในรูปแบบ Open Neural Network Exchange (ONNX) ซึ่งเป็นมาตรฐานสำหรับการแทนโมเดลแมชชีนเลิร์นนิง ONNX Runtime ช่วยให้ประสบการณ์ลูกค้าเร็วขึ้นและลดต้นทุน รองรับโมเดลจากเฟรมเวิร์กการเรียนรู้เชิงลึกอย่าง PyTorch และ TensorFlow/Keras รวมถึงไลบรารีแมชชีนเลิร์นนิงทั่วไป เช่น scikit-learn, LightGBM, XGBoost เป็นต้น ONNX Runtime รองรับฮาร์ดแวร์ ไดรเวอร์ และระบบปฏิบัติการที่ต่างกัน และมอบประสิทธิภาพที่ดีที่สุดโดยใช้ตัวเร่งฮาร์ดแวร์ควบคู่ไปกับการปรับแต่งและแปลงกราฟ

## Generative AI คืออะไร?
Generative AI หมายถึงระบบ AI ที่สามารถสร้างเนื้อหาใหม่ เช่น ข้อความ รูปภาพ หรือดนตรี จากข้อมูลที่ได้รับการฝึกมา ตัวอย่างเช่น โมเดลภาษาอย่าง GPT-3 และโมเดลสร้างภาพเช่น Stable Diffusion ไลบรารี ONNX Runtime for GenAI จัดเตรียมวงจร Generative AI สำหรับโมเดล ONNX รวมถึงการสรุปผลด้วย ONNX Runtime, การประมวลผล logits, การค้นหาและการสุ่มตัวอย่าง, และการจัดการแคช KV

## ONNX Runtime for GENAI
ONNX Runtime for GENAI ขยายความสามารถของ ONNX Runtime เพื่อรองรับโมเดล Generative AI นี่คือคุณสมบัติสำคัญบางประการ:

- **รองรับแพลตฟอร์มหลากหลาย:** ใช้งานได้บนหลายแพลตฟอร์ม รวมถึง Windows, Linux, macOS, Android และ iOS
- **รองรับโมเดล:** รองรับโมเดล Generative AI ชื่อดังหลายตัว เช่น LLaMA, GPT-Neo, BLOOM และอื่นๆ
- **การปรับแต่งประสิทธิภาพ:** มีการปรับแต่งสำหรับตัวเร่งฮาร์ดแวร์ต่างๆ เช่น NVIDIA GPUs, AMD GPUs และอื่นๆ
- **ใช้งานง่าย:** มี API สำหรับการผนวกรวมอย่างง่ายดายในแอปพลิเคชัน ช่วยให้คุณสร้างข้อความ รูปภาพ และเนื้อหาอื่นๆ ด้วยโค้ดน้อยที่สุด
- ผู้ใช้สามารถเรียกใช้เมธอด generate() ระดับสูง หรือรันแต่ละรอบของโมเดลในลูปโดยสร้างโทเค็นทีละตัว และอัปเดตพารามิเตอร์การสร้างภายในลูปตามต้องการ
- ONNX runtime ยังรองรับการค้นหาแบบ greedy/beam และการสุ่มตัวอย่าง TopP, TopK สำหรับสร้างลำดับโทเค็น รวมถึงการประมวลผล logits ในตัว เช่น การลงโทษการทำซ้ำ คุณยังสามารถเพิ่มการให้คะแนนแบบกำหนดเองได้ง่ายๆ

## เริ่มต้นใช้งาน
เพื่อเริ่มต้นใช้งาน ONNX Runtime for GENAI คุณสามารถทำตามขั้นตอนเหล่านี้:

### ติดตั้ง ONNX Runtime:
```Python
pip install onnxruntime
```
### ติดตั้งส่วนขยาย Generative AI:
```Python
pip install onnxruntime-genai
```

### รันโมเดล: นี่คือตัวอย่างง่ายๆ ใน Python:
```Python
import onnxruntime_genai as og

model = og.Model('path_to_your_model.onnx')

tokenizer = og.Tokenizer(model)

input_text = "Hello, how are you?"

input_tokens = tokenizer.encode(input_text)

output_tokens = model.generate(input_tokens)

output_text = tokenizer.decode(output_tokens)

print(output_text) 
```
### ตัวอย่างสาธิต: ใช้ ONNX Runtime GenAI เรียก Phi-3.5-Vision


```python

import onnxruntime_genai as og

model_path = './Your Phi-3.5-vision-instruct ONNX Path'

img_path = './Your Image Path'

model = og.Model(model_path)

processor = model.create_multimodal_processor()

tokenizer_stream = processor.create_stream()

text = "Your Prompt"

prompt = "<|user|>\n"

prompt += "<|image_1|>\n"

prompt += f"{text}<|end|>\n"

prompt += "<|assistant|>\n"

image = og.Images.open(img_path)

inputs = processor(prompt, images=image)

params = og.GeneratorParams(model)

params.set_inputs(inputs)

params.set_search_options(max_length=3072)

generator = og.Generator(model, params)

while not generator.is_done():

    generator.compute_logits()
    
    generator.generate_next_token()

    new_token = generator.get_next_tokens()[0]
    
    output = tokenizer_stream.decode(new_token)
    
    print(tokenizer_stream.decode(new_token), end='', flush=True)

```
