# พื้นฐานการออกแบบคำสั่ง (Prompt Engineering Fundamentals)

[![พื้นฐานการออกแบบคำสั่ง](../../../translated_images/th/04-lesson-banner.a2c90deba7fedacd.webp)](https://youtu.be/GElCu2kUlRs?si=qrXsBvXnCW12epb8)

## บทนำ  
โมดูลนี้ครอบคลุมแนวคิดและเทคนิคที่จำเป็นสำหรับการสร้างคำสั่งที่มีประสิทธิภาพในโมเดล AI สร้างสรรค์ วิธีการเขียนคำสั่งของคุณกับ LLM ก็มีความสำคัญเช่นกัน คำสั่งที่ถูกสร้างอย่างรอบคอบสามารถทำให้ได้คุณภาพของคำตอบที่ดีกว่า แต่แล้วคำว่า _prompt_ และ _prompt engineering_ หมายความว่าอย่างไร? และเราจะพัฒนาคำสั่ง _input_ ที่เราส่งไปยัง LLM ได้อย่างไร? คำถามเหล่านี้เราจะพยายามตอบภายในบทนี้และบทต่อไป

_Generative AI_ มีความสามารถในการสร้างเนื้อหาใหม่ (เช่น ข้อความ รูปภาพ เสียง โค้ด ฯลฯ) ตามคำขอของผู้ใช้ โดยใช้ _Large Language Models_ เช่น ชุด GPT ของ OpenAI ("Generative Pre-trained Transformer") ที่ได้รับการฝึกฝนให้ใช้ภาษาธรรมชาติและโค้ด

ผู้ใช้สามารถโต้ตอบกับโมเดลเหล่านี้โดยใช้รูปแบบที่คุ้นเคยอย่างการแชท โดยไม่จำเป็นต้องมีความเชี่ยวชาญหรือการฝึกอบรมทางเทคนิค โมเดลเหล่านี้เป็น _prompt-based_ - ผู้ใช้ส่งข้อความเข้า (คำสั่ง) และได้รับคำตอบจาก AI (completion) จากนั้นสามารถแชทกับ AI ได้อย่างต่อเนื่องในหลายรอบ เพื่อตกแต่งคำสั่งของตนจนกว่าคำตอบจะตรงตามความคาดหวัง

"คำสั่ง" กลายเป็น _อินเทอร์เฟซโปรแกรมมิ่ง_ หลักสำหรับแอป AI สร้างสรรค์ บอกโมเดลว่าต้องทำอะไรและส่งผลต่อคุณภาพของคำตอบที่ได้รับ "การออกแบบคำสั่ง (Prompt Engineering)" เป็นสาขาการศึกษาที่เติบโตอย่างรวดเร็วซึ่งมุ่งเน้นที่ _การออกแบบและปรับแต่ง_ คำสั่งเพื่อให้ได้คำตอบที่สม่ำเสมอและมีคุณภาพในวงกว้าง

## เป้าหมายการเรียนรู้

ในบทเรียนนี้ เราจะเรียนรู้ว่า Prompt Engineering คืออะไร ทำไมจึงสำคัญ และเราจะสร้างคำสั่งที่มีประสิทธิภาพมากขึ้นสำหรับโมเดลและวัตถุประสงค์ของแอปพลิเคชันอย่างไร เราจะเข้าใจแนวคิดหลักและแนวทางปฏิบัติที่ดีที่สุดสำหรับการออกแบบคำสั่ง — และเรียนรู้เกี่ยวกับสภาพแวดล้อม "sandbox" ของ Jupyter Notebooks ที่สามารถดูการประยุกต์ใช้แนวคิดเหล่านี้กับตัวอย่างจริง  

เมื่อจบบทเรียนนี้ เราจะสามารถ:

1. อธิบายว่า prompt engineering คืออะไรและทำไมจึงสำคัญ  
2. บรรยายองค์ประกอบของคำสั่งและวิธีการใช้งาน  
3. เรียนรู้แนวทางปฏิบัติและเทคนิคที่ดีที่สุดสำหรับการออกแบบคำสั่ง  
4. ประยุกต์ใช้เทคนิคที่เรียนรู้กับตัวอย่างจริง โดยใช้ OpenAI endpoint  

## คำศัพท์สำคัญ  

Prompt Engineering: การปฏิบัติของการออกแบบและปรับแต่งอินพุตเพื่อชี้นำโมเดล AI ให้สร้างผลลัพธ์ที่ต้องการ  
Tokenization: กระบวนการแปลงข้อความเป็นหน่วยเล็ก ๆ เรียกว่า token ซึ่งโมเดลสามารถเข้าใจและประมวลผลได้  
Instruction-Tuned LLMs: โมเดลภาษาขนาดใหญ่ที่ได้รับการปรับแต่งด้วยคำสั่งเฉพาะเพื่อปรับปรุงความแม่นยำและความเกี่ยวข้องของคำตอบ  

## Learning Sandbox

การออกแบบคำสั่งในปัจจุบันยังเป็นศิลปะมากกว่าวิทยาศาสตร์ วิธีที่ดีที่สุดในการพัฒนาความเข้าใจคือ _ฝึกฝนมากขึ้น_ และใช้วิธีลองผิดลองถูกที่รวมความรู้เชิงลึกของโดเมนกับเทคนิคที่แนะนำและการปรับแต่งเฉพาะโมเดล

Jupyter Notebook ที่แนบมากับบทเรียนนี้มีสภาพแวดล้อม _sandbox_ ที่คุณสามารถลองใช้สิ่งที่เรียนรู้ได้ทันทีหรือเป็นส่วนหนึ่งของความท้าทายโค้ดตอนท้าย ในการรันแบบฝึกหัด คุณจะต้องมี:

1. **กุญแจ API Azure OpenAI** - endpoint สำหรับโมเดล LLM ที่ใช้งาน  
2. **Runtime Python** - สำหรับรัน Notebook  
3. **ตัวแปร Env ในเครื่อง** - _ทำตามขั้นตอน [SETUP](./../00-course-setup/02-setup-local.md?WT.mc_id=academic-105485-koreyst) ตอนนี้เพื่อเตรียมตัว_  

โน้ตบุ๊กมาพร้อมกับแบบฝึกหัด _เริ่มต้น_ แต่คุณสามารถเพิ่ม _Markdown_ (คำอธิบาย) และ _Code_ (คำขอคำสั่ง) ของคุณเองเพื่อลองตัวอย่างหรือไอเดียเพิ่มเติม — และสร้างความคุ้นเคยสำหรับการออกแบบคำสั่ง

- **นักเรียน** อาจขอให้ AI _สอนเรื่องที่ยาก_ AI สามารถแนะนำบทเรียน พร้อมคำแนะนำและตัวอย่างที่เหมาะสมกับระดับของนักเรียน

## Prompt Engineering คืออะไร?

เราเริ่มบทเรียนนี้ด้วยการนิยาม **Prompt Engineering** ว่าเป็นกระบวนการของการ _ออกแบบและปรับแต่ง_ อินพุตข้อความ (คำสั่ง) เพื่อส่งมอบคำตอบ (completion) ที่มีคุณภาพและสม่ำเสมอสำหรับวัตถุประสงค์ของแอปพลิเคชันและโมเดลนั้น ๆ เราสามารถคิดได้ว่าเป็นกระบวนการ 2 ขั้นตอน:

- _ออกแบบ_ คำสั่งเริ่มต้นสำหรับโมเดลและวัตถุประสงค์ที่กำหนด  
- _ปรับแต่ง_ คำสั่งแบบซ้ำ ๆ เพื่อปรับปรุงคุณภาพของคำตอบ  

นี่จำเป็นต้องเป็นกระบวนการลองผิดลองถูกที่ต้องใช้สัญชาตญาณและความพยายามของผู้ใช้เพื่อให้ได้ผลลัพธ์ที่เหมาะสม แล้วทำไมถึงสำคัญ? เพื่ออธิบาย เราต้องเข้าใจสามแนวคิด:

- _Tokenization_ = โมเดล "มอง" คำสั่งอย่างไร  
- _Base LLMs_ = โมเดลพื้นฐาน "ประมวลผล" คำสั่งอย่างไร  
- _Instruction-Tuned LLMs_ = โมเดลที่ "รับรู้ภารกิจ"  

### Tokenization

LLM มองคำสั่งเป็น _ลำดับของ token_ ซึ่งโมเดลแต่ละตัว (หรือเวอร์ชันของโมเดล) อาจ tokenize คำสั่งเดียวกันแตกต่างกัน การที่ LLM ถูกฝึกด้วย token (ไม่ใช่ข้อความดิบ) วิธีที่คำสั่งถูก tokenize จะส่งผลโดยตรงต่อคุณภาพของคำตอบที่สร้างขึ้น

เพื่อให้เข้าใจวิธีการทำงานของ tokenization ลองใช้เครื่องมือเช่น [OpenAI Tokenizer](https://platform.openai.com/tokenizer?WT.mc_id=academic-105485-koreyst) ด้านล่าง วางคำสั่งของคุณ — แล้วดูว่าแปลงเป็น token อย่างไร โดยสังเกตว่าความว่างและเครื่องหมายวรรคตอนถูกจัดการอย่างไร โปรดทราบว่าตัวอย่างนี้ใช้โมเดล LLM รุ่นเก่า (GPT-3) — ดังนั้นการลองกับโมเดลใหม่อาจได้ผลลัพธ์ต่างกัน  

![Tokenization](../../../translated_images/th/04-tokenizer-example.e71f0a0f70356c5c.webp)

### แนวคิด: โมเดลพื้นฐาน (Foundation Models)

เมื่อคำสั่งถูก tokenize แล้ว ฟังก์ชันหลักของ ["Base LLM"](https://blog.gopenai.com/an-introduction-to-base-and-instruction-tuned-large-language-models-8de102c785a6?WT.mc_id=academic-105485-koreyst) (หรือโมเดลพื้นฐาน) คือการทำนาย token ถัดไปในลำดับนั้น เนื่องจาก LLM ได้รับการฝึกบนชุดข้อมูลข้อความจำนวนมาก พวกเขาจะมีความเข้าใจความสัมพันธ์เชิงสถิติระหว่าง token และสามารถทำนายได้ด้วยความมั่นใจระดับหนึ่ง โปรดทราบว่าโมเดลไม่ได้เข้าใจ _ความหมาย_ ของคำหรือ token ในคำสั่งพวกเขาแค่เห็นรูปแบบที่สามารถ "เติมให้ครบ" ด้วยการทำนายถัดไป พวกเขาสามารถทำนายลำดับต่อไปจนกว่าจะถูกหยุดโดยผู้ใช้หรือตามเงื่อนไขบางอย่างที่ตั้งไว้ล่วงหน้า  

อยากเห็นการทำงานของ completion แบบ prompt-based ใช่ไหม? ลองใส่ prompt ด้านบนใน Azure OpenAI Studio [_Chat Playground_](https://oai.azure.com/playground?WT.mc_id=academic-105485-koreyst) ด้วยการตั้งค่าเริ่มต้น ระบบจะถูกตั้งให้จัดการคำสั่งเป็นคำขอข้อมูล — คุณจะเห็นคำตอบที่ตรงกับบริบทนี้  

แต่ถ้าผู้ใช้ต้องการคำตอบเฉพาะที่ตรงตามเกณฑ์หรืองานหนึ่งนี่คือจุดที่ _instruction-tuned_ LLMs เข้ามามีบทบาท  

![Base LLM Chat Completion](../../../translated_images/th/04-playground-chat-base.65b76fcfde0caa67.webp)

### แนวคิด: Instruction Tuned LLMs

[Instruction Tuned LLM](https://blog.gopenai.com/an-introduction-to-base-and-instruction-tuned-large-language-models-8de102c785a6?WT.mc_id=academic-105485-koreyst) เริ่มจากโมเดลพื้นฐานและปรับแต่งเพิ่มเติมด้วยตัวอย่างหรือคู่ input/output (เช่น "ข้อความ" หลายรอบ) ที่มีคำสั่งชัดเจน — คำตอบจาก AI จะพยายามทำตามคำสั่งนั้น

ใช้เทคนิคเช่น Reinforcement Learning with Human Feedback (RLHF) ที่ช่วยฝึกโมเดลให้ _ปฏิบัติตามคำสั่ง_ และ _เรียนรู้จากผลตอบรับ_ เพื่อสร้างคำตอบที่เหมาะสมกับการใช้งานจริงและสอดคล้องกับวัตถุประสงค์ของผู้ใช้มากขึ้น  

ลองดู — กลับไปที่ prompt ข้างบน แต่เปลี่ยน _ข้อความระบบ_ เพื่อให้คำสั่งดังนี้เป็นบริบท:  

> _สรุปเนื้อหาที่ให้มาให้เด็กนักเรียนชั้นประถมปีที่ 2 ฟัง จำกัดผลลัพธ์ให้อยู่ในย่อหน้าเดียว พร้อมประเด็นย่อย 3-5 ข้อ_  

ดูว่าผลลัพธ์ตอนนี้ถูกปรับแต่งให้ตรงกับเป้าหมายและรูปแบบที่ต้องการอย่างไร ครูผู้สอนสามารถใช้คำตอบนี้โดยตรงในสไลด์สำหรับชั้นเรียนนั้นได้  

![Instruction Tuned LLM Chat Completion](../../../translated_images/th/04-playground-chat-instructions.b30bbfbdf92f2d05.webp)

## ทำไมเราต้องการ Prompt Engineering?

ตอนนี้ที่เรารู้ว่าคำสั่งถูกประมวลผลโดย LLM อย่างไร มาเจาะลึกว่า _ทำไม_ เราต้องการออกแบบคำสั่ง คำตอบอยู่ที่ว่า LLM ปัจจุบันมีความท้าทายหลายประการที่ทำให้ _การสร้าง completion ที่น่าเชื่อถือและสม่ำเสมอ_ เป็นเรื่องยากถ้าไม่มีความพยายามในการสร้างและปรับแต่งคำสั่ง เช่น:

1. **คำตอบของโมเดลมีความไม่แน่นอน**  _คำสั่งเดียวกัน_ อาจให้คำตอบที่แตกต่างกันกับโมเดลหรือเวอร์ชันของโมเดลที่ต่างกัน และอาจได้ผลลัพธ์แตกต่างกับ _โมเดลเดียวกัน_ ในเวลาต่างกันด้วย _เทคนิค prompt engineering ช่วยลดความแปรปรวนเหล่านี้โดยจัดการขอบเขตที่ดีกว่า_

1. **โมเดลอาจสร้างคำตอบที่ไม่ถูกต้อง** โมเดลถูกฝึกด้วยชุดข้อมูล _ขนาดใหญ่แต่จำกัด_ หมายความว่าไม่มีความรู้เกี่ยวกับแนวคิดนอกเหนือจากขอบเขตนั้น ทำให้โมเดลอาจสร้างคำตอบที่ไม่แม่นยำ จินตนาการ หรือขัดแย้งกับข้อเท็จจริงที่ทราบอยู่แล้ว _เทคนิค prompt engineering ช่วยให้ผู้ใช้ระบุและลดข้อผิดพลาดประเภทนี้ เช่น การขอให้ AI อ้างอิงแหล่งข้อมูลหรือเหตุผล_

1. **ความสามารถของโมเดลแตกต่างกัน** โมเดลใหม่หรือรุ่นใหม่จะมีความสามารถมากขึ้น แต่ก็มีลักษณะเฉพาะและข้อแลกเปลี่ยนในเรื่องต้นทุนและความซับซ้อน _Prompt engineering ช่วยให้เราพัฒนาวิธีปฏิบัติที่ดีที่สุดและเวิร์กโฟลว์ที่ซ่อนความแตกต่างและปรับให้เข้ากับความต้องการเฉพาะของโมเดลอย่างไร้รอยต่อและขยายได้_

ลองดูตัวอย่างการทำงานใน OpenAI หรือ Azure OpenAI Playground:

- ใช้คำสั่งเดียวกันกับการปรับใช้ LLM ต่างกัน (เช่น OpenAI, Azure OpenAI, Hugging Face) — คุณเห็นความแตกต่างไหม?  
- ใช้คำสั่งเดียวกันซ้ำกับ LLM เดิม (เช่น Azure OpenAI playground) — ความแตกต่างเหล่านี้เป็นอย่างไร?  

### ตัวอย่าง Fabrications

ในหลักสูตรนี้ เราใช้คำว่า **"fabrication"** เพื่ออ้างถึงปรากฏการณ์ที่ LLM บางครั้งสร้างข้อมูลที่ไม่ถูกต้องตามข้อเท็จจริงเนื่องจากข้อจำกัดในการฝึกหรือข้อจำกัดอื่น คุณอาจเคยได้ยินคำนี้เรียกว่า _"hallucinations"_ ในบทความหรือเอกสารวิจัยยอดนิยม อย่างไรก็ตาม เราขอแนะนำให้ใช้คำว่า _"fabrication"_ เพื่อหลีกเลี่ยงการตีความพฤติกรรมของโมเดลเป็นลักษณะมนุษย์ ซึ่งช่วยเสริม [แนวทางปฏิบัติ AI อย่างรับผิดชอบ](https://www.microsoft.com/ai/responsible-ai?WT.mc_id=academic-105485-koreyst) ในแง่คำศัพท์ด้วย โดยตัดคำที่อาจไม่เหมาะสมหรือไม่ครอบคลุมในบางบริบทออก  

อยากเข้าใจว่า fabrications เป็นอย่างไร? ลองคิดถึงคำสั่งที่บอก AI ให้สร้างเนื้อหาเกี่ยวกับหัวข้อที่ไม่มีอยู่จริง (เพื่อให้แน่ใจว่าไม่พบในชุดฝึก) ตัวอย่างเช่น — ฉันลองใช้คำสั่งนี้:  

> **Prompt:** สร้างแผนบทเรียนเกี่ยวกับสงครามมาร์ชันในปี 2076.
การค้นหาเว็บแสดงให้เห็นว่ามีเรื่องแต่ง (เช่น ซีรีส์โทรทัศน์หรือนวนิยาย) เกี่ยวกับสงครามดาวอังคาร — แต่ไม่มีเรื่องในปี 2076 ซึ่งความรู้ทั่วไปก็บอกเราว่าปี 2076 เป็น _ในอนาคต_ ดังนั้นจึงไม่สามารถเกี่ยวข้องกับเหตุการณ์จริงได้

แล้วจะเกิดอะไรขึ้นเมื่อเรารันพรอมต์นี้กับผู้ให้บริการ LLM ต่าง ๆ?

ตามที่คาดไว้แต่ละโมเดล (หรือเวอร์ชันโมเดล) ให้คำตอบที่แตกต่างกันเล็กน้อยเนื่องจากพฤติกรรมแบบสุ่มและความสามารถของโมเดลที่แตกต่างกัน ยกตัวอย่างเช่น โมเดลหนึ่งตั้งเป้าหมายสำหรับผู้ฟังชั้นประถมปีที่ 8 ในขณะที่อีกโมเดลหนึ่งสมมติว่าผู้ใช้เป็นนักเรียนมัธยมปลาย แต่ทั้งสามโมเดลก็สร้างคำตอบที่สามารถโน้มน้าวให้ผู้ใช้ที่ไม่มีข้อมูลเชื่อว่าเหตุการณ์นั้นเป็นเรื่องจริงได้

เทคนิคการออกแบบพรอมต์ เช่น _metaprompting_ และ _การตั้งค่าอุณหภูมิ_ อาจช่วยลดการสร้างข้อมูลเท็จของโมเดลในระดับหนึ่ง สถาปัตยกรรมใหม่ของการออกแบบพรอมต์ยังผสมผสานเครื่องมือและเทคนิคใหม่ ๆ เข้าไปในลำดับของพรอมต์อย่างไร้รอยต่อ เพื่อลดหรือบรรเทาผลกระทบบางอย่างเหล่านี้

## การสร้างพรอมต์

เราได้เห็นแล้วว่าทำไมการออกแบบพรอมต์ถึงสำคัญ — ตอนนี้มาทำความเข้าใจว่าพรอมต์ถูก _สร้างขึ้น_ อย่างไร เพื่อที่เราจะได้ประเมินเทคนิคต่าง ๆ สำหรับการออกแบบพรอมต์ที่มีประสิทธิภาพมากขึ้น

### พรอมต์พื้นฐาน

เริ่มต้นด้วยพรอมต์พื้นฐาน: ข้อความนำเข้าที่ส่งไปยังโมเดลโดยไม่มีบริบทอื่น ๆ นี่คือตัวอย่าง — เมื่อเราส่งคำไม่กี่คำแรกของเพลงชาติสหรัฐอเมริกาไปยัง OpenAI [Completion API](https://platform.openai.com/docs/api-reference/completions?WT.mc_id=academic-105485-koreyst) มันจะเติมข้อความต่อทันทีด้วยบรรทัดถัดไป แสดงพฤติกรรมการทำนายพื้นฐาน

| Prompt (Input)     | Completion (Output)                                                                                                                        |
| :----------------- | :----------------------------------------------------------------------------------------------------------------------------------------- |
| Oh say can you see | ฟังดูเหมือนว่าคุณเริ่มต้นด้วยเนื้อเพลง “The Star-Spangled Banner” เพลงชาติของสหรัฐอเมริกา เนื้อเพลงทั้งหมดคือ ... |

### พรอมต์ซับซ้อน

ตอนนี้เรามาเพิ่มบริบทและคำสั่งลงในพรอมต์พื้นฐานนั้น [Chat Completion API](https://learn.microsoft.com/azure/ai-services/openai/how-to/chatgpt?WT.mc_id=academic-105485-koreyst) ให้เราสร้างพรอมต์ซับซ้อนในรูปแบบของชุด _ข้อความ_ ด้วย:

- คู่อินพุต/เอาต์พุตสะท้อนการป้อนข้อมูลของ _ผู้ใช้_ และการตอบของ _ผู้ช่วย_
- ข้อความระบบที่ตั้งบริบทสำหรับพฤติกรรมหรือบุคลิกภาพของผู้ช่วย

คำขอตอนนี้อยู่ในรูปแบบดังต่อไปนี้ ซึ่ง _การตัดคำ_ (tokenization) จับข้อมูลสำคัญจากบริบทและบทสนทนาได้อย่างมีประสิทธิภาพ ตอนนี้ การเปลี่ยนบริบทของระบบจะมีผลต่อคุณภาพของคำตอบ ไม่แพ้การเปลี่ยนอินพุตของผู้ใช้

```python
response = openai.chat.completions.create(
    model="gpt-3.5-turbo",
    messages=[
        {"role": "system", "content": "You are a helpful assistant."},
        {"role": "user", "content": "Who won the world series in 2020?"},
        {"role": "assistant", "content": "The Los Angeles Dodgers won the World Series in 2020."},
        {"role": "user", "content": "Where was it played?"}
    ]
)
```

### พรอมต์แนวคำสั่ง

ในตัวอย่างข้างต้น พรอมต์ของผู้ใช้เป็นข้อความค้นหาง่าย ๆ ที่ตีความได้ว่าเป็นการร้องขอข้อมูล ด้วยพรอมต์ _แนวคำสั่ง_ เราสามารถใช้ข้อความนั้นระบุภารกิจได้ละเอียดขึ้น เพื่อให้คำแนะนำที่ดีกว่าแก่ AI ดังนี้คือตัวอย่าง:

| Prompt (Input)                                                                                                                                                                                                                         | Completion (Output)                                                                                                        | ประเภทคำสั่ง       |
| :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :------------------------------------------------------------------------------------------------------------------------- | :------------------ |
| เขียนคำอธิบายเกี่ยวกับสงครามกลางเมือง                                                                                                                                                                                            | _คืนย่อหน้าธรรมดา_                                                                                                       | ง่าย                |
| เขียนคำอธิบายเกี่ยวกับสงครามกลางเมือง พร้อมบอกวันที่และเหตุการณ์สำคัญ พร้อมบรรยายความสำคัญของเหตุการณ์เหล่านั้น                                                                                                                                           | _คืนย่อหน้าพร้อมรายการวันสำคัญและคำอธิบาย_                                                                               | ซับซ้อน             |
| เขียนคำอธิบายเกี่ยวกับสงครามกลางเมืองใน 1 ย่อหน้า พร้อมจุดหัวข้อ 3 จุดกับวันที่สำคัญและความสำคัญของแต่ละเหตุการณ์ ให้จุดหัวข้อเพิ่มอีก 3 จุดเกี่ยวกับบุคคลสำคัญทางประวัติศาสตร์และผลงานของพวกเขา ส่งผลลัพธ์เป็นไฟล์ JSON | _คืนคำอธิบายรายละเอียดมากขึ้นในกล่องข้อความ รูปแบบ JSON ให้สามารถคัดลอกไปวางในไฟล์และตรวจสอบได้_                          | ซับซ้อน. รูปแบบ.   |

## เนื้อหาหลัก

ในตัวอย่างข้างต้น พรอมต์ยังเปิดกว้างพอที่จะให้ LLM เลือกว่าข้อมูลในชุดฝึกสอนที่เรียนมาส่วนไหนสำคัญ ด้วยรูปแบบการออกแบบ _เนื้อหาหลัก_ ข้อความนำเข้าจะแบ่งเป็นสองส่วน:

- คำสั่ง (การกระทำ)
- เนื้อหาที่เกี่ยวข้อง (ที่มีอิทธิพลต่อการกระทำ)

สามารถใช้ส่วนเนื้อหาหลักได้หลายวิธีเพื่อสร้างคำสั่งที่มีประสิทธิภาพมากขึ้น:

- **ตัวอย่าง** — แทนที่จะบอกโมเดลให้ทำงานด้วยคำสั่งตรง ๆ ให้ป้อนตัวอย่างว่าต้องการอย่างไรและปล่อยให้โมเดลสรุปรูปแบบ
- **สัญญาณ** — ตามหลังคำสั่งด้วย “สัญญาณ” ที่เป็นการกระตุ้นให้คำตอบอยู่ในรูปแบบที่ต้องการ ช่วยชี้นำโมเดลให้ตอบสนองที่เกี่ยวข้องมากขึ้น
- **แม่แบบ** — คือ ‘สูตร’ ที่ใช้ซ้ำได้สำหรับพรอมต์ที่มีช่องว่าง (ตัวแปร) ซึ่งปรับแต่งด้วยข้อมูลเฉพาะสำหรับกรณีการใช้งานต่าง ๆ

### การใช้ตัวอย่าง

นี่คือแนวทางที่ใช้เนื้อหาหลักเพื่อ "ป้อนตัวอย่าง" ผลลัพธ์ที่ต้องการสำหรับคำสั่งที่กำหนด และให้มันสรุปรูปแบบจากตัวอย่าง โดยขึ้นอยู่กับจำนวนตัวอย่าง จะมี zero-shot prompting, one-shot prompting, few-shot prompting เป็นต้น

พรอมต์ตอนนี้ประกอบด้วยสามส่วน:

- คำอธิบายงาน
- ตัวอย่างผลลัพธ์ที่ต้องการสักสองสามตัวอย่าง
- จุดเริ่มต้นของตัวอย่างใหม่ (ซึ่งกลายเป็นคำอธิบายงานโดยปริยาย)

สังเกตว่าต้องให้คำสั่งที่ชัดเจน ("แปลเป็นภาษาสเปน") ใน zero-shot prompting แต่ในตัวอย่าง one-shot คำสั่งนั้นจะถูกสรุปออกมาเอง และตัวอย่าง few-shot แสดงว่าการเพิ่มตัวอย่างมากขึ้นช่วยให้โมเดลสรุปได้แม่นยำขึ้นโดยไม่ต้องเพิ่มคำสั่งใหม่

### Prompt Templates

แม่แบบพรอมต์คือ _สูตรล่วงหน้าสำหรับพรอมต์_ ที่สามารถเก็บไว้และนำกลับมาใช้ใหม่เมื่อจำเป็น เพื่อสร้างประสบการณ์ผู้ใช้ที่สม่ำเสมอมากขึ้นในวงกว้าง ในรูปแบบที่ง่ายที่สุด มันคือการรวบรวมตัวอย่างพรอมต์เช่น [ตัวอย่างนี้จาก OpenAI](https://cookbook.openai.com/examples/gpt4-1_prompting_guide?WT.mc_id=academic-105485-koreyst) ที่ให้ทั้งส่วนประกอบพรอมต์แบบโต้ตอบ (ข้อความของผู้ใช้และระบบ) และรูปแบบคำขอที่ขับเคลื่อนด้วย API - เพื่อรองรับการนำกลับมาใช้ใหม่

ในรูปแบบที่ซับซ้อนกว่า เช่น [ตัวอย่างนี้จาก LangChain](https://python.langchain.com/docs/concepts/prompt_templates/?WT.mc_id=academic-105485-koreyst) จะมี _ตัวแทน_ ที่สามารถแทนที่ด้วยข้อมูลจากแหล่งข้อมูลหลากหลาย (ข้อมูลผู้ใช้, บริบทระบบ, แหล่งข้อมูลภายนอก ฯลฯ) เพื่อสร้างพรอมต์อย่างไดนามิก ซึ่งช่วยให้เราสร้างคลังพรอมต์ที่นำกลับมาใช้ใหม่ได้เพื่อขับเคลื่อนประสบการณ์ผู้ใช้ที่สม่ำเสมอ **เชิงโปรแกรม** ในวงกว้าง

สุดท้าย ค่าที่แท้จริงของแม่แบบคือความสามารถในการสร้างและเผยแพร่ _คลังพรอมต์_ สำหรับโดเมนแอปพลิเคชันเฉพาะด้าน - ที่ซึ่งแม่แบบพรอมต์นั้นถูก _ปรับให้เหมาะสม_ เพื่อสะท้อนบริบทเฉพาะแอปพลิเคชันหรือตัวอย่างที่ทำให้การตอบสนองมีความเกี่ยวข้องและแม่นยำมากขึ้นสำหรับกลุ่มเป้าหมายผู้ใช้ ตัวอย่างที่ดีของแนวทางนี้คือ [Prompts For Edu](https://github.com/microsoft/prompts-for-edu?WT.mc_id=academic-105485-koreyst) ซึ่งรวบรวมคลังพรอมต์สำหรับโดเมนการศึกษาโดยเน้นวัตถุประสงค์สำคัญอย่างการวางแผนบทเรียน การออกแบบหลักสูตร การติวเตอร์นักเรียน ฯลฯ

## Supporting Content

ถ้าเราคิดถึงการสร้างพรอมต์เหมือนมีคำสั่ง (งาน) และเป้าหมาย (เนื้อหาหลัก) เนื้อหารองจะเหมือนกับบริบทเพิ่มเติมที่เรามอบให้เพื่อ **มีอิทธิพลต่อผลลัพธ์ในบางด้าน** อาจเป็นค่าการปรับแต่ง คำแนะนำการจัดรูปแบบ หมวดหมู่หัวข้อ ฯลฯ ที่ช่วยให้โมเดล _ปรับ_ การตอบให้ออกมาตรงตามวัตถุประสงค์หรือความคาดหวังของผู้ใช้

ตัวอย่างเช่น: หากมีกลุ่มข้อมูลหลักสูตรพร้อมเมตาดาต้าที่กว้างขวาง (ชื่อ, คำอธิบาย, ระดับ, แท็กเมตาดาต้า, ผู้สอน ฯลฯ) ของหลักสูตรทั้งหมดในหลักสูตรการเรียน:

- เราสามารถกำหนดคำสั่งให้ "สรุปรายชื่อหลักสูตรสำหรับฤดูใบไม้ร่วง 2023"
- ใช้เนื้อหาหลักเพื่อให้ตัวอย่างผลลัพธ์บางส่วนที่ต้องการ
- ใช้เนื้อหารองเพื่อระบุ 5 แท็กที่สนใจมากที่สุด

ตอนนี้ โมเดลสามารถให้สรุปในรูปแบบที่แสดงในตัวอย่าง แต่ถ้าผลลัพธ์มีหลายแท็ก มันจะให้ความสำคัญกับ 5 แท็กที่ระบุในเนื้อหารอง

## Prompting Best Practices

เมื่อเรารู้วิธีสร้างพรอมต์แล้ว เราสามารถเริ่มคิดเกี่ยวกับการ _ออกแบบ_ พรอมต์ให้สอดคล้องกับแนวทางปฏิบัติที่ดีที่สุด เราสามารถพิจารณาเป็นสองส่วน - มี _ทัศนคติ_ ที่ถูกต้องและใช้ _เทคนิค_ ที่เหมาะสม

### Prompt Engineering Mindset

การสร้างพรอมต์เป็นกระบวนการลองผิดลองถูก ดังนั้นให้จำสามปัจจัยครอบคลุมนี้ไว้:

1. **ความเข้าใจโดเมนสำคัญ** ความแม่นยำและความเกี่ยวข้องของการตอบขึ้นอยู่กับ _โดเมน_ ที่แอปพลิเคชันหรือผู้ใช้ทำงาน ปรับเทคนิคตามความรู้สึกและความชำนาญของโดเมน เช่น กำหนด _บุคลิกภาพเฉพาะโดเมน_ ในพรอมต์ระบบ หรือใช้ _แม่แบบเฉพาะโดเมน_ ในพรอมต์ผู้ใช้ ให้เนื้อหารองสะท้อนบริบทเฉพาะโดเมน หรือใช้ _สัญญาณและตัวอย่างเฉพาะโดเมน_ เพื่อชี้นำโมเดลเข้าสู่รูปแบบการใช้งานที่คุ้นเคย

2. **ความเข้าใจโมเดลสำคัญ** เรารู้ว่าโมเดลมีความสุ่มโดยธรรมชาติ แต่การนำโมเดลไปใช้ก็แตกต่างกันในเรื่องชุดข้อมูลเทรนนิ่ง (ความรู้ที่ฝึกไว้ล่วงหน้า) ความสามารถที่ให้ (เช่น ผ่าน API หรือ SDK) และประเภทรูปแบบเนื้อหาที่เหมาะ (เช่น โค้ด, ภาพ หรือข้อความ) เข้าใจจุดแข็งและข้อจำกัดของโมเดลที่ใช้ แล้วใช้ความรู้นั้นเพื่อ _จัดลำดับความสำคัญงาน_ หรือสร้าง _แม่แบบที่ปรับแต่ง_ เพื่อให้เหมาะกับความสามารถของโมเดลมากขึ้น

3. **การวนซ้ำและตรวจสอบสำคัญ** โมเดลพัฒนาเร็วมาก เช่นเดียวกับเทคนิคการสร้างพรอมต์ ในฐานะผู้เชี่ยวชาญโดเมน คุณอาจมีบริบทหรือเกณฑ์เฉพาะสำหรับ _แอปพลิเคชันของคุณเอง_ ซึ่งอาจไม่เหมือนกับของชุมชนทั่วไป ใช้เครื่องมือและเทคนิคสร้างพรอมต์เพื่อ "เริ่มต้น" การสร้างพรอมต์ จากนั้นวนซ้ำและตรวจสอบผลลัพธ์ด้วยความรู้สึกและความเชี่ยวชาญของคุณเอง บันทึกข้อมูลเชิงลึกและสร้าง **ฐานความรู้** (เช่น คลังพรอมต์) ที่ผู้อื่นสามารถใช้เป็นฐานใหม่สำหรับการวนซ้ำที่เร็วขึ้นในอนาคต

## Best Practices

มาดูแนวทางปฏิบัติที่แนะนำโดย [OpenAI](https://help.openai.com/en/articles/6654000-best-practices-for-prompt-engineering-with-openai-api?WT.mc_id=academic-105485-koreyst) และผู้เชี่ยวชาญ [Azure OpenAI](https://learn.microsoft.com/azure/ai-services/openai/concepts/prompt-engineering#best-practices?WT.mc_id=academic-105485-koreyst) กัน

| What                              | Why                                                                                                                                                                                                                                               |
| :-------------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| ประเมินโมเดลล่าสุด              | รุ่นโมเดลใหม่มักมีฟีเจอร์และคุณภาพดีขึ้น แต่ก็อาจมีต้นทุนสูงกว่า ประเมินผลกระทบแล้วตัดสินใจย้ายรุ่น                                                                                                                           |
| แยกคำสั่งกับบริบท              | ตรวจสอบว่าโมเดล/ผู้ให้บริการมีการกำหนด _ตัวแบ่ง_ เพื่อแยกคำสั่ง เนื้อหาหลัก และเนื้อหารองอย่างชัดเจน ซึ่งช่วยให้โมเดลกำหนดน้ำหนักโทเค็นได้แม่นยำยิ่งขึ้น                                                                                  |
| ชัดเจนและเฉพาะเจาะจง          | ให้รายละเอียดเกี่ยวกับบริบท ผลลัพธ์ ความยาว รูปแบบ สไตล์ ฯลฯ มากขึ้น ซึ่งจะช่วยปรับปรุงคุณภาพและความสม่ำเสมอของการตอบสนอง เก็บสูตรในแม่แบบที่นำกลับมาใช้ซ้ำได้                                                                     |
| ให้คำอธิบาย ใช้ตัวอย่าง        | โมเดลตอบสนองได้ดีขึ้นกับวิธี "แสดงและบอก" เริ่มจากวิธี `zero-shot` คือให้คำสั่งโดยไม่มีตัวอย่าง จากนั้นลอง `few-shot` เพื่อปรับปรุง โดยให้ตัวอย่างผลลัพธ์ที่ต้องการ ใช้การเปรียบเปรย                                                               |
| ใช้สัญญาณเพื่อเริ่มต้นคำตอบ    | กระตุ้นให้ไปถึงผลลัพธ์ที่ต้องการโดยให้คำหรือวลีเริ่มต้นที่โมเดลใช้เป็นจุดเริ่มสำหรับคำตอบ                                                                                                                                                |
| ทำซ้ำ                          | บางครั้งคุณอาจต้องพูดซ้ำกับโมเดล ให้คำสั่งก่อนและหลังเนื้อหาหลัก ใช้คำสั่งและสัญญาณ ฯลฯ วนซ้ำและตรวจสอบดูว่าอะไรได้ผล                                                                                                               |
| ลำดับมีความสำคัญ              | ลำดับที่นำเสนอข้อมูลให้โมเดลมีผลกับผลลัพธ์ แม้ในตัวอย่างการเรียนรู้ ก็เพราะอคติความสดใหม่ ลองตัวเลือกต่าง ๆ เพื่อดูว่าอะไรได้ผลดีที่สุด                                                                                                      |
| ให้ทางเลือกสำหรับโมเดล         | ให้โมเดลมี _คำตอบสำรอง_ ที่จะใช้หากไม่สามารถทำงานให้เสร็จได้ด้วยเหตุผลใดก็ตาม วิธีนี้ช่วยลดโอกาสที่โมเดลจะตอบผิดหรือประดิษฐ์ข้อมูล                                                                                                           |
|                                   |                                                                                                                                                                                                                                                   |

เช่นเดียวกับแนวปฏิบัติที่ดีที่สุดใด ๆ อย่าลืมว่า _ประสบการณ์ของคุณอาจแตกต่าง_ ขึ้นอยู่กับโมเดล งาน และโดเมน ใช้เป็นจุดเริ่มต้นแล้ววนซ้ำเพื่อค้นหาสิ่งที่ดีที่สุดสำหรับคุณ ประเมินกระบวนการสร้างพรอมต์ของคุณอย่างสม่ำเสมอเมื่อมีโมเดลและเครื่องมือใหม่ ๆ พร้อมใช้งาน โดยมุ่งเน้นที่ความสามารถในการขยายกระบวนการและคุณภาพการตอบสนอง

## Assignment

ยินดีด้วย! คุณมาถึงตอนจบบทเรียนแล้ว ถึงเวลาทดลองใช้แนวคิดและเทคนิคเหล่านั้นกับตัวอย่างจริง!

สำหรับงานที่มอบหมาย เราจะใช้ Jupyter Notebook กับแบบฝึกหัดที่คุณทำได้แบบโต้ตอบ คุณยังสามารถขยาย Notebook ด้วยเซลล์ Markdown และโค้ดของตัวเองเพื่อสำรวจไอเดียและเทคนิคเพิ่มเติมได้

### เริ่มต้นโดยการโคลน repo แล้ว

- (แนะนำ) เปิดใช้งาน GitHub Codespaces
- (อีกทางเลือก) โคลน repo ลงอุปกรณ์ของคุณแล้วใช้กับ Docker Desktop
- (อีกทางเลือก) เปิด Notebook ด้วยสภาพแวดล้อมรันไทม์ Notebook ที่คุณชื่นชอบ

### ถัดไป ตั้งค่าตัวแปรสภาพแวดล้อมของคุณ

- คัดลอกไฟล์ `.env.copy` ในโฟลเดอร์รากของ repo เป็น `.env` แล้วกรอกค่า `AZURE_OPENAI_API_KEY`, `AZURE_OPENAI_ENDPOINT` และ `AZURE_OPENAI_DEPLOYMENT` กลับไปที่ส่วน [Learning Sandbox](#learning-sandbox) เพื่อเรียนรู้วิธีตั้งค่า

### ตอนนี้ เปิด Jupyter Notebook

- เลือกเคอร์เนลรันไทม์ หากใช้ตัวเลือก 1 หรือ 2 ให้เลือกเคอร์เนล Python 3.10.x ที่มีอยู่ใน dev container ได้เลย

พร้อมแล้วสำหรับการรันแบบฝึกหัด โปรดทราบว่าไม่มีคำตอบที่ถูกหรือผิดอย่างชัดเจน มีเพียงการสำรวจตัวเลือกด้วยการลองผิดลองถูกและสร้างความรู้สึกว่าอะไรเหมาะกับโมเดลและโดเมนแอปพลิเคชันใด

_ด้วยเหตุนี้จึงไม่มีเซกเมนต์โค้ดตัวอย่างในบทเรียนนี้ แต่ Notebook จะมีเซลล์ Markdown ชื่อ "My Solution:" ที่แสดงตัวอย่างผลลัพธ์เพื่ออ้างอิง_
