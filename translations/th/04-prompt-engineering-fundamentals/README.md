# พื้นฐานการออกแบบ Prompt

## บทนำ
โมดูลนี้ครอบคลุมแนวคิดและเทคนิคที่จำเป็นในการสร้าง prompt ที่มีประสิทธิภาพสำหรับโมเดล AI ที่สร้างสรรค์ วิธีที่คุณเขียน prompt ให้กับ LLM ก็มีความสำคัญเช่นกัน การสร้าง prompt อย่างรอบคอบสามารถทำให้ได้คุณภาพของการตอบสนองที่ดียิ่งขึ้น แต่คำว่า _prompt_ และ _prompt engineering_ หมายถึงอะไร? และเราจะปรับปรุงการป้อนข้อมูล prompt ที่เราส่งไปยัง LLM ได้อย่างไร? นี่คือคำถามที่เราจะพยายามตอบในบทนี้และบทต่อไป

_Generative AI_ สามารถสร้างเนื้อหาใหม่ (เช่น ข้อความ รูปภาพ เสียง โค้ด เป็นต้น) เพื่อตอบสนองต่อคำขอของผู้ใช้ โดยใช้ _Large Language Models_ เช่น ซีรีส์ GPT ของ OpenAI ("Generative Pre-trained Transformer") ที่ได้รับการฝึกอบรมสำหรับการใช้ภาษาธรรมชาติและโค้ด

ผู้ใช้สามารถโต้ตอบกับโมเดลเหล่านี้ได้โดยใช้แนวทางที่คุ้นเคย เช่น การแชท โดยไม่จำเป็นต้องมีความเชี่ยวชาญทางเทคนิคหรือการฝึกอบรม โมเดลเหล่านี้เป็น _prompt-based_ - ผู้ใช้ส่งข้อความป้อนข้อมูล (prompt) และได้รับการตอบสนองจาก AI (completion) จากนั้นพวกเขาสามารถ "แชทกับ AI" ได้อย่างต่อเนื่องในบทสนทนาแบบหลายรอบ โดยปรับแต่ง prompt จนกว่าการตอบสนองจะตรงตามความคาดหวัง

"Prompts" กลายเป็น _อินเทอร์เฟซการเขียนโปรแกรม_ หลักสำหรับแอปพลิเคชัน AI ที่สร้างสรรค์ โดยบอกโมเดลว่าต้องทำอะไรและมีอิทธิพลต่อคุณภาพของการตอบสนองที่ส่งกลับ "Prompt Engineering" เป็นสาขาการศึกษาที่เติบโตอย่างรวดเร็วซึ่งมุ่งเน้นไปที่ _การออกแบบและการเพิ่มประสิทธิภาพ_ ของ prompts เพื่อให้ได้การตอบสนองที่สม่ำเสมอและมีคุณภาพในระดับที่ใหญ่ขึ้น

## เป้าหมายการเรียนรู้

ในบทเรียนนี้ เราจะเรียนรู้ว่า Prompt Engineering คืออะไร ทำไมถึงสำคัญ และเราจะสร้าง prompt ที่มีประสิทธิภาพมากขึ้นสำหรับโมเดลและวัตถุประสงค์ของแอปพลิเคชันที่กำหนดได้อย่างไร เราจะเข้าใจแนวคิดหลักและแนวทางปฏิบัติที่ดีที่สุดสำหรับ prompt engineering - และเรียนรู้เกี่ยวกับสภาพแวดล้อม "sandbox" ของ Jupyter Notebooks ที่เราสามารถเห็นแนวคิดเหล่านี้ถูกนำไปใช้กับตัวอย่างจริง

เมื่อจบบทเรียนนี้ เราจะสามารถ:

1. อธิบายว่า prompt engineering คืออะไรและทำไมถึงสำคัญ
2. อธิบายองค์ประกอบของ prompt และวิธีการใช้งาน
3. เรียนรู้แนวทางปฏิบัติที่ดีที่สุดและเทคนิคสำหรับ prompt engineering
4. นำเทคนิคที่เรียนรู้ไปใช้กับตัวอย่างจริงโดยใช้ OpenAI endpoint

## คำศัพท์สำคัญ

Prompt Engineering: การปฏิบัติในการออกแบบและปรับแต่งการป้อนข้อมูลเพื่อชี้นำโมเดล AI ให้ผลิตผลลัพธ์ที่ต้องการ
Tokenization: กระบวนการแปลงข้อความเป็นหน่วยย่อยที่เรียกว่า tokens ที่โมเดลสามารถเข้าใจและประมวลผลได้
Instruction-Tuned LLMs: Large Language Models (LLMs) ที่ได้รับการปรับแต่งด้วยคำสั่งเฉพาะเพื่อปรับปรุงความแม่นยำและความเกี่ยวข้องของการตอบสนอง

## พื้นที่เรียนรู้

Prompt engineering ปัจจุบันยังเป็นศิลปะมากกว่าวิทยาศาสตร์ วิธีที่ดีที่สุดในการปรับปรุงสัญชาตญาณของเราคือ _ฝึกฝนมากขึ้น_ และใช้วิธีการทดลองและข้อผิดพลาดที่รวมความเชี่ยวชาญด้านโดเมนแอปพลิเคชันกับเทคนิคที่แนะนำและการเพิ่มประสิทธิภาพเฉพาะโมเดล

Jupyter Notebook ที่มาพร้อมกับบทเรียนนี้ให้สภาพแวดล้อม _sandbox_ ที่คุณสามารถลองใช้สิ่งที่คุณเรียนรู้ - ตามที่คุณไปหรือเป็นส่วนหนึ่งของความท้าทายด้านโค้ดในตอนท้าย ในการดำเนินการแบบฝึกหัด คุณจะต้องการ:

1. **คีย์ API ของ Azure OpenAI** - จุดบริการสำหรับ LLM ที่ปรับใช้แล้ว
2. **Python Runtime** - ที่สามารถเรียกใช้ Notebook ได้
3. **ตัวแปรสภาพแวดล้อมในเครื่อง** - _ทำตามขั้นตอน [SETUP](./../00-course-setup/SETUP.md?WT.mc_id=academic-105485-koreyst) ตอนนี้เพื่อเตรียมความพร้อม_

Notebook มาพร้อมกับแบบฝึกหัด _เริ่มต้น_ - แต่คุณได้รับการสนับสนุนให้เพิ่มส่วน _Markdown_ (คำอธิบาย) และ _Code_ (คำขอ prompt) ของคุณเองเพื่อลองใช้ตัวอย่างหรือไอเดียเพิ่มเติม - และสร้างสัญชาตญาณของคุณสำหรับการออกแบบ prompt

## คู่มือภาพประกอบ

ต้องการเห็นภาพรวมของสิ่งที่บทเรียนนี้ครอบคลุมก่อนที่คุณจะดำน้ำหรือไม่? ดูคู่มือภาพประกอบนี้ซึ่งให้ความรู้สึกถึงหัวข้อหลักที่ครอบคลุมและข้อคิดสำคัญที่คุณควรพิจารณาในแต่ละหัวข้อ แผนที่เส้นทางของบทเรียนจะพาคุณจากการทำความเข้าใจแนวคิดหลักและความท้าทายไปสู่การแก้ไขปัญหาด้วยเทคนิคการออกแบบ prompt ที่เกี่ยวข้องและแนวทางปฏิบัติที่ดีที่สุด โปรดทราบว่าส่วน "เทคนิคขั้นสูง" ในคู่มือนี้อ้างถึงเนื้อหาที่ครอบคลุมในบท _ถัดไป_ ของหลักสูตรนี้

## สตาร์ทอัพของเรา

ตอนนี้เรามาพูดถึงว่า _หัวข้อนี้_ เกี่ยวข้องกับพันธกิจสตาร์ทอัพของเราในการ [นำ AI มาสู่การศึกษา](https://educationblog.microsoft.com/2023/06/collaborating-to-bring-ai-innovation-to-education?WT.mc_id=academic-105485-koreyst) อย่างไร เราต้องการสร้างแอปพลิเคชันที่ใช้ AI สำหรับ _การเรียนรู้แบบส่วนบุคคล_ - ดังนั้นลองคิดดูว่าผู้ใช้ที่แตกต่างกันของแอปพลิเคชันของเราอาจ "ออกแบบ" prompt อย่างไร:

- **ผู้ดูแลระบบ** อาจขอให้ AI _วิเคราะห์ข้อมูลหลักสูตรเพื่อระบุช่องว่างในความครอบคลุม_ AI สามารถสรุปผลลัพธ์หรือแสดงภาพด้วยโค้ด
- **ครูผู้สอน** อาจขอให้ AI _สร้างแผนการสอนสำหรับกลุ่มเป้าหมายและหัวข้อ_ AI สามารถสร้างแผนที่ปรับแต่งในรูปแบบที่ระบุ
- **นักเรียน** อาจขอให้ AI _สอนพวกเขาในวิชาที่ยาก_ AI สามารถแนะนำบทเรียน คำแนะนำ & ตัวอย่างที่ปรับให้เหมาะกับระดับของพวกเขาได้

นั่นเป็นเพียงส่วนหนึ่งของภูเขาน้ำแข็งเท่านั้น ลองดู [Prompts For Education](https://github.com/microsoft/prompts-for-edu/tree/main?WT.mc_id=academic-105485-koreyst) - ห้องสมุด prompts แบบโอเพ่นซอร์สที่คัดสรรโดยผู้เชี่ยวชาญด้านการศึกษา - เพื่อให้คุณเห็นภาพรวมของความเป็นไปได้ที่กว้างขึ้น! _ลองใช้บางส่วนของ prompts เหล่านั้นใน sandbox หรือใช้ OpenAI Playground เพื่อดูว่าจะเกิดอะไรขึ้น!_

## Prompt Engineering คืออะไร?

เราเริ่มบทเรียนนี้โดยการกำหนด **Prompt Engineering** ว่าเป็นกระบวนการของ _การออกแบบและเพิ่มประสิทธิภาพ_ ข้อความป้อนข้อมูล (prompts) เพื่อให้ได้การตอบสนองที่สม่ำเสมอและมีคุณภาพ (completions) สำหรับวัตถุประสงค์ของแอปพลิเคชันและโมเดลที่กำหนด เราสามารถคิดว่านี่เป็นกระบวนการสองขั้นตอน:

- _การออกแบบ_ prompt เริ่มต้นสำหรับโมเดลและวัตถุประสงค์ที่กำหนด
- _การปรับแต่ง_ prompt อย่างต่อเนื่องเพื่อปรับปรุงคุณภาพของการตอบสนอง

นี่เป็นกระบวนการที่ต้องใช้การลองผิดลองถูกซึ่งต้องใช้สัญชาตญาณและความพยายามของผู้ใช้เพื่อให้ได้ผลลัพธ์ที่ดีที่สุด แล้วทำไมมันถึงสำคัญ? เพื่อตอบคำถามนั้น เราต้องเข้าใจสามแนวคิดนี้ก่อน:

- _Tokenization_ = วิธีที่โมเดล "เห็น" prompt
- _Base LLMs_ = วิธีที่โมเดลพื้นฐาน "ประมวลผล" prompt
- _Instruction-Tuned LLMs_ = วิธีที่โมเดลสามารถเห็น "งาน" ได้

### Tokenization

LLM เห็น prompt เป็น _ลำดับของ tokens_ ซึ่งโมเดลต่างๆ (หรือเวอร์ชันของโมเดล) สามารถ tokenized prompt เดียวกันได้ในหลายวิธี เนื่องจาก LLMs ได้รับการฝึกอบรมบน tokens (และไม่ใช่บนข้อความดิบ) วิธีที่ prompt ถูก tokenized มีผลโดยตรงต่อคุณภาพของการตอบสนองที่สร้างขึ้น

เพื่อให้ได้สัญชาตญาณว่า tokenization ทำงานอย่างไร ลองใช้เครื่องมือเช่น [OpenAI Tokenizer](https://platform.openai.com/tokenizer?WT.mc_id=academic-105485-koreyst) ที่แสดงด้านล่าง คัดลอก prompt ของคุณ - และดูว่ามันถูกแปลงเป็น tokens อย่างไร โดยให้ความสนใจกับวิธีการจัดการอักขระเว้นวรรคและเครื่องหมายวรรคตอน โปรดทราบว่าตัวอย่างนี้แสดง LLM เก่า (GPT-3) - ดังนั้นการลองใช้กับโมเดลใหม่กว่าอาจให้ผลลัพธ์ที่แตกต่างออกไป

### แนวคิด: โมเดลพื้นฐาน

เมื่อ prompt ถูก tokenized ฟังก์ชันหลักของ ["Base LLM"](https://blog.gopenai.com/an-introduction-to-base-and-instruction-tuned-large-language-models-8de102c785a6?WT.mc_id=academic-105485-koreyst) (หรือ Foundation model) คือการทำนาย token ในลำดับนั้น เนื่องจาก LLMs ได้รับการฝึกอบรมบนชุดข้อมูลข้อความขนาดใหญ่ พวกเขามีความเข้าใจที่ดีเกี่ยวกับความสัมพันธ์ทางสถิติระหว่าง tokens และสามารถทำการทำนายนั้นด้วยความมั่นใจบางอย่าง โปรดทราบว่าพวกเขาไม่เข้าใจ _ความหมาย_ ของคำใน prompt หรือ token; พวกเขาแค่เห็นรูปแบบที่สามารถ "ทำให้เสร็จ" ด้วยการทำนายครั้งต่อไป พวกเขาสามารถทำนายลำดับต่อไปได้จนกว่าจะถูกยุติโดยการแทรกแซงของผู้ใช้หรือเงื่อนไขที่กำหนดไว้ล่วงหน้า

ต้องการดูว่าการทำให้เสร็จตาม prompt ทำงานอย่างไร? ป้อน prompt ข้างต้นลงใน Azure OpenAI Studio [_Chat Playground_](https://oai.azure.com/playground?WT.mc_id=academic-105485-koreyst) ด้วยการตั้งค่าเริ่มต้น ระบบถูกกำหนดค่าให้ปฏิบัติต่อ prompt เป็นคำขอข้อมูล - ดังนั้นคุณควรเห็นการทำให้เสร็จที่ตอบสนองบริบทนี้

แต่ถ้าผู้ใช้ต้องการเห็นสิ่งที่เฉพาะเจาะจงที่ตรงตามเกณฑ์หรือวัตถุประสงค์ของงานล่ะ? นี่คือที่ที่ _instruction-tuned_ LLMs เข้ามามีบทบาท

### แนวคิด: Instruction Tuned LLMs

[Instruction Tuned LLM](https://blog.gopenai.com/an-introduction-to-base-and-instruction-tuned-large-language-models-8de102c785a6?WT.mc_id=academic-105485-koreyst) เริ่มต้นด้วยโมเดลพื้นฐานและปรับแต่งด้วยตัวอย่างหรือคู่ป้อนข้อมูล/ผลลัพธ์ (เช่น "ข้อความ" หลายรอบ) ที่สามารถมีคำสั่งที่ชัดเจน - และการตอบสนองจาก AI พยายามทำตามคำสั่งนั้น

สิ่งนี้ใช้เทคนิคเช่น Reinforcement Learning with Human Feedback (RLHF) ที่สามารถฝึกโมเดลให้ _ทำตามคำสั่ง_ และ _เรียนรู้จากข้อเสนอแนะ_ เพื่อให้มันผลิตการตอบสนองที่เหมาะสมกับการใช้งานจริงและเกี่ยวข้องกับวัตถุประสงค์ของผู้ใช้มากขึ้น

ลองใช้ดู - กลับไปที่ prompt ข้างต้น แต่ตอนนี้เปลี่ยน _ข้อความระบบ_ เพื่อให้คำสั่งต่อไปนี้เป็นบริบท:

> _สรุปเนื้อหาที่คุณได้รับสำหรับนักเรียนชั้นประถมศึกษาปีที่สอง เก็บผลลัพธ์ให้เป็นหนึ่งย่อหน้าพร้อม 3-5 ข้อ_

ดูว่าผลลัพธ์ตอนนี้ได้รับการปรับให้สะท้อนถึงเป้าหมายและรูปแบบที่ต้องการหรือไม่? ครูสามารถใช้การตอบสนองนี้ในสไลด์สำหรับชั้นเรียนได้โดยตรง

## ทำไมเราถึงต้องการ Prompt Engineering?

ตอนนี้ที่เรารู้ว่า prompts ถูกประมวลผลโดย LLMs อย่างไร ลองพูดถึง _ทำไม_ เราต้องการ prompt engineering คำตอบอยู่ในความจริงที่ว่า LLMs ปัจจุบันมีความท้าทายหลายประการที่ทำให้ _การทำให้เสร็จที่เชื่อถือได้และสม่ำเสมอ_ ยากขึ้นในการบรรลุผลโดยไม่ต้องใส่ความพยายามในการสร้างและเพิ่มประสิทธิภาพ prompt ตัวอย่างเช่น:

1. **การตอบสนองของโมเดลเป็นแบบสุ่ม** prompt _เดียวกัน_ มีแนวโน้มที่จะผลิตการตอบสนองที่แตกต่างกันด้วยโมเดลต่างๆ หรือเวอร์ชันของโมเดล และอาจผลิตผลลัพธ์ที่แตกต่างกันกับ _โมเดลเดียวกัน_ ในเวลาที่ต่างกัน _เทคนิค prompt engineering สามารถช่วยเราลดความแปรปรวนเหล่านี้ได้โดยการให้กรอบการทำงานที่ดีกว่า_

2. **โมเดลสามารถสร้างการตอบสนองขึ้นมาเอง** โมเดลถูกฝึกด้วยชุดข้อมูลขนาดใหญ่แต่มีขอบเขต ซึ่งหมายความว่าพวกเขาขาดความรู้เกี่ยวกับแนวคิดนอกเหนือจากขอบเขตการฝึกอบรม ส่งผลให้พวกเขาสามารถสร้างการตอบสนองที่ไม่ถูกต้องหรือขัดแย้งกับข้อเท็จจริงที่รู้จัก _เทคนิค prompt engineering ช่วยให้ผู้ใช้ระบุและลดการสร้างขึ้นมาเองได้ เช่น โดยการขอ AI สำหรับการอ้างอิงหรือการให้เหตุผล_

3. **ความสามารถของโมเดลจะแตกต่างกัน** โมเดลใหม่หรือรุ่นของโมเดลจะมีความสามารถที่หลากหลายขึ้นแต่ก็นำเสนอความเฉพาะตัวและข้อแลกเปลี่ยนในด้านต้นทุนและความซับซ้อน _prompt engineering สามารถช่วยเราพัฒนาแนวทางปฏิบัติที่ดีที่สุดและเวิร์กโฟลว์ที่สามารถลดความแตกต่างและปรับให้เข้ากับข้อกำหนดเฉพาะโมเดลในลักษณะที่ปรับขนาดได้และไร้รอยต่อ_

ลองดูสิ่งนี้ใน OpenAI หรือ Azure OpenAI Playground:

- ใช้ prompt เดียวกันกับการติดตั้ง LLM ที่แตกต่างกัน (เช่น OpenAI, Azure OpenAI, Hugging Face) - คุณเห็นความแปรปรวนหรือไม่?
- ใช้ prompt เดียวกันซ้ำๆ กับการติดตั้ง LLM _เดียวกัน_ (เช่น Azure OpenAI playground) - ความแปรปรวนเหล่านี้แตกต่างกันอย่างไร?

### ตัวอย่างการสร้างขึ้นมาเอง

ในหลักสูตรนี้ เราใช้คำว่า **"การสร้างขึ้นมาเอง"** เพื่ออ้างถึงปรากฏการณ์ที่ LLMs บางครั้งสร้างข้อมูลที่ไม่ถูกต้องเนื่องจากข้อจำกัดในการฝึกอบรมหรือข้อจำกัดอื่นๆ คุณอาจเคยได้ยินว่ามีการอ้างถึงสิ่งนี้ว่า _"การสร้างขึ้นมาเอง"_ ในบทความหรือเอกสารวิจัยยอดนิยม อย่างไรก็ตาม เราขอแนะนำให้ใช้ _"การสร้างขึ้นมาเอง"_ เป็นคำเพื่อที่เราจะไม่เผลอให้ลักษณะมนุษย์กับผลลัพธ์ที่เกิดจากเครื่องจักร นี่เป็นการเสริมสร้างแนวทาง [Responsible AI guidelines](https://www.microsoft.com/ai/responsible-ai?WT.mc_id=academic-105485-koreyst) จากมุมมองของคำศัพท์ การลบคำที่อาจถูกมองว่าเป็นการล่วงละเมิดหรือไม่ครอบคลุมในบางบริบท

ต้องการเห็นภาพการทำงานของการสร้างขึ้นมาเองหรือไม่? ลองคิดถึง prompt ที่สั่งให้ AI สร้างเนื้อหาสำหรับหัวข้อที่ไม่มีอยู่จริง (เพื่อให้แน่ใจว่าไม่มีอยู่ในชุดข้อมูลการฝึกอบรม) ตัวอย่างเช่น - ฉันลองใช้ prompt นี้:

> **Prompt:** สร้างแผนการสอนเกี่ยวกับสงครามบนดาวอังคารในปี 2076

การค้นหาเว็บแสดงให้ฉันเห็นว่ามีเรื่องราวสมมุติ (เช่น ซีรีส์โทรทัศน์หรือหนังสือ) เกี่ยวกับสงครามบนดาวอังคาร - แต่ไม่มีในปี 2076 สามัญสำนึกยังบอกเราว่า 2076 เป็น _อนาคต_ และดังนั้นจึงไม่สามารถเกี่ยวข้อง
ในที่สุด ความสำคัญที่แท้จริงของแม่แบบคือความสามารถในการสร้างและเผยแพร่ _ห้องสมุดของคำสั่ง_ สำหรับโดเมนแอปพลิเคชันเฉพาะ ซึ่งแม่แบบคำสั่งได้รับการ _ปรับแต่ง_ ให้สะท้อนบริบทหรือตัวอย่างเฉพาะของแอปพลิเคชันที่ทำให้การตอบสนองมีความเกี่ยวข้องและแม่นยำมากขึ้นสำหรับกลุ่มผู้ใช้ที่ตั้งเป้าหมายไว้ [Prompts For Edu](https://github.com/microsoft/prompts-for-edu?WT.mc_id=academic-105485-koreyst) เป็นตัวอย่างที่ดีของวิธีการนี้ โดยการรวบรวมห้องสมุดของคำสั่งสำหรับโดเมนการศึกษา โดยเน้นไปที่วัตถุประสงค์หลัก เช่น การวางแผนการสอน การออกแบบหลักสูตร การสอนนักเรียน เป็นต้น

## เนื้อหาที่สนับสนุน

หากเราคิดเกี่ยวกับการสร้างคำสั่งว่าเป็นการมีคำแนะนำ (งาน) และเป้าหมาย (เนื้อหาหลัก) แล้ว _เนื้อหาทางเลือก_ ก็เหมือนบริบทเพิ่มเติมที่เรามอบให้เพื่อ **มีอิทธิพลต่อผลลัพธ์ในบางวิธี** อาจเป็นการปรับค่าพารามิเตอร์ คำแนะนำการจัดรูปแบบ การจัดประเภทหัวข้อ เป็นต้น ที่สามารถช่วยให้โมเดล _ปรับแต่ง_ การตอบสนองให้เหมาะสมกับวัตถุประสงค์หรือความคาดหวังของผู้ใช้ที่ต้องการ

ตัวอย่างเช่น: ให้แคตตาล็อกหลักสูตรที่มีเมตาดาต้ามากมาย (ชื่อ คำอธิบาย ระดับ แท็กเมตาดาต้า ผู้สอน เป็นต้น) ของหลักสูตรทั้งหมดในหลักสูตร:

- เราสามารถกำหนดคำแนะนำให้ "สรุปแคตตาล็อกหลักสูตรสำหรับฤดูใบไม้ร่วงปี 2023"
- เราสามารถใช้เนื้อหาหลักเพื่อให้ตัวอย่างบางส่วนของผลลัพธ์ที่ต้องการ
- เราสามารถใช้เนื้อหาทางเลือกเพื่อระบุ 5 "แท็ก" ที่สนใจที่สุด

ตอนนี้ โมเดลสามารถให้สรุปในรูปแบบที่แสดงโดยตัวอย่างบางส่วน - แต่หากผลลัพธ์มีแท็กหลายแท็ก ก็สามารถจัดลำดับความสำคัญของ 5 แท็กที่ระบุในเนื้อหาทางเลือกได้

## แนวทางปฏิบัติที่ดีที่สุดสำหรับการใช้คำสั่ง

ตอนนี้ที่เรารู้วิธีการ _สร้าง_ คำสั่งแล้ว เราสามารถเริ่มคิดเกี่ยวกับวิธีการ _ออกแบบ_ คำสั่งเพื่อสะท้อนแนวทางปฏิบัติที่ดีที่สุด เราสามารถคิดเกี่ยวกับเรื่องนี้ในสองส่วน - การมี _ทัศนคติ_ ที่ถูกต้องและการใช้ _เทคนิค_ ที่เหมาะสม

### ทัศนคติในการออกแบบคำสั่ง

การออกแบบคำสั่งเป็นกระบวนการลองผิดลองถูก ดังนั้นควรจำไว้เสมอถึงสามปัจจัยหลักที่ชี้นำ:

1. **ความเข้าใจในโดเมนสำคัญ.** ความแม่นยำและความเกี่ยวข้องของการตอบสนองเป็นผลของ _โดเมน_ ที่แอปพลิเคชันหรือผู้ใช้ดำเนินการอยู่ ใช้สัญชาตญาณและความเชี่ยวชาญในโดเมนของคุณเพื่อ **ปรับแต่งเทคนิค** เพิ่มเติม เช่น กำหนด _บุคลิกเฉพาะโดเมน_ ในคำสั่งของระบบ หรือใช้ _แม่แบบเฉพาะโดเมน_ ในคำสั่งของผู้ใช้ ให้เนื้อหาทางเลือกที่สะท้อนบริบทเฉพาะของโดเมน หรือใช้ _สัญญาณและตัวอย่างเฉพาะโดเมน_ เพื่อแนะนำโมเดลไปสู่รูปแบบการใช้งานที่คุ้นเคย

2. **ความเข้าใจในโมเดลสำคัญ.** เรารู้ว่าโมเดลมีลักษณะสุ่มโดยธรรมชาติ แต่การใช้งานโมเดลอาจแตกต่างกันในแง่ของชุดข้อมูลการฝึกอบรมที่ใช้ (ความรู้ที่ผ่านการฝึกอบรม) ความสามารถที่มีให้ (เช่น ผ่าน API หรือ SDK) และประเภทของเนื้อหาที่ปรับแต่งให้เหมาะสม (เช่น โค้ด vs. รูปภาพ vs. ข้อความ) เข้าใจจุดแข็งและข้อจำกัดของโมเดลที่คุณใช้งาน และใช้ความรู้นั้นเพื่อ _จัดลำดับความสำคัญของงาน_ หรือสร้าง _แม่แบบที่ปรับแต่ง_ ที่เหมาะสมกับความสามารถของโมเดล

3. **การวนซ้ำและการตรวจสอบสำคัญ.** โมเดลกำลังพัฒนาอย่างรวดเร็ว และเทคนิคสำหรับการออกแบบคำสั่งก็เช่นกัน ในฐานะผู้เชี่ยวชาญในโดเมน คุณอาจมีบริบทหรือเกณฑ์อื่น ๆ _แอปพลิเคชันเฉพาะของคุณ_ ที่อาจไม่ใช้กับชุมชนทั่วไป ใช้เครื่องมือและเทคนิคการออกแบบคำสั่งเพื่อ "เริ่มต้น" การสร้างคำสั่ง จากนั้นวนซ้ำและตรวจสอบผลลัพธ์โดยใช้สัญชาตญาณและความเชี่ยวชาญในโดเมนของคุณ บันทึกข้อมูลเชิงลึกของคุณและสร้าง **ฐานความรู้** (เช่น ห้องสมุดของคำสั่ง) ที่สามารถใช้เป็นฐานใหม่โดยผู้อื่น เพื่อการวนซ้ำที่เร็วขึ้นในอนาคต

ตอนนี้มาดูแนวทางปฏิบัติที่ดีที่สุดที่เป็นที่แนะนำโดยผู้ปฏิบัติงานของ [OpenAI](https://help.openai.com/en/articles/6654000-best-practices-for-prompt-engineering-with-openai-api?WT.mc_id=academic-105485-koreyst) และ [Azure OpenAI](https://learn.microsoft.com/azure/ai-services/openai/concepts/prompt-engineering#best-practices?WT.mc_id=academic-105485-koreyst)

| สิ่งที่ควรทำ                        | เหตุผล                                                                                                                                                                                                                                                |
| :-------------------------------- | :-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| ประเมินโมเดลล่าสุด.                | การสร้างโมเดลรุ่นใหม่อาจมีคุณสมบัติและคุณภาพที่ดีขึ้น - แต่ก็อาจมีค่าใช้จ่ายที่สูงขึ้นด้วย ประเมินผลกระทบ แล้วตัดสินใจเกี่ยวกับการโยกย้าย                                                                                                       |
| แยกคำแนะนำและบริบท               | ตรวจสอบว่าโมเดล/ผู้ให้บริการของคุณกำหนด _ตัวแบ่ง_ เพื่อแยกคำแนะนำ เนื้อหาหลักและเนื้อหาทางเลือกให้ชัดเจนขึ้น สิ่งนี้สามารถช่วยให้โมเดลกำหนดน้ำหนักให้กับโทเค็นได้แม่นยำมากขึ้น                                                             |
| ระบุให้ชัดเจนและชัดเจน            | ให้รายละเอียดเพิ่มเติมเกี่ยวกับบริบทที่ต้องการ ผลลัพธ์ ความยาว รูปแบบ สไตล์ เป็นต้น สิ่งนี้จะปรับปรุงทั้งคุณภาพและความสม่ำเสมอของการตอบสนอง จับสูตรในแม่แบบที่ใช้ซ้ำได้                                                           |
| อธิบายและใช้ตัวอย่าง              | โมเดลอาจตอบสนองได้ดีกับวิธีการ "แสดงและบอก" เริ่มต้นด้วย `zero-shot` approach where you give it an instruction (but no examples) then try `few-shot` as a refinement, providing a few examples of the desired output. Use analogies. |
| Use cues to jumpstart completions | Nudge it towards a desired outcome by giving it some leading words or phrases that it can use as a starting point for the response.                                                                                                               |
| Double Down                       | Sometimes you may need to repeat yourself to the model. Give instructions before and after your primary content, use an instruction and a cue, etc. Iterate & validate to see what works.                                                         |
| Order Matters                     | The order in which you present information to the model may impact the output, even in the learning examples, thanks to recency bias. Try different options to see what works best.                                                               |
| Give the model an “out”           | Give the model a _fallback_ completion response it can provide if it cannot complete the task for any reason. This can reduce chances of models generating false or fabricated responses.                                                         |
|                                   |                                                                                                                                                                                                                                                   |

As with any best practice, remember that _your mileage may vary_ based on the model, the task and the domain. Use these as a starting point, and iterate to find what works best for you. Constantly re-evaluate your prompt engineering process as new models and tools become available, with a focus on process scalability and response quality.

1. แสดงภาพรถยนต์สีแดงให้ฉันดู
2. แสดงภาพรถยนต์สีแดงยี่ห้อ Volvo และรุ่น XC90 จอดอยู่ข้างหน้าผาโดยมีพระอาทิตย์ตกดิน
3. แสดงภาพรถยนต์สีแดงยี่ห้อ Volvo และรุ่น XC90

A: 2, เป็นคำสั่งที่ดีที่สุดเนื่องจากให้รายละเอียดเกี่ยวกับ "อะไร" และเจาะจงลงไป (ไม่ใช่แค่รถยนต์ใด ๆ แต่เป็นยี่ห้อและรุ่นเฉพาะ) และยังอธิบายบรรยากาศโดยรวมด้วย 3 เป็นอันดับถัดไปเนื่องจากมีคำอธิบายมากเช่นกัน
