# พื้นฐานการออกแบบ Prompt

[![พื้นฐานการออกแบบ Prompt](../../../translated_images/th/04-lesson-banner.a2c90deba7fedacd.webp)](https://youtu.be/GElCu2kUlRs?si=qrXsBvXnCW12epb8)

## บทนำ
โมดูลนี้ครอบคลุมแนวคิดและเทคนิคพื้นฐานสำหรับการสร้าง prompt ที่มีประสิทธิภาพในโมเดล AI สร้างสรรค์ วิธีการเขียน prompt ให้กับ LLM ก็มีความสำคัญเช่นกัน Prompt ที่ออกแบบอย่างรอบคอบสามารถทำให้ได้ผลลัพธ์ที่มีคุณภาพดีกว่า แต่คำว่า _prompt_ และ _prompt engineering_ นั้นหมายถึงอะไรจริง ๆ? และเราจะปรับปรุง prompt _input_ ที่ส่งไปยัง LLM ได้อย่างไร? นี่คือคำถามที่เราจะพยายามตอบในบทนี้และบทถัดไป

_Generative AI_ สามารถสร้างเนื้อหาใหม่ ๆ (เช่น ข้อความ รูปภาพ เสียง โค้ด ฯลฯ) เพื่อตอบสนองคำขอของผู้ใช้ โดยใช้ _Large Language Models_ อย่างชุด GPT ("Generative Pre-trained Transformer") ของ OpenAI ที่ได้รับการฝึกฝนให้ใช้ภาษาธรรมชาติและโค้ดได้

ผู้ใช้สามารถโต้ตอบกับโมเดลเหล่านี้ผ่านรูปแบบที่คุ้นเคยเช่นแชท โดยไม่ต้องมีความชำนาญหรือการฝึกอบรมทางเทคนิค โมเดลเป็นแบบ _prompt-based_ - ผู้ใช้ส่งข้อความ (prompt) และรับคำตอบจาก AI กลับมา จากนั้นสามารถ "แชทกับ AI" ได้หลายรอบ โดยปรับแต่ง prompt จนผลลัพธ์ตรงกับความคาดหวัง

"Prompt" กลายเป็น _โปรแกรมมิ่งอินเทอร์เฟซ_ หลักสำหรับแอป AI สร้างสรรค์ บอกโมเดลว่าต้องทำอะไรและมีอิทธิพลต่อคุณภาพของคำตอบที่ได้ "Prompt Engineering" คือสาขาที่เติบโตเร็วซึ่งเน้นการ _ออกแบบและปรับปรุง_ prompt เพื่อให้ได้คำตอบที่สม่ำเสมอและมีคุณภาพในระดับกว้าง

## เป้าหมายการเรียนรู้

ในบทเรียนนี้ เราจะเรียนรู้ว่า Prompt Engineering คืออะไร สำคัญอย่างไร และเราจะสร้าง prompt ที่มีประสิทธิภาพมากขึ้นสำหรับโมเดลและวัตถุประสงค์ของแอปพลิเคชันอย่างไร เราจะเข้าใจแนวคิดหลักและแนวทางปฏิบัติที่ดีที่สุดสำหรับการออกแบบ prompt และเรียนรู้เกี่ยวกับสภาพแวดล้อม "sandbox" ของ Jupyter Notebooks แบบโต้ตอบ ที่เราจะได้เห็นแนวคิดเหล่านี้ประยุกต์ใช้กับตัวอย่างจริง

เมื่อเรียนจบบทนี้ เราจะสามารถ:

1. อธิบายว่า prompt engineering คืออะไร และทำไมจึงสำคัญ
2. บอกส่วนประกอบของ prompt และวิธีการใช้
3. เรียนรู้แนวทางปฏิบัติที่ดีที่สุดและเทคนิคการออกแบบ prompt
4. นำเทคนิคที่เรียนรู้ไปใช้กับตัวอย่างจริงโดยใช้ OpenAI endpoint

## คำศัพท์สำคัญ

Prompt Engineering: การปฏิบัติในการออกแบบและปรับแต่งอินพุตเพื่อชี้นำโมเดล AI ให้สร้างผลลัพธ์ตามที่ต้องการ
Tokenization: กระบวนการแปลงข้อความเป็นหน่วยเล็ก ๆ เรียกว่า tokens ซึ่งโมเดลสามารถเข้าใจและประมวลผลได้
Instruction-Tuned LLMs: โมเดลภาษาใหญ่ (LLMs) ที่ได้รับการปรับแต่งเฉพาะด้วยคำสั่งที่ชัดเจนเพื่อปรับปรุงความแม่นยำและความเกี่ยวข้องของคำตอบ

## สนามฝึกเรียนรู้

การออกแบบ prompt ยังเป็นศิลปะมากกว่าวิทยาศาสตร์ วิธีที่ดีที่สุดในการเพิ่มความเข้าใจคือ _ฝึกฝนบ่อยๆ_ และใช้แนวทางทดลองผิดทดลองถูกผสมกับความเชี่ยวชาญด้านโดเมนและเทคนิคที่แนะนำพร้อมการปรับแต่งเฉพาะโมเดล

Jupyter Notebook ที่แนบมากับบทเรียนนี้เป็นสภาพแวดล้อม _sandbox_ ที่คุณสามารถทดลองใช้สิ่งที่เรียนรู้ได้ทั้งระหว่างเรียนหรือใช้ทำแบบฝึกหัดท้ายบท ในการทำแบบฝึกหัดนี้คุณต้องมี:

1. **กุญแจ API ของ Azure OpenAI** - จุดเชื่อมต่อสำหรับ LLM ที่ปรับใช้แล้ว
2. **รันไทม์ Python** - เพื่อรัน Notebook
3. **ตัวแปรสภาพแวดล้อมท้องถิ่น** - _ทำตามขั้นตอน [SETUP](./../00-course-setup/02-setup-local.md?WT.mc_id=academic-105485-koreyst) เพื่อเตรียมพร้อม_

โน้ตบุ๊กมาพร้อมแบบฝึกหัด _เริ่มต้น_ แต่คุณสามารถเพิ่มเติม _Markdown_ (คำอธิบาย) และ _Code_ (คำขอ prompt) เพื่อทดลองตัวอย่างหรือไอเดียอื่น ๆ และสร้างความเข้าใจในการออกแบบ prompt

## คู่มือภาพประกอบ

ต้องการเห็นภาพรวมสิ่งที่บทเรียนนี้ครอบคลุมก่อนลงมือทำไหม? ดูคู่มือภาพประกอบนี้ที่จะช่วยให้คุณเข้าใจหัวข้อหลักและใจความสำคัญที่ควรคิดในแต่ละหัวข้อ เส้นทางเรียนนี้จะพาคุณจากการเข้าใจแนวคิดและความท้าทาย ไปสู่การแก้ไขด้วยเทคนิคและแนวทางปฏิบัติที่ดีที่สุดสำหรับ prompt engineering หมายเหตุ: ส่วน "เทคนิคขั้นสูง" ในคู่มือนี้หมายถึงเนื้อหาในบทถัดไปของหลักสูตรนี้

## สตาร์ทอัพของเรา

ตอนนี้ มาเล่าถึง _หัวข้อนี้_ ว่ามีความเกี่ยวข้องกับพันธกิจสตาร์ทอัพของเราในการ [นำ AI มาปรับใช้ในการศึกษาขั้นสูง](https://educationblog.microsoft.com/2023/06/collaborating-to-bring-ai-innovation-to-education?WT.mc_id=academic-105485-koreyst) อย่างไร เราต้องการสร้างแอปเรียนรู้เฉพาะบุคคลด้วยพลัง AI ดังนั้นลองคิดว่าใคร ๆ ที่ใช้แอปเราอาจ "ออกแบบ" prompt อย่างไรบ้าง:

- **ผู้ดูแลระบบ** อาจขอให้ AI _วิเคราะห์ข้อมูลหลักสูตรเพื่อหาช่องว่างในการสอน_ AI สามารถสรุปผลหรือสร้างภาพข้อมูลด้วยโค้ด
- **ครูผู้สอน** อาจขอให้ AI _สร้างแผนการสอนสำหรับกลุ่มเป้าหมายและหัวข้อเฉพาะ_ AI สามารถจัดทำแผนเฉพาะรายในรูปแบบที่กำหนดได้
- **นักเรียน** อาจขอให้ AI _ติวเตอร์ในวิชาที่ยาก_ AI สามารถแนะนำบทเรียน คำใบ้ และตัวอย่างที่เหมาะสมกับระดับของนักเรียน

นี่เป็นเพียงส่วนเล็ก ๆ ของความเป็นไปได้ ลองดู [Prompts For Education](https://github.com/microsoft/prompts-for-edu/tree/main?WT.mc_id=academic-105485-koreyst) - ห้องสมุด prompt แบบโอเพนซอร์สที่ผู้เชี่ยวชาญด้านการศึกษาได้คัดสรรมา แล้วลองรัน prompt เหล่านั้นใน sandbox หรือ OpenAI Playground ดูสิ่งที่จะเกิดขึ้น!

## Prompt Engineering คืออะไร?

เราเริ่มบทเรียนนี้ด้วยการกำหนด **Prompt Engineering** ว่าเป็นกระบวนการ _ออกแบบและปรับแต่ง_ อินพุตข้อความ (prompt) เพื่อให้ออกคำตอบ (completion) ที่สม่ำเสมอและมีคุณภาพสำหรับวัตถุประสงค์ของแอปและโมเดลที่ระบุ เราสามารถคิดเป็นสองขั้นตอน:

- _ออกแบบ_ prompt แรกสำหรับโมเดลและวัตถุประสงค์ที่กำหนด
- _ปรับปรุง_ prompt ทีละรอบเพื่อเพิ่มคุณภาพคำตอบ

นี่คือกระบวนการทดลองผิดลองถูกที่ต้องใช้สัญชาตญาณและความพยายามของผู้ใช้เพื่อให้ได้ผลดีสุด แล้วทำไมจึงสำคัญ? เราต้องเข้าใจสามแนวคิดก่อน:

- _Tokenization_ = โมเดล "มองเห็น" prompt อย่างไร
- _Base LLMs_ = โมเดลพื้นฐาน "ประมวลผล" prompt อย่างไร
- _Instruction-Tuned LLMs_ = โมเดลที่มองเห็น "งาน" อย่างไร

### Tokenization

LLM มอง prompt เป็น _ลำดับของ tokens_ ซึ่งแต่ละโมเดล (หรือแต่ละเวอร์ชันของโมเดล) อาจแบ่ง tokens ของ prompt เดียวกันได้แตกต่างกัน เนื่องจาก LLM ถูกฝึกด้วย tokens (ไม่ใช่ข้อความดิบ) วิธีที่ prompt ถูกแบ่ง tokens มีผลโดยตรงกับคุณภาพคำตอบที่สร้างขึ้น

เพื่อให้เข้าใจ tokenization ลองใช้เครื่องมือเช่น [OpenAI Tokenizer](https://platform.openai.com/tokenizer?WT.mc_id=academic-105485-koreyst) ที่แสดงด้านล่าง คัดลอก prompt ของคุณไปใส่ แล้วดูว่ามันถูกแปลงเป็น tokens อย่างไร ให้สังเกตการจัดการช่องว่างและเครื่องหมายวรรคตอน ตัวอย่างนี้ใช้ LLM เก่า (GPT-3) ดังนั้นถ้าลองกับโมเดลใหม่อาจได้ผลลัพธ์ต่างกัน

![Tokenization](../../../translated_images/th/04-tokenizer-example.e71f0a0f70356c5c.webp)

### แนวคิด: โมเดลพื้นฐาน (Foundation Models)

เมื่อ prompt ถูกแบ่ง tokens แล้ว ฟังก์ชันหลักของ ["Base LLM"](https://blog.gopenai.com/an-introduction-to-base-and-instruction-tuned-large-language-models-8de102c785a6?WT.mc_id=academic-105485-koreyst) (หรือโมเดลพื้นฐาน) คือการทำนาย token ถัดไปในลำดับนั้น LLM ถูกฝึกด้วยชุดข้อมูลข้อความขนาดใหญ่จึงมีความเข้าใจความสัมพันธ์เชิงสถิติระหว่าง tokens และสามารถทำนายได้ด้วยความมั่นใจ อย่างไรก็ตาม โมเดลไม่ได้เข้าใจ _ความหมาย_ ของคำใน prompt หรือ tokens เพียงแค่เห็นรูปแบบที่สามารถ "เติมเต็ม" ด้วยการทำนายถัดไปได้ และจะทำนายไปเรื่อย ๆ จนกว่าจะหยุดโดยผู้ใช้หรือเงื่อนไขที่ตั้งไว้

อยากดูการทำงานของการเติมคำตอบด้วย prompt ไหม? ลองนำ prompt ด้านบนไปใส่ใน [Microsoft Foundry playground](https://ai.azure.com?WT.mc_id=academic-105485-koreyst) พร้อมตั้งค่าตามค่าเริ่มต้น ระบบจะมอง prompt เป็นคำขอข้อมูล ดังนั้นคุณควรจะเห็นคำตอบในบริบทนั้น

แต่ถ้าผู้ใช้ต้องการดูอะไรเฉพาะเจาะจงตามเกณฑ์หรือวัตถุประสงค์ของงาน? นี่คือที่มาของ _instruction-tuned_ LLMs

### แนวคิด: Instruction Tuned LLMs

[Instruction Tuned LLM](https://blog.gopenai.com/an-introduction-to-base-and-instruction-tuned-large-language-models-8de102c785a6?WT.mc_id=academic-105485-koreyst) เริ่มจากโมเดลพื้นฐานและปรับแต่งด้วยตัวอย่างหรือตัวคู่ป้อนเข้า/ออก (เช่น "ข้อความ" หลายรอบ) ที่มีคำสั่งชัดเจน และคำตอบของ AI จะพยายามทำตามคำสั่งนั้น

ใช้เทคนิคเช่น Reinforcement Learning with Human Feedback (RLHF) เพื่อฝึกโมเดลให้ _ทำตามคำสั่ง_ และ _เรียนรู้จากฟีดแบ็ค_ ทำให้ได้คำตอบที่เหมาะสมกับการใช้งานจริงและเกี่ยวข้องกับเป้าหมายของผู้ใช้มากขึ้น

ลองดู - กลับไปดู prompt ด้านบนอีกครั้ง แต่เปลี่ยน _ข้อความระบบ_ เพื่อให้มีคำสั่งนี้เป็นบริบท:

> _สรุปเนื้อหาที่ให้ไว้สำหรับนักเรียนชั้นประถมศึกษาปีที่ 2 จำกัดผลลัพธ์เป็นย่อหน้าเดียวพร้อมลูกศร 3-5 ข้อ_

ดูไหมว่าคำตอบตอนนี้ถูกปรับแต่งให้สะท้อนเป้าหมายและรูปแบบที่ต้องการ ครูสามารถนำคำตอบนี้ไปใช้ในสไลด์สำหรับชั้นเรียนได้ทันที

![Instruction Tuned LLM Chat Completion](../../../translated_images/th/04-playground-chat-instructions.b30bbfbdf92f2d05.webp)

## ทำไมเราต้อง Prompt Engineering?

ตอนนี้ที่เราเข้าใจแล้วว่า LLM ประมวลผล prompt อย่างไร ลองมาพูดถึง _เหตุผล_ ว่าทำไมเราต้อง prompt engineering คำตอบอยู่ที่ว่า LLM ปัจจุบันมีความท้าทายหลายประการที่ทำให้การได้คำตอบที่ _น่าเชื่อถือและสม่ำเสมอ_ เป็นเรื่องยากถ้าไม่ลงทุนเวลาออกแบบและปรับแต่ง prompt เช่น:

1. **คำตอบของโมเดลมีความเป็นสุ่ม** _prompt เดียวกัน_ อาจให้คำตอบต่างกันระหว่างโมเดลหรือเวอร์ชันของโมเดล และอาจเปลี่ยนแปลงแม้ใช้โมเดล _เดียวกัน_ ในเวลาต่างกัน เทคนิค prompt engineering ช่วยลดความแปรปรวนนี้ได้ด้วยการสร้างกรอบการใช้งานที่ชัดเจนขึ้น

1. **โมเดลอาจสร้างคำตอบที่ผิด** โมเดลได้รับการฝึกด้วยชุดข้อมูล _ขนาดใหญ่แต่จำกัด_ ซึ่งทำให้ไม่รู้เรื่องราวนอกเหนือจากขอบเขตฝึก ผลลัพธ์คืออาจสร้างคำตอบที่ไม่ถูกต้อง จินตนาการ หรือขัดแย้งกับข้อเท็จจริง เทคนิค prompt engineering ช่วยให้ผู้ใช้สังเกตและลดปัญหานี้ เช่น ขอให้ AI อ้างอิงแหล่งข้อมูลหรือเหตุผลประกอบ

1. **ความสามารถของโมเดลจะแตกต่างกัน** โมเดลหรือรุ่นใหม่กว่ามีความสามารถมากกว่า แต่นำปัญหาเฉพาะตัวและต้นทุนเพิ่มขึ้นมา เทคนิค prompt engineering ช่วยพัฒนาแนวทางปฏิบัติที่ทำให้ความแตกต่างนี้เป็นอัตโนมัติและรองรับความต้องการเฉพาะของโมเดลในรูปแบบที่ขยายผลได้และราบรื่น

ลองดูตัวอย่างนี้ใน OpenAI หรือ Azure OpenAI Playground:

- ใช้ prompt เดียวกันกับ LLM จากหลายที่ (เช่น OpenAI, Azure OpenAI, Hugging Face) - เห็นความแตกต่างไหม?
- ใช้ prompt เดิมซ้ำกับ LLM _ที่เดียวกัน_ (เช่น Azure OpenAI playground) - ความแตกต่างแต่ละครั้งเป็นอย่างไร?

### ตัวอย่างการสร้างข้อมูลปลอม (Fabrications)

ในหลักสูตรนี้ เราใช้คำว่า **"fabrication"** เพื่ออ้างถึงปรากฏการณ์ที่ LLM บางครั้งสร้างข้อมูลที่ไม่ถูกต้องตามข้อเท็จจริงเนื่องจากข้อจำกัดในการฝึกหรือข้อจำกัดอื่น ๆ คุณอาจเคยได้ยินคำนี้ในชื่อ _"hallucinations"_ ในบทความหรือวิจัยต่าง ๆ แต่เราแนะนำให้ใช้คำว่า _"fabrication"_ เพื่อไม่ให้เกิดการเหมารวมว่าพฤติกรรมนี้เหมือนมนุษย์ และเพื่อสนับสนุนแนวทาง [Responsible AI](https://www.microsoft.com/ai/responsible-ai?WT.mc_id=academic-105485-koreyst) โดยใช้คำที่เหมาะสมและไม่ก่อให้เกิดความเข้าใจผิดหรือละเมิดในบางบริบท

อยากรู้ว่าการสร้างข้อมูลปลอมทำงานอย่างไร? ลองคิดถึง prompt ที่สั่งให้ AI สร้างเนื้อหาเกี่ยวกับหัวข้อที่ไม่มีจริง (เพื่อให้แน่ใจว่าไม่มีในชุดข้อมูลฝึก) เช่น ฉันลองใช้ prompt นี้:

> **Prompt:** สร้างแผนการสอนเกี่ยวกับสงครามดาวอังคารในปี 2076

การค้นหาทางเว็บแสดงให้เห็นว่ามีเรื่องราวสมมติ (เช่น ซีรีส์ทีวีหรือหนังสือ) เกี่ยวกับสงครามดาวอังคาร แต่ไม่มีในปี 2076 ข้อมูลสามัญสำนึกบอกเราว่า 2076 อยู่ใน _อนาคต_ จึงไม่อาจเกี่ยวข้องกับเหตุการณ์จริงได้

## การสร้างพรอมต์

เราเห็นแล้วว่าทำไมการออกแบบพรอมต์จึงสำคัญ — ตอนนี้มาทำความเข้าใจวิธีการ _สร้าง_ พรอมต์เพื่อที่เราจะได้ประเมินเทคนิคต่าง ๆ สำหรับการออกแบบพรอมต์ที่มีประสิทธิภาพมากขึ้น

### พรอมต์พื้นฐาน

เรามาเริ่มต้นด้วยพรอมต์พื้นฐาน: ข้อความอินพุตที่ส่งไปยังโมเดลโดยไม่มีบริบทอื่น ๆ นี่คือตัวอย่าง — เมื่อเราส่งคำไม่กี่คำแรกของเพลงชาติสหรัฐอเมริกาไปยัง OpenAI [Completion API](https://platform.openai.com/docs/api-reference/completions?WT.mc_id=academic-105485-koreyst) มันจะ _เติมเต็ม_ คำตอบทันทีด้วยบรรทัดถัดไปเล็กน้อย แสดงพฤติกรรมการทำนายพื้นฐาน

| พรอมต์ (อินพุต) | การเติมเต็ม (เอาต์พุต)                                                                                                                     |
| :----------------- | :----------------------------------------------------------------------------------------------------------------------------------------- |
| Oh say can you see | ฟังดูเหมือนคุณกำลังเริ่มต้นเนื้อเพลงของ "The Star-Spangled Banner" เพลงชาติของสหรัฐอเมริกา เนื้อเพลงฉบับเต็มคือ ...                   |

### พรอมต์ซับซ้อน

ตอนนี้เรามาเพิ่มบริบทและคำแนะนำลงในพรอมต์พื้นฐานนั้น [Chat Completion API](https://learn.microsoft.com/azure/ai-foundry/openai/how-to/chatgpt?WT.mc_id=academic-105485-koreyst) ช่วยให้เราสร้างพรอมต์ซับซ้อนได้ในรูปแบบของ _ข้อความ_ หลายชิ้นที่ประกอบด้วย:

- คู่ของอินพุต/เอาต์พุตที่สะท้อนการป้อนข้อมูลของ _ผู้ใช้_ และการตอบกลับของ _ผู้ช่วย_
- ข้อความระบบที่ตั้งค่าบริบทสำหรับพฤติกรรมหรือบุคลิกของผู้ช่วย

คำขออยู่ในรูปแบบด้านล่าง ซึ่ง _การตัดโทเคน_ จะดึงข้อมูลที่เกี่ยวข้องจากบริบทและการสนทนาออกมาได้อย่างมีประสิทธิภาพ การเปลี่ยนบริบทของระบบจึงส่งผลกระทบต่อคุณภาพของการเติมเต็มไม่ต่างจากการป้อนข้อมูลของผู้ใช้

```python
response = client.responses.create(
    model="gpt-5-mini",
    input=[
        {"role": "system", "content": "You are a helpful assistant."},
        {"role": "user", "content": "Who won the world series in 2020?"},
        {"role": "assistant", "content": "The Los Angeles Dodgers won the World Series in 2020."},
        {"role": "user", "content": "Where was it played?"}
    ]
)
```

### พรอมต์คำสั่ง

ในตัวอย่างข้างต้น พรอมต์ของผู้ใช้เป็นการถามข้อความง่าย ๆ ที่ตีความได้ว่าเป็นคำขอข้อมูล ด้วยพรอมต์ _คำสั่ง_ เราสามารถใช้ข้อความนั้นเพื่อกำหนดงานให้ละเอียดขึ้น ให้คำแนะนำที่ดีกับ AI นี่คือตัวอย่าง:

| พรอมต์ (อินพุต)                                                                                                                                                                                                                         | การเติมเต็ม (เอาต์พุต)                                                                                                        | ประเภทคำสั่ง      |
| :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :------------------------------------------------------------------------------------------------------------------------- | :------------------ |
| Write a description of the Civil War                                                                                                                                                                                                   | _ส่งกลับเป็นย่อหน้าเรียบง่าย_                                                                                              | ง่าย               |
| Write a description of the Civil War. Provide key dates and events and describe their significance                                                                                                                                     | _ส่งกลับเป็นย่อหน้าตามด้วยรายการวันที่เหตุการณ์สำคัญพร้อมคำอธิบาย_                                                     | ซับซ้อน            |
| Write a description of the Civil War in 1 paragraph. Provide 3 bullet points with key dates and their significance. Provide 3 more bullet points with key historical figures and their contributions. Return the output as a JSON file | _ส่งกลับข้อมูลรายละเอียดมากขึ้นในกล่องข้อความ รูปแบบ JSON ที่คุณสามารถคัดลอกวางลงในไฟล์และตรวจสอบได้ตามต้องการ_        | ซับซ้อน รูปแบบ JSON |

ส่วนเนื้อหาหลักสามารถใช้ได้หลายวิธีเพื่อขับเคลื่อนคำสั่งที่มีประสิทธิภาพมากขึ้น:

- **ตัวอย่าง** - แทนที่จะสั่งงานโมเดลด้วยคำสั่งชัดเจน ให้มันดูตัวอย่างว่างานควรเป็นอย่างไรแล้วปล่อยให้มันสรุปแบบแผนผังได้เอง
- **สัญญาณนำทาง** - ต่อคำสั่งด้วย "สัญญาณนำทาง" ที่กระตุ้นการเติมเต็ม ช่วยชี้นำโมเดลไปยังคำตอบที่เกี่ยวข้องมากขึ้น
- **เทมเพลต** - สูตรซ้ำ ๆ สำหรับพรอมต์ที่มีตัวแปรแบบแทนที่ (placeholders) ซึ่งสามารถปรับแต่งด้วยข้อมูลเฉพาะสำหรับการใช้งานจริง

มาลองดูวิธีเหล่านี้ในการใช้งานจริง

### การใช้ตัวอย่าง

วิธีนี้ใช้เนื้อหาหลักเป็น "อาหาร" ให้โมเดล โดยให้ตัวอย่างผลลัพธ์ที่ต้องการสำหรับคำสั่งที่กำหนด และปล่อยให้โมเดลสรุปรูปแบบของผลลัพธ์นั้นตามจำนวนตัวอย่างที่ให้ไป เราจึงมี zero-shot prompting, one-shot prompting, few-shot prompting เป็นต้น

### แบบฟอร์มพร้อมใช้สำหรับ Prompt

แบบฟอร์มพร้อมใช้สำหรับ prompt คือ _สูตรสำเร็จที่กำหนดไว้ล่วงหน้าสำหรับ prompt_ ที่สามารถเก็บไว้และนำกลับมาใช้ใหม่ได้ตามต้องการ เพื่อสร้างประสบการณ์ผู้ใช้ที่สอดคล้องกันในขนาดใหญ่ ในรูปแบบที่ง่ายที่สุด มันเป็นเพียงชุดตัวอย่าง prompt เช่น [ตัวอย่างนี้จาก OpenAI](https://cookbook.openai.com/examples/gpt4-1_prompting_guide?WT.mc_id=academic-105485-koreyst) ที่ให้ทั้งส่วนประกอบ interactive prompt (ข้อความผู้ใช้และระบบ) และรูปแบบคำขอ API - เพื่อสนับสนุนการนำกลับมาใช้ใหม่

ในรูปแบบที่ซับซ้อนขึ้น เช่น [ตัวอย่างนี้จาก LangChain](https://python.langchain.com/docs/concepts/prompt_templates/?WT.mc_id=academic-105485-koreyst) จะมี _ตัวแทนที่ถือที่ว่าง_ ที่สามารถแทนที่ด้วยข้อมูลจากแหล่งข้อมูลหลากหลาย (ข้อมูลผู้ใช้, บริบทของระบบ, แหล่งข้อมูลภายนอก ฯลฯ) เพื่อสร้าง prompt ได้อย่างไดนามิก ซึ่งช่วยให้เราสร้างห้องสมุด prompt ที่นำกลับมาใช้ใหม่ได้เพื่อขับเคลื่อนประสบการณ์ผู้ใช้ที่สอดคล้องกัน **อย่างเป็นโปรแกรมเมติก** ในขนาดใหญ่

สุดท้าย คุณค่าจริงของแบบฟอร์มอยู่ที่ความสามารถในการสร้างและเผยแพร่ _ห้องสมุด prompt_ สำหรับโดเมนการใช้งานเฉพาะ - ซึ่งแบบฟอร์ม prompt นั้นถูก _ปรับให้เหมาะสม_ เพื่อสะท้อนบริบทหรือคำอธิบายเฉพาะของแอปพลิเคชันที่ทำให้คำตอบมีความเกี่ยวข้องและแม่นยำมากขึ้นสำหรับกลุ่มผู้ใช้เป้าหมาย เช่น ห้องสมุด [Prompts For Edu](https://github.com/microsoft/prompts-for-edu?WT.mc_id=academic-105485-koreyst) เป็นตัวอย่างที่ดีของแนวทางนี้ โดยรวบรวมห้องสมุดของ prompt สำหรับโดเมนการศึกษาโดยเน้นที่วัตถุประสงค์หลัก เช่น การวางแผนบทเรียน การออกแบบหลักสูตร การติวเตอร์นักเรียน ฯลฯ

## เนื้อหาสนับสนุน

ถ้าเราคิดว่าการสร้าง prompt มีคำสั่ง (งาน) และเป้าหมาย (เนื้อหาหลัก) แล้ว _เนื้อหารอง_ เหมือนบริบทเพิ่มเติมที่เรามอบให้เพื่อ **มีอิทธิพลต่อผลลัพธ์ในทางใดทางหนึ่ง** อาจเป็นพารามิเตอร์การปรับจูน คำแนะนำการจัดรูปแบบ หมวดหมู่หัวข้อ ฯลฯ ที่ช่วยให้โมเดล _ปรับแต่ง_ การตอบสนองให้เหมาะสมกับวัตถุประสงค์หรือความคาดหวังของผู้ใช้ที่ต้องการ

ตัวอย่างเช่น: สมมติมีแคตตาล็อกหลักสูตรที่มีเมตาดาต้าหลากหลาย (ชื่อ คำอธิบาย ระดับ แท็กเมตาดาต้า ผู้สอน ฯลฯ) สำหรับหลักสูตรทั้งหมดในหลักสูตร

- เราสามารถกำหนดคำสั่งให้ "สรุปแคตตาล็อกหลักสูตรสำหรับฤดูใบไม้ร่วง 2023"
- เราสามารถใช้เนื้อหาหลักเพื่อให้ตัวอย่างบางส่วนของผลลัพธ์ที่ต้องการ
- เราสามารถใช้เนื้อหารองเพื่อระบุ 5 "แท็ก" ยอดนิยมที่น่าสนใจ

ตอนนี้ โมเดลสามารถให้สรุปในรูปแบบที่แสดงจากตัวอย่างจำนวนหนึ่ง - แต่ถ้าผลลัพธ์มีแท็กหลายแท็ก มันสามารถจัดลำดับความสำคัญ 5 แท็กที่ระบุในเนื้อหารองได้

## แนวทางปฏิบัติที่ดีที่สุดสำหรับการสร้าง Prompt

ตอนนี้ที่เรารู้ว่าการสร้าง prompt สามารถ _สร้างขึ้น_ ได้อย่างไร เราสามารถเริ่มคิดเกี่ยวกับวิธีการ _ออกแบบ_ เพื่อสะท้อนแนวทางปฏิบัติที่ดีที่สุดได้ เราสามารถคิดเรื่องนี้เป็นสองส่วน - มี _กรอบความคิด_ ที่ถูกต้องและใช้ _เทคนิค_ ที่ถูกต้อง

### กรอบความคิดสำหรับการสร้าง Prompt

การสร้าง prompt เป็นกระบวนการลองผิดลองถูก ดังนั้นให้คำนึงถึงปัจจัยชี้นำกว้าง ๆ สามประการนี้:

1. **ความเข้าใจโดเมนสำคัญ** ความแม่นยำและความเกี่ยวข้องของคำตอบขึ้นอยู่กับ _โดเมน_ ที่แอปพลิเคชันหรือผู้ใช้ทำงาน ทำความเข้าใจและความเชี่ยวชาญในโดเมนเพื่อ **ปรับแต่งเทคนิค** ต่อไป เช่น กำหนด _บุคลิกเฉพาะโดเมน_ ใน prompt ระบบของคุณ ใช้ _แบบฟอร์มเฉพาะโดเมน_ ใน prompt ผู้ใช้ ให้เนื้อหารองที่สะท้อนบริบทเฉพาะโดเมน หรือใช้ _สัญญาณและตัวอย่างเฉพาะโดเมน_ เพื่อชี้นำโมเดลไปสู่รูปแบบการใช้งานที่คุ้นเคย

2. **ความเข้าใจโมเดลสำคัญ** เรารู้อยู่แล้วว่าโมเดลเป็นแบบสุ่มโดยธรรมชาติ แต่อิมพลีเมนเทชันของโมเดลอาจแตกต่างกันในแง่ของชุดข้อมูลการฝึกอบรมที่ใช้ (ความรู้ที่ได้รับการฝึกมาแล้ว), ความสามารถที่มี (เช่น ผ่าน API หรือ SDK) และประเภทเนื้อหาที่ถูกปรับให้เหมาะสม (เช่น โค้ด เทียบกับภาพ เทียบกับข้อความ) เข้าใจจุดแข็งและข้อจำกัดของโมเดลที่คุณใช้ และใช้ความรู้นั้นเพื่อ _จัดลำดับความสำคัญของงาน_ หรือสร้าง _แบบฟอร์มที่ปรับแต่ง_ ซึ่งเหมาะกับความสามารถของโมเดล

3. **การทำซ้ำและการตรวจสอบสำคัญ** โมเดลกำลังพัฒนาอย่างรวดเร็ว เช่นเดียวกับเทคนิคสำหรับการสร้าง prompt ในฐานะผู้เชี่ยวชาญโดเมนอาจมีบริบทหรือเกณฑ์อื่นสำหรับ _แอปพลิเคชันเฉพาะของคุณ_ ที่อาจไม่เกี่ยวข้องกับชุมชนกว้าง ใช้เครื่องมือและเทคนิคการสร้าง prompt เพื่อ "เริ่มต้นอย่างรวดเร็ว" ในการสร้าง prompt จากนั้นทำซ้ำและตรวจสอบผลลัพธ์โดยใช้สัญชาตญาณและความเชี่ยวชาญในโดเมนของคุณ บันทึกข้อสังเกตและสร้าง **ฐานความรู้** (เช่น ห้องสมุด prompt) ที่สามารถใช้เป็นฐานใหม่โดยผู้อื่น เพื่อให้เกิดการทำซ้ำรวดเร็วขึ้นในอนาคต

| ระบุชัดเจนและชัดเจน             | ให้รายละเอียดมากขึ้นเกี่ยวกับบริบท ผลลัพธ์ ความยาว รูปแบบ สไตล์ เป็นต้น สิ่งนี้จะช่วยปรับปรุงคุณภาพและความสอดคล้องของคำตอบ จดบันทึกสูตรในแบบฟอร์มที่นำกลับมาใช้ได้                                                             |
| อธิบายใช้ตัวอย่าง               | โมเดลอาจตอบสนองได้ดีกับแนวทาง "แสดงและบอก" เริ่มด้วยแบบ `zero-shot` ที่ให้คำสั่ง (แต่ไม่มีตัวอย่าง) แล้วลองแบบ `few-shot` ที่ให้ตัวอย่างบางส่วนของผลลัพธ์ที่ต้องการ ใช้การเปรียบเทียบหรืออนาล็อกี           |
| ใช้สัญญาณเพื่อเริ่มต้นคำตอบ    | กระตุ้นให้โมเดลไปสู่ผลลัพธ์ที่ต้องการโดยให้คำหรือวลีที่เริ่มต้นที่โมเดลสามารถใช้เป็นจุดเริ่มต้นของคำตอบ                                                                           |
| ทำซ้ำอีกครั้ง                   | บางครั้งคุณอาจต้องย้ำคำสั่งกับโมเดล ให้คำสั่งก่อนและหลังเนื้อหาหลัก ใช้คำสั่งกับสัญญาณ ฯลฯ ทำซ้ำและตรวจสอบเพื่อดูว่าอะไรใช้ได้ผล                                                            |
| ลำดับสำคัญ                     | ลำดับที่คุณนำเสนอข้อมูลกับโมเดลอาจส่งผลต่อผลลัพธ์ แม้ในการเรียนรู้ด้วยตัวอย่าง เนื่องจากอคติเรื่องความสดใหม่ ลองตัวเลือกต่าง ๆ เพื่อดูว่าอะไรดีที่สุด                                                    |
| ให้โมเดลมี “ทางออก”           | ให้โมเดลมีคำตอบ _สำรอง_ ที่สามารถให้ได้หากไม่สามารถทำงานให้เสร็จด้วยเหตุผลใดก็ตาม ช่วยลดโอกาสที่โมเดลจะสร้างคำตอบผิดหรือข่าวสารที่แต่งขึ้น                                                        |
|                                   |                                                                                                                                                                                                                                                   |

เช่นเดียวกับแนวทางปฏิบัติที่ดีที่สุดใด ๆ จำไว้ว่าผลลัพธ์อาจแตกต่างกันตามโมเดล งาน และโดเมน ใช้สิ่งเหล่านี้เป็นจุดเริ่มต้น และทำซ้ำเพื่อค้นหาสิ่งที่ดีที่สุดสำหรับคุณ ประเมินกระบวนการสร้าง prompt ของคุณอย่างต่อเนื่องเมื่อมีโมเดลและเครื่องมือใหม่ ๆ พร้อมใช้งาน โดยเน้นที่ความสามารถในการขยายกระบวนการและคุณภาพคำตอบ

## การบ้าน

ยินดีด้วย! คุณมาถึงจุดสิ้นสุดของบทเรียนแล้ว! ถึงเวลาทดสอบแนวคิดและเทคนิคเหล่านั้นด้วยตัวอย่างจริง!

สำหรับการบ้านของเรา เราจะใช้ Jupyter Notebook ที่มีแบบฝึกหัดที่คุณสามารถทำได้แบบอินเทอร์แอกทีฟ คุณยังสามารถขยายโน้ตบุ๊กด้วย Markdown และ Code เซลล์ของคุณเองเพื่อสำรวจแนวคิดและเทคนิคด้วยตัวเอง

### เพื่อเริ่มต้น ให้ทำการ fork รีโปแล้ว

- (แนะนำ) เปิดใช้งาน GitHub Codespaces
- (อีกทางเลือก) โคลนรีโปไปยังอุปกรณ์ของคุณและใช้งานกับ Docker Desktop
- (อีกทางเลือก) เปิดโน้ตบุ๊กด้วยสภาพแวดล้อมรันไทม์โน้ตบุ๊กที่คุณชอบ

### ถัดไป ตั้งค่าตัวแปรสภาพแวดล้อมของคุณ

- คัดลอกไฟล์ `.env.copy` ในโฟลเดอร์รากรีโปมาเป็น `.env` แล้วกรอกค่า `AZURE_OPENAI_API_KEY`, `AZURE_OPENAI_ENDPOINT` และ `AZURE_OPENAI_DEPLOYMENT` กลับไปที่[ส่วน Learning Sandbox](#สนามฝึกเรียนรู้) เพื่อเรียนรู้วิธีการทำ

### ถัดไป เปิด Jupyter Notebook

- เลือกเคอร์เนลรันไทม์ หากใช้ตัวเลือก 1 หรือ 2 ให้เลือกเคอร์เนล Python 3.10.x ค่าเริ่มต้นที่จัดเตรียมโดย dev container

คุณพร้อมที่จะรันแบบฝึกหัดแล้ว โปรดทราบว่าไม่มีคำตอบที่ _ถูกหรือผิด_ ในที่นี้ แค่สำรวจทางเลือกโดยการลองผิดลองถูกและสร้างสัญชาตญาณว่าสิ่งใดเหมาะสมกับโมเดลและโดเมนการใช้งานที่กำหนด

_ด้วยเหตุนี้จึงไม่มีส่วนคำตอบโค้ดในบทเรียนนี้ แทนที่จะเป็นเชลล์ Markdown ในโน้ตบุ๊กที่มีชื่อว่า "My Solution:" ซึ่งแสดงตัวอย่างผลลัพธ์หนึ่งตัวอย่างเพื่ออ้างอิง_

## การตรวจสอบความรู้

ข้อใดต่อไปนี้เป็น prompt ที่ดีตามแนวทางปฏิบัติที่สมเหตุสมผล?

1. แสดงภาพรถสีแดงให้ฉัน
2. แสดงภาพรถสีแดง ยี่ห้อ Volvo รุ่น XC90 จอดอยู่ริมหน้าผาในขณะที่พระอาทิตย์กำลังตกดิน
3. แสดงภาพรถสีแดง ยี่ห้อ Volvo รุ่น XC90

ตอบ: 2 เป็น prompt ที่ดีที่สุดเพราะให้รายละเอียดเกี่ยวกับ "อะไร" และเจาะจงมาก (ไม่ใช่แค่รถใด ๆ แต่เป็นยี่ห้อและรุ่นเฉพาะ) และยังอธิบายบริบทโดยรวมด้วย 3 เป็นอันดับสองที่ดีที่สุดเพราะมีคำอธิบายมากมายเช่นกัน
