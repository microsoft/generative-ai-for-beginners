# สำรวจและเปรียบเทียบ LLMs ที่แตกต่างกัน

[![สำรวจและเปรียบเทียบ LLMs ที่แตกต่างกัน](../../../translated_images/th/02-lesson-banner.ef94c84979f97f60.webp)](https://youtu.be/KIRUeDKscfI?si=8BHX1zvwzQBn-PlK)

> _คลิกที่ภาพด้านบนเพื่อดูวิดีโอบทเรียนนี้_

จากบทเรียนก่อนหน้านี้ เราได้เห็นว่า Generative AI กำลังเปลี่ยนแปลงภูมิทัศน์ทางเทคโนโลยีอย่างไร, โมเดลภาษาขนาดใหญ่ (LLMs) ทำงานอย่างไร และธุรกิจ - เช่น สตาร์ทอัพของเรา - สามารถนำไปใช้ในกรณีใช้งานและเติบโตได้อย่างไร! ในบทนี้ เราจะพิจารณาเปรียบเทียบประเภทโมเดลภาษาขนาดใหญ่ (LLMs) ต่างๆ เพื่เข้าใจข้อดีและข้อเสียของพวกเขา

ขั้นตอนถัดไปในเส้นทางของสตาร์ทอัพของเราคือการสำรวจภูมิทัศน์ปัจจุบันของ LLMs และทำความเข้าใจว่าโมเดลใดที่เหมาะสมกับกรณีการใช้งานของเรา

## บทนำ

บทเรียนนี้จะครอบคลุม:

- ประเภทต่างๆ ของ LLMs ในภูมิทัศน์ปัจจุบัน
- การทดสอบ, การปรับและเปรียบเทียบโมเดลต่างๆ สำหรับกรณีการใช้งานของคุณใน Azure
- วิธีการนำ LLM มาใช้งาน

## เป้าหมายการเรียนรู้

หลังจากเสร็จสิ้นบทเรียนนี้ คุณจะสามารถ:

- เลือกโมเดลที่เหมาะสมสำหรับกรณีการใช้งานของคุณ
- เข้าใจวิธีการทดสอบ, ปรับและพัฒนาประสิทธิภาพของโมเดลของคุณ
- รู้วิธีที่ธุรกิจนำเสนอโมเดลไปใช้งานจริง

## ทำความเข้าใจประเภทของ LLMs ที่แตกต่างกัน

LLMs สามารถจัดประเภทได้หลายแบบตามสถาปัตยกรรม, ข้อมูลการฝึก และกรณีการใช้งาน การเข้าใจความแตกต่างเหล่านี้จะช่วยให้สตาร์ทอัพของเราเลือกโมเดลที่เหมาะสมสำหรับสถานการณ์นั้น และเข้าใจวิธีการทดสอบ ปรับ และพัฒนาประสิทธิภาพ

มีโมเดล LLM หลายประเภทมาก การเลือกโมเดลของคุณขึ้นอยู่กับสิ่งที่คุณตั้งใจจะใช้, ข้อมูลของคุณ, งบประมาณที่คุณพร้อมจ่าย และอื่นๆ

ขึ้นอยู่กับว่าคุณตั้งใจจะใช้โมเดลเพื่อข้อความ, เสียง, วิดีโอ, การสร้างภาพ และอื่นๆ คุณอาจเลือกโมเดลชนิดที่แตกต่างกัน

- **การรู้จำเสียงและคำพูด** โมเดลสไตล์ Whisper ยังคงมีประโยชน์ในฐานะโมเดลรู้จำเสียงทั่วไป แต่ตัวเลือกในการใช้งานจริงในปัจจุบันยังรวมถึงโมเดลพูดเป็นข้อความใหม่ เช่น `gpt-4o-transcribe`, `gpt-4o-mini-transcribe` และรุ่น diarization ประเมินขอบเขตภาษาที่รองรับ, การแยกเสียงตามผู้พูด, การสนับสนุนแบบเรียลไทม์, ความหน่วงเวลา และค่าใช้จ่ายสำหรับสถานการณ์ของคุณ เรียนรู้เพิ่มเติมใน [เอกสาร OpenAI speech-to-text](https://platform.openai.com/docs/guides/speech-to-text?WT.mc_id=academic-105485-koreyst)

- **การสร้างภาพ** DALL-E และ Midjourney เป็นตัวเลือกการสร้างภาพที่เป็นที่รู้จักดี, แต่ API ภาพของ OpenAI ในปัจจุบันเน้นที่โมเดลภาพ GPT เช่น `gpt-image-2` ในขณะที่ Stable Diffusion, Imagen, Flux และครอบครัวโมเดลอื่นๆ ก็เป็นตัวเลือกที่นิยม เปรียบเทียบความถูกต้องของการตอบสนองคำสั่ง, การสนับสนุนการแก้ไข, การควบคุมสไตล์, ข้อกำหนดด้านความปลอดภัย และลิขสิทธิ์ เรียนรู้เพิ่มเติมใน [คู่มือการสร้างภาพของ OpenAI](https://platform.openai.com/docs/guides/images?WT.mc_id=academic-105485-koreyst) และบทที่ 9 ของหลักสูตรนี้

- **การสร้างข้อความ** โมเดลข้อความขณะนี้ครอบคลุมโมเดลแนวหน้า, โมเดลเชิงเหตุผล, โมเดลขนาดเล็กที่มีความหน่วงต่ำ, และโมเดลน้ำหนักเปิด ตัวอย่างปัจจุบันรวมถึงโมเดล OpenAI GPT-5.x, โมเดล Anthropic Claude 4.x, โมเดล Google Gemini 3.x, โมเดล Meta Llama 4, และโมเดล Mistral อย่าเลือกแค่ตามวันปล่อยหรือราคา; เปรียบเทียบคุณภาพงานที่ทำ, ความหน่วง, หน้าต่างบริบท, การใช้เครื่องมือ, พฤติกรรมความปลอดภัย, การให้บริการในภูมิภาค, และต้นทุนรวม [แคตตาล็อกโมเดล Microsoft Foundry](https://ai.azure.com/catalog?WT.mc_id=academic-105485-koreyst) เป็นสถานที่ที่ดีสำหรับเปรียบเทียบโมเดลที่มีใน Azure

- **มัลติ-โมดัล** โมเดลปัจจุบันจำนวนมากสามารถประมวลผลได้มากกว่าข้อความ บางโมเดลรับภาพ, เสียง, หรือวิดีโอเป็นข้อมูลเข้า; บางโมเดลสามารถเรียกใช้เครื่องมือได้; และโมเดลเฉพาะทางสามารถสร้างภาพ, เสียง, หรือวิดีโอ ตัวอย่างเช่น โมเดล OpenAI ปัจจุบันรองรับข้อความและภาพ, โมเดล Gemini รองรับข้อความ, โค้ด, ภาพ, เสียง, และวิดีโอขึ้นอยู่กับเวอร์ชัน และ Llama 4 Scout และ Maverick เป็นโมเดลน้ำหนักเปิดที่รองรับมัลติโมดัลโดยธรรมชาติ ตรวจสอบการ์ดโมเดลแต่ละใบสำหรับข้อมูลเข้าและออกที่รองรับก่อนสร้างเวิร์กโฟลว์รอบๆ โมเดลนั้น

การเลือกโมเดลหมายความว่าคุณจะได้รับความสามารถพื้นฐานบางอย่าง แต่นั่นอาจยังไม่เพียงพอ หลายครั้งคุณจะมีข้อมูลเฉพาะบริษัทที่คุณต้องแจ้งให้ LLM รับรู้ มีตัวเลือกหลายแบบว่าจะจัดการกับเรื่องนี้อย่างไร จะพูดถึงเพิ่มเติมในส่วนที่ตามมา

### Foundation Models เทียบกับ LLMs

คำว่า Foundation Model ถูก [นิยามโดยนักวิจัยที่ Stanford](https://arxiv.org/abs/2108.07258?WT.mc_id=academic-105485-koreyst) และกำหนดว่าเป็นโมเดล AI ที่ตรงตามเกณฑ์บางข้อ เช่น:

- **ถูกฝึกโดยใช้การเรียนรู้แบบไม่มีผู้ควบคุมหรือแบบเรียนรู้ด้วยตนเอง** หมายความว่าโมเดลถูกฝึกบนข้อมูลมัลติโมดัลที่ไม่มีป้ายกำกับ และไม่ต้องการการแปะป้ายหรือการตีความข้อมูลโดยมนุษย์ในกระบวนการฝึก
- **เป็นโมเดลขนาดใหญ่มาก** อิงจากโครงข่ายประสาทเทียมลึกที่ฝึกบนพารามิเตอร์นับพันล้าน
- **โดยปกติมีวัตถุประสงค์เพื่อใช้เป็น ‘ฐาน’ สำหรับโมเดลอื่นๆ** หมายความว่าสามารถนำไปใช้เป็นจุดเริ่มต้นเพื่อสร้างโมเดลอื่นๆ ต่อจากนั้นโดยการปรับแต่งเพิ่มเติม

### โมเดลน้ำหนักเปิด/โอเพนซอร์ส เทียบกับ โมเดลลิขสิทธิ์

อีกวิธีหนึ่งในการจัดประเภท LLMs คือการดูว่าโมเดลนั้นเป็นน้ำหนักเปิด, โอเพนซอร์ส หรือเป็นลิขสิทธิ์

โมเดลโอเพนซอร์สและน้ำหนักเปิดเปิดโครงสร้างโมเดลให้ตรวจสอบ, ดาวน์โหลด หรือปรับแต่งได้ แต่ลิขสิทธิ์อาจแตกต่างกัน บางตัวเป็นโอเพนซอร์สเต็มรูปแบบ ขณะที่บางตัวเป็นโมเดลน้ำหนักเปิดที่มีข้อจำกัดการใช้งาน เหมาะสำหรับธุรกิจที่ต้องการควบคุมการปรับใช้, การเก็บข้อมูลในพื้นที่, ต้นทุน หรือการปรับแต่งมากขึ้น อย่างไรก็ตาม ทีมงานยังต้องตรวจสอบเงื่อนไขลิขสิทธิ์, ค่าใช้จ่ายการให้บริการ, การบำรุงรักษา, การอัปเดตความปลอดภัย และคุณภาพการประเมินก่อนนำไปใช้ในงานจริง ตัวอย่าง ได้แก่ [Meta Llama 4](https://ai.meta.com/blog/llama-4-multimodal-intelligence/?WT.mc_id=academic-105485-koreyst), บางโมเดลของ [Mistral](https://docs.mistral.ai/models/overview?WT.mc_id=academic-105485-koreyst), และโมเดลมากมายที่โฮสต์บน [Hugging Face](https://huggingface.co/models?WT.mc_id=academic-105485-koreyst)

โมเดลลิขสิทธิ์เป็นของผู้ให้บริการและโฮสต์โดยผู้ให้บริการ โมเดลเหล่านี้มักถูกปรับให้เหมาะกับการใช้งานในผลิตภัณฑ์ที่จัดการแล้ว และสามารถให้การสนับสนุนที่แข็งแกร่ง, ระบบความปลอดภัย, การผนวกเครื่องมือ และการปรับขนาดได้ อย่างไรก็ตาม ลูกค้ามักไม่สามารถตรวจสอบหรือแก้ไขน้ำหนักโมเดลได้ และต้องตรวจสอบเงื่อนไขผู้ให้บริการเกี่ยวกับความเป็นส่วนตัว, การเก็บข้อมูล, การปฏิบัติตามกฎระเบียบ และการใช้งานที่ยอมรับได้ ตัวอย่างได้แก่ [โมเดล OpenAI](https://platform.openai.com/docs/models?WT.mc_id=academic-105485-koreyst), [Google Gemini](https://deepmind.google/models/gemini/pro/?WT.mc_id=academic-105485-koreyst), และ [Anthropic Claude](https://platform.claude.com/docs/en/about-claude/models/overview?WT.mc_id=academic-105485-koreyst)

### การสร้าง Embedding เทียบกับ การสร้างภาพ เทียบกับ การสร้างข้อความและโค้ด

LLMs ยังสามารถจัดประเภทโดยผลลัพธ์ที่สร้างขึ้น

Embeddings เป็นชุดโมเดลที่สามารถแปลงข้อความเป็นรูปแบบตัวเลขที่เรียกว่า embedding ซึ่งเป็นตัวแทนเชิงตัวเลขของข้อความเข้า Embeddings ช่วยให้เครื่องจักรเข้าใจความสัมพันธ์ระหว่างคำหรือประโยคได้ง่ายขึ้น และสามารถใช้เป็นข้อมูลเข้าโดยโมเดลอื่น เช่น โมเดลจัดประเภท หรือโมเดลจัดกลุ่มซึ่งมีประสิทธิภาพดีกับข้อมูลเชิงตัวเลข โมเดล embedding มักใช้สำหรับการถ่ายโอนความรู้ (transfer learning) โดยโมเดลจะถูกสร้างขึ้นสำหรับงานตัวแทนที่มีข้อมูลจำนวนมากก่อน และจากนั้นจึงนำรายหนักโมเดล (embedding) ไปใช้ซ้ำกับงานอื่น ตัวอย่างในหมวดนี้ได้แก่ [OpenAI embeddings](https://platform.openai.com/docs/models/embeddings?WT.mc_id=academic-105485-koreyst)

![Embedding](../../../translated_images/th/Embedding.c3708fe988ccf760.webp)

โมเดลสร้างภาพเป็นโมเดลที่สร้างภาพ โมเดลเหล่านี้มักถูกใช้สำหรับการแก้ไขภาพ, สังเคราะห์ภาพ และแปลงภาพ โมเดลสร้างภาพมักได้รับการฝึกบนชุดข้อมูลภาพขนาดใหญ่ เช่น [LAION-5B](https://laion.ai/blog/laion-5b/?WT.mc_id=academic-105485-koreyst) และสามารถใช้สร้างภาพใหม่หรือแก้ไขภาพที่มีอยู่ด้วยเทคนิคเช่น inpainting, super-resolution และการเติมสี ตัวอย่างได้แก่ [โมเดล GPT Image](https://platform.openai.com/docs/guides/images?WT.mc_id=academic-105485-koreyst), [โมเดล Stable Diffusion](https://github.com/Stability-AI/StableDiffusion?WT.mc_id=academic-105485-koreyst), และโมเดล Imagen

โมเดลสร้างข้อความและโค้ดคือโมเดลที่สร้างข้อความหรือโค้ด โมเดลเหล่านี้มักถูกใช้สำหรับการสรุปข้อความ, การแปลภาษา, และการตอบคำถาม โมเดลสร้างข้อความมักได้รับการฝึกบนชุดข้อมูลข้อความขนาดใหญ่ เช่น [BookCorpus](https://www.cv-foundation.org/openaccess/content_iccv_2015/html/Zhu_Aligning_Books_and_ICCV_2015_paper.html?WT.mc_id=academic-105485-koreyst) และสามารถใช้สร้างข้อความใหม่หรือตอบคำถาม โมเดลสร้างโค้ด เช่น [CodeParrot](https://huggingface.co/codeparrot?WT.mc_id=academic-105485-koreyst) มักได้รับการฝึกบนชุดข้อมูลโค้ดจำนวนมาก เช่น GitHub และสามารถสร้างโค้ดใหม่ หรือตรวจแก้ข้อผิดพลาดในโค้ดที่มีอยู่

### Encoder-Decoder เทียบกับ Decoder-only

เพื่อพูดถึงประเภทของสถาปัตยกรรม LLMs ลองใช้การเปรียบเทียบแบบง่ายๆ

สมมติว่าผู้จัดการของคุณมอบหมายงานให้คุณเขียนแบบทดสอบสำหรับนักเรียน คุณมีเพื่อนร่วมงานสองคน; คนหนึ่งรับผิดชอบสร้างเนื้อหาและอีกคนหนึ่งรับผิดชอบตรวจทาน

ผู้สร้างเนื้อหาเหมือนกับโมเดล decoder-only: พวกเขาสามารถดูหัวข้อ, ดูสิ่งที่คุณเขียนไปแล้ว แล้วสร้างเนื้อหาต่อจากบริบทนั้น พวกเขาเก่งในการเขียนเนื้อหาที่น่าสนใจและให้ข้อมูล แต่ไม่ใช่ตัวเลือกที่ดีที่สุดเมื่อภารกิจคือการจัดประเภท, ค้นคืน หรือเข้ารหัสข้อมูล ตัวอย่างครอบครัวโมเดล decoder-only ได้แก่ GPT และ Llama

ผู้ตรวจทานเหมือนกับโมเดล encoder-only, พวกเขาดูเนื้อหาที่เขียนและคำตอบ, สังเกตความสัมพันธ์ระหว่างกันและเข้าใจบริบท แต่ไม่เก่งในการสร้างเนื้อหา ตัวอย่างโมเดล encoder-only คือ BERT

สมมติว่าเราสามารถมีคนที่สามารถสร้างและตรวจทานแบบทดสอบได้ ซึ่งเป็นโมเดล encoder-decoder ตัวอย่างเช่น BART และ T5

### บริการ เทียบกับ โมเดล

ตอนนี้มาพูดถึงความแตกต่างระหว่างบริการกับโมเดล บริการคือผลิตภัณฑ์ที่ให้โดยผู้ให้บริการคลาวด์ มักเป็นการรวมตัวของโมเดล, ข้อมูล และส่วนประกอบอื่นๆ โมเดลเป็นส่วนประกอบหลักของบริการ และมักเป็น foundation model เช่น LLM

บริการมักถูกปรับแต่งสำหรับการใช้งานจริงและมักใช้งานง่ายกว่าโมเดลผ่านส่วนติดต่อผู้ใช้แบบกราฟิก อย่างไรก็ตาม บริการไม่ได้มีให้ใช้ฟรีเสมอไป และอาจต้องสมัครสมาชิกหรือชำระเงิน เพื่อแลกกับการใช้เครื่องมือและทรัพยากรของเจ้าของบริการ, การปรับปรุงต้นทุน และการปรับขนาดได้ง่าย ตัวอย่างบริการคือ [Azure OpenAI Service](https://learn.microsoft.com/azure/ai-services/openai/overview?WT.mc_id=academic-105485-koreyst) ที่มีแผนจ่ายตามการใช้งานจริง หมายความว่าผู้ใช้จะถูกเรียกเก็บเงินตามปริมาณการใช้ Azure OpenAI Service ยังให้ความปลอดภัยระดับองค์กรและกรอบงาน AI ที่รับผิดชอบชั้นบนของความสามารถโมเดล

โมเดลคือสิ่งประดิษฐ์ของโครงข่ายประสาท: พารามิเตอร์, น้ำหนัก, สถาปัตยกรรม, ตัวแปลงโทเค็น และการตั้งค่าที่สนับสนุน การรันโมเดลบนเครื่องภายในหรือในสภาพแวดล้อมส่วนตัวต้องมีฮาร์ดแวร์ที่เหมาะสม, โครงสร้างพื้นฐานสำหรับให้บริการ, การตรวจสอบ และต้องมีใบอนุญาตโอเพนซอร์ส/น้ำหนักเปิดที่เข้ากันได้ หรือใบอนุญาตเชิงพาณิชย์ โมเดลน้ำหนักเปิดเช่น Llama 4 หรือโมเดล Mistral สามารถโฮสต์เองได้ แต่ยังคงต้องการพลังการประมวลผลและความเชี่ยวชาญด้านการปฏิบัติการ

## การปรับปรุงผลลัพธ์ LLM

เราได้สำรวจ LLMs ประเภทต่างๆ กับทีมสตาร์ทอัพของเราและแพลตฟอร์มคลาวด์ (Microsoft Foundry) ที่ช่วยให้เราสามารถเปรียบเทียบโมเดลต่างๆ ประเมินผลบนข้อมูลทดสอบ ปรับปรุงประสิทธิภาพ และนำไปใช้บนปลายทางการสรุปผล

แต่เมื่อใดที่ควรพิจารณาปรับแต่งโมเดลแทนการใช้โมเดลที่ผ่านการฝึกมาแล้ว? มีแนวทางอื่นๆ ในการปรับปรุงประสิทธิภาพโมเดลบนภาระงานเฉพาะหรือไม่?

มีหลายแนวทางที่ธุรกิจสามารถใช้เพื่อให้ได้ผลลัพธ์ที่ต้องการจาก LLM คุณสามารถเลือกโมเดลประเภทต่างๆ ที่มีระดับการฝึกอบรมแตกต่างกันเมื่อนำ LLM ไปใช้ในสภาพแวดล้อมจริง โดยมีความซับซ้อน ค่าใช้จ่าย และคุณภาพที่ต่างกัน นี่คือแนวทางที่แตกต่างกัน:

- **การออกแบบ prompt พร้อมบริบท** แนวคิดคือให้บริบทเพียงพอเมื่อคุณส่ง prompt เพื่อให้แน่ใจว่าคุณได้รับคำตอบที่ต้องการ

- **การสร้างเสริมการดึงข้อมูล (Retrieval Augmented Generation, RAG)** ข้อมูลของคุณอาจอยู่ในฐานข้อมูลหรือจุดเชื่อมต่อเว็บ ตัวอย่างเช่น เพื่อให้ข้อมูลนี้ หรือส่วนย่อยของข้อมูลนี้ รวมอยู่ในเวลาที่ส่ง prompt คุณสามารถดึงข้อมูลที่เกี่ยวข้องและทำให้เป็นส่วนหนึ่งของ prompt ของผู้ใช้ได้

- **โมเดลที่ปรับแต่งแล้ว** ที่นี่ คุณได้ฝึกโมเดลเพิ่มเติมด้วยข้อมูลของคุณเอง ซึ่งทำให้โมเดลแม่นยำและตอบสนองความต้องการได้มากขึ้น แต่ค่าใช้จ่ายอาจสูง

![LLMs deployment](../../../translated_images/th/Deploy.18b2d27412ec8c02.webp)

### การออกแบบ Prompt พร้อมบริบท

LLM ที่ผ่านการฝึกอบรมล่วงหน้าทำงานได้ดีมากกับงานภาษาธรรมชาติทั่วไป แม้แต่การเรียกใช้งานด้วย prompt สั้นๆ เช่น ประโยคให้เติมหรือคำถาม – เรียกว่า “การเรียนรู้แบบ zero-shot”

อย่างไรก็ตาม ยิ่งผู้ใช้สามารถกำหนดคำถามของพวกเขาให้ชัดเจนขึ้น โดยการร้องขออย่างละเอียดและตัวอย่าง – บริบท – ตอบสนองจะถูกต้องและใกล้เคียงกับความคาดหวังของผู้ใช้มากขึ้น ในกรณีนี้ เราพูดถึง “การเรียนรู้แบบ one-shot” หาก prompt มีตัวอย่างเพียงหนึ่งตัวอย่าง และ “การเรียนรู้แบบ few-shot” หากมีหลายตัวอย่าง
การออกแบบ prompt พร้อมบริบทเป็นแนวทางที่คุ้มค่าที่สุดในการเริ่มต้น

### การสร้างเสริมการดึงข้อมูล (RAG)

LLM มีข้อจำกัดที่สามารถใช้ได้เฉพาะข้อมูลที่ใช้ในการฝึกอบรมเท่านั้นในการสร้างคำตอบ ซึ่งหมายความว่าพวกเขาจะไม่ทราบข้อมูลใดๆ ที่เกิดขึ้นหลังจากกระบวนการฝึก และไม่สามารถเข้าถึงข้อมูลที่ไม่เปิดเผยสาธารณะ (เช่น ข้อมูลบริษัท)
ปัญหานี้สามารถแก้ไขได้โดยใช้เทคนิค RAG ซึ่งเพิ่มข้อมูลจากภายนอกในรูปแบบของชิ้นส่วนเอกสาร โดยพิจารณาข้อจำกัดความยาวของ prompt เทคนิคนี้ได้รับการสนับสนุนโดยเครื่องมือฐานข้อมูลเวกเตอร์ (เช่น [Azure Vector Search](https://learn.microsoft.com/azure/search/vector-search-overview?WT.mc_id=academic-105485-koreyst)) ที่ดึงชิ้นส่วนที่มีประโยชน์จากแหล่งข้อมูลที่กำหนดไว้ล่วงหน้าและเพิ่มเข้าในบริบทของ prompt

เทคนิคนี้มีประโยชน์มากเมื่อธุรกิจไม่มีข้อมูล เวลาหรือทรัพยากรเพียงพอสำหรับการปรับแต่ง LLM แต่ยังคงต้องการปรับปรุงประสิทธิภาพในภาระงานเฉพาะและลดความเสี่ยงของคำตอบที่ผิดพลาด ล้าสมัย หรือไม่มีการสนับสนุน

### โมเดลที่ปรับแต่ง

การปรับแต่งเป็นกระบวนการที่ใช้การเรียนรู้แบบถ่ายโอนเพื่อ ‘ปรับ’ โมเดลให้เหมาะกับงานล่างหรือแก้ปัญหาเฉพาะ แตกต่างจากการเรียนรู้แบบ few-shot และ RAG จะส่งผลให้เกิดโมเดลใหม่ที่มีน้ำหนักและไบแอสอัปเดต ต้องใช้ชุดตัวอย่างฝึกที่ประกอบด้วยอินพุตเดียว (prompt) และเอาต์พุตที่เกี่ยวข้อง (การเติมคำ)
นี่คือแนวทางที่แนะนำหาก:

- **ใช้โมเดลขนาดเล็กที่เฉพาะเจาะจงงาน** ธุรกิจอาจต้องการปรับแต่งโมเดลขนาดเล็กสำหรับงานเฉพาะ มากกว่าการส่ง prompt ซ้ำ ๆ ไปยังโมเดลขนาดใหญ่กว่า ซึ่งทำให้ประหยัดค่าใช้จ่ายและรวดเร็วกว่า

- **พิจารณาความหน่วงเวลา** ความหน่วงเวลาสำคัญสำหรับกรณีใช้งานเฉพาะ ดังนั้นจึงไม่สามารถใช้ prompt ที่ยาวมากหรือจำนวนตัวอย่างที่ควรเรียนรู้ไม่สอดคล้องกับข้อจำกัดความยาวของ prompt ได้

- **ปรับพฤติกรรมให้คงที่** ธุรกิจมีตัวอย่างคุณภาพสูงจำนวนมากและต้องการให้โมเดลปฏิบัติตามรูปแบบงาน รูปแบบเอาต์พุต น้ำเสียง หรือสไตล์ที่เฉพาะเจาะจงโดเมนอย่างสม่ำเสมอ หากปัญหาหลักคือข้อเท็จจริงปัจจุบันหรือความรู้ส่วนตัวที่เปลี่ยนแปลงบ่อย ให้ใช้ RAG แทนการพึ่งพาการปรับแต่งเพียงอย่างเดียว

### โมเดลที่ฝึกอบรม

การฝึก LLM ตั้งแต่ต้นเป็นแนวทางที่ยากและซับซ้อนที่สุด ต้องใช้ข้อมูลจำนวนมหาศาล ทรัพยากรที่มีทักษะและพลังการประมวลผลที่เหมาะสม ตัวเลือกนี้ควรพิจารณาเฉพาะในกรณีที่ธุรกิจมีกรณีการใช้งานเฉพาะโดเมนและข้อมูลขนาดใหญ่ที่เน้นโดเมน

## การตรวจสอบความรู้

วิธีการที่ดีในการปรับปรุงผลลัพธ์การเติมคำของ LLM คืออะไร?

1. การออกแบบ prompt พร้อมบริบท
1. RAG
1. โมเดลที่ปรับแต่ง

ตอบ: ทั้งสามวิธีช่วยได้ เริ่มด้วยการออกแบบ prompt และบริบทเพื่อปรับปรุงอย่างรวดเร็ว และใช้ RAG เมื่อโมเดลต้องการข้อเท็จจริงปัจจุบันหรือข้อมูลส่วนตัวของธุรกิจ เลือกการปรับแต่งเมื่อคุณมีตัวอย่างคุณภาพสูงเพียงพอและต้องการให้โมเดลปฏิบัติตามงาน รูปแบบ น้ำเสียง หรือรูปแบบโดเมนอย่างสม่ำเสมอ
