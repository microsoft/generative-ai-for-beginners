## مقدمه

عامل‌های هوش مصنوعی یک پیشرفت هیجان‌انگیز در هوش مصنوعی تولیدی هستند که مدل‌های زبان بزرگ (LLMها) را از دستیاران به عامل‌هایی تبدیل می‌کنند که قادر به انجام اقدامات هستند. چارچوب‌های عامل هوش مصنوعی به توسعه‌دهندگان این امکان را می‌دهند تا برنامه‌هایی ایجاد کنند که به LLMها دسترسی به ابزارها و مدیریت وضعیت ارائه می‌دهند. این چارچوب‌ها همچنین دیدپذیری را افزایش می‌دهند تا کاربران و توسعه‌دهندگان بتوانند اقدامات برنامه‌ریزی شده توسط LLMها را نظارت کرده و مدیریت تجربه را بهبود بخشند.

این درس موارد زیر را پوشش خواهد داد:

- درک اینکه عامل هوش مصنوعی چیست - عامل هوش مصنوعی دقیقاً چیست؟
- بررسی پنج چارچوب عامل هوش مصنوعی مختلف - چه چیزی آنها را منحصربه‌فرد می‌کند؟
- استفاده از این عامل‌ها در موارد کاربردی مختلف - چه زمانی باید از عامل‌های هوش مصنوعی استفاده کنیم؟

## عامل‌های هوش مصنوعی چیستند؟

عامل‌های هوش مصنوعی حوزه‌ای بسیار هیجان‌انگیز در دنیای هوش مصنوعی تولیدی هستند. با این هیجان گاهی اوقات سردرگمی در اصطلاحات و کاربرد آنها به وجود می‌آید. برای ساده و شامل بودن اکثر ابزارهایی که به عامل‌های هوش مصنوعی اشاره دارند، ما از این تعریف استفاده می‌کنیم:

عامل‌های هوش مصنوعی به مدل‌های زبان بزرگ (LLMها) اجازه می‌دهند تا با دسترسی به **وضعیت** و **ابزارها** وظایف را انجام دهند.

![مدل عامل](../../../translated_images/fa/what-agent.21f2893bdfd01e6a.webp)

بیایید این اصطلاحات را تعریف کنیم:

**مدل‌های زبان بزرگ** - این‌ها مدل‌هایی هستند که در کل این دوره به آنها اشاره شده است مانند GPT-5، GPT-40 و Llama 3.3 و غیره.

**وضعیت** - این به زمینه‌ای اشاره دارد که LLM در آن کار می‌کند. LLM از زمینه اقدامات گذشته و زمینه فعلی استفاده می‌کند تا تصمیم‌گیری برای اقدامات بعدی را هدایت کند. چارچوب‌های عامل هوش مصنوعی این زمینه را برای توسعه‌دهندگان ساده‌تر نگه می‌دارند.

**ابزارها** - برای انجام کاری که کاربر درخواست کرده و LLM برنامه‌ریزی کرده است، LLM نیاز به دسترسی به ابزارها دارد. برخی نمونه‌های ابزار می‌تواند پایگاه داده، API، نرم‌افزار خارجی یا حتی یک LLM دیگر باشد!

امیدواریم این تعاریف پایه خوبی به شما بدهند تا با نگاه به چگونگی پیاده‌سازی آنها، چند چارچوب عامل هوش مصنوعی مختلف را بررسی کنیم:

## عامل‌های LangChain

[عامل‌های LangChain](https://python.langchain.com/docs/how_to/#agents?WT.mc_id=academic-105485-koreyst) پیاده‌سازی تعاریفی است که ما در بالا ارائه دادیم.

برای مدیریت **وضعیت**، از تابع داخلی به نام `AgentExecutor` استفاده می‌کند. این تابع `agent` تعریف شده و `tools` در دسترس را می‌پذیرد.

`AgentExecutor` همچنین تاریخچه چت را ذخیره می‌کند تا زمینه چت را فراهم کند.

![عامل‌های LangChain](../../../translated_images/fa/langchain-agents.edcc55b5d5c43716.webp)

LangChain یک [فهرست ابزارها](https://integrations.langchain.com/tools?WT.mc_id=academic-105485-koreyst) ارائه می‌دهد که می‌توانند در برنامه شما وارد شده و LLM به آنها دسترسی پیدا کند. این ابزارها توسط جامعه و تیم LangChain ساخته شده‌اند.

شما سپس می‌توانید این ابزارها را تعریف کرده و به `AgentExecutor` بفرستید.

دیدپذیری بخش مهم دیگری هنگام صحبت درباره عامل‌های هوش مصنوعی است. برای توسعه‌دهندگان برنامه مهم است که بفهمند LLM از کدام ابزار استفاده می‌کند و چرا.. برای این منظور، تیم LangChain ابزار LangSmith را توسعه داده است.

## AutoGen

چارچوب عامل هوش مصنوعی بعدی که بررسی می‌کنیم [AutoGen](https://microsoft.github.io/autogen/?WT.mc_id=academic-105485-koreyst) است. تمرکز اصلی AutoGen بر روی مکالمات است. عامل‌ها هم **قابل گفتگو** و هم **قابل سفارشی‌سازی** هستند.

**قابل گفتگو -** LLMها می‌توانند مکالمه را با یک LLM دیگر آغاز و ادامه دهند تا یک وظیفه را به انجام برسانند. این با ایجاد `AssistantAgents` و دادن پیام سیستم خاصی به آنها انجام می‌شود.

```python

autogen.AssistantAgent( name="Coder", llm_config=llm_config, ) pm = autogen.AssistantAgent( name="Product_manager", system_message="Creative in software product ideas.", llm_config=llm_config, )

```

**قابل سفارشی‌سازی** - عامل‌ها نه تنها می‌توانند به عنوان LLM تعریف شوند بلکه می‌توانند کاربر یا ابزار باشند. به عنوان یک توسعه‌دهنده، شما می‌توانید `UserProxyAgent` تعریف کنید که مسئول تعامل با کاربر برای بازخورد در تکمیل یک وظیفه است. این بازخورد می‌تواند اجرای وظیفه را ادامه دهد یا متوقف کند.

```python
user_proxy = UserProxyAgent(name="user_proxy")
```

### وضعیت و ابزارها

برای تغییر و مدیریت وضعیت، یک عامل دستیار کدهای پایتون تولید می‌کند تا وظیفه را انجام دهد.

در اینجا نمونه‌ای از فرآیند آمده است:

![AutoGen](../../../translated_images/fa/autogen.dee9a25a45fde584.webp)

#### LLM تعریف شده با پیام سیستم

```python
system_message="For weather related tasks, only use the functions you have been provided with. Reply TERMINATE when the task is done."
```

این پیام سیستم LLM خاص را هدایت می‌کند که کدام توابع برای وظیفه‌اش مرتبط هستند. به یاد داشته باشید، با AutoGen می‌توانید چندین AssistantAgent با پیام‌های سیستمی مختلف داشته باشید.

#### چت توسط کاربر شروع می‌شود

```python
user_proxy.initiate_chat( chatbot, message="I am planning a trip to NYC next week, can you help me pick out what to wear? ", )

```

این پیام از user_proxy (انسان) شروع‌کننده روند عامل برای بررسی توابع ممکن است که باید اجرا شوند.

#### تابع اجرا می‌شود

```bash
chatbot (to user_proxy):

***** Suggested tool Call: get_weather ***** Arguments: {"location":"New York City, NY","time_periond:"7","temperature_unit":"Celsius"} ******************************************************** --------------------------------------------------------------------------------

>>>>>>>> EXECUTING FUNCTION get_weather... user_proxy (to chatbot): ***** Response from calling function "get_weather" ***** 112.22727272727272 EUR ****************************************************************

```

پس از پردازش چت اولیه، عامل ابزار پیشنهاد شده را برای فراخوانی ارسال می‌کند. در این مورد، تابعی به نام `get_weather` است. بسته به پیکربندی شما، این تابع می‌تواند به طور خودکار توسط عامل اجرا و خوانده شود یا با ورودی کاربر اجرا شود.

شما می‌توانید فهرستی از [نمونه‌های کد AutoGen](https://microsoft.github.io/autogen/docs/Examples/?WT.mc_id=academic-105485-koreyst) را برای کاوش بیشتر روش شروع به ساخت بیابید.

## چارچوب عامل مایکروسافت

[چارچوب عامل مایکروسافت](https://learn.microsoft.com/agent-framework/?WT.mc_id=academic-105485-koreyst) SDK متن‌باز مایکروسافت برای ساخت عامل‌های هوش مصنوعی و سیستم‌های چندعامله در هر دو زبان **پایتون** و **.NET** است. این چارچوب قدرت دو پروژه قبلی مایکروسافت — ویژگی‌های سازمانی **Semantic Kernel** و ارکستراسیون چندعامله **AutoGen** — را در یک چارچوب واحد و پشتیبانی‌شده گرد هم می‌آورد. اگر امروز پروژه عاملی جدیدی را شروع می‌کنید، این جانشین پیشنهادی AutoGen است.

این چارچوب از یک **عامل چت** ساده تا **روندهای کاری چندعامله** پیچیده مقیاس می‌یابد و مستقیماً با Microsoft Foundry، Azure OpenAI و OpenAI ادغام می‌شود. همچنین دیدپذیری داخلی از طریق OpenTelemetry را ارائه می‌دهد تا دقیقاً بتوانید رد عملکرد عامل‌های خود را دنبال کنید.

### وضعیت و ابزارها

**وضعیت** - چارچوب زمینه مکالمات را از طریق **رشته‌ها** مدیریت می‌کند. یک عامل تاریخچه پیام‌ها (درخواست‌های کاربر، فراخوانی ابزارها و نتایج آنها) را دنبال می‌کند، بنابراین هر قدم بر پایه قدم‌های قبلی ساخته می‌شود. رشته‌ها همچنین می‌توانند ذخیره شوند تا مکالمه بعداً متوقف و از سر گرفته شود.

**ابزارها** - به عامل ابزارها را با ارسال توابع ساده پایتون می‌دهید. پارامترهای دارای نوع به طور خودکار به یک طرح تبدیل می‌شوند تا مدل بداند چه زمانی و چگونه آنها را فراخوانی کند (فراخوانی تابع). این چارچوب همچنین از سرورهای پروتکل زمینه مدل (MCP) و ابزارهای میزبانی‌شده مانند مفسر کد پشتیبانی می‌کند.

در اینجا نمونه‌ای از یک عامل واحد با یک ابزار سفارشی آمده است:

```python
import asyncio
from typing import Annotated

from pydantic import Field
from agent_framework import Agent
from agent_framework.openai import OpenAIChatClient


def get_weather(
    location: Annotated[str, Field(description="The location to get the weather for.")],
) -> str:
    """Get the weather for a given location."""
    return f"The weather in {location} is sunny with a high of 22°C."


async def main():
    agent = Agent(
        client=OpenAIChatClient(),
        instructions="You are a helpful assistant that can answer weather questions.",
        tools=[get_weather],
    )

    response = await agent.run("What's the weather in Amsterdam?")
    print(response)


asyncio.run(main())
```

برای اتصال به Azure OpenAI در Microsoft Foundry به جای آن، نقطه پایان و مدارک خود را به مشتری ارسال کنید:

```python
from azure.identity.aio import AzureCliCredential
from agent_framework.openai import OpenAIChatClient

client = OpenAIChatClient(
    model="my-gpt-5-mini-deployment",
    azure_endpoint="https://my-resource.openai.azure.com",
    credential=AzureCliCredential(),
)
```

### روندهای کاری چندعامله

جایی که این چارچوب واقعاً برجسته است ارکستراسیون چند عامل به صورت همزمان است. برای مثال، می‌توانید عوامل را یکی پس از دیگری اجرا کنید (هر کدام زمینه خود را به بعدی منتقل کنند) یا بازوهای موازی به چند عامل باز کنید و نتایج آنها را جمع‌آوری کنید:

```python
from agent_framework.orchestrations import SequentialBuilder, ConcurrentBuilder

# اجرای نمایندگان به صورت پشت سر هم، با انتقال زمینه گفتگو در طول زنجیره
sequential = SequentialBuilder(participants=[researcher, writer, editor]).build()

# اجرای همزمان نمایندگان به صورت موازی، سپس تجمیع پاسخ‌های آن‌ها
concurrent = ConcurrentBuilder(participants=[analyst_a, analyst_b, analyst_c]).build()
```

## Taskweaver

چارچوب عامل بعدی که بررسی می‌کنیم [Taskweaver](https://microsoft.github.io/TaskWeaver/?WT.mc_id=academic-105485-koreyst) است. این چارچوب به عنوان یک عامل "کد‌محور" شناخته می‌شود زیرا به جای کار صرف با `رشته‌ها`، می‌تواند با DataFrameها در پایتون کار کند. این برای تحلیل و تولید داده بسیار مفید است. این می‌تواند شامل ایجاد نمودارها و چارت‌ها یا تولید اعداد تصادفی باشد.

### وضعیت و ابزارها

برای مدیریت وضعیت مکالمه، TaskWeaver از مفهوم `Planner` استفاده می‌کند. `Planner` یک LLM است که درخواست کاربران را می‌گیرد و وظایفی را که باید برای انجام درخواست تکمیل شود ترسیم می‌کند.

برای به انجام رساندن وظایف، `Planner` به مجموعه‌ای از ابزارها به نام `Plugins` دسترسی دارد. این می‌تواند کلاس‌های پایتون یا یک مفسر کد عمومی باشد. این پلاگین‌ها به صورت تعبیه (embedding) ذخیره می‌شوند تا LLM بهتر بتواند پلاگین صحیح را جستجو کند.

![Taskweaver](../../../translated_images/fa/taskweaver.da8559999267715a.webp)

اینجا نمونه‌ای از یک پلاگین برای تشخیص ناهنجاری است:

```python
class AnomalyDetectionPlugin(Plugin): def __call__(self, df: pd.DataFrame, time_col_name: str, value_col_name: str):
```

کد قبل از اجرا تأیید می‌شود. یکی دیگر از ویژگی‌های مدیریت زمینه در Taskweaver، `تجربه` است. تجربه اجازه می‌دهد زمینه یک مکالمه برای مدت‌زمان طولانی‌تر در یک فایل YAML ذخیره شود. این می‌تواند به گونه‌ای پیکربندی شود که LLM در طول زمان در وظایف خاصی با توجه به مواجه‌شدن با مکالمات قبلی بهبود یابد.

## JARVIS

آخرین چارچوب عاملی که بررسی می‌کنیم [JARVIS](https://github.com/microsoft/JARVIS?tab=readme-ov-file&WT.mc_id=academic-105485-koreyst) است. فرق JARVIS این است که از یک LLM برای مدیریت `وضعیت` مکالمه استفاده می‌کند و `ابزارها` مدل‌های هوش مصنوعی دیگر هستند. هر کدام از این مدل‌های تخصصی برای انجام کارهای خاصی مانند تشخیص اشیاء، رونویسی یا توصیف تصویر طراحی شده‌اند.

![JARVIS](../../../translated_images/fa/jarvis.762ddbadbd1a3a33.webp)

این LLM که یک مدل چندمنظوره است، درخواست کاربر را دریافت می‌کند و وظیفه خاص و هرگونه آرگومان/داده لازم برای انجام آن وظیفه را شناسایی می‌کند.

```python
[{"task": "object-detection", "id": 0, "dep": [-1], "args": {"image": "e1.jpg" }}]
```

سپس LLM درخواست را به شکلی فرمت می‌کند که مدل تخصصی هوش مصنوعی بتواند آن را تفسیر کند، مانند JSON. پس از اینکه مدل هوش مصنوعی پیش‌بینی خود را بر اساس وظیفه انجام داد، پاسخ به LLM ارسال می‌شود.

اگر انجام کار نیازمند چندین مدل باشد، LLM همچنین پاسخ‌های آن مدل‌ها را تفسیر کرده و سپس آنها را با هم ترکیب می‌کند تا پاسخ نهایی به کاربر داده شود.
