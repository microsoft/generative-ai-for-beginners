## مقدمه

عامل‌های هوش مصنوعی یک پیشرفت هیجان‌انگیز در هوش مصنوعی مولد هستند که امکان می‌دهند مدل‌های بزرگ زبانی (LLMها) از دستیار به عامل‌هایی که قادر به انجام اقدامات هستند تکامل یابند. چارچوب‌های عامل هوش مصنوعی به توسعه‌دهندگان اجازه می‌دهند برنامه‌هایی بسازند که دسترسی LLMها به ابزارها و مدیریت وضعیت را فراهم کنند. این چارچوب‌ها همچنین دیدپذیری را بهبود می‌بخشند، به طوری که کاربران و توسعه‌دهندگان می‌توانند اقدامات برنامه‌ریزی‌شده توسط LLMها را نظارت کنند و بدین ترتیب مدیریت تجربه را بهبود بخشند.

درس موارد زیر را پوشش خواهد داد:

- درک اینکه عامل هوش مصنوعی چیست - عامل هوش مصنوعی دقیقاً چیست؟
- بررسی پنج چارچوب مختلف عامل هوش مصنوعی - چه ویژگی‌هایی آنها را متمایز می‌کند؟
- کاربرد این عامل‌های هوش مصنوعی در موارد مختلف - چه زمانی باید از عامل‌های هوش مصنوعی استفاده کرد؟

- تفاوت‌های بین برخی از چارچوب‌های محبوب عامل هوش مصنوعی و نحوه تمایز آنها را بفهمید.

## عامل‌های هوش مصنوعی چیستند؟

عامل‌های هوش مصنوعی حوزه بسیار هیجان‌انگیزی در دنیای هوش مصنوعی مولد هستند. با این هیجان، گاهی اوقات گیجی در تعاریف و کاربردشان وجود دارد. برای ساده و جامع بودن اکثر ابزارهایی که به عامل‌های هوش مصنوعی اشاره می‌کنند، تعریف زیر را استفاده می‌کنیم:

عامل‌های هوش مصنوعی به مدل‌های بزرگ زبانی (LLMها) اجازه می‌دهند با در اختیار گذاشتن **وضعیت** و **ابزارها**، کارها را انجام دهند.

![مدل عامل](../../../translated_images/fa/what-agent.21f2893bdfd01e6a.webp)

بیایید این اصطلاحات را تعریف کنیم:

**مدل‌های بزرگ زبانی** - این مدل‌هایی هستند که در این دوره ذکر شده‌اند مانند GPT-3.5، GPT-4، Llama-2 و غیره.

**وضعیت** - به زمینه‌ای اشاره دارد که LLM در آن کار می‌کند. LLM از زمینه‌ی اقدامات گذشته و زمینه‌ی جاری استفاده می‌کند تا تصمیم‌گیری برای اقدامات بعدی را هدایت کند. چارچوب‌های عامل هوش مصنوعی به توسعه‌دهندگان اجازه می‌دهند این زمینه را راحت‌تر حفظ کنند.

**ابزارها** - برای انجام کاری که کاربر درخواست کرده و LLM برنامه‌ریزی کرده است، LLM به ابزارها نیاز دارد. برخی نمونه‌های ابزار می‌توانند پایگاه داده، API، برنامه خارجی یا حتی یک LLM دیگر باشند!

امید است این تعاریف مبنای خوبی برای ادامه موضوع فراهم کنند تا ببینیم چگونه پیاده‌سازی می‌شوند. بیایید چند چارچوب عامل هوش مصنوعی مختلف را بررسی کنیم:

## عامل‌های LangChain

[عامل‌های LangChain](https://python.langchain.com/docs/how_to/#agents?WT.mc_id=academic-105485-koreyst) پیاده‌سازی تعاریف ارائه شده در بالا هستند.

برای مدیریت **وضعیت**، از تابع داخلی به نام `AgentExecutor` استفاده می‌کند. این تابع `agent` تعریف شده و `ابزار`های در دسترس را می‌پذیرد.

`AgentExecutor` همچنین تاریخچه چت را ذخیره می‌کند تا زمینه چت را فراهم کند.

![عامل‌های LangChain](../../../translated_images/fa/langchain-agents.edcc55b5d5c43716.webp)

LangChain یک [کاتالوگ ابزار](https://integrations.langchain.com/tools?WT.mc_id=academic-105485-koreyst) ارائه می‌دهد که می‌توان آنها را در برنامه خود وارد کرد تا LLM به آنها دسترسی داشته باشد. این ابزارها توسط جامعه و تیم LangChain ساخته شده‌اند.

سپس می‌توانید این ابزارها را تعریف کرده و به `AgentExecutor` ارسال کنید.

دیدپذیری جنبه مهم دیگری است وقتی درباره عامل‌های هوش مصنوعی صحبت می‌کنیم. برای توسعه‌دهندگان برنامه اهمیت دارد که بفهمند LLM کدام ابزار را استفاده می‌کند و چرا. برای این منظور، تیم LangChain محصولی به نام LangSmith توسعه داده است.

## AutoGen

چارچوب عامل هوش مصنوعی بعدی که بررسی می‌کنیم [AutoGen](https://microsoft.github.io/autogen/?WT.mc_id=academic-105485-koreyst) است. تمرکز اصلی AutoGen بر مکالمات است. عامل‌ها هم **قابل گفتگو** و هم **قابل سفارشی‌سازی** هستند.

**قابل گفتگو -** مدل‌های بزرگ زبانی می‌توانند گفتگویی را با LLM دیگر شروع و ادامه دهند تا کاری را انجام دهند. این با ایجاد `AssistantAgents` و دادن پیام سیستمی مشخص به آنها انجام می‌شود.

```python

autogen.AssistantAgent( name="Coder", llm_config=llm_config, ) pm = autogen.AssistantAgent( name="Product_manager", system_message="Creative in software product ideas.", llm_config=llm_config, )

```

**قابل سفارشی‌سازی** - عامل‌ها را نمی‌توان فقط به عنوان LLM تعریف کرد بلکه می‌توان آنها را کاربر یا ابزار هم تعریف کرد. به عنوان توسعه‌دهنده، شما می‌توانید `UserProxyAgent` تعریف کنید که مسئول تعامل با کاربر برای بازخورد در انجام کار است. این بازخورد یا اجرای کار را ادامه می‌دهد یا متوقف می‌کند.

```python
user_proxy = UserProxyAgent(name="user_proxy")
```

### وضعیت و ابزارها

برای تغییر و مدیریت وضعیت، یک عامل دستیار کد پایتون تولید می‌کند تا کار را انجام دهد.

مثال فرایند به شرح زیر است:

![AutoGen](../../../translated_images/fa/autogen.dee9a25a45fde584.webp)

#### LLM با پیام سیستمی تعریف شده

```python
system_message="For weather related tasks, only use the functions you have been provided with. Reply TERMINATE when the task is done."
```

این پیام سیستمی LLM خاص را راهنمایی می‌کند که کدام توابع مرتبط با کارش است. به یاد داشته باشید با AutoGen می‌توانید چندین AssistantAgent با پیام‌های سیستمی متفاوت داشته باشید.

#### گفت‌وگو توسط کاربر آغاز می‌شود

```python
user_proxy.initiate_chat( chatbot, message="I am planning a trip to NYC next week, can you help me pick out what to wear? ", )

```

این پیام از user_proxy (انسان) فرایند عامل را برای کاوش توابعی که باید اجرا شوند شروع می‌کند.

#### تابع اجرا می‌شود

```bash
chatbot (to user_proxy):

***** Suggested tool Call: get_weather ***** Arguments: {"location":"New York City, NY","time_periond:"7","temperature_unit":"Celsius"} ******************************************************** --------------------------------------------------------------------------------

>>>>>>>> EXECUTING FUNCTION get_weather... user_proxy (to chatbot): ***** Response from calling function "get_weather" ***** 112.22727272727272 EUR ****************************************************************

```

بعد از پردازش چت اولیه، عامل ابزار پیشنهادی برای فراخوانی را ارسال می‌کند. در اینجا تابعی به نام `get_weather` است. بسته به پیکربندی، این تابع ممکن است به طور خودکار توسط عامل اجرا و خوانده شود یا بر اساس ورودی کاربر اجرا گردد.

می‌توانید نمونه‌های کد [AutoGen](https://microsoft.github.io/autogen/docs/Examples/?WT.mc_id=academic-105485-koreyst) را برای شروع ساختن بیشتر بررسی کنید.

## چارچوب عامل مایکروسافت

[چارچوب عامل مایکروسافت](https://learn.microsoft.com/agent-framework/?WT.mc_id=academic-105485-koreyst) SDK متن‌باز مایکروسافت برای ساخت عامل‌های هوش مصنوعی و سیستم‌های چندعامله در هر دو زبان **پایتون** و **.NET** است. این چارچوب قدرت دو پروژه قبلی مایکروسافت — ویژگی‌های سازمانی **Semantic Kernel** و ارکستراسیون چندعامله **AutoGen** — را در یک چارچوب پشتیبانی شده گرد هم آورده است. اگر امروز پروژه عاملی جدیدی را شروع می‌کنید، این گزینه جایگزین توصیه‌شده AutoGen است.

این چارچوب از یک **عامل چت** واحد تا **جریان‌های کاری چندعامله** پیچیده مقیاس‌پذیر است و به طور مستقیم با Microsoft Foundry، Azure OpenAI و OpenAI ادغام می‌شود. همچنین از طریق OpenTelemetry قابلیت مشاهده داخلی دارد تا بتوانید دقیقا دنبال کنید عامل‌های شما چه کاری انجام می‌دهند.

### وضعیت و ابزارها

**وضعیت** - چارچوب مدیریت زمینه مکالمه را از طریق **رشته‌ها** برای شما انجام می‌دهد. یک عامل تاریخچه پیام‌ها (درخواست‌های کاربر، فراخوانی ابزارها و نتایج آنها) را پیگیری می‌کند، بنابراین هر نوبت بر مبنای نوبت‌های قبلی ساخته می‌شود. رشته‌ها همچنین می‌توانند حفظ شوند، به‌طوری که یک مکالمه می‌تواند متوقف و بعدا ادامه یابد.

**ابزارها** - شما ابزارها را به عامل با ارسال توابع ساده پایتون می‌دهید. پارامترهای دارای نوع به طور خودکار به یک طرح تبدیل می‌شوند تا مدل بداند چگونه و چه زمانی آنها را فراخوانی کند (فراخوانی تابع). چارچوب همچنین از سرورهای Model Context Protocol (MCP) و ابزارهای میزبانی شده مانند مفسر کد پشتیبانی می‌کند.

اینجا یک نمونه از یک عامل واحد با ابزار سفارشی آمده است:

```python
import asyncio
from typing import Annotated

from pydantic import Field
from agent_framework import Agent
from agent_framework.openai import OpenAIChatClient


def get_weather(
    location: Annotated[str, Field(description="The location to get the weather for.")],
) -> str:
    """Get the weather for a given location."""
    return f"The weather in {location} is sunny with a high of 22°C."


async def main():
    agent = Agent(
        client=OpenAIChatClient(),
        instructions="You are a helpful assistant that can answer weather questions.",
        tools=[get_weather],
    )

    response = await agent.run("What's the weather in Amsterdam?")
    print(response)


asyncio.run(main())
```

برای اتصال به Azure OpenAI در Microsoft Foundry، به جای آن نقطه پایانی و اعتبارنامه‌های خود را به کلاینت ارسال کنید:

```python
from azure.identity.aio import AzureCliCredential
from agent_framework.openai import OpenAIChatClient

client = OpenAIChatClient(
    model="my-gpt-4o-deployment",
    azure_endpoint="https://my-resource.openai.azure.com",
    credential=AzureCliCredential(),
)
```

### جریان‌های کاری چندعامله

جایی که این چارچوب واقعاً برجسته می‌شود، ارکستراسیون چندین عامل با هم است. برای مثال، می‌توانید عامل‌ها را یکی پس از دیگری اجرا کنید (هرکدام زمینه خود را به بعدی منتقل می‌کند) یا همزمان به چند عامل واگذار کرده و نتایج را جمع‌آوری کنید:

```python
from agent_framework.orchestrations import SequentialBuilder, ConcurrentBuilder

# اجرای عوامل به صورت متوالی، انتقال زمینه گفتگو در طول زنجیره
sequential = SequentialBuilder(participants=[researcher, writer, editor]).build()

# ارسال موازی به عوامل مختلف، سپس تجمیع پاسخ‌های آن‌ها
concurrent = ConcurrentBuilder(participants=[analyst_a, analyst_b, analyst_c]).build()
```

## Taskweaver

چارچوب عامل بعدی که بررسی می‌کنیم [Taskweaver](https://microsoft.github.io/TaskWeaver/?WT.mc_id=academic-105485-koreyst) است. به عنوان عامل «کد-اول» شناخته می‌شود زیرا به جای کار صرف با `رشته`‌ها، می‌تواند با DataFrames در پایتون کار کند. این برای تحلیل داده و تولید داده بسیار مفید است. این می‌تواند شامل ایجاد نمودارها یا تولید اعداد تصادفی باشد.

### وضعیت و ابزارها

برای مدیریت وضعیت مکالمه، TaskWeaver از مفهوم `Planner` استفاده می‌کند. `Planner` یک LLM است که درخواست کاربران را گرفته و کارهایی که باید برای انجام آن درخواست انجام شوند را نقشه‌برداری می‌کند.

برای انجام این کارها، `Planner` به مجموعه‌ای از ابزارها به نام `Plugins` دسترسی دارد. این می‌تواند کلاس‌های پایتون یا مفسر کد عمومی باشد. این افزونه‌ها به صورت embedding ذخیره می‌شوند تا LLM بهتر بتواند افزونه مناسب را جستجو کند.

![Taskweaver](../../../translated_images/fa/taskweaver.da8559999267715a.webp)

اینجا نمونه‌ای از یک افزونه برای تشخیص ناهنجاری آورده شده است:

```python
class AnomalyDetectionPlugin(Plugin): def __call__(self, df: pd.DataFrame, time_col_name: str, value_col_name: str):
```

کد قبل از اجرا تأیید می‌شود. ویژگی دیگری برای مدیریت زمینه در Taskweaver `تجربه` است. تجربه اجازه می‌دهد زمینه یک مکالمه در درازمدت در یک فایل YAML ذخیره شود. این می‌تواند طوری پیکربندی شود که LLM در گذر زمان در برخی کارها بهبود یابد، به شرطی که با مکالمات قبلی مواجه شود.

## JARVIS

آخرین چارچوب عاملی که بررسی می‌کنیم [JARVIS](https://github.com/microsoft/JARVIS?tab=readme-ov-file&WT.mc_id=academic-105485-koreyst) است. چیزی که JARVIS را منحصر به فرد می‌کند این است که از یک LLM برای مدیریت `وضعیت` مکالمه استفاده می‌کند و `ابزارها` مدل‌های هوش مصنوعی دیگر هستند. هر یک از مدل‌های هوش مصنوعی مدل‌های تخصصی هستند که وظایف خاصی مانند تشخیص اشیاء، رونویسی یا کپشن‌گذاری تصویر را انجام می‌دهند.

![JARVIS](../../../translated_images/fa/jarvis.762ddbadbd1a3a33.webp)

LLM، که یک مدل چندمنظوره است، درخواست کاربر را دریافت کرده و کار مشخص و هر آرگومان/داده‌ای که برای انجام کار لازم است را شناسایی می‌کند.

```python
[{"task": "object-detection", "id": 0, "dep": [-1], "args": {"image": "e1.jpg" }}]
```

سپس LLM درخواست را به فرمت قابل تفسیر مدل تخصصی مثل JSON تبدیل می‌کند. وقتی مدل AI پیش‌بینی خود را بر اساس کار برمی‌گرداند، LLM پاسخ را دریافت می‌کند.

اگر چند مدل برای انجام کار لازم باشد، ابتدا پاسخ آن مدل‌ها را تفسیر می‌کند و بعد آنها را با هم ترکیب می‌کند تا پاسخ را به کاربر ارائه دهد.

مثال زیر نشان می‌دهد چگونه این کار وقتی یک کاربر درخواست توضیحات و شمارش اشیاء در تصویر را می‌دهد انجام می‌شود:
