# מסגרות עבודה לרשתות נוירונים

כפי שלמדנו כבר, כדי לאמן רשתות נוירונים בצורה יעילה עלינו לעשות שני דברים:

* לפעול על טנזורים, למשל לכפל, חיבור וחישוב של פונקציות כמו סיגמואיד או סופטמקס
* לחשב גרדיאנטים של כל הביטויים, כדי לבצע אופטימיזציה של ירידת גרדיאנט

בעוד שספריית `numpy` יכולה לבצע את החלק הראשון, אנו זקוקים למנגנון כלשהו לחישוב גרדיאנטים. במסגרת העבודה שפיתחנו בסעיף הקודם היינו צריכים לתכנת ידנית את כל פונקציות הנגזרת בתוך השיטה `backward`, שמבצעת backpropagation. באופן אידיאלי, מסגרת עבודה צריכה לתת לנו את ההזדמנות לחשב גרדיאנטים של *כל ביטוי* שנוכל להגדיר.

דבר חשוב נוסף הוא היכולת לבצע חישובים על GPU, או כל יחידת חישוב מתמחה אחרת, כמו TPU. אימון רשתות נוירונים עמוקות דורש *הרבה* חישובים, והיכולת לבצע חישובים במקביל על GPUs היא חשובה מאוד.

> ✅ המונח 'parallelize' מתייחס להפצת החישובים על פני מכשירים מרובים.

כיום, שתי המסגרות הנפוצות ביותר לרשתות נוירונים הן: TensorFlow ו-PyTorch. שתיהן מספקות API ברמה נמוכה לפעולה עם טנזורים על CPU ו-GPU. בנוסף ל-API ברמה נמוכה, יש גם API ברמה גבוהה, הנקראים בהתאמה Keras ו-PyTorch Lightning.

API ברמה נמוכה | TensorFlow| PyTorch
--------------|-------------------------------------|--------------------------------
API ברמה גבוהה | Keras| Pytorch

**APIs ברמה נמוכה** בשתי המסגרות מאפשרים לבנות את מה שנקרא **גרפים חישוביים**. גרף זה מגדיר כיצד לחשב את הפלט (בדרך כלל פונקציית האובדן) עם פרמטרי הקלט הנתונים, וניתן לדחוף אותו לחישוב על GPU, אם הוא זמין. ישנן פונקציות להבדיל בין גרף חישובי זה ולחשב גרדיאנטים, שניתן להשתמש בהם לאחר מכן לאופטימיזציה של פרמטרי המודל.

**APIs ברמה גבוהה** מתייחסים לרשתות נוירונים כאל **רצף של שכבות**, ומקלים על בניית רוב הרשתות הנוירוניות. אימון המודל בדרך כלל דורש הכנת הנתונים ואז קריאה לפונקציה `fit` כדי לבצע את העבודה.

ה-API ברמה גבוהה מאפשר לבנות רשתות נוירונים טיפוסיות במהירות רבה ללא צורך לדאוג להרבה פרטים. במקביל, API ברמה נמוכה מציעים שליטה רבה יותר על תהליך האימון, ולכן הם משמשים הרבה במחקר, כאשר מתמודדים עם ארכיטקטורות חדשות של רשתות נוירונים.

חשוב גם להבין שניתן להשתמש בשני ה-APIs יחד, למשל, ניתן לפתח את ארכיטקטורת השכבה של הרשת שלך באמצעות API ברמה נמוכה, ולאחר מכן להשתמש בה בתוך רשת גדולה יותר שנבנתה ואומנה עם API ברמה גבוהה. או שניתן להגדיר רשת באמצעות API ברמה גבוהה כרצף של שכבות, ולאחר מכן להשתמש בלולאת אימון ברמה נמוכה משלך כדי לבצע אופטימיזציה. שני ה-APIs משתמשים באותם מושגים בסיסיים, והם מתוכננים לעבוד טוב יחד.

## למידה

בקורס זה, אנו מציעים את רוב התוכן הן עבור PyTorch והן עבור TensorFlow. אתה יכול לבחור את המסגרת המועדפת עליך ולעבור רק על המחברות המתאימות. אם אינך בטוח איזו מסגרת לבחור, קרא כמה דיונים באינטרנט על **PyTorch לעומת TensorFlow**. אתה יכול גם להסתכל על שתי המסגרות כדי לקבל הבנה טובה יותר.

כאשר ניתן, נשתמש ב-APIs ברמה גבוהה לשם פשטות. עם זאת, אנו מאמינים שחשוב להבין כיצד רשתות נוירונים פועלות מהבסיס, ולכן בהתחלה נתחיל בעבודה עם API ברמה נמוכה וטנזורים. עם זאת, אם אתה רוצה להתחיל מהר ולא רוצה להקדיש הרבה זמן ללמידת הפרטים האלה, אתה יכול לדלג עליהם ולעבור ישירות למחברות ה-API ברמה גבוהה.

# התאמת יתר

התאמת יתר היא מושג חשוב ביותר בלמידת מכונה, וחשוב מאוד להבין אותו נכון!

התבונן בבעיה הבאה של קירוב 5 נקודות (מיוצגות על ידי `x` בגרפים למטה):

!ליניארי | התאמת יתר
-------------------------|--------------------------
**מודל ליניארי, 2 פרמטרים** | **מודל לא ליניארי, 7 פרמטרים**
שגיאת אימון = 5.3 | שגיאת אימון = 0
שגיאת אימות = 5.1 | שגיאת אימות = 20

* משמאל, אנו רואים קירוב קו ישר טוב. מכיוון שמספר הפרמטרים מתאים, המודל תופס נכון את הרעיון שמאחורי התפלגות הנקודות.
* מימין, המודל חזק מדי. מכיוון שיש לנו רק 5 נקודות והמודל כולל 7 פרמטרים, הוא יכול להתאים כך שיעבור דרך כל הנקודות, ויגרום לשגיאת האימון להיות 0. עם זאת, זה מונע מהמודל להבין את התבנית הנכונה מאחורי הנתונים, ולכן שגיאת האימות גבוהה מאוד.

חשוב מאוד למצוא את האיזון הנכון בין עושר המודל (מספר הפרמטרים) לבין מספר דגימות האימון.

## מדוע מתרחשת התאמת יתר

  * לא מספיק נתוני אימון
  * מודל חזק מדי
  * יותר מדי רעש בנתוני הקלט

## כיצד לזהות התאמת יתר

כפי שניתן לראות מהגרף למעלה, ניתן לזהות התאמת יתר על ידי שגיאת אימון נמוכה מאוד ושגיאת אימות גבוהה. בדרך כלל במהלך האימון נראה את שגיאות האימון והאימות מתחילות לרדת, ואז בנקודה מסוימת שגיאת האימות עשויה להפסיק לרדת ולהתחיל לעלות. זה יהיה סימן להתאמת יתר, והאינדיקטור לכך שכנראה כדאי להפסיק את האימון בנקודה זו (או לפחות לבצע צילום מצב של המודל).

## כיצד למנוע התאמת יתר

אם אתה רואה שההתאמת יתר מתרחשת, אתה יכול לעשות אחד מהדברים הבאים:

 * להגדיל את כמות נתוני האימון
 * להקטין את המורכבות של המודל
 * להשתמש בטכניקת רגולריזציה כלשהי, כגון Dropout, שנשקול בהמשך.

## התאמת יתר והמסחר בין הטיה לשונות

התאמת יתר היא למעשה מקרה של בעיה כללית יותר בסטטיסטיקה הנקראת המסחר בין הטיה לשונות. אם נבחן את מקורות השגיאה האפשריים במודל שלנו, נוכל לראות שני סוגי שגיאות:

* **שגיאות הטיה** נגרמות מכך שהאלגוריתם שלנו לא מצליח לתפוס את הקשר בין נתוני האימון בצורה נכונה. זה יכול לנבוע מכך שהמודל שלנו לא חזק מספיק (**התאמה חסרה**).
* **שגיאות שונות**, הנגרמות מכך שהמודל מתאים רעש בנתוני הקלט במקום קשר משמעותי (**התאמת יתר**).

במהלך האימון, שגיאת הטיה יורדת (כשהמודל שלנו לומד לקירוב את הנתונים), ושגיאת שונות עולה. חשוב להפסיק את האימון - או ידנית (כשאנו מזהים התאמת יתר) או אוטומטית (על ידי הכנסת רגולריזציה) - כדי למנוע התאמת יתר.
