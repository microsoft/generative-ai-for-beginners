## יצירה מוגברת על ידי שליפה (RAG)

צ'אטבוט שמופעל באמצעות LLM מעבד פקודות משתמש ליצירת תגובות. הוא מתוכנן להיות אינטראקטיבי ומתקשר עם משתמשים בנושאים מגוונים. עם זאת, תגובותיו מוגבלות להקשר המסופק ולנתוני האימון הבסיסיים שלו. לדוגמה, החיתוך של הידע ב-GPT-4 הוא בספטמבר 2021, כלומר הוא חסר ידע על אירועים שהתרחשו לאחר תקופה זו. בנוסף, הנתונים בהם משתמשים לאימון LLM אינם כוללים מידע חסוי כגון הערות אישיות או מדריך מוצר של חברה.

### כיצד עובדים RAGים (יצירה מוגברת על ידי שליפה)

![שרטוט המראה כיצד RAGים עובדים](../../../translated_images/he/how-rag-works.f5d0ff63942bd3a6.webp)

נניח שאתה רוצה לפרוס צ'אטבוט שיוצר מבחנים מההערות שלך, תזדקק לחיבור לבסיס הידע. כאן RAG נכנס לתמונה. RAGים פועלים כך:

- **בסיס הידע:** לפני השליפה, יש לבצע עיבוד והכנסה של מסמכים, בדרך כלל פירוק מסמכים גדולים לחלקים קטנים יותר, המרתם לאמבדינג טקסט ואחסונם במסד הנתונים.

- **שאילתת משתמש:** המשתמש שואל שאלה

- **שליפה:** כשמשתמש שואל שאלה, מודל האמבדינג מושך מידע רלוונטי מבסיס הידע כדי לספק הקשר שיכלל בפקודה.

- **יצירה מוגברת:** ה-LLM משפר את תגובתו בהתבסס על הנתונים שנשלפו. הוא מאפשר שהתשובה תהיה לא רק מבוססת על הנתונים שאומנו מראש, אלא גם על מידע רלוונטי מההקשר שנוסף. הנתונים שנמשכים משמשים להרחבת התגובות של ה-LLM. לאחר מכן ה-LLM מחזיר תשובה לשאלה של המשתמש.

![שרטוט המראה את ארכיטקטורת RAG](../../../translated_images/he/encoder-decode.f2658c25d0eadee2.webp)

הארכיטקטורה של RAGים מיושמת באמצעות טרנספורמרים הכוללים שני חלקים: מקודד ופיענוח. לדוגמה, כאשר משתמש שואל שאלה, הטקסט הקלט מומר לוקטורים המתארים את משמעות המילים והוקטורים 'מתקודדים' לאינדקס המסמכים שלנו ויוצרים טקסט חדש בהתבסס על שאילתת המשתמש. ה-LLM משתמש במודל מקודד-מפענח כדי ליצור את הפלט.

יש שתי גישות ליישום RAG לפי המאמר המוצע: [Retrieval-Augmented Generation for Knowledge intensive NLP Tasks](https://arxiv.org/pdf/2005.11401.pdf?WT.mc_id=academic-105485-koreyst):

- **_RAG-Sequence_** המשתמש במסמכים שנשלפו כדי לנבא את התשובה הטובה ביותר לשאילתת משתמש

- **RAG-Token** המשתמש במסמכים כדי ליצור את הטוקן הבא, ואז מוציא אותם כדי לענות על השאילתה של המשתמש

### למה להשתמש ב-RAGים? 

- **עושר מידע:** מבטיחה שתשובות הטקסט מעודכנות ונוכחיות. לכן היא משפרת את הביצועים במשימות ספציפיות לתחום על ידי גישה לבסיס הידע הפנימי.

- מפחיתה המצאות על ידי שימוש בנתונים **ניתנים לאימות** בבסיס הידע כדי לספק הקשר לשאלות המשתמש.

- **חסכוני** כי הוא זול יותר מהתאמת LLM באופן ספציפי (fine-tuning)

### מטקסט לאמבדינגים

לפני שאוחסן את הנתונים שלנו, נצטרך להמיר אותם לאמבדינגים וקטוריים לפני אחסונם במסד הנתונים. אם אתם עובדים עם מסמכים גדולים או טקסטים ארוכים, ניתן לפצל אותם בהתאם לשאילתות שאתם מצפים להן. הפיצול יכול להיעשות ברמת משפט, או ברמת פסקה. מאחר שהפיצול מבוסס על המשמעות מהסביבה המילולית, ניתן להוסיף הקשר נוסף לחתיכה, לדוגמה על ידי הוספת כותרת המסמך או הכללת טקסט לפני או אחרי החתיכה. ניתן לפצל את הנתונים כך:

```python
def split_text(text, max_length, min_length):
    words = text.split()
    chunks = []
    current_chunk = []

    for word in words:
        current_chunk.append(word)
        if len(' '.join(current_chunk)) < max_length and len(' '.join(current_chunk)) > min_length:
            chunks.append(' '.join(current_chunk))
            current_chunk = []

    # אם הקטע האחרון לא הגיע לאורך המינימלי, הוסף אותו בכל מקרה
    if current_chunk:
        chunks.append(' '.join(current_chunk))

    return chunks
```

לאחר הפיצול, נוכל להטמיע את הטקסט באמצעות מודלים שונים לאמבדינג. כמה מודלים שניתן להשתמש בהם כוללים: word2vec, ada-002 של OpenAI, Azure Computer Vision ועוד רבים. הבחירה במודל תלויה בשפות שבהן אתה עובד, סוג התוכן (טקסט/תמונות/אודיו), גודל הקלט שהוא יכול לקודד ואורך פלט האמבדינג.

## שליפה וחיפוש וקטורי

כאשר משתמש שואל שאלה, המערכת המשלפת ממירה אותה לוקטור באמצעות מקודד השאילתות, ואז מחפשת באינדקס חיפוש המסמכים שלנו וקטורים רלוונטיים במסמכים שקשורים לקלט. לאחר מכן, הן וקטור הקלט והווקטורים של המסמכים מומרות לטקסט ומועברות דרך ה-LLM.

### שליפה

השליפה מתרחשת כשהמערכת מנסה במהירות למצוא את המסמכים באינדקס שמתאימים לקריטריוני החיפוש. מטרת המשלף היא לקבל מסמכים שישמשו לספק הקשר ולעגן את ה-LLM על הנתונים שלך.

קיימות מספר דרכים לביצוע חיפוש במסד הנתונים שלנו כגון:

- **חיפוש לפי מילות מפתח** - משמש לחיפושים בטקסט

- **חיפוש וקטורי** - ממיר מסמכים מייצוג טקסט לייצוג וקטורי באמצעות מודלים של אמבדינג, מאפשר **חיפוש סמנטי** על פי משמעות המילים. השליפה מתבצעת על ידי שאילת מסמכים שהייצוג הוקטורי שלהם הוא הקרוב ביותר לשאלת המשתמש.

- **היברידי** - שילוב של חיפוש מילות מפתח ווקטורי.

אתגר בשליפה מתגלה כאשר אין תגובה דומה לשאילתה בבסיס הנתונים, אז המערכת תחזיר את המידע הטוב ביותר שהיא יכולה להשיג, אך ניתן להשתמש בטקטיקות כמו קביעת מרחק מקסימלי לרלוונטיות או להשתמש בחיפוש היברידי המשלב מילות מפתח וחיפוש וקטורי. בשיעור זה נשתמש בחיפוש היברידי, שילוב של חיפוש וקטורי ומילות מפתח. נאחסן את הנתונים שלנו במסגרת נתונים עם עמודות שכוללות את החתיכות וכן את האמבדינגים.

### דמיון וקטורי

המשלף יחפש דרך מסד הידע אחר אמבדינגים שהם קרובים זה לזה, השכן הקרוב ביותר, כי הם טקסטים דומים. בתרחיש שבו המשתמש שואל שאלה, היא מוטמעת ראשון ואז מותאמת עם אמבדינגים דומים. המדד הנפוץ שמשמש לבדוק עד כמה וקטורים דומים הוא דמיון קוסינוס, המבוסס על הזווית בין שני וקטורים.

אנו יכולים למדוד דמיון גם באמצעות חלופות אחרות שביכולתנו להשתמש בהן: מרחק אוקלידי שהוא הקו הישר בין קצוות הוקטורים ומכפלה סקלרית שמודדת את סכום המוצרים של מרכיבים מתאימים בשני הוקטורים.

### אינדקס חיפוש

כשמבצעים שליפה, נצטרך לבנות אינדקס חיפוש עבור בסיס הידע שלנו לפני שנבצע חיפוש. אינדקס יאחסן את האמבדינגים שלנו ויוכל במהירות לשלוף את החתיכות הכי דומות גם במסד נתונים גדול. ניתן ליצור את האינדקס באופן מקומי באמצעות:

```python
from sklearn.neighbors import NearestNeighbors

embeddings = flattened_df['embeddings'].to_list()

# צור את מדד החיפוש
nbrs = NearestNeighbors(n_neighbors=5, algorithm='ball_tree').fit(embeddings)

# כדי לשאול את המדד, תוכל להשתמש בשיטת kneighbors
distances, indices = nbrs.kneighbors(embeddings)
```

### סידור מחדש (Re-ranking)

לאחר ששאלת את מסד הנתונים, ייתכן שתצטרך למיין את התוצאות מהרלוונטיות ביותר. LLM לסידור מחדש משתמש בלמידת מכונה כדי לשפר את הרלוונטיות של תוצאות החיפוש על ידי מיון מהכי רלוונטי. בשימוש עם Azure AI Search, הסידור מחדש מתבצע אוטומטית באמצעות מארגן סמנטי. דוגמה לאופן פעולת הסידור מחדש דרך שכנות קרובות:

```python
# מצא את המסמכים הדומים ביותר
distances, indices = nbrs.kneighbors([query_vector])

index = []
# הדפס את המסמכים הדומים ביותר
for i in range(3):
    index = indices[0][i]
    for index in indices[0]:
        print(flattened_df['chunks'].iloc[index])
        print(flattened_df['path'].iloc[index])
        print(flattened_df['distances'].iloc[index])
    else:
        print(f"Index {index} not found in DataFrame")
```

## הערכת היישום שלנו

### מדדי הערכה

- איכות התגובות המתקבלות, להבטיח שהן נשמעות טבעיות, שוטפות ודמויות אנוש.

- עיגון הנתונים: הערכת האם התגובה מגיעה מהמסמכים שנמסרו.

- רלוונטיות: הערכת התאמת התגובה ושייכותה לשאלה שנשאלה.

- שטף - האם התגובה הגיונית דקדוקית.

## מקרים לשימוש ב-RAG ומסדי נתונים וקטוריים

קיימים שימושים שונים בהם קריאות פונקציה יכולות לשפר את האפליקציה שלך כמו:

- שאלות ותשובות: עיגון הנתונים של החברה שלך לצ'אט שניתן להשתמש בו על ידי עובדים לשאלת שאלות.

- מערכות המלצה: בהן ניתן ליצור מערכת התואמת את הערכים הדומים ביותר לדוגמה סרטים, מסעדות ועוד.

- שירותי צ'אטבוט: ניתן לאחסן היסטוריית שיחות ולהתאים אישית את השיחה על פי נתוני המשתמש.

- חיפוש תמונות בהתבסס על אמבדינג וקטורי, שימושי בזיהוי תמונות וזיהוי חריגות.

## סיכום

סקרנו את התחומים הבסיסיים של RAG מהוספת הנתונים לאפליקציה, שאלת המשתמש ויציאת התגובה. כדי לפשט יצירת RAG, ניתן להשתמש במסגרת כמו Semanti Kernel, Langchain או Autogen.
