## יצירה מוגברת על ידי שליפה (RAG)

צ'טבוט מונע על ידי LLM מעבד פקודות משתמש כדי ליצור תגובות. הוא מעוצב להיות אינטראקטיבי ומעורבב עם משתמשים על מגוון רחב של נושאים. עם זאת, התגובות שלו מוגבלות להקשר הניתן ולנתוני האימונים הבסיסיים שלו. לדוגמה, הידע של GPT-4 מוגבל לספטמבר 2021, כלומר, הוא חסר ידע על אירועים שהתרחשו לאחר תקופה זו. נוסף על כך, הנתונים בהם משתמשים לאימון LLM אינם כוללים מידע סודי כגון הערות אישיות או מדריך מוצר של חברה.

### איך RAGs (יצירה מוגברת על ידי שליפה) עובדים

![תמונה שמראה איך RAGs עובדים](../../../translated_images/he/how-rag-works.f5d0ff63942bd3a6.webp)

נניח שאתם רוצים לפרוס צ'טבוט שיוצר שאלונים מההערות שלכם, תצטרכו חיבור לבסיס הידע. כאן RAG נכנס לפעולה. RAGs פועלים כך:

- **בסיס ידע:** לפני השליפה, יש לעבד ולפרק את המסמכים, בדרך כלל לפירוק למסמכים קטנים יותר, המרה לטקסט מוטמע ואחסון בבסיס נתונים.

- **שאילתת משתמש:** המשתמש שואל שאלה

- **שליפה:** כאשר המשתמש שואל שאלה, מודל ההטמעות שולף מידע רלוונטי מבסיס הידע שלנו כדי לספק הקשר שישולב בבקשה.

- **יצירה מוגברת:** ה-LLM משדרג את התגובה שלו בהתבסס על הנתונים שנשלפו. זה מאפשר שתגובה שנוצרה לא תהיה מבוססת רק על נתוני האימון, אלא גם על מידע רלוונטי מההקשר שנוסף. הנתונים שנשלפו משמשים לשדרוג תגובות ה-LLM. לאחר מכן ה-LLM מחזיר תשובה לשאלת המשתמש.

![איור שמראה את האדריכלות של RAGs](../../../translated_images/he/encoder-decode.f2658c25d0eadee2.webp)

האדריכלות של RAGs מיושמת באמצעות טרנספורמרים הכוללים שני חלקים: מקודד ופענח. לדוגמה, כאשר משתמש שואל שאלה, הטקסט הקלט 'מקודד' לוקטורים הכוללים את משמעות המילים ווקטורים אלו 'מפוענחים' לאינדקס המסמכים שלנו ומייצרים טקסט חדש בהתבסס על שאילתת המשתמש. ה-LLM משתמש במודל מקודד-פענח ליצירת הפלט.

ישנן שתי גישות לביצוע RAG לפי המאמר המוצע: [יצירה מוגברת על ידי שליפה למשימות NLP אינטנסיביות בידע](https://arxiv.org/pdf/2005.11401.pdf?WT.mc_id=academic-105485-koreyst):

- **_RAG-Sequence_** המשתמש במסמכים שנשלפו כדי לנבא את התשובה הטובה ביותר לשאילתת משתמש

- **RAG-Token** המשתמש במסמכים ליצירת הטוקן הבא, ואז שולף אותם כדי לענות על שאלת המשתמש

### למה כדאי להשתמש ב-RAGs?

- **עושר מידע:** מבטיח שהתשובות הטקסטואליות מעודכנות ונכונות. לכן, משפר ביצועים במשימות ספציפיות לפי תחום בעזרת גישה לבסיס הידע הפנימי.

- מצמצם המצאות (fabrication) על ידי שימוש ב**נתונים שניתן לאמת** בבסיס הידע כדי לספק הקשר לשאלות המשתמשים.

- הוא **חסכוני** שכן הוא זול יותר בהשוואה לכוונון עדין של LLM

### מטקסט להטמעות

לפני שנאחסן את הנתונים, נצטרך להמירם להטמעות וקטוריות לפני האחסון בבסיס הנתונים. אם אתם עובדים עם מסמכים גדולים או טקסטים ארוכים, ניתן לפרק אותם לפי השאילתות שאתם מצפים להם. פילוח יכול להתבצע ברמת משפט או ברמת פסקה. מאחר שפילוח שואב משמעות מהמילים סביב, ניתן להוסיף הקשר לחלק, למשל, על ידי הוספת כותרת המסמך או כלול טקסט לפני או אחרי החלק. ניתן לפרק את הנתונים כך:

```python
def split_text(text, max_length, min_length):
    words = text.split()
    chunks = []
    current_chunk = []

    for word in words:
        current_chunk.append(word)
        if len(' '.join(current_chunk)) < max_length and len(' '.join(current_chunk)) > min_length:
            chunks.append(' '.join(current_chunk))
            current_chunk = []

    # אם הקטע האחרון לא הגיע לאורך המינימלי, הוסף אותו בכל מקרה
    if current_chunk:
        chunks.append(' '.join(current_chunk))

    return chunks
```

לאחר הפילוח, נוכל להטמיע את הטקסט באמצעות דגמי הטמעות שונים. בין הדגמים שניתן להשתמש בהם: word2vec, ada-002 של OpenAI, Azure Computer Vision ועוד רבים. בחירת מודל תתבסס על השפות בהן אתם משתמשים, סוג התוכן המקודד (טקסט/תמונות/אודיו), גודל הקלט אותו ניתן לקודד ואורך הפלט של ההטמעה.

## שליפה וחיפוש וקטורי

כאשר משתמש שואל שאלה, המאתר ממיר אותה לוקטור באמצעות מקודד השאילתה, ואז מחפש באינדקס החיפוש שלנו וקטורים רלוונטיים במסמכים הקשורים לקלט. לאחר מכן הוא ממיר גם את וקטור הקלט וגם את וקטורי המסמכים לטקסט ומעבירם דרך ה-LLM.

### שליפה

שליפה מתבצעת כאשר המערכת מנסה במהירות לאתר את המסמכים מהאינדקס העונים לקריטריוני החיפוש. מטרת המאתר היא להשיג מסמכים שישמשו לספק הקשר ולעגן את ה-LLM על הנתונים שלכם.

קיימות מספר דרכים לבצע חיפוש בתוך מאגר הנתונים שלנו כגון:

- **חיפוש מילות מפתח** - משמש לחיפושי טקסט

- **חיפוש וקטורי** - ממיר מסמכים מטקסט לייצוגים וקטוריים באמצעות דגמי הטמעות, ומאפשר **חיפוש סמנטי** באמצעות משמעות המילים. השליפה תתבצע על ידי שאילת מסמכים שהייצוג הוקטורי שלהם קרוב ביותר לשאלת המשתמש.

- **השילוב** - שילוב של חיפוש מילות מפתח וחיפוש וקטורי.

אתגר בשליפה מתעורר כאשר אין תגובה דומה לשאילתה בבסיס הנתונים, אז המערכת תחזיר את המידע הטוב ביותר שיכולה להשיג, עם זאת, ניתן להשתמש בטקטיקות כמו הגדרת מרחק מקסימלי לרלוונטיות או שימוש בחיפוש משולב המשלב מילות מפתח ווקטורים. בשיעור זה נשתמש בחיפוש משולב, שילוב של חיפוש וקטורי ומילות מפתח. נאחסן את המידע שלנו במסגרת נתונים עם עמודות שמכילות את החלקים כמו גם את ההטמעות.

### דמיון וקטורי

המאחד יחפש במאגר הידע הטמעות שהן קרובות אחת לשנייה, השכן הקרוב ביותר, שכן אלו טקסטים דומים. בתרחיש בו המשתמש שואל שאלה, היא תחילה מוטמעת ואז תתאים לה הטמעות דומות. המדידה המקובלת להשוואת דמיון בין וקטורים שונים היא הדמיון הקוסינוסי המבוסס על הזווית בין שני וקטורים.

אנו יכולים למדוד דמיון באמצעות חלופות אחרות כגון מרחק אוקלידי שהוא הקו הישר בין נקודות קצה של וקטורים, ומכפלה סקלרית שמודדת את סכום מכפלות הרכיבים המתאימים בשני הוקטורים.

### אינדקס חיפוש

בעת ביצוע שליפה, נצטרך לבנות אינדקס חיפוש עבור בסיס הידע שלנו לפני ביצוע החיפוש. אינדקס יאחסן את ההטמעות שלנו ויוכל לשלוף במהירות את החלקים הדומים ביותר גם במאגר מידע גדול. נוכל ליצור את האינדקס שלנו מקומית באמצעות:

```python
from sklearn.neighbors import NearestNeighbors

embeddings = flattened_df['embeddings'].to_list()

# צור את אינדקס החיפוש
nbrs = NearestNeighbors(n_neighbors=5, algorithm='ball_tree').fit(embeddings)

# כדי לשאול את האינדקס, ניתן להשתמש בשיטת kneighbors
distances, indices = nbrs.kneighbors(embeddings)
```

### דירוג מחדש

לאחר ששאלתם את בסיס הנתונים, ייתכן ותצטרכו למיין את התוצאות לפי הרלוונטיות הגבוהה ביותר. LLM מדרג מחדש משתמש בלמידת מכונה לשפר את רלוונטיות תוצאות החיפוש על ידי סידור מהכי רלוונטי. באמצעות Azure AI Search, הדירוג מחודש נעשה אוטומטית עבורכם בעזרת דירוג סמנטי מחדש. דוגמה לאופן פעולת הדירוג מחדש באמצעות שכנים קרובים:

```python
# מצא את המסמכים הכי דומים
distances, indices = nbrs.kneighbors([query_vector])

index = []
# הדפס את המסמכים הכי דומים
for i in range(3):
    index = indices[0][i]
    for index in indices[0]:
        print(flattened_df['chunks'].iloc[index])
        print(flattened_df['path'].iloc[index])
        print(flattened_df['distances'].iloc[index])
    else:
        print(f"Index {index} not found in DataFrame")
```

user_input = "what is a perceptron?"

## הערכת האפליקציה שלנו

### מדדי הערכה

- איכות התגובות שסופקו - להבטיח שהן נשמעות טבעיות, שוטפות ובנות אדם

- עיגון הנתונים: הערכת האם התגובה נובעת מהמסמכים שסופקו

- רלוונטיות: הערכת התאמת התגובה והקשר לשאלה שנשאלה

- שוטפות - האם התגובה הגיונית מבחינה דקדוקית

## מקרים לשימוש ב-RAG ומאגרי וקטורים

קיימים שימושים רבים ושונים שבהם קריאות לפונקציות יכולות לשפר את האפליקציה שלכם, כגון:

- שאלות ותשובות: עיגון נתוני החברה שלכם לצ'ט שניתן להשתמש בו על ידי עובדים לשאילת שאלות.

- מערכות המלצה: שם ניתן ליצור מערכת המתאימה ערכים דומים ביותר, למשל סרטים, מסעדות ועוד רבים.

- שירותי צ'טבוט: ניתן לאחסן היסטוריית שיחה ולהתאים את השיחה באופן אישי בהתבסס על נתוני המשתמש.

- חיפוש תמונות מבוסס הטמעות וקטוריות, שימושי בזיהוי תמונות וזיהוי חריגות.

## סיכום
