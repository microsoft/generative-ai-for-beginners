# מבוא למודלים קטנים של שפה לבינה מלאכותית גנרטיבית למתחילים
בינה מלאכותית גנרטיבית היא תחום מרתק של אינטליגנציה מלאכותית המתמקד ביצירת מערכות המסוגלות לייצר תוכן חדש. תוכן זה יכול לנוע מטקסט ותמונות למוזיקה ואפילו לסביבות וירטואליות שלמות. אחת מהיישומים המרגשים ביותר של בינה מלאכותית גנרטיבית היא בתחום מודלי השפה.

## מה הם מודלים קטנים של שפה?

מודל שפה קטן (SLM) מייצג גרסה מקטנת של מודל שפה גדול (LLM), המשתמשת בהרבה מהעקרונות והטכניקות האדריכליות של LLMים, תוך הצגת טביעת חישובית מצומצמת משמעותית.

SLMים הם תת-קבוצה של מודלי שפה המיועדים לייצר טקסט המדמה טקסט אנושי. בניגוד לעמיתיהם הגדולים יותר, כמו GPT-4, SLMים הם קומפקטיים ויעילים יותר, מה שהופך אותם לאידיאליים ליישומים שבהם משאבי החישוב מוגבלים. על אף גודלם הקטן יותר, הם עדיין יכולים לבצע מגוון משימות. בדרך כלל, SLMים נבנים על ידי דחיסה או זיקוק של LLMים, במטרה לשמר חלק משמעותי מהפונקציונליות ויכולות השפה של המודל המקורי. הפחתה זו בגודל המודל מפחיתה את המורכבות הכוללת, מה שהופך את ה-SLMים ליעילים יותר מבחינת שימוש בזיכרון ודרישות חישוב. למרות האופטימיזציות הללו, SLMים יכולים עדיין לבצע מגוון רחב של משימות עיבוד שפה טבעית (NLP):

- יצירת טקסט: יצירת משפטים או פסקאות קוהרנטיים ורלוונטיים להקשר.
- השלמת טקסט: חיזוי והשלמת משפטים על בסיס הנחיה נתונה.
- תרגום: המרת טקסט משפה אחת לאחרת.
- סיכום: קיצור טקסטים ארוכים לתמציות קצרות ומובנות יותר.

אף על פי שישנן האטות בביצועים או בעומק ההבנה בהשוואה לעמיתיהם הגדולים יותר.

## איך מודלים קטנים של שפה עובדים?
SLMים מאומנים על כמויות עצומות של נתוני טקסט. במהלך האימון הם לומדים את התבניות והמבנים של השפה, המאפשרים להם ליצור טקסט שהוא גם דקדוקי נכון וגם מתאים להקשר. תהליך האימון כולל:

- איסוף נתונים: איסוף מערכות נתונים גדולות של טקסט ממקורות שונים.
- קדם עיבוד: ניקוי וארגון הנתונים כדי להפוך אותם מתאימים לאימון.
- אימון: שימוש באלגוריתמים של למידת מכונה ללמד את המודל כיצד להבין וליצור טקסט.
- כיוונון עדין: התאמת המודל לשיפור הביצועים במשימות ספציפיות.

פיתוח ה-SLMים מתיישב עם הצורך הגובר במודלים שניתן לפרוס בסביבות עם משאבים מוגבלים, כגון מכשירים ניידים או פלטפורמות מחשוב בקצה, שבהן LLMים בקנה מידה מלא עשויים להיות בלתי מעשיים עקב דרישות המשאבים הגבוהות שלהם. בהתמקדות ביעילות, SLMים מאוזנים בין ביצועים לזמינות, ומאפשרים שימוש נרחב יותר בתחומים שונים.

## ההבדלים בין מודלים גדולים של שפה (LLMs) למודלים קטנים של שפה (SLMs)

שני סוגי המודלים, LLMים ו-SLMים, מבוססים על עקרונות יסודיים של למידת מכונה הסתברותית, ופועלים בהתאם לגישות דומות בעיצוב אדריכלי, מתודולוגיות אימון, תהליכי יצירת נתונים וטכניקות הערכת מודל. עם זאת, קיימים כמה גורמים מרכזיים שמבדילים בין שני סוגי המודלים.

## יישומים של מודלים קטנים של שפה

ל-SLMים יש טווח רחב של יישומים, הכולל:

- צ'אטבוטים: מתן תמיכה ללקוח ותקשורת עם משתמשים בצורה שיחתית.
- יצירת תוכן: סיוע לכותבים ביצירת רעיונות ואפילו בהכנת מאמרים שלמים.
- חינוך: עזרה לסטודנטים במשימות כתיבה או בלמידת שפות חדשות.
- נגישות: יצירת כלים לאנשים עם מוגבלויות, כמו מערכות טקסט לדיבור.

**גודל**
  
הבחנה ראשית בין LLMים ל-SLMים היא בקנה המידה של המודלים. LLMים, כמו ChatGPT (GPT-4), יכולים להכיל כ-1.76 טריליון פרמטרים, בעוד של-SLMים בקוד פתוח כמו Mistral 7B יש פרמטרים משמעותית פחותים — כ-7 מיליארד. פער זה נובע בעיקר מהבדלים באדריכלות המודל ותהליכי האימון. לדוגמה, ChatGPT מפעיל מנגנון תשומת לב עצמית במסגרת מקודד-פותר, בעוד ש-Mistral 7B משתמש בתשומת לב בחלון נגלל, המאפשרת אימון יעיל יותר במסגרת מודל שמבוסס רק על פותר. שונות אדריכלית זו משפיעה עמוקות על המורכבות והביצועים של המודלים.

**הבנה**

בדרך כלל, SLMים מותאמים לביצועים בתוך תחומים ספציפיים, מה שהופך אותם למומחים מאוד אך מוגבלים ביכולתם לספק הבנה הקשרית רחבה בתחומים מגוונים. לעומת זאת, LLMים שואפים לדמות אינטליגנציה דמוית-אנושית ברמה מקיפה יותר. מאומנים על מערכות נתונים מגוונות וגדולות, LLMים מיועדים לפעול היטב במגוון תחומים, ומציעים גמישות והתאמה מרבית. כתוצאה מכך, LLMים מתאימים יותר למגוון רחב של משימות מקצה אחורי, כגון עיבוד שפה טבעית ותכנות.

**חישוב**

אימון ופריסה של LLMים הם תהליכים עתירי משאבים, שלרוב דורשים תשתיות חישוביות משמעותיות, כולל אשכולות GPU בקנה מידה גדול. לדוגמה, אימון מודל כמו ChatGPT מאפס עשוי לדרוש אלפי GPUs לאורך תקופות ממושכות. לעומת זאת, SLMים, עם מספר פרמטרים קטן יותר, נגישים יותר מבחינת משאבי חישוב. מודלים כמו Mistral 7B יכולים להיות מאומנים ומופעלים במכונות מקומיות המצוידות ביכולות GPU מתונות, למרות שאימון עדיין דורש כמה שעות על פני מרובות GPUs.

**הטיה**

הטיה היא בעיה ידועה ב-LLMים, בעיקר בגלל טיב נתוני האימון. מודלים אלה לרוב מסתמכים על נתונים גלויים גולמיים מהאינטרנט, שעשויים לייצג בקושי או באופן מוטעה קבוצות מסוימות, להציג תיוג שגוי, או לשקף הטיות לשוניות המושפעות מדיאלקטים, וריאציות גאוגרפיות וכללי דקדוק. בנוסף, מורכבות אדריכלות ה-LLM יכולה להחריף את ההטיה, שלעיתים אינה נראית בלי כיוונון עדין. לעומת זאת, SLMים, המתווכים על מערכות נתונים מצומצמות וממוקדות תחום יותר, רגישים פחות להטיות כאלו, אם כי לא חסינים לגמרי מהן.

**אינפרנס**

גודלם המצומצם של ה-SLMים מעניק להם יתרון משמעותי במהירות האינפרנס, שמאפשר להם ליצור פלטים ביעילות על חומרה מקומית וללא הצורך בעיבוד מקביל נרחב. לעומת זאת, LLMים, בשל גדלם ומורכבותם, לעיתים מצריכים משאבים חישוביים מקבילים משמעותיים כדי להגיע לזמני אינפרנס מקובלים. נוכחות משתמשים רבים בו זמנית אף מאטה את זמני התגובה של LLMים, במיוחד בפריסה בקנה מידה גדול.

לסיכום, בעוד שני סוגי המודלים, LLMים ו-SLMים, מבוססים על יסודות למידת מכונה, הם שונים באופן משמעותי בגודל המודל, דרישות המשאבים, ההבנה ההקשרית, הפגיעות להטיה ומהירות האינפרנס. הבדלים אלה משקפים את ההתאמה שלהם לשימושים שונים, כאשר LLMים מגוונים אך דורשים משאבים כבדים, ו-SLMים מציעים יעילות תחומית מוגברת עם דרישות חישוב מופחתות.

***הערה: בשיעור זה נציג את ה-SLM באמצעות Microsoft Phi-3 / 3.5 כדוגמה.***

## הצגת משפחת Phi-3 / Phi-3.5

משפחת Phi-3 / 3.5 מתמקדת בעיקר בתרחישי שימוש בטקסט, ראייה ו-Agent (MoE):

### Phi-3 / 3.5 Instruct

בעיקר ליצירת טקסט, השלמה בשיחה, וחילוץ מידע מתוכן, ועוד.

**Phi-3-mini**

מודל שפה 3.8B זמין ב-Microsoft Foundry, Hugging Face ו-Ollama. מודלי Phi-3 מתפקדים טוב בהרבה ממודלי שפה בגודל שווה וגדולים יותר במבחנים מרכזיים (ראו את המספרים מטה, מספרים גבוהים יותר טובים יותר). Phi-3-mini מתעלה על מודלים בגודל כפול, בעוד ש-Phi-3-small ו-Phi-3-medium מתעלים אף על מודלים גדולים יותר, כולל GPT-3.5.

**Phi-3-small & medium**

עם רק 7 מיליארד פרמטרים, Phi-3-small מנצח את GPT-3.5T במגוון מבחני שפה, היגיון, קידוד ומתמטיקה.

Phi-3-medium עם 14 מיליארד פרמטרים ממשיך מגמה זו ומתעלה על Gemini 1.0 Pro.

**Phi-3.5-mini**

ניתן לראות בו שדרוג של Phi-3-mini. הפרמטרים נשארים ללא שינוי, אך הוא משפר את היכולת לתמוך בשפות רבות (תומך ב-20+ שפות: ערבית, סינית, צ'כית, דנית, הולנדית, אנגלית, פינית, צרפתית, גרמנית, עברית, הונגרית, איטלקית, יפנית, קוריאנית, נורווגית, פולנית, פורטוגזית, רוסית, ספרדית, שוודית, תאילנדית, טורקית, אוקראינית) ומוסיף תמיכה חזקה יותר להקשרים ארוכים.

Phi-3.5-mini עם 3.8 מיליארד פרמטרים מתעלה על מודלי שפה בגודל זהה ומשווה למודלים בגודל כפול.

### Phi-3 / 3.5 Vision

ניתן לראות במודל ה-Instruct של Phi-3/3.5 את יכולתו של Phi להבין, ו-Vision היא מה שנותנת ל-Phi עיניים להבין את העולם.


**Phi-3-Vision**

Phi-3-vision, עם רק 4.2 מיליארד פרמטרים, ממשיך מגמה זו ומתעלה על מודלים גדולים יותר כגון Claude-3 Haiku ו-Gemini 1.0 Pro V במשימות היגיון חזותי כלליות, OCR, והבנת טבלאות ודיאגרמות.


**Phi-3.5-Vision**

Phi-3.5-Vision הוא גם שדרוג של Phi-3-Vision, המוסיף תמיכה בתמונות מרובות. ניתן לראות בו שיפור בראייה — לא רק ראייה של תמונות, אלא גם של וידאו.

Phi-3.5-vision מתעלה על מודלים גדולים כגון Claude-3.5 Sonnet ו-Gemini 1.5 Flash במשימות OCR, הבנת טבלאות ותרשימים, ומשווה במטלות היגיון של ידע חזותי כללי. תומך בקלט רב-מסגרתי, כלומר מבצע היגיון על מספר תמונות קלט.


### Phi-3.5-MoE

***Mixture of Experts (MoE)*** מאפשר למודלים להתאמן עם הרבה פחות חישוב, מה שאומר שאפשר להגדיל משמעותית את גודל המודל או מערך הנתונים עם תקציב חישוב זהה למודל צפוף. בפרט, מודל MoE אמור להגיע לאותו איכות כמו מקבילו הצפוף הרבה יותר מהר באימון מוקדם.

Phi-3.5-MoE מורכב מ-16 מודולים מומחים של 3.8 מיליארד פרמטרים כל אחד. Phi-3.5-MoE עם רק 6.6 מיליארד פרמטרים פעילים משיג רמה דומה של היגיון, הבנת שפה ומתמטיקה למודלים גדולים בהרבה.

ניתן להשתמש במודל משפחת Phi-3/3.5 בהתאם לתרחישים שונים. בניגוד ל-LLM, ניתן לפרוס את Phi-3/3.5-mini או Phi-3/3.5-Vision במכשירי קצה.

בנוסף לקטלוג המודלים מבוססי הענן של Microsoft Foundry, ניתן גם להשתמש ב-[NVIDIA NIM](https://developer.nvidia.com/nim?WT.mc_id=academic-105485-koreyst) להשלמת הקריאות הרלוונטיות. ניתן לבקר ב-NVIDIA NIM כדי להשלים את קריאות ה-API של משפחת Phi-3/3.5. NVIDIA NIM (NVIDIA Inference Microservices) הוא אוסף של מיקרו-שירותי אינפרנס מואצים, שנועד לסייע למפתחים לפרוס דגמי בינה מלאכותית ביעילות בסביבות שונות, כולל עננים, מרכזי נתונים ותחנות עבודה.

הנה כמה תכונות מפתח של NVIDIA NIM:

- **קלות פריסה:** NIM מאפשר פריסה של דגמי AI עם פקודה אחת, מה שהופך את ההטמעה לפשוטה בתהליכים קיימים.

NIM הוא חלק מ-NVIDIA AI Enterprise, שמטרתו לפשט את פריסת המודלים התפעולית של AI, ולהבטיח שהם ירוצו ביעילות על GPUs של NVIDIA.

1. **מודלים מאומנים מראש**: היא מציעה אלפי מודלים מאומנים מראש שניתן להשתמש בהם למשימות שונות כגון סיווג טקסט, זיהוי ישויות שמיות, מענה על שאלות, סיכום, תרגום, ויצירת טקסט.

2. **תאימות למסגרות עבודה:** הספרייה תומכת במספר מסגרות למידה עמוקה, כולל PyTorch, TensorFlow, ו-JAX. זה מאפשר לך לאמן מודל במסגרת עבודה אחת ולהשתמש בו באחרת.

3. **יכולות מולטימודל:** מלבד NLP, Hugging Face Transformers תומכת גם במשימות בראייה ממוחשבת (כגון סיווג תמונות, זיהוי עצמים) ועיבוד אודיו (כגון זיהוי דיבור, סיווג אודיו).

4. **קלות שימוש:** הספרייה מציעה API וכלים להורדה וכיול מודלים בקלות, מה שהופך אותה לנגישה הן למתחילים והן למומחים.

5. **קהילה ומשאבים:** ל-Hugging Face יש קהילה פעילה ותיעוד מקיף, מדריכים, והדרכות לעזור למשתמשים להתחיל ולהפיק את המרב מהספרייה.

**ONNX Runtime עבור GenAI**

[ONNX Runtime](https://github.com/microsoft/onnxruntime-genai?WT.mc_id=academic-105485-koreyst) הוא מאיץ למידה מכונה חוצה פלטפורמות עבור הסקה ואימון. ONNX Runtime עבור בינה מלאכותית יצירתית (GENAI) הוא כלי חזק שעוזר להריץ מודלים של AI יצירתי ביעילות על גבי פלטפורמות שונות.

## מה זה ONNX Runtime?
ONNX Runtime הוא פרויקט קוד פתוח שמאפשר הסקה בעלת ביצועים גבוהים של מודלים ללמידת מכונה. הוא תומך במודלים בפורמט Open Neural Network Exchange (ONNX), שהוא תקן לייצוג מודלים. ONNX Runtime מאפשר חוויות לקוח מהירות יותר והפחתת עלויות, ותומך במודלים ממסגרות למידה עמוקה כמו PyTorch ו-TensorFlow/Keras וכן מספריות למידת מכונה קלאסיות כמו scikit-learn, LightGBM, XGBoost ועוד. ONNX Runtime תואם לחומרות, דרייברים ומערכות הפעלה שונות, ומספק ביצועים מיטביים על ידי ניצול מאיצי חומרה לצד אופטימיזציות וטרנספורמציות גרפיות.

## מה זה בינה מלאכותית יצירתית?
בינה מלאכותית יצירתית מתייחסת למערכות AI שמסוגלות לייצר תוכן חדש, כגון טקסט, תמונות, או מוזיקה, בהתבסס על הנתונים שעליהם אומנו. דוגמאות לכך הן מודלי שפה כמו GPT-3 ומודלי יצירת תמונות כמו Stable Diffusion. ספריית ONNX Runtime עבור GenAI מספקת את הלולאה היצירתית למודלי ONNX, כולל הסקה עם ONNX Runtime, עיבוד לוגיטים, חיפוש ודגימה, וניהול זיכרון KV.

## ONNX Runtime עבור GENAI
ONNX Runtime עבור GENAI מרחיב את יכולות ONNX Runtime לתמוך במודלים של בינה מלאכותית יצירתית. הנה כמה תכונות מרכזיות:

- **תמיכה רחבה בפלטפורמות:** הוא פועל על פלטפורמות שונות, כולל Windows, Linux, macOS, Android ו-iOS.
- **תמיכה במודלים:** תומך במגוון מודלים פופולריים של בינה מלאכותית יצירתית, כגון LLaMA, GPT-Neo, BLOOM ועוד.
- **אופטימיזציית ביצועים:** כולל אופטימיזציות למאיצי חומרה שונים כמו NVIDIA GPUs, AMD GPUs ועוד.
- **קלות שימוש:** מספק API לאינטגרציה קלה ביישומים, ומאפשר יצירת טקסט, תמונות ותוכן נוסף עם קוד מינימלי.
- משתמשים יכולים לקרוא למתודה ברמה גבוהה generate(), או להריץ כל איטרציה של המודל בלולאה, וליצור תג אחד בכל פעם, עם אפשרות לעדכון פרמטרים בתוך הלולאה.
- ONNX Runtime גם תומך בחיפוש greedy/beam ובדגימות TopP, TopK ליצירת רצפי תגיות, ועיבוד לוגיטים מובנה כמו ענישה על חזרות. אפשר גם להוסיף ניקוד מותאם אישית בקלות.

```Python
import onnxruntime_genai as og

model = og.Model('path_to_your_model.onnx')

tokenizer = og.Tokenizer(model)

input_text = "Hello, how are you?"

input_tokens = tokenizer.encode(input_text)

output_tokens = model.generate(input_tokens)

output_text = tokenizer.decode(output_tokens)

print(output_text) 
```
### הדגמה: שימוש ב-ONNX Runtime GenAI לקריאה ל-Phi-3.5-Vision


```python

import onnxruntime_genai as og

model_path = './Your Phi-3.5-vision-instruct ONNX Path'

img_path = './Your Image Path'

model = og.Model(model_path)

processor = model.create_multimodal_processor()

tokenizer_stream = processor.create_stream()

text = "Your Prompt"

prompt = "<|user|>\n"

prompt += "<|image_1|>\n"

prompt += f"{text}<|end|>\n"

prompt += "<|assistant|>\n"

image = og.Images.open(img_path)

inputs = processor(prompt, images=image)

params = og.GeneratorParams(model)

params.set_inputs(inputs)

params.set_search_options(max_length=3072)

generator = og.Generator(model, params)

while not generator.is_done():

    generator.compute_logits()
    
    generator.generate_next_token()

    new_token = generator.get_next_tokens()[0]
    
    output = tokenizer_stream.decode(new_token)
    
    print(tokenizer_stream.decode(new_token), end='', flush=True)

```
